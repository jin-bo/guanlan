## 未决项：蒸馏工作法住哪个 skill？

两个选项，共同前提都成立：**蒸馏正文由 Agent 用文件工具写 `dist/`（散文本就是 Agent 的活，无 P3.7 式"谁来写"张力）；wrapper 只事后零-LLM 校验 + 写 marketplace**。分歧只在"怎么把蒸馏工作法喂给 Agent"：

- **Option A —— 扩既有 `guanlan-wiki` 维护引擎**，加一个 `distill` 工作流
  - 优：单一 skill；不增安装/可发现性面（`SKILL_NAME=guanlan-wiki` 的构造期强激活门禁原样可用，distill 命令直接复用已激活的引擎）。
  - 缺：把**两种相反 posture** 塞进一份 SKILL.md——"维护我的 wiki"（写 wiki、就地标 `## ⚠️ 矛盾与存疑`、可建外链）vs"产出给别人的自洽 skill"（只读 wiki、**禁**外链、写 dist/）。可能扰乱 Agent 对当前任务 posture 的判断。

- **Option B —— 新发布专用 `guanlan-distill` skill**（倾向）
  - 优：posture 干净隔离（OpenKB 正是用独立 `prompts/skill_create.md`，不与其维护逻辑混）。
  - 缺：`BUNDLED_SKILL_NAMES` 要加它、`pyproject` force-include 要加一行 `skills/guanlan-distill → guanlan/_skill/guanlan-distill`、且须定**激活方式**子决策——构造期强激活（like `guanlan-wiki`）还是按需激活（like `pdf-to-markdown`）。distill 是**显式命令触发**，倾向**按需激活**（命令内对该次 Agentao 调用强激活 guanlan-distill，平时不激活），但这需确认 `run_agent_task`/runtime 的 skill 激活注入口支持"本次指定激活某 skill"。

**倾向：B（posture 隔离、长期更干净）。** 但因其扩了安装面 + 含一项"激活方式"子决策，留作未决待人工拍板。
