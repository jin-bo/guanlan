## 3. 关键洞察：一道边界决策、两件事（两半均已拍，取 A）

最高杠杆的发现——**`P3.9 源撤回` 与 `P3.7 语义审计` 曾卡在同一道架构决策上**：**宿主（wrapper/host）能否写 / 删 wiki *内容*页**。**两半现均已拍板取 Option A**：

- **P3.7-3a（✅ 已拍板取 A，2026-06-15）**：wrapper 在内容页 stamp `raw_digest`（**头一回 wrapper 碰内容页**——以往只 verify，仅 `reindex` 写过 `index.md` 那种 *config* 页）。**定论**：宿主**可**对内容页做**纯 provenance 的 append-only 确定性写**。
- **P3.9-1（✅ 已拍板取 A，同日）**：`remove` 删 / 移 wiki *内容*页（越"Agent 写 wiki"线比 P3.7 更远——P3.7 只 stamp、不删）。**定论**：宿主**可**确定性删 / 移内容页，但须**默认预览 + 显式 `--yes` + `.trash/` 回收区留档**（核心 = 回收区把"删内容页"从不可逆变**可逆**）。一期经评审收窄（决策P3.9-10）**只移 1:1 摘要页**、独源衍生页只 advisory 不删（删它 = `--cascade`，降二期 `prune-orphans`）。被否：B（宿主只改 provenance、绝不删内容页 → 撤回不闭环）/ C（走 Agent → 删非生成、违分档）。设计已转定稿 [`P3.9-源撤回.md`](../../P3.9-源撤回.md) §8.2。

两条 backlog 笔记当初**都建议"一并拍"**：先拍了 P3.7 这半（窄判例：宿主写 provenance ✅），后单独拍 P3.9 这半（删内容页 ✅，靠恢复窗越这道更重边界）。**P3.7-3a 不自动延伸到 P3.9 的"删"**——删 / 移内容页比 append-only stamp 重，故各自评估（共同前提稳：纯 provenance / 人发起纠正都是 Agent 结构上做不到或不该做的——LLM 抄不准 sha256、撤回是人意图，落点类比 `reindex` 写 config 页的先例；P3.9 多加恢复窗这道安全垫）。**这道"宿主能否写/删内容页"的边界至此两半全部落定并均已实现（`guanlan audit` / `guanlan remove`），地图上不再有架构级未决（仅余 P6-1 的 skill 归属一项实现取舍）。**
