# CJK 检索增强备选（backlog）

> 状态：**非 MVP，未排期**。仅在中文漏召回**确有实证**后再评估；届时才敲定具体规则，不预先实现。
> 关联：DESIGN.md §8「CJK 检索质量」、§4.4 query、§4.5 CJK 约定。

## MVP 现状（不在本 backlog 范围）

query 用 `index.md` + 2-gram 粗召回；召回不中时 Agent 扫相关目录或请用户补关键词兜底。这是已采纳的最小方案，本笔记只记录**之后**可能的逐级增强。

## 逐级增强备选

1. **别名字段（首选第一步）** —— 给 entity/concept 页加 frontmatter `aliases`，例如 `aliases: ["大语言模型", "LLM"]`。对齐 Obsidian 同名键，便于用户直接拿 Obsidian 打开。要发挥作用须接入：
   - **wikilink/断链解析**：`[[别名]]` 解析到声明该别名的页，不再误报断链（在现有 stem 大小写不敏感解析之上扩展）。
   - **2-gram 匹配面**：别名串纳入可匹配文本，补中文同义召回。
   - **ingest 去重纪律**：建实体页前先查现有别名，命中则追加别名而非新建重复页。
   - **校验**：别名与页面 stem 同一命名空间，`check.py` 校验别名全局唯一、不撞 title（纯算术、零 LLM）。
2. **同义词表** —— 别名仍不够时的集中式映射，按需引入。
3. **轻量检索 / 向量检索** —— 上述都不够再上；向量检索天然跨分词边界，是比分词更合适的下一步（nashsu/gbrain 均用向量+混合检索）。

## 分词为何不优先

知识库实体多为词典外词（OOV），分词器最易漏标错切；分词结果随词典版本漂移、不确定、且引入词典依赖，违背"脚本零 LLM"。真要补精度，向量检索优于词典分词。
