> 关联：DESIGN.md §8「CJK 检索质量」、§7 路线表（P5「多格式 + 检索」/ E1）、§4.4 query、§4.5 CJK 约定。

## MVP 现状

query 用 `index.md` + 2-gram 粗召回；召回不中时 Agent 扫相关目录 / 现有页 `aliases` 或请用户补关键词兜底（P3.1 别名已纳入匹配面）。这是已采纳的最小方案；P5.0 在其上加**整页正文**的确定性 BM25 召回作为 query/skill 的召回前端（不替代综合）。

## 逐级增强备选（次序与现状）

1. **别名字段** —— ✅ **已落 P3.1**（[`../../P3.1-别名解析.md`](../../P3.1-别名解析.md)）。给 entity/concept 页加 frontmatter `aliases`，已接入：`[[别名]]` 解析（消假断链）、2-gram 匹配面、ingest 去重纪律、`check` 全局唯一校验（纯算术、零 LLM）。
2. **同义词表** —— ⏸️ **按需、非前置**（未排期）。集中式 term→term 映射（如「LLM」⇄「大语言模型」）。**被 P5.0 提前越过**：P3.1 别名 + P5.0 整页 BM25 召回已部分顶替其作用，故不再作为③的前置门槛；仅当别名 + BM25 仍漏召回**确有实证**时再评估。残值：纯 BM25/2-gram 对"查询词不出现在文档、但与文档词同义"仍可能漏，同义词表能补这一缺口。
3. **轻量检索 / 向量检索** —— 拆为两半：
   - **轻量检索（确定性 BM25 + 2-gram）** —— 📐 **已细化为 P5.0**（[`../../P5.0-检索层.md`](../../P5.0-检索层.md)，设计已出、待实现）。零 LLM、零依赖、查询时即时计算、无持久化派生；title/aliases 字段加权；`search_pages()` 内核供未来 web/mcp 复用。**提前做**的理由（P5.0 §0）：Karpathy 数百页封顶 + 三个对照实现齐投检索 + P5 多格式会放大语料——不待"②都不够"。
   - **向量检索** —— ⏸️ 仍留 **E1**（存储与检索升级，选型另文）。天然跨分词边界，是 BM25 仍不够时的下一步（nashsu/gbrain 均用向量+混合检索）；属重型基础设施，不进 P5.0 的零基建边界。

## 分词为何不优先（不变）

知识库实体多为词典外词（OOV），分词器最易漏标错切；分词结果随词典版本漂移、不确定、且引入词典依赖，违背"脚本零 LLM"。真要补精度，向量检索优于词典分词。**P5.0 据此用 2-gram 而非分词器**（决策 P5.0-7）——本戒律在 P5.0 落地后依然成立、未被突破。
