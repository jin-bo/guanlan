## 1. 下一里程碑候选与排序

按"补全当前里程碑 + 不破坏不变量 + 已有脚手架"三维度：

| 候选 | 是什么 | 契合度 | 风险 |
|---|---|---|---|
| **P5.2 多格式摄入** ⭐ | `guanlan convert` 把 PDF/DOCX/… 转 md 落 `raw/` 再 ingest | **补全 P5 正题**（P5 = 多格式+检索，检索那半 P5.0/P5.1 已完）；**脚手架已有**：`pdf-to-markdown` skill（P4.6，随 wheel 全局装）+ `workspace/{uploads,parsed}/` | 低-中：**复用 skill convert.py、无新依赖**；独立命令、ingest 不动（见 spec 决策P5.2-5） |
| 语义 lint | 矛盾复检/过期论断/资料缺口（需 LLM、按需触发，DESIGN §8） | 深化 lint 诊断线 | 高：**首次把 LLM 引入 lint**，破 lint 零-LLM 纪律 |

| LLM 推断边 | `INFERRED`/`AMBIGUOUS` 边 | 图谱继续深化 | **最高**：破"markdown 唯一真相/图可重建"核心不变量，DESIGN 明确"需另开" |

## 2. 推荐：P5.2 多格式摄入

理由集中且有据：

1. **它是当前里程碑的"另一半正题"**。P5 全名「语料规模化与自动化（多格式 + 检索）」，P5.0/P5.1 只交付了**检索**；**多格式摄入**仍是 P5 交付栏白纸黑字的待做项。先收口 P5，比横跳 E 轨或 LLM-lint 更顺。
2. **脚手架已就位**，非从零起。P4.6 已落 `pdf-to-markdown` skill + `workspace/uploads/`（暂存）+ `workspace/parsed/`（解析产物）+ Web 晋级写 `raw/`。P5.2 本质 = 把这条"上传→解析→人审→晋级"的 **Web-only 手动链路，沉淀成 CLI 侧独立命令 `guanlan convert <file>`**，**直接复用 skill 的 `convert.py` 当转换引擎**（无新依赖），落 `raw/<slug>.md` 后由既有 `guanlan ingest` 入库（见 spec 决策P5.2-1/5）。
3. **守得住不变量**。它是 **ingest 边界处的格式转换 adapter**——不碰"markdown 唯一真相"、不引 LLM-lint、不破图可重建；多格式→md 后下游 ingest/check/graph/search 全部原样复用。和语义-lint / 推断边那种"动核心契约"不是一个量级。

**形态草案**：`guanlan/adapters/` 子包（gated by `guanlan-wiki[ingest]` extra，markitdown 兜底 + 可选更高质 PDF backend），把非-md 源**确定性、零-LLM**地转成 `raw/` 下 md（保留原文件指针/出处），LLM 只在随后既有 ingest 出现；与 P4.6 解析路径共用同一归口。

## 3. 图可视化：暂不做（被推迟两次、至今仍空的坑）

### 3.1 现状：交互可视化"承诺给 P4"，但 P4 没做

- 决策P3-7 把力导向/缩放/SVG 绘图整体**推给"P4 Web UI"**（P3 只承诺 `graph.json` 稳定，`graph.html` 是零-JS 静态邻接列表）。
- 但 P4 真正落地时，Web「graph」按钮只是 `window.open("/graph")` → 重建后 302 跳到**同一份零-JS 静态 `graph.html`**。`app.js` 里**无任何** d3/cytoscape/力导向/canvas。

⇒ 交互式图可视化是**被推迟两次、至今仍空的坑**，不是已完成项。

### 3.2 判断：值得做，但不是现在，且不排在多格式摄入之前

结论 **暂不必要**，价值与成本都和"语料规模"强绑定：

- **可行动信号已交付**：孤儿/过载枢纽/割边/割点/孤岛——P3.5/P3.6 已以 **lint 文字 finding + 静态 html** 列出。可视化加的是"探索/概览"体验，**不新增可行动信号**；正是 DESIGN 警惕的"把确定性 graph 数据退化成前端渲染项目"。
- **收益随节点数才显现**：小库（多数用户区间）力导向是毛球或三五点，静态列表够用；几百节点以上才开始比列表强。
- **顺序错了**：可视化回报恰要靠**多格式摄入**把语料做大才解锁。先做 viz = 造一个没通电的功能。

**但它有正当位置（故"推迟"非"否决"）：**
- 决策P3-7 只约束**零网络静态 `graph.html`**，并**显式豁免**"图形化留 P4 Web UI"——在 P4 Web 内引 JS 图库**不违反任何已立决策**。
- P3.5/P3.6 刚塞进 graph 的社区/枢纽/割边割点数据**天生是空间/视觉的**：按社区着色 + 高亮割边割点的力导向图能让这批数据"一眼可读"——**一旦语料够大**，边际价值才抬上来。

### 3.3 建议排法

定为**可选后置 P4.x 半阶段**（原拟 `P4.11 Web 图可视化`，但 `P4.11` 已被信任边界、`P4.12` 已被 Web 语义审计占用 → **须另取空号 P4.13+**），触发条件"语料规模上来后"，非下一里程碑。形态那时再定：P4 Web 内嵌 **vendored（零 CDN，守自包含）** 力导向视图，**复用现成 `graph.json`**（数据已全：含 community/bridges/cut_vertices），纯前端渲染、不动后端、不碰 `graph.html` 零-JS 承诺。

## 4. 一句话

可行动的图谱诊断已经有了；**下一里程碑推荐 P5.2 多格式摄入**（补全 P5、脚手架已就位、守不变量），**图可视化是等语料做大后放进 P4 Web 的锦上添花**，不该插到多格式摄入前面。
