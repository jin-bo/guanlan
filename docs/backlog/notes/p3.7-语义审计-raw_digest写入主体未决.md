## 背景一句话

P3.7 给 `lint.py:19` 推迟的「语义 lint（矛盾复检/过期论断，需 LLM）」动工：新开 `guanlan audit` 命令，**确定性粗筛（零 LLM）→ LLM 复核（复用 P2 门禁）**两层，复刻 heal。第一类跑通"过期论断 / source-drift"。它的拱心石是一项新 provenance——`source` 摘要页 frontmatter 上的 `raw_digest: 'raw/<file>@sha256:<hex>'`，记本页所摘 raw 的版本指纹；源被 `convert --overwrite`/人手替换后，靠它比对出"页该重综合"。

`raw_digest` 约定**绝大部分已定稿**（字段在 source 页一份、单标量含路径、经 `sources:` slug 图传播、`check` 不可见、向后兼容跳过）。**唯一卡住的是写入主体。**

## 未决项：`raw_digest` 由谁写？

两个选项，共同前提都成立：**只 stamp 本次摄入那一张 source 页，绝不顺手刷新别页**（否则抹平别处真实漂移，致命）。分歧只在写入主体：

- **Option A —— wrapper 门禁后直写 source 页**
  - 优：值由代码算、零腐败、零 LLM、幂等确定。
  - 缺：**头一回让 wrapper 碰内容页**——以往 wrapper 只 verify，仅 `reindex` 写过 `index.md` 那种 *config* 页；这越过"Agent 写 wiki、wrapper 只验"默契一小步。
  - 正当性：纯 provenance（不可变 raw 的 hash）、Agent 结构上做不到（`INGEST_PROMPT` 禁 shell + LLM 不会/抄不准 sha256）、类比 reindex。

- **Option B —— wrapper 算值塞进 prompt、Agent 文件工具写 + wrapper 回读校验**
  - 优：守住"Agent 是 wiki 唯一写者"。
  - 缺：要 LLM 逐字抄 64 位 hex，**易腐**；须加一层"写后回读比对、不符则 wrapper 兜底改正或报错"。

**倾向：A（可靠性优先）。** 但因其越"Agent 写 wiki"默契，留作未决待人工拍板。
