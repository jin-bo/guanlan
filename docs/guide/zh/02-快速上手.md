# 快速上手

本页用最少的命令跑通一个知识库:**初始化 → 投喂 → 提问 → 维护**。命令细节见 [CLI 命令](03-cli-命令.md)。

> 前置:已 `pip install guanlan-wiki`(见 [安装](01-安装.md))。`ingest` / `query` 需配置模型;其余命令零 LLM、可离线。

## 1. 初始化一个知识库

```bash
guanlan init my-wiki     # 在新目录初始化
guanlan init             # 或就地初始化当前目录
```

`init` 是**确定性的(零 LLM)**,已存在的文件**不会被覆盖**,可安全重复运行。

生成结构:

```
my-wiki/
├── AGENTAO.md       # Agent 行为约束 + 指针
├── SCHEMA.md        # 本库 Schema:领域 / 启用页面类型 / 自定义规则
├── raw/             # 原始资料(只读,事实来源)
└── wiki/            # Agent 全权生成的知识层
    ├── index.md     # 全量页面目录
    ├── log.md       # append-only 时间线
    └── overview.md  # 跨资料活体综述
```

> **`-C` 指定知识库根**:除 `init` 外,其余命令都用 `guanlan -C my-wiki <命令>` 指向知识库目录(也可先 `cd my-wiki` 再省略 `-C`)。

### 1.1 让 LLM 辅助更新 `SCHEMA.md`

`SCHEMA.md` 是本库的人面约定:领域边界、页面类型用法、自定义规则、演进中的组织偏向。它不是机器解析的 API schema。知识库积累一段时间后,如果主题范围、标签、命名、章节结构或组织偏向已经发生变化,可以让 LLM 先分析当前库状态,再更新 `SCHEMA.md`。

可直接把下面这段交给可写入当前目录的 LLM:

```text
你正在维护一个 Guanlan 知识库。请分析当前知识库的真实状态,并据此更新根目录的 SCHEMA.md。

任务目标:
让 SCHEMA.md 准确反映当前知识库的领域边界、页面类型使用方式、命名/标签/章节规则,以及已经形成的组织偏向。SCHEMA.md 是给人和 Agent 阅读的自由 Markdown 约定,不是机器解析的 API schema;不要把它写成程序配置文件。

请按以下步骤执行:

1. 阅读并理解这些文件/目录:
   - SCHEMA.md
   - wiki/index.md
   - wiki/overview.md
   - wiki/log.md
   - wiki/ 下各目录的页面样本,优先覆盖 source/entity/concept/synthesis 四类
   - 如有必要,抽样查看 raw/ 中的源资料标题或 frontmatter 来源

2. 统计和归纳当前知识库状态:
   - 当前主要领域/主题是什么
   - 哪些主题或资料类型已经形成规模
   - source/entity/concept/synthesis 实际分别怎么使用
   - 常见 tags 有哪些,是否存在同义、重复、引号混用、粒度不一致
   - 页面标题、文件名、wikilink、章节结构有哪些实际约定
   - 哪些组织偏向已经稳定出现,值得写进“演进中的论点 / 偏向”
   - 哪些旧规则已经不符合现状

3. 更新 SCHEMA.md:
   - 保留它作为“本库约定”的性质
   - 不要新增机器不可执行的伪配置字段
   - 不要承诺当前工具链并不支持的能力,例如“SCHEMA.md 会被程序解析”
   - 不要虚构不存在的主题、标签或规则
   - 对已有规则进行收敛、澄清、去重
   - 如果发现只是个别坏页,不要把坏页写成规则;应把主流、稳定、值得延续的实践写入 SCHEMA.md

4. 更新内容应覆盖这些章节:
   - ## 领域 / 主题
   - ## 启用的页面类型
   - ## 本库自定义规则
   - ## 演进中的论点 / 偏向

5. 写作要求:
   - 使用中文
   - 用简洁、可执行的规则表达
   - 保持 Markdown 可读
   - 表格可以保留,但不要让表格承载过细或易过期的事实
   - 对标签词表、命名规则、章节规则要尽量具体
   - 对“演进中的论点 / 偏向”要明确这是工作假设,遇到反例应在相关页面记录矛盾与存疑

6. 完成后请输出:
   - 你对现有库状态的简短分析
   - SCHEMA.md 的主要修改点
   - 更新后的 SCHEMA.md 全文
   - 建议后续人工复核的点

重要边界:
- SCHEMA.md 不被 Guanlan Python 代码解析;它是人/Agent 约定。
- 不要修改 raw/。
- 不要批量改 wiki 页面。
- 除非用户明确要求,否则本任务只更新 SCHEMA.md。
- 如果发现页面结构问题,只在最后列为建议,不要顺手迁移页面。
```

如果要让 LLM 直接落盘,可在提示词末尾补一句:

```text
请直接编辑根目录 SCHEMA.md。完成后运行:
uv run guanlan check
uv run guanlan health

如果检查失败,请只修正 SCHEMA.md 引起的问题;不要修改 raw/ 或批量改 wiki 页面。
```

## 2. 投喂资料

把一篇 markdown 资料喂给 Agent,它会读 `raw/` 里的源、生成/更新 `wiki/` 里的页面:

```bash
guanlan -C my-wiki ingest path/to/source.md
```

> 非 `.md` 文件(PDF / DOCX / PPTX / …)先用 [`guanlan convert`](07-多格式转换.md) 转成 `raw/<slug>.md`,再 `ingest`。
>
> **`raw/` 只读不可变**:Agent 只读原始资料、永不修改,写门禁会在每次写操作前后对 `raw/` 做快照校验。

## 3. 提问

```bash
guanlan -C my-wiki query "你的问题"
```

默认是只读问答(基于已建好的 wiki)。若想把这次综合**沉淀回 wiki**(一次受门禁约束的写入),用回填:

```bash
guanlan -C my-wiki query "你的问题" --backfill
```

## 4. 维护(零 LLM、可离线)

```bash
guanlan -C my-wiki check     # frontmatter / 断链 / 来源校验
guanlan -C my-wiki health    # stub 页面 + index↔disk 同步(--strict → 退出码 6)
guanlan -C my-wiki lint      # 孤儿页 / 断链 / 缺失实体
guanlan -C my-wiki graph     # 写出 graph/graph.json + graph.html(--json-only 跳过 html)
```

`health` / `lint` 是 **advisory**(建议性,不阻断);`check` 是确定性校验。各命令的退出码语义见 [维护:体检与图谱](04-维护-体检-图谱.md)。

## 5. 检索

```bash
guanlan -C my-wiki search "关键词"     # 零 LLM 确定性 BM25 + CJK 全页召回
```

## 可选:在浏览器里用

```bash
pip install 'guanlan-wiki[web]'
guanlan -C my-wiki web                 # 起本地 Web 宿主,仅监听 127.0.0.1,默认开浏览器
guanlan -C my-wiki web --port 9000 --no-browser
```

浏览器里可:浏览 wiki 并跟随 `[[wikilink]]` 导航、跑 check·health·lint、看 graph、从 `raw/` 触发 ingest、与 agent 只读多轮对话。**仅供本机单用户——绝不要把端口暴露到网络。** 详见 [Web 宿主](05-web-宿主.md)。

## 可选:暴露给 MCP 客户端

```bash
pip install 'guanlan-wiki[mcp]'
guanlan -C my-wiki mcp                 # stdio 上起只读 MCP 服务端
```

详见 [MCP 宿主](06-mcp-宿主.md)。
