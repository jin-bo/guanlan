# P5.1 — Web 检索接入与长驻缓存

> 本文承接 [`P5.0-检索层.md`](P5.0-检索层.md)：P5.0 已交付 `guanlan search` 与
> `guanlan.search` 内核（BM25 + CJK 2-gram、零 LLM、零写盘），但当前 Web UI 的搜索框仍只是前端对
> `/api/pages` 返回的 `title/path` 做本地过滤；而 Web 问答里**只读 Agent 没有 shell、根本跑不了
> `guanlan search`**（仅 workspace-write 会话能跑、且每轮冷 shell out），P5.0 加的“先 `guanlan search`”
> 召回步在只读姿态下今天是死指令。P5.1 的目标是把 P5.0 内核接入 Web 长驻进程：既让浏览器搜索吃到正文
> BM25 召回，又用一个 `is_read_only=True` 的宿主工具让**只读 Web 会话第一次拥有确定性召回入口**，并复用
> 同一个 `CorpusCache` 减少每轮读盘/分词。
>
> **不是 E1。** P5.1 不引入向量、重排、持久化倒排索引、多租户隔离或外部检索服务；仍保持 P5.0 的
> “markdown 是唯一事实来源、索引可即时重建、不落盘派生”的边界。它只是把 P5.0 已有的进程内
> `CorpusCache` 用在 P4 Web 宿主这个长驻进程里。

---

## 1. 现状与问题

### 1.1 Web UI 搜索不是 P5.0

当前右栏 `#wiki-search` 的行为是：

1. 启动/刷新时 `GET /api/pages` 拿全量页面清单；
2. 前端在 `allPages` 里用 `p.title.toLowerCase().includes(q)` 或 `p.path...includes(q)` 过滤；
3. 只展示标题列表，不搜正文、不返回 snippet、不打 BM25 分数。

所以它只能找“标题或路径里出现关键词”的页，找不到正文里的命中，也吃不到 P5.0 的 title/aliases boost、
CJK 2-gram、BM25 排序。

### 1.2 只读 Agent 今天根本调不了 `guanlan search`，P5.0 的召回步在只读姿态下是死指令

P5.0 已把 `query.py` 与 `skills/guanlan-wiki/SKILL.md` 改成“先 `guanlan search "<关键词>"` 召回候选页”。
但 `guanlan search` 是一条 **shell 命令**，而 Web chat **默认 read-only**（`--reader` 更是**强制** read-only，
见 `app.py` 的 reader 钳制）。只读姿态下 agentao 把 `run_shell_command`/`shell`/`bash` 一律 DENY
（`chat.py` 的 `_SHELL_TOOL_NAMES` + agentao `tool_planning._decide`：`readonly_mode and not is_read_only → DENY`）；
CLI 的 `guanlan query` 默认路径同样是 `permission_mode="read-only"`（`query.py:_run_readonly`）。

所以现状是两件事，**都与“重复冷算”相反**：

- **只读会话（默认 + reader，主流）**：Agent **没有 shell、跑不了 `guanlan search`**。它只能用
  `grep`/`search_file_content` 做裸关键词扫，吃不到 P5.0 的 CJK 2-gram / 别名 boost / BM25 排序。
  即 **P5.0 在 SKILL/`QUERY_PROMPT` 里加的“先 `guanlan search`”召回步，在只读姿态下今天是死指令。**
- **workspace-write 会话（少数，需 `/mode` 显式切）**：Agent 才有 shell，能跑 `guanlan search`；此时它走
  一次性子进程冷算——

  ```text
  Web workspace-write 会话
    -> shell 命令 `guanlan search "..."` -> 新 CLI 进程
    -> search_pages(root/wiki, ...) -> build_corpus(...) 冷算：遍历 wiki、读每页、分词、打分
  ```

  P5.0 的 `CorpusCache` 只活在 Python 进程内、不跨 shell out 复用，故**写姿态会话每轮重建语料**。

结论：P5.1 要解决的**不只**是“写姿态重复冷算”，**更主要**是“**只读姿态根本没有确定性召回入口**”——
这正是 §5 宿主工具的价值（给只读 Agent 一个 `is_read_only=True` 的召回工具）。

---

## 2. 目标与范围

**P5.1 = 让 Web 长驻进程直接调用 P5.0 检索内核，并复用同一个 `CorpusCache`。**

纳入 P5.1：

- 新增 `GET /api/search?q=...&limit=10`，直接调用 `guanlan.search.score(cache.corpus(root / "wiki"), q, limit=...)`。
- `create_app()` 初始化 `app.state.search_cache = CorpusCache()`；缓存进程内、不落盘、按 `(mtime_ns,size)` 增量失效。
- 前端右栏搜索从“本地 title/path 过滤”升级为“正文 BM25 检索结果列表”：展示 `title/path/type/score/snippet`，点击结果打开页。
- Web chat 增加宿主侧 `is_read_only=True` 的 cached search 工具：让**只读会话**也能用 P5.0 召回（今天只读
  Agent 无 shell、跑不了 `guanlan search`），并让 workspace-write 会话不再每轮 shell out 冷算。
- P5.0 CLI 保持冷算权威：`guanlan search` 仍不读 Web cache、不依赖 Web 服务。

不纳入 P5.1：

- 不做盘上索引、倒排索引文件、SQLite、向量库或 embedding；
- 不让 CLI query 依赖 Web server；
- 不改变 `query` 的综合、引用、`--backfill` 门禁；
- 不把 Web `/api/search` 做成写端点，不写 `raw/`、`wiki/`、`log.md`、`graph/`；
- 不做多用户权限隔离或 per-user search scope，仍按整个 KB 搜索。

## 3. 后端设计

### 3.1 App 级缓存

在 `guanlan/web/app.py:create_app()` 中初始化：

```python
from ..search import CorpusCache

app.state.search_cache = CorpusCache()
```

缓存生命周期与 Web 进程一致：

- 重启 Web 后缓存为空，首次搜索按需重建；
- 同一进程内后续搜索只对变更/新增页面重跑 `build_doc`；
- 删除页由 `CorpusCache.corpus()` 的 stale 剪枝处理；
- 不落任何磁盘文件，不改变 P5.0 “无盘上派生”的承诺；
- **内存随库大小线性**：cache 常驻持**全语料 body 文本**于 RAM（O(KB)、非按用户；不像 P4.9 会话有 idle 回收，
  它是新的常驻持有者）；极大库另议（盘上索引留 E1，决策 P5.1-2）。

**`/api/search` 与 chat 必须吃**同一个** `app.state.search_cache`**（§2 共享缓存目标的落点）。`/api/search`
端点直接闭包引用 `app.state.search_cache` 即可；但 chat 侧要把它注进**每个 `Conversation`**才共享——当前
`create_app()` 只把 `root/model/write_gate` 传给 `ConversationStore`，`Conversation.__init__` 没有 cache/工具参数、
`build_from_environment(**opts)` 也没传工具（`chat.py:332-345`）。故 P5.1 须**显式新增改造点**：
`ConversationStore(..., search_cache=app.state.search_cache)` → 透传 `Conversation.__init__(..., search_cache=...)`
→ 在构造 agent 时**每会话用 `make_guanlan_search_tool(search_cache, wiki=root/"wiki")` 新建一个工具实例、只共享
同一个 `CorpusCache`**（§5；切忌跨会话复用同一 Tool 实例）。**新建与懒恢复两条路径都要带上**——懒恢复同样
`build_from_environment` 重建 agent（`chat.py` restore 路径），漏掉则恢复出的会话没有召回工具。缺这一步，
实现极易退化成“每会话一个 cache”，破坏 P5.1 的核心目标。

### 3.2 HTTP 契约

新增只读端点：

| 方法 路径 | 入参 | 返回 | 错误 |
| --- | --- | --- | --- |
| `GET /api/search` | `q: str`，`limit: int = 10` | `{ok:true, query, pages_searched, results}` | `422` 空/纯标点 query 或 `limit < 1` |

响应体与 P5.0 JSON 契约保持同形，便于前端、未来 MCP、测试共用：

```jsonc
{
  "ok": true,
  "query": "去中心化金融",
  "pages_searched": 795,
  "results": [
    {
      "page": "wiki/entities/DeFi.md",
      "title": "DeFi",
      "type": "entity",
      "score": 8.4213,
      "snippet": "DeFi（去中心化金融）指建立在公链上的..."
    }
  ]
}
```

端点伪代码：

```python
@app.get("/api/search")
async def api_search(
    q: str = Query(..., min_length=1),
    limit: int = Query(10, ge=1),
) -> dict:
    if not tokenize(q):
        raise HTTPException(status_code=422, detail="query 为空或无可检索词。")

    def _run() -> SearchResult:
        docs = app.state.search_cache.corpus(root / "wiki")
        return score(docs, q, limit=limit)

    result = await anyio.to_thread.run_sync(_run)
    return search_result_dict(result)
```

`search_result_dict(result)` **必须**新增在 `guanlan/search.py` 作为**单一归口**，且 P5.0 的 CLI `_render`
JSON 分支也改调它（不再手写映射）——CLI / Web / 未来 MCP 三处共用同一份字段+取整映射，杜绝漂移（这正是
P5.0 用 `SearchResult` 单一归口的初衷，决策 P5.1-4）。**不保留“各自保持同形”的退路。**

**错误体形态**：成功体与 P5.0 `--json` 同形；但**错误体走 HTTP 原生** `status_code`(422) + `{"detail": ...}`，
**不**复刻 P5.0 CLI 的 `{"ok":false,"error":...}`（决策 P5.0-21 是 CLI 传输的约定，HTTP 用 status 码表达错误）。
前端/下游按 **HTTP 状态码**判错、成功时才按 `ok`/字段取值——两种传输各自自洽，不强求错误体同形。

### 3.3 线程安全

FastAPI 同步工作经 `anyio.to_thread.run_sync` 进入线程池；多个 `/api/search` 可并发进入同一个
`CorpusCache.corpus()`。P5.0 已要求 `CorpusCache` 内部持 `threading.Lock`，因此 P5.1 不再在 Web 层另加全局搜索锁。

端点只把“阻塞的 stat/读盘/分词/打分”卸到线程，事件循环不直接跑 `build_corpus`。

**临界区收窄（决策P5.1-8，已落地）**：`corpus()` 把**重活（glob + 全库 stat + `build_doc`）放在锁外**，
`threading.Lock` 只护两段纯字典操作——拍桶快照、合并重建 + 剪枝。代价是放弃「全程单锁」的双重-build 抑制：
并发线程可能对同页各 build 一次、合并 last-writer-wins，但 `(mtime_ns,size)` 键保证下次 stat 必重建，最坏多
算一次、绝不永久陈旧；返回值只取「本调用 stat 对齐的命中/新建 DocBag」，故与冷算字节等价、确定性不变。

**实测后的真热点订正**：起初以为瓶颈是「锁内 O(N) stat」，profile（10000 页）证否——**stat 仅占整次
`corpus()` 的 ~9%（18ms）**，真正的大头是按页算相对库根 key `path.relative_to(root).as_posix()`：纯 Python、
受 GIL 串行，**占整次 ~55%（111ms）**。故 P5.1-8 的实质优化是把该 key 改用**字符串切片**（`os.fspath(path)`
切掉 `root` 前缀，POSIX 上字节等价于 `as_posix`，实测 111ms→1ms），使热缓存单查 corpus() 从 ~200ms 降到
~80ms（Web 热路径单查 ~217ms→~94ms，**降 ~57%**）。**GIL 天花板**：因 91% 是 Python 串行工作，并发
`/api/search` 仍无法真正并行（`score()`/stat 的 syscall 段可释放 GIL，但 glob+字典段不行）；但单查 GIL 占用
减半 ≈ 竞争下吞吐翻倍。**残余热点**：`iter_pages` 的 `rglob+sorted`（~65ms，现占 ~82%）是下一个杠杆，但它是
check/health/lint/graph 共享归口，改它（如 `os.scandir` / 目录-mtime 失效）超出 P5.1 范围、另议。

## 4. 前端设计

### 4.1 右栏搜索升级

`#wiki-search` 继续作为右栏搜索入口，但语义从“标题过滤”升级为“全文召回”：

- 输入为空：保留现有 index 列表（按 type 分组展示所有页）；
- 输入非空：debounce 后调用 `/api/search?q=...&limit=20`；
- 展示结果列表：标题、路径/type、score、snippet；
- 点击结果仍走现有 `navigate({kind:"page", path})`；
- `/api/search` 失败时显示错误空态，不清空 `allPages`。

**必须防迟到响应覆盖（与单页渲染 `renderToken` 同纪律）**：debounce 只压频、**不防乱序**——“先搜 `A`、
再搜 `AB`，`A` 的响应后到”会用旧词结果盖掉新词。每次发起搜索须带一个**单调 token**（或 `AbortController`
取消上一发），响应回来时**只在 token 仍是最新时**才渲染、否则丢弃；切到空输入/导航离开时也作废在飞请求。
不加这条，远端搜索必现 stale-overwrite 的 UI bug。

搜索态建议作为新的 view kind：

```js
{ kind: "search", query: "去中心化金融", results: [...] }
```

这样浏览器式 back/forward 可以恢复搜索词与结果，不把搜索结果混进现有 `{kind:"index"}` 的本地过滤语义。

### 4.2 与 `/api/pages` 的关系

`/api/pages` 仍保留，职责是“页面目录/空 query 首页/语言切换后重绘列表”。`/api/search` 只服务非空查询。

为消除“搜索框找不到正文”的错觉，P5.1 **钉死**：**非空输入一律走 `/api/search`**（不再对非空输入做本地
`title/path` 过滤）；本地列表只服务**空输入**（按 type 分组的目录首页）。即“本地过滤”语义只保留给空输入的目录
展示，不再与正文检索并存——避免两套搜索语义按代码路径分叉、保留它自己点名的那个错觉。

## 5. Web 问答接入（P5.1 的核心价值，非附属）

§1.2 已说明：只读 Agent 没有 shell、跑不了 `guanlan search`，所以这一步**不是“去掉重复 shell out”那么简单**——
它是**让只读 Web 会话第一次拥有确定性召回入口**（CJK 2-gram / 别名 boost / BM25），并顺带让 workspace-write
会话不再每轮冷算。

路径：给 Web 嵌入 Agent 注册一个**宿主侧只读工具** `guanlan_search`：

```text
guanlan_search(query: str, limit: int = 10)   # 工具元数据必须 is_read_only=True
  -> limit = max(1, int(limit))                # 工具路径自校验：score 对 limit<1 raise（决策 P5.0-15）
  -> docs = 共享 search_cache.corpus(wiki)      # 注入的同一个 CorpusCache（§3.1）；wiki 由工厂捕获
  -> result = score(docs, query, limit=limit)   # 阻塞活：须卸到线程、勿堵事件循环（见下「不堵 loop」）
  -> return json.dumps(search_result_dict(result), ensure_ascii=False)   # 工具 execute 返回 str
```

**返回类型（务必区分两条出口）**：agentao 工具契约是 `execute()`/`async_execute() -> str`，执行器把返回值当
`result_text` 往下传。故**Agent 工具返回字符串**——`json.dumps(search_result_dict(result), ensure_ascii=False)`，
**不是 dict**；而 **HTTP `/api/search` 返回 dict**（FastAPI 序列化）。两者同源于 `search_result_dict`、但出口类型不同，
照搬伪代码把 dict 塞进 tool result 会出问题。

**关键约束（否则白注册）**：

- **`is_read_only=True` 是硬要求**。否则 agentao 在只读姿态下按 `tool_planning._decide`
  （`readonly_mode and not is_read_only → DENY`）会把它和 shell 一起拦掉（镜像 `chat.py:_blocked_in_readonly`），
  只读会话依旧召回不了——正是本阶段要解的问题。该工具纯读 `wiki/`、不写盘，标 `is_read_only=True` 名正言顺。
- **共享 cache、但每会话独立 Tool 实例**：工具后端必须是 §3.1 注进 `Conversation` 的那个 `app.state.search_cache`，
  不能在工具内自建 `CorpusCache()`（否则每会话一个 cache，回到 P5.1 要消除的问题）。但**Tool 实例不可跨会话复用**——
  agentao 注册 `extra_tools` 时会把 `working_directory`/`filesystem`/shell 等**绑定写到工具实例上**，复用同一实例会
  让后注册会话**覆盖**前一会话的绑定。故用工厂 `make_guanlan_search_tool(search_cache, wiki=root/"wiki")`（两参在
  `create_app` 期已知、对全会话恒定）**每个 `Conversation` 新建一个实例**，闭包捕获共享 `CorpusCache` 与固定 `wiki/`
  路径——**不**依赖 agentao 绑到实例的 `working_directory` 取路径（少一层隐式耦合）。即：**共享 cache、不共享 Tool 实例。**
- **阻塞活的线程归属（别误判为必然钉死事件循环）**：`guanlan_search` 的 `corpus()/score()` 是与 `/api/search`
  同样的阻塞活、且要拿 `CorpusCache` 的 `threading.Lock`。但 agentao 的 `.arun()` **已把整轮 `chat()` 放进
  `loop.run_in_executor()`**——同步工具的 `execute()` 跑在**该会话 turn 的 chat-executor 线程**里，**不**直接跑在
  FastAPI 事件循环上，故**不会**“卡住整个 loop / SSE”。代价是它**占住该 turn 的 executor 线程**整段时长（全库 stat +
  重建 + BM25），并持 `CorpusCache` 锁、与并发 `/api/search` 的全库重建串行。实现成 `AsyncToolBase.async_execute` +
  内部 `await anyio.to_thread.run_sync(...)` 可把实际 stat/score 再挪到 anyio 线程池、**确保阻塞活不在 event loop 上
  执行**；**但这不释放该 turn 的 executor 线程**——agentao 的 async bridge 把 coroutine 调回 host loop，chat-executor
  线程仍卡在 `run_coroutine_threadsafe(...).result()` 上**等工具完成**（无论 `execute` 还是 `async_execute`，该 turn 都
  占着一个 worker，直到召回返回）。spike 顺带确认线程归属，据此决定是否值得上 `async_execute`。
- **工具路径自校验 `limit`**：HTTP 有 `Query(ge=1)` 兜底，工具路径是 LLM 填参、无此门。`score` 对 `limit<1`
  **raise ValueError**（决策 P5.0-15），故工具体先 `limit = max(1, int(limit))`（或 catch `ValueError` 回错误串）；
  空/纯标点 query 不必额外管——`score` 对零 query-token 短路回空、安全。
- 注册后须同步进 P4.4 的 `/tools` 自省与 `/context` token 计量（新工具改 tools schema、影响 token breakdown），
  否则自省面与实际工具集漂移。

提示词调整为“在 Web 宿主中优先调用 `guanlan_search` 工具召回候选页”。**注意 CLI fallback 的可达性**：
“工具不可用时退回 `guanlan search` CLI”**只在 workspace-write 会话成立**（那里才有 shell）；**只读会话没有 shell、
无 fallback 可退**——故 `guanlan_search` 工具在只读姿态下是**唯一**召回入口，不能把 CLI 当通用安全网。

**消解与共享 SKILL 的指令冲突**：嵌入会话激活的是**共享** `skills/guanlan-wiki/SKILL.md`（`chat.py`
`activate_skill(SKILL_NAME)`），而 P5.0 把其 query step 1 改成了“先 `guanlan search "<词>"`（shell 命令）”。只读
Web Agent 读到这条会去 shell（被 DENY、白费一轮），却不知有 `guanlan_search` 工具。必须二选一并写进实现：
**(a)** 把召回措辞改成**传输中立**的“用可用的 search 入口（宿主 `guanlan_search` 工具 / `guanlan search` CLI）召回，
都不可用则退回扫目录/补词”；**或 (b)** Web system prompt 显式覆盖 SKILL 的 CLI 指令。倾向 **(a)**。

**但 (a) 涉及两处硬编码召回措辞**——`skills/guanlan-wiki/SKILL.md` step 1 **与** `guanlan/query.py` 的 `QUERY_PROMPT`
（P5.0 同样把它改成了“先用 `guanlan search`”）。要兑现“顺带修好只读 CLI `query` 的死指令”（`query.py:_run_readonly`
无 shell、宿主工具又不在 CLI 里），**两处都得改成传输中立措辞**；**只改 SKILL、漏改 `QUERY_PROMPT`，CLI `query` 的
死指令仍在、是验收漏项**。若选 (b)（仅 Web prompt 覆盖）——它只修 Web、不碰 CLI，那就**别承诺修 CLI `query`**（或仍
单独改 `QUERY_PROMPT`）。

这样同一 Web 进程内：右栏 `/api/search` 与 chat 召回共享同一个 `CorpusCache`，一次会话多轮问答不重读全库。

**真实未知不是“能否注册”，而是“怎么注入”（spike 方向据此校准）**：agentao **已有**构造期 `extra_tools=`
（`build_from_environment(**opts)` 可带）与运行期 `add_tool()` 两条注入面，注册本身不是问题。P5.1 要定的是：

- **构造期 `extra_tools=` 还是运行期 `add_tool()`**：观澜的 `Conversation` 在 `__init__` 里一次性
  `build_from_environment(**opts)`（`chat.py:332-345`）。倾向**构造期 `opts["extra_tools"]=[guanlan_search]`**——
  与现有 transport/filesystem 同在构造期注入，一处装齐、无“构造后再补挂”的时序窗口。
- **如何传共享 cache**：工具需绑定 §3.1 注进 `Conversation` 的 `search_cache`（闭包/工厂捕获该实例），不在工具内自建。
- **restored conversation 同样带工具**：懒恢复也走 `build_from_environment` 重建 agent，注入逻辑必须**对新建与
  恢复两条路径一致生效**，否则恢复出的会话丢召回工具。

**建议切片 3/3 先做一个窄 spike**：用 `extra_tools=[make_guanlan_search_tool(search_cache, wiki=root/"wiki")]`
构造期注入（**每会话新实例、只共享 cache**）一个 `is_read_only=True` 的 `guanlan_search`，验证“只读会话能调到、
返回 `str`、`limit=0` 被 clamp 不抛、确认线程归属（`chat()` turn 在 `run_in_executor` 线程、非事件循环——据此选
`execute` 还是 `async_execute`）、`/tools` 能列出、恢复会话也带它、两会话工具实例相异但 cache 同一”，再定最终契约与提示词。

**不建议**让 `guanlan query` CLI 反过来请求 Web `/api/search`。CLI 应继续独立、可离线、冷算权威；Web cache 是长驻宿主优化，不是全局服务依赖。

## 6. 决策

- **决策P5.1-1（Web 直接复用 P5.0 内核，不 shell out CLI）**：`/api/search` 必须 import
  `guanlan.search` 内核并调用 `score(cache.corpus(...))`。shell out `guanlan search --json` 虽然省代码，但每次新进程冷算，正是 P5.1 要解决的问题。
- **决策P5.1-2（缓存只在 Web 进程内，不落盘）**：沿用 `CorpusCache` 的 `(mtime_ns,size)` memo；不新增 `.guanlan/search-cache`、SQLite 或倒排文件。盘上持久化索引留给 E1 或实测后另文。
- **决策P5.1-3（CLI 冷算保持权威）**：`guanlan search` 不依赖 Web server、不读 Web cache。冷算路径仍是可复现实验与排错基准。
- **决策P5.1-4（成功体经 `search_result_dict` 单一归口；错误体走 HTTP 原生）**：成功体字段+取整由
  `guanlan/search.py` 新增的 `search_result_dict(result)` **单一归口**产出，CLI `_render` 的 JSON 分支也改调它，
  CLI/Web/MCP 三处不再各写映射（消漂移）。**错误体**走 HTTP `status`(422)+`{"detail":...}`，不复刻 CLI 的
  `{"ok":false,"error":...}`——下游按状态码判错、成功才按 `ok`/字段取值。
- **决策P5.1-5（空 query 不走检索）**：前端空输入展示页面目录；后端空/纯标点 query 返回 422。避免“空输入触发全库 BM25”与现有首页语义冲突。
- **决策P5.1-6（Web chat 用 `is_read_only=True` 宿主工具召回；CLI fallback 只对写姿态可达）**：只读会话没有
  shell、跑不了 `guanlan search`，故宿主工具是其**唯一**召回入口、且必须 `is_read_only=True` 才不被只读 DENY；
  “退回 CLI”仅在 workspace-write 会话成立，不是通用安全网。绕开 shell out 不只是省冷算，更是让只读会话首次具备
  确定性召回。
- **决策P5.1-7（只读端点，不参与写线）**：`/api/search` 不入 `JobQueue`、不取 raw 快照、不 bump wiki generation。它只读 `wiki/`，与 `/api/pages`、`/api/page` 同属读线。并发 ingest/heal 子进程正写 `wiki/` 时，`corpus()` 可能 stat 到半写页——**接受**（最终一致、best-effort 召回；下次搜索按 `mtime` 自动重建该页），**不**为此加写锁。
- **决策P5.1-8（`corpus()` 临界区收窄 + key 切片，实测驱动）**：把 glob/stat/`build_doc` 挪出锁、`Lock` 只护
  「拍桶快照」「合并重建 + 剪枝」两段纯字典操作；剪枝按 `snapshot - seen`（不动并发线程在拍快照后新加的页）。
  返回值取「本调用 stat 对齐的命中/新建 DocBag」、不从合并后的共享 cache 取，保字节等价与确定性。**关键订正**：
  profile 证明 stat 仅占 ~9%、真热点是 `relative_to().as_posix()` 算 key（~55%），故改用 `os.fspath` 字符串切片
  （POSIX 上字节等价、快 ~100×），热缓存单查 ~200ms→~80ms。**不**追求并发并行（GIL 串行 91% 的 Python 工作，
  锁外也不并行）——本决策是**降单查延迟**（连带 GIL 占用减半→竞争吞吐改善），非解 GIL。盘上索引/scandir 留 E1。

## 7. 测试要点

- `/api/search?q=正文词` 能召回标题/路径不含该词但正文含该词的页面；旧本地过滤做不到。
- `search_result_dict` 单一归口：CLI `guanlan search --json` 与 `/api/search` 成功体**字段/结构同形**——JSON
  解析后相等、同一 `search_result_dict` 映射产出；**非字节级**（HTTP 经 FastAPI 紧凑序列化、CLI 经 `indent=2`，
  缩进/编码不同，不强求字节一致）。字段为 `ok/query/pages_searched/results[].page/title/type/score/snippet`。
- `limit < 1`、空白 query、纯标点 query 返回 **422 + `{"detail":...}`（HTTP 原生错误体，非 `ok:false`）**；
  有词但无命中返回 `ok:true, results:[]`。
- `CorpusCache` 被复用：同一 app 内连续两次搜索，未变页面不重建；修改一页后只该页重建，结果与冷算 `search_pages` 等价。
- 并发两个 `/api/search` 不损坏 cache，不抛竞态错误。
- reader 模式下 `/api/search` 仍可用，且不写 KB、不开日志、不注册任何写端点。
- 前端空输入仍显示页面目录；非空输入展示 BM25 结果，点击结果打开对应页面。
- **前端防迟到覆盖**：连发两次搜索（旧词响应人为后到），UI 只渲染最新词的结果、丢弃迟到响应（token/AbortController）。
- 只读会话能调到 `guanlan_search` 宿主工具（`is_read_only=True`、不被只读 DENY），复用 `CorpusCache`、不执行
  `guanlan search` shell；CLI fallback 仅 workspace-write 会话可达（只读无 shell、无 fallback）。
- **指令不冲突（含 CLI prompt）**：走 (a) 时 `skills/guanlan-wiki/SKILL.md` step 1 **与** `guanlan/query.py`
  `QUERY_PROMPT` 均改为传输中立措辞、**不含**硬编码“先用 `guanlan search`” CLI 串（可加 prompt-substring 断言防回归）——
  只读 Web Agent 去召回时调 `guanlan_search` 工具、不白费一轮试 shell，且只读 CLI `query` 的死指令一并消除。
- **工具返回 `str`**：`guanlan_search` 的 `execute` 返回 `json.dumps(search_result_dict(...))` 字符串（非 dict），
  执行器拿到的 `result_text` 可被 `json.loads` 还原成 P5.0 同形对象。
- **工具路径 `limit` 自校验**：工具收到 `limit=0`/负数时 clamp 到 ≥1（不让 `score` 的 `ValueError` 冒泡成工具崩溃）。
- **工具不堵事件循环 / SSE**：`chat()` turn 在 `run_in_executor` 线程跑，长 `/api/search` 全库重建并发期间调工具，
  FastAPI 事件循环与 SSE token 流仍保持响应。**注**：无论 `execute` 还是 `async_execute`，该 turn 的 chat-executor
  线程都被工具占用至完成——`async_execute` 只是把阻塞活挪出 event loop（到 anyio 线程池），**不**释放该 executor 线程。
- **共享 cache、不共享 Tool 实例（覆盖新建 + 懒恢复两条路径）**：同一 app 内新建会话与懒恢复会话的 `guanlan_search`
  工具后端 `is` 同一个 `app.state.search_cache`（不每会话自建 cache）；但两会话的 **Tool 实例相异**（`is not`，
  避免 agentao 把 per-agent 绑定写到共享实例上互相覆盖）。
- 新工具出现在 P4.4 `/tools` 列表且标 `is_read_only`，`/context` token breakdown 含其 schema。
