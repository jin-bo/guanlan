## 0. 一句话

`guanlan remove <源>`：一条**人发起、零-LLM、确定性、小表面积**的源撤回命令（一期目标 = 像 `reindex` 那样易测的维护命令，不是回收站系统）——把一个误摄 / 已撤稿源的"自身落盘物"（`raw/<slug>.md` + 可选 `raw/images/<slug>/` 图片、`wiki/sources/<slug>.md` 摘要页）**整体移入回收区 `<kb>/.trash/<slug>@ts>/` 而非 `rm`**（gbrain 软删形状 + `manifest.json` 留档，给反悔窗），多源衍生页里的 `<slug>` 引用**确定性摘除**（provenance 编辑、正文一字不改），独源衍生页 / 撤回后留下的悬链**只报 advisory、不删不改**。默认**预览不写**（比 `reindex`/`convert` 更保守），显式 `--yes` 才落。**宿主不起 Agentao、不二次综合**——失源页的正文重综合是 LLM 的活，留给后续 `ingest` / 人工。自动 `restore` / 回收区 GC / 级联删独源页均为**后续项**（决策P3.9-10）。

## 1. 目标与范围

### 1.1 真实缺口

观澜当前**完全无路**撤回一个源。一旦 `raw/foo.md` 落盘 + `ingest` 建了页，它就**只增不减**：

- `raw/` 不可变 + 快照门禁（`gate.snapshot_raw` → `EXIT_RAW_MUTATED`）——任何对 `raw/` 的改动都被 Agentao 写窗里的快照拦死；
- 源不回退（P2.1，`run_guarded_write` 的 `sources.dropped` 阻断）——既有页的 `sources` 集**只增不减**。

这两道闸是**对的**（防 Agent 在摄入里悄悄抹掉来源），但它们约束的是 ***Agent* 的写**。运营现实里，源会**误摄**（投错文件）或**已撤稿**（上游撤回了那篇论文 / 报告），此时需要一条**人发起的纠正路径**。`remove` 就是这条路径，与 P4.1 投喂（人加源）/ `convert`（宿主写 `raw/`）**结构对称**——只是方向相反。

### 1.2 一个"源"在盘上的三层结构

撤回一个 slug 为 `<s>` 的源，须同时处理三层（**任一层可能缺**，命令对"现存的那几层"操作、缺的跳过）：

| 层 | 落点 | 性质 | 撤回动作 |
|---|---|---|---|
| **① 原始素材** | `raw/<s>.md`（+ 可选 `raw/images/<s>/`，见 [`P5.2.1-图片落盘.md`](P5.2.1-图片落盘.md)） | 不可变源 | **移入 `.trash/`**（人发起，越过 raw 不可变闸——见 §6） |
| **② 摘要页** | `wiki/sources/<s>.md`（`type: source`，可能带 `raw_digest`） | 内容页（图节点） | **移入 `.trash/`**（删内容页 ← 决策P3.9-1 核心） |
| **③ 衍生引用** | `entities/`·`concepts/`·`syntheses/` 各页 frontmatter `sources: [..., <s>, ...]` | 内容页 | **从 `sources` 摘除 `<s>`**（provenance 编辑，正文不动） |

三层的对应关系由**既有口径**确定、不新造解析：raw stem ≡ source slug ≡ 摘要页 stem（投喂 / `convert` / `ingest` 的命名约定）；衍生页 → 源的引用解析复用 `check._check_sources`（`check.py:114`：`sources` 里每个 slug 必须解析为 `wiki/sources/<slug>.md` 的**直接**文件，禁 `/`·`\`·`..`）。

### 1.3 为什么能零-LLM —— 以及刻意不做的那半

撤回的**结构 / provenance 部分全是确定性的**，无须 LLM：

- **定位**三层 = 字符串 slug + 既有路径约定 + `iter_pages` 反扫；
- **移动**三层 = 文件系统 `os.replace` 移入 `.trash/`；
- **摘除** `<s>` = 读 frontmatter → 从 `sources` 列表去掉一项 → 重序列化（`yaml.safe_dump`，复用 `rawio.apply_origin` 同款"`split_frontmatter` → 改 → dump"模式，**绝不裸拼**）；
- **修 index** = 删摘要页的登记行（复用 `reindex._prune_dangling` 的精确单行匹配口径，`reindex.py:154`）。

**唯一需要"生成"的——一个多源衍生页失去某源后正文该如何重写——是 LLM 的活，`remove` 明确不做**（§7）。这与 `reindex` 的立场同构：`reindex` 把页"登记可见"、把人话摘要留空给后续 `ingest`；`remove` 把源"结构性撤干净"、把"失源页的正文重综合"留空给后续 `ingest` / 人工（产一条 advisory 提示，见 §3.4）。**薄壳纪律：宿主做机械的 provenance / 搬迁，不碰语义重写。**

### 1.4 为什么不照搬 OpenKB

OpenKB `remove` 的**形状**可借（删源 + 清派生 + 修 index + `--dry-run`），但两点实现**不照搬**：

1. **OpenKB 是直接清理（`rm`）**；本设计取 **gbrain 软删的"先移回收区"形状**（一期：移 `.trash/` + `manifest.json` 留档；TTL/`--purge`/自动 `restore` 留二期，决策P3.9-10）——撤回不立即硬删，与"`raw/` 不可变"张力更小（不是改写源、是把源**整体可逆地移到回收区**），且给误删一个反悔窗。
2. **OpenKB 无匹配时 strip 链接**（去括号留文本）——**绝不借**：删前向引用占位与决策8「断链是会自我消除的生长信号、警告非阻断」对撞（[`broken-link-handling-survey.md`](backlog/notes/broken-link-handling-survey.md) §4.5）。撤回留下的悬链 / 孤儿一律 **advisory，不自动改写**。

`recompile`（OpenKB）同样**别借**：覆盖人工编辑、悖增量 ingest（[`openkb-反向评审结论.md`](backlog/notes/openkb-反向评审结论.md) §1）。

## 2. 命令契约

**一期（核心闭环）：**

```bash
guanlan -C <kb> remove <源>          # 默认：只打印撤回 worklist，零写盘（人审预览）
guanlan -C <kb> remove <源> --yes    # 确认执行：移源自身落盘物入 .trash/ + 摘多源页引用 + 修 index + 写 manifest
guanlan -C <kb> remove <源> --json   # 结构化回执（worklist / 已执行）
```

`<源>` 三种写法**归一到同一 slug**（复用 `rawio` slug 归口 + §1.2 路径约定）：`<slug>` / `raw/<slug>.md` / `wiki/sources/<slug>.md`。

**后续项（不在一期，决策P3.9-10）**：`restore`（自动逆转——manifest 已留好配方，冲突/合并语义留二期；一期靠人工从 `.trash/` 取回 + 按 manifest 补回 slug）、`remove --purge`/回收区 GC（→ 未来 `trash purge`；一期靠人工清 `.trash/`）、`--cascade`/独源页删除（→ 未来 `prune-orphans`）、`--keep-raw`（部分撤回模式）。

**契约要点（守"半阶段不引新退出码 / 新数据模型"纪律）：**

- **默认预览、显式 `--yes` 才写**（决策P3.9-1 / 决策P3.9-9，§8）：`remove` 删内容页、比 `reindex`/`convert` 的默认即写更具破坏性，故采 OpenKB"预览→人审→执行"的更保守默认。**不另设 `--dry-run` 别名**（不带 `--yes` 即预览，按简明路线不扩参数面，评审发现4）。
- **无新退出码**：成功恒 `EXIT_OK(0)`；非知识库根 / 找不到源 / `<源>` 越界等用法错 `EXIT_USAGE(1)`；落盘 IO 异常 → `EXIT_USAGE`（命令壳 `except OSError`，镜像 `convert`/`graph`）。**不**引 `EXIT_RAW_MUTATED`（`remove` 是 raw 的**合法写者**，不经快照门禁——§6）。
- **不起 Agentao、不写 `log.md`**（决策P3.9-7）：与 `reindex` 同（决策P3.4-6）——零-LLM 宿主维护不碰 `log.md`，`.trash/<id>/manifest.json` 本身就是审计轨迹。
- **不承诺事务级回滚，改 manifest-先写 + 幂等向前收敛**（评审发现6，决策P3.9-10）：文件搬迁 + frontmatter 重写 + index 重写跨多文件，难做简单可靠的全量回滚。故执行序为 ① 写 `manifest.json`（记下全部意图，先落） → ② 摘多源页 slug → ③ 删 index 行 → ④ **最后**移源自身落盘物入 `.trash/`，**每步幂等**（slug 已不在 → no-op、行已删 → no-op、源位已空 → 跳过）。**移源放最后是关键**：源文件是 locate 的锚，若先移走、中断后重跑会 locate 失败而无法收敛；放最后则中断后锚仍在、`<slug>` 仍可被定位，**重跑同命令即向前收敛**（同 OpenKB"commit point + 之前步骤幂等可重跑" / `reindex`·`heal` 的幂等纪律）。中途 IO 失败 → **报 partial 状态**、不假装回滚。manifest 先写只保证**恢复所需的意图记录存在**——**不**保证"恢复永远可行"（评审发现1）：源可能尚未移入 `.trash/`、部分 wiki 页可能已改、原路径后续可能被占，实际恢复仍需检查文件现状。

## 3. 撤回流程（四步确定性）

`run_remove_result(root, slug) -> RemovePlan`（**纯函数、只读、不写盘**，仿 `reindex.run_reindex` / `heal.run_heal_result` 的"算 plan / entrypoint 决定写"分立；一期无 `cascade`/`keep_raw` 入参）：

1. **定位**：解析 `<源>`→`<slug>`；探源自身落盘物是否存在（`raw/<slug>.md` / `raw/images/<slug>/` / `wiki/sources/<slug>.md`）。全缺 → `EXIT_USAGE`「找不到源 `<slug>`」。
2. **反扫衍生引用**：`iter_pages(wiki)` 一遍，对每页用 `gate._trusted_sources`（`gate.py:220`，复用，不重写 `sources` 解析）取其可信 `sources` 集；命中 `<slug>` 的页分两类——
   - **多源页**（`sources` ⊋ `{<slug>}`）：列入"摘除"清单（只去 `<slug>`、页留）；
   - **独源页**（`sources` == `{<slug>}`，且非摘要页本身）：列入"内容孤儿" advisory `remove.orphaned_page`——**一期只报不删**（删独源内容页 = `--cascade`，后续项；见决策P3.9-3/10）。
3. **算 index 改动**：摘要页 ② 在 `index.md` 的登记行——精确调 `reindex._prune_dangling(lines, {f"sources/{slug}.md"})`（评审发现5：入参是相对 `wiki/` 的链接目标集合、非页面对象；该函数删"整行链接目标 ⊆ 给定集合"的行，单行单链接故精确删该行、不误删邻行；不另造 index 删除逻辑）。
4. **回执**：汇成 `RemovePlan`（`relocate`[] / `drop_slug`[] / `orphans`[] / `index_lines`[]）。

> **撤回后留下的悬链 = 交既有 `lint`/`check`，不在 `remove` 内算**（评审发现4，决策P3.9-10）：别页 `[[link]]` 指向被撤摘要页将悬空，但**精确入链需解析 stem→path，且 `Graph.Node.id` 是小写 stem、`sources/foo` 与 `entities/foo` 同 stem 会歧义**——不值得在 `remove` 内重造。`check.wikilink.broken` / `lint.broken_link` 已用 P3.8 resolve_owner 的正确口径全库扫断链，故 `remove` 回执只打一句"撤回后请跑 `guanlan lint` 复核断链"，不自算入链列表（决策P3.9-4：悬链一律 advisory、绝不 auto-strip 仍成立）。

`remove_entrypoint`：`--yes` 时按 §2 的 manifest-先写 + 幂等执行序落盘（写 manifest → 摘多源页 slug → 删 index 行 → **最后**移源落盘物），失败报 partial、重跑收敛（**不**承诺事务回滚）；否则 `format_plan` 打印预览。

### 3.4 失源页的正文不一致（advisory，不修）

多源页摘掉某 `<slug>` 后，**正文可能仍引用 / 转述已撤源的材料**——宿主无法（也不该）判定并重写。故对每个被摘除的多源页产一条 advisory：`remove.stale_body`「页 X 已摘除源 `<slug>`，正文可能仍含其内容，建议重 `ingest` 复核」。这把"重综合"明确交回 LLM 工作流，宿主只保证 **provenance 字段一致**（`sources` 不再列已撤源）。

## 4. 回收区 `.trash/` 与一期恢复路径

- **落点 `<kb>/.trash/`**：与 `raw/`·`wiki/` 平级的点目录（同 `.agentao/` sessions，P4.2）——**必须在 `raw/` 与 `wiki/` 之外**，否则污染 `iter_pages`/graph/lint 扫描、且重新触发 raw 快照门禁。
- **结构**：每次撤回一个 `.trash/<slug>@<时间戳>/`，原样保留被移走文件的相对结构（`raw/<slug>.md`、`raw/images/<slug>/`、`wiki/sources/<slug>.md`）+ 一份 **`manifest.json`**（先写、§2）记：撤回时间、slug、被移文件原始路径、**对每个多源页施加的 slug 摘除**（每条形状 `{page, dropped_slug, before_sources}`——`before_sources` 记摘除**前**的整张 `sources` 列表，评审发现2）、**独源孤儿页清单**（`orphaned`——它们未被改动、不进恢复，但记入 blast-radius 审计）、删除的 index 行原文。manifest 是**纯记录**（无校验和 / 无冲突状态 / 无 TTL），**一职三用**——审计轨迹（决策P3.9-7）+ 人工恢复配方 + 二期自动 `restore` 的输入；`before_sources` 既供人工 / 二期 union 并回，又让二期 restore 能**比对页现 `sources` 是否已被改**（冲突检测），故一期就把字段定准、免二期改 manifest 格式。
- **一期恢复 = 人工 + manifest 配方**（评审发现1，决策P3.9-10）：一期**不出自动 `restore` 命令**（其原位冲突 / 与人工改动的合并语义复杂、留二期）。需要反悔时：把 `.trash/<id>/` 下文件移回原位（→ 跑 `guanlan reindex` 复原 index），**再按 manifest 各条 `{page, dropped_slug}` 把 slug 并回对应多源页**（`before_sources` 可辅助核对该页是否被改）。
  > **为何坚持留 manifest（独立于评审的细化）**：评审建议的"人工取回 + reindex"恢复路径**漏了多源页的 slug 并回**——`reindex` 只复原 index、不动衍生页 frontmatter，故没有 manifest 时多源页会永久缺那个已撤源的 slug，"可逆"打折。manifest 是一次廉价 JSON 写（不引入评审担心的 restore 复杂度），却让"恢复"**完整**、并兑现决策P3.9-1 赖以成立的"可逆"前提。故 manifest 留一期，仅 `restore` *命令* 降二期。
- **`markdown 唯一真相`不破**：`.trash/` 非权威、不被任何索引/图扫描；恢复从被移走的真文件重建，index/graph 一律可从 `wiki/` 幂等重建。
- **回收区 GC 降二期**：一期靠人工删 `.trash/<id>/`（文档化即可）；自动按龄硬删（原 `--purge` + TTL + mtime 语义 + 硬删测试）属独立关注点，留未来 `trash purge`（决策P3.9-10）。

## 6. 红线对照（换线不丢的信心锚）

| 红线 | `remove` 怎么守 |
|---|---|
| **`raw/` 不可变 + 快照门禁** | 快照门禁（`gate.snapshot_raw`→`EXIT_RAW_MUTATED`）拦的是 **Agentao 写窗内**对 `raw/` 的改动。`remove` 是**人发起的宿主确定性写**，**不经那条子进程 / 快照路径**（同 `convert` 决策P5.2-3：纯宿主写 `raw/`、不取快照）。它是 `raw/` 的**合法删者**，与 `convert` 的合法增者对称。门禁对 Agent 的保护一字不动。 |
| **源不回退（P2.1）** | P2.1 的 `sources.dropped` 阻断绑在 `run_guarded_write`（**Agent** 摄入只增不减）。`remove` **不走** `run_guarded_write`，故不误触该闸。语义边界澄清：源不回退 = "Agent 写不回退"，**非**"人不可删"。`remove` 是人发起纠正的**唯一受批删者**，与 P4.1 投喂对称。 |
| **零-LLM / LLM 干净分档** | `remove` 全零-LLM、不起 Agentao；落 P3 确定性维护族（`init`/`check`/`health`/`lint`/`graph`/`reindex`/`convert` 同档）。 |
| **markdown 唯一真相、派生可重建** | `.trash/` 非权威、不被扫描；index/graph 一律可从 `wiki/` 幂等重建；恢复从 `.trash/` 真文件 + manifest 重建。 |
| **薄壳（wrapper 不携带业务智能）** | 宿主只做机械 provenance / 搬迁；"失源页正文重综合"是语义活，留 `ingest`/人（§3.4）。`remove` 不判定"该不该撤"（人发起）、不重写正文。 |

## 7. 明确不做（范围闸）

- **不删衍生内容页（一期）**：独源衍生页只报 `remove.orphaned_page`、**不删**。删 Agent 所造的衍生内容页 = `--cascade`，是文档自承的"最重一步"，一期**整条砍掉**、留未来独立 `prune-orphans`（评审发现2，决策P3.9-3/10）——这样一期"宿主删内容页"只触及 1:1 摘要页（§8.2）。
- **不二次综合**：多源页失源后的正文重写——LLM 的活，只产 `remove.stale_body` advisory（§3.4）。
- **不 auto-strip 悬链**：撤回留下的 `[[断链]]` 只报 advisory，绝不去括号 / 改写（决策8 / `broken-link-handling-survey.md` §4.5；OpenKB strip 半不借）；且**不在 `remove` 内自算入链**，交既有 `lint`/`check`（发现4）。
- **不自动 `restore` / 不回收区 GC（一期）**：自动逆转（冲突 / 合并语义）+ 按龄硬删（TTL/`--purge`）均降二期；一期回收靠人工 + manifest 配方、清回收区靠人工（决策P3.9-10）。
- **不承诺事务级回滚**：改 manifest-先写 + 幂等向前收敛 + partial 报告（发现6，§2/§3）。
- **不写 `log.md`**：manifest 即审计轨迹（同 `reindex` 决策P3.4-6）。
- **不起 Agentao、不嵌 LLM client**：纯确定性。
- **不做 `recompile`**：覆盖人工编辑、悖增量 ingest，别借（OpenKB §1）。
- **Web / MCP 侧不做（本阶段）**：`remove` 是删 `raw/`+内容页的写操作，与 P4.10 MCP 只读 posture 冲突（同 `convert` 决策P5.2-9，MCP 不做 convert）；Web 侧若要，属 P4.x 机会性优化、单独评估，不在本阶段。

## 8. 决策

### 8.1 红线推得（决策P3.9-2…9）

- **决策P3.9-2（多源页只摘 slug、不删）**：衍生页若被多源建成，撤回一源仅从其 frontmatter `sources` 摘掉该 slug、**正文一字不改、页保留**——这是 provenance 编辑（`sources` 列表去一项），比 P3.7 的 append-only stamp 略重但同属"宿主对内容页做确定性 provenance 写"，且可逆（manifest 记 `before_sources`、可人工并回；自动 `restore` 留二期）。
- **决策P3.9-3（独源页一期只 advisory、不删；`--cascade` 降二期）**：仅此一源建成的衍生页只报 `remove.orphaned_page`、**一期不删**。"宿主删 Agent 所造衍生内容页"是最重一步，整条降为后续独立 `prune-orphans`（评审发现2 / 决策P3.9-10），不挡核心 `remove`。
- **决策P3.9-4（悬链 advisory、绝不 auto-strip；且不在 `remove` 内自算入链）**：见 §3 旁注 / §7 / 决策8——撤回后断链交既有 `lint`/`check`（发现4）。
- **决策P3.9-5（`.trash/` + manifest 留档；`restore`/`--purge` 降二期）**：一期移 `.trash/<slug>@<ts>/` + 写纯记录 `manifest.json`（非 `rm`，gbrain 软删形状的"先移回收区"半）。自动 `restore`（冲突/合并语义）与按龄硬删 `--purge`(TTL/mtime) 均降二期（评审发现1/3 / 决策P3.9-10）；**manifest 仍留一期**——它是审计 + 恢复配方 + 二期 restore 输入，廉价却兑现决策P3.9-1 的"可逆"（§4 旁注）。
- **决策P3.9-6（人发起的宿主确定性写、不经 Agentao / 快照门禁）**：见 §6。
- **决策P3.9-7（不写 `log.md`、manifest 即审计）**：同 `reindex` 决策P3.4-6。
- **决策P3.9-8（无新退出码、无新数据模型）**：复用 `EXIT_OK`/`EXIT_USAGE`；`.trash/` 是文件系统约定、非新模型。
- **决策P3.9-9（默认预览、`--yes` 才写）**：比 `reindex`/`convert` 默认即写更保守，因删内容页破坏性更大；与回收区留档共同把"删内容页"的风险压到可逆 + 须人显式确认两道。
- **决策P3.9-10（评审收窄 / phasing，2026-06-15）**：反向评审指出原设计已超"简明维护命令"复杂度、漂成回收站/恢复/级联系统。故**一期收窄为核心闭环**（§0 / 状态块的"一期保留"清单），让 P3.9 像 `reindex`——确定性、小表面积、易测。**降为后续项**：自动 `restore`（发现1，manifest 已留配方）、`--purge`/回收区 GC（发现3 → `trash purge`）、`--cascade`/独源页删除（发现2 → `prune-orphans`）、精确入链 advisory（发现4 → 交 `lint`/`check`）、事务级回滚承诺（发现6 → manifest-先写 + 幂等收敛）。**决策P3.9-1（取 A）与红线推得的 P3.9-2/4/6/7/8/9 不变**；收窄只缩**一期实现表面积**、不改架构判定。

### 8.2 决策P3.9-1（CRUX · 2026-06-15 取 A）—— 宿主删 / 移 wiki *内容*页的执行主体与确认级别

[`未来工作计划-反向评审收口.md`](backlog/notes/未来工作计划-反向评审收口.md) §3 / §7 点名"待拍的最高杠杆单点决策"。摘要页 ② 必然要随源撤掉（撤源的全部意义就在此）、独源衍生页可能随 `--cascade` 撤掉——二者都是"**宿主删 / 移内容页**"，比 P3.7-3a 的 append-only stamp 重，**不由该判例自动延伸**，故须单独拍"谁执行、要不要人确认"。

**拍板取 Option A——宿主确定性执行 + 默认预览 + 显式 `--yes` + `.trash/` 回收区留档。** 即（一期收窄后）：

> `remove` = 移①raw + 移②摘要页（1:1 随源、可逆）+ 改③多源页 provenance（摘 slug）；**独源衍生页只 advisory、一期不删**（删它属 `--cascade`，降二期 `prune-orphans`，决策P3.9-3/10）。

**取 A 原委**：人是决策者（发起命令 + 看 worklist + `--yes` 确认），宿主只做**确定性 move**（非生成）；`.trash/` + manifest 留档使其**可逆**，把"删内容页"的不可逆风险压到接近零——这正是 P3.7 stamp 没有、为越过这道更重边界而设的安全垫。撤回是**人意图的结构操作**、不是创作，落点类比 `reindex` 写 config 页 / `convert` 写 `raw/` 的宿主确定性写先例，只是对象是内容页 + 可恢复。**评审收窄反而收紧了越线**（决策P3.9-10）：把最重那步（删 Agent 所造衍生内容页 = `--cascade`）整条移出一期后，**一期"宿主删内容页"只触及 ② 摘要页**——它与单一 raw 1:1、近机械（一篇 raw 的 wiki 形态）、且回收区可逆，是这道边界**最窄、最可辩护**的一步，可视作 P3.7-3a 判例的**可逆延伸**；独源衍生页（可能含 Agent 的高阶综合）一期碰都不碰。

**被否选项（存档）**：

- **Option B（更保守，否）—— 宿主只改 provenance + 移 `raw/`，绝不删任何 wiki 内容页**：宿主做①移 raw + ③摘衍生 slug，②摘要页 / 独源页一律只报 advisory 交人手删。否因撤回**不闭环**（摘要页留在 `wiki/sources/`、仍是图节点 / 仍被 `[[link]]`），把"运营刚需"做成"半自动"；有恢复窗兜底后，B 的额外保守无谓。
- **Option C（否）—— 删页走 Agent 门禁写**：删除是结构操作非生成任务，Agent 加不了价值（结构上做不到 / 不该做，同 P3.7-3a 否其 Option B 的逻辑）；且把零-LLM 确定性清理硬塞进 LLM 写路径，违分档。

## 10. 与 P3.7-3a 判例的关系（为何"删"半要单独拍）

P3.7-3a 拍板"宿主**可**对内容页做**纯 provenance 的确定性写**"（在 source 页 stamp `raw_digest`），但其判例边界**窄**：那是 **append-only、不可逆、但极小**的一步。`remove` 的"删 / 移内容页"本可在两个维度上更远——**(a) 是删除/搬迁而非追加**、**(b) 涉及 Agent 所造的衍生内容**（独源衍生页）。故 P3.7-3a **不自动延伸**到 P3.9，须单独拍板（即 决策P3.9-1）。**评审收窄（决策P3.9-10）把 (b) 整条移出一期**（`--cascade` 降二期 `prune-orphans`），故**一期实际只迈出 (a)、且只对 1:1 摘要页**——这是这道边界最窄的一步。

但两者**共同前提稳固**：纯 provenance / 人发起的纠正都是 **Agent 结构上做不到或不该做**的事（LLM 抄不准 sha256；撤回是人意图）。P3.9 在此之上**多加一道 P3.7 没有的安全垫——回收区留档**（`.trash/` + `manifest.json`，把"删内容页"从不可逆变可逆；自动 `restore`/GC 是其上的便利、留二期），正是为越过这道更重的边界而设计。未决P3.9-1 已于 2026-06-15 取 A（决策P3.9-1），随后按评审收窄一期表面积（决策P3.9-10）；P3.9 至此**决策齐备、待实现**（同 P3.7 拍板后转定稿）。
