## 1. 目标与范围

**P4.6 = 让「上传文件 → ①当附件问 Agent / ②解析后人确认晋级为源 → ingest」两条用途都在浏览器里跑通,且全程不破「Agent 永不写 `raw/`」。**

### 端到端流(一次上传、双重用途;三个写者各司其职)

```
                                ┌─①当附件─▶ chat：<attachment> 标签 / 图像走 arun(images=) → Agent 当场读，不成源
①上传                            │                                                          （读路径，无写）
PDF/任意文件 ──POST /api/upload──▶ workspace/uploads/x.pdf
   宿主写                         │ ②解析(P4.5)：可写会话 Agent 跑 shell(pdftotext/marker) / 读附件
 (workspace)                     ▼   Agent 写 workspace/parsed/（绝不写 raw/）
                               workspace/parsed/*.md ⇄人审阅/修订(拆分·合并)── POST /api/raw{source}─▶ raw/*.md ─POST /api/ingest─▶ wiki/
                                  ②′修订循环（P4.5，Agent 写 parsed/）  ③晋级（人背书，宿主写 raw/）       ④入库（ingest 子进程，gated）
```

> **②′ 修订循环**:解析出 `parsed/x.md` 后,人可在浏览器里**预览**,并按需要让 Agent **修订**——含
> **拆分**(一篇 → 多篇,各自将来晋级为独立源)、**合并**(多篇 → 一篇)、精修(改/删某节)。修订仍是 **P4.5
> 可写会话里 Agent 写 `workspace/parsed/`**(同解析,**非新写边界**、Agent 仍绝不写 `raw/`);人满意后再按
> **最终 parsed 集逐文件**晋级。详见 §6.1。

**三个写者、三条边界,谁都不越界**(决策P4.6-1):

| 写者 | 写哪 | 怎么触发 | 边界依据 |
|------|------|---------|---------|
| 宿主上传 | `workspace/uploads/` | 人上传文件(`POST /api/upload`) | 暂存区、非源;多格式不限扩展名;同名覆盖(scratch) |
| **Agent** | `workspace/parsed/`(及 `workspace/` 其余) | 人在可写会话里驱动解析(P4.5) | **绝不写 `raw/`**(P4.5 §2:层①结构化写拒 + 层③shell `curl` 宿主写端点活跃期 423;**shell 直写 `raw/` 仍是 P4.5 残留——不扫、不回滚,待网络 sandbox**,层②仅快照还原 `AGENTAO.md` 一个文件、不覆盖 `raw/`,P4.5 §2.3/决策P4.5-3/c) |
| 宿主晋级 | `raw/`(`.md`) | 人审阅派生物后点「晋级为源」(`POST /api/raw {source}`) | 人投喂源、非 gated agent 写(P4.1 决策P4.1-1);单写者作业 |
| ingest 子进程 | `wiki/` | 人点 ingest(`POST /api/ingest`,P4) | gated 写 + `raw/` 快照(P2/P4 不变) |

> **「当附件」是读路径,不在上表**——它不写任何持久状态(base64 连会话快照都不进,见 §3 / 决策P4.6-8),只是把
> `workspace/uploads/` 里的文件以 `<attachment>` 标签喂进**单轮** chat。它和晋级是同一份上传文件的两种用途,**不互替**(决策P4.6-7)。

### 为什么「Agent 解析后不直接写 `raw/`」——原则 1 的底线(决策P4.6-1)

`raw/` 只读不可变的目的是**可追溯**:源不能被 Agent 改写/伪造,否则「造一个 source 再引用」会击穿引用信任。
故 **`raw/` 对 Agent 只读、对人/宿主可追加**(P4.1 决策P4.1-1 已定:投喂是人投喂源、非 gated agent 写)。
P4.6 把这条贯彻到上传流:Agent 只在 `workspace/` 产**派生物**(或在会话里**读**附件),由**人审阅后、宿主**把它
**晋级**为源——人在为「这份解析忠实代表我上传的 PDF」背书。**「解析出的 Markdown 当源」**这条模型比「PDF
当源」简单:复用现有 `.md` 单格式 ingest 不动,代价只是多一步「人确认晋级」(决策P4.6-2)。

### 「当附件」与「晋级为源」的分工(决策P4.6-7)

| 用途 | 入口 | 含义 | 落点 / 持久性 |
|------|------|------|--------------|
| **①当附件** | composer 📎(`ChatBody.attachments`) | 随消息发给 Agent 当场读(问答/摘要),**不成源** | 读 `workspace/uploads/`;base64 仅进本轮、不落快照 |
| **②晋级为源** | 「暂存区」弹层(`POST /api/raw {source}`) | Agent 解析成 `.md` → 人审 → 升为永久源 → ingest | `workspace/parsed/` → `raw/`(人背书、可追溯) |

上传端点与落点(`workspace/uploads/`)**两用途共用**;分叉只在「之后怎么用」。同一份上传文件两条路都能走。

### 纳入 P4.6

- **`POST /api/upload`**(§2,**已实现**):multipart 文件上传 → `workspace/uploads/<安全名>`(**多格式、不限扩展名、保留原扩展名、同名覆盖**——暂存非源;二进制许可;单写者落盘,决策P4.6-3)。返回 `{saved, name, bytes, kind}`。
- **聊天附件**(§3,**已实现**):`ChatBody.attachments` → 宿主按 agentao 约定追加 `<attachment uri=… [mimetype=…]/>` 标签;图像额外读盘 base64 走 `arun(images=)` 视觉通道;非视觉模型由 agentao 自动降级;base64 不入会话快照(决策P4.6-8)。
- **`POST /api/raw` 扩「源自 workspace」**(§4,**已实现**):body 增 `source`(与 `content` 互斥)+ 可选 `origin`(出处,前端/人提供)——`source` 指向 `workspace/` 内一个 `.md`,宿主读其内容、**过文本准入(UTF-8/`MAX_RAW_BYTES`/拒控制字符,同投喂)**、写 `origin` frontmatter(`yaml.safe_dump`)、经单写者作业晋级为 `raw/<安全名>.md`(端点默认不覆盖 `409`、人确认,决策P4.6-4/10)。
- **`GET /api/workspace` / `GET /api/workspace/file` / `DELETE /api/workspace/file`**(§5,**已实现**):列 `workspace/uploads/` + `workspace/parsed/`(只读审阅,决策P4.6-5);预览 `workspace/` 内 `.md`(复用 `render_page`);删 `workspace/` 内 scratch 文件(`uploads/` 堆积**与** `parsed/` 草稿/合并后原件皆可删;单写者 + 层③ 423、path-contained,决策P4.6-11)。
- **审阅—修订—晋级回路**(§6.1,**已实现 / 复用 P4.5,宿主零新增**):待晋级文件可预览,并由人下指令让 Agent 在 `workspace/parsed/` 内**修订**(含**拆分/合并**);晋级按最终 parsed 集逐文件触发(决策P4.6-9)。
- **前端「暂存区」弹层**(§7,**已实现**):上传(已在 composer 📎)+ `workspace/` 浏览/预览 +「让 agent 修订」(拆分/合并/精修,回填 composer)+「晋级为源」按钮 + 串联「立即 ingest」(复用既有 `triggerIngest`)。

### 不纳入 P4.6(按路线图推后)

- **PDF→Markdown 解析本身** → 由 P4.5 可写会话里 Agent 跑 shell / 读附件完成;**多格式自动解析管线 / nightly 富化**属 P5(决策P4.6-6)。
- **「PDF 原件留 `raw/`」模型**(`AGENTAO.full.md` 另一种措辞)→ 不采;它要 ingest 读二进制/多格式,是更大的改动。P4.6 采「解析 Markdown 当源」(决策P4.6-2)。
- **自动晋级 / 跳过人确认** → 不做:晋级是人对「解析忠实」的背书,**必须有人在环**(决策P4.6-1)。

## 2. 上传端点 `POST /api/upload`(宿主写 `workspace/uploads/`，已实现)

| 项 | 约定 |
|----|------|
| 协议 | `multipart/form-data`,单文件字段 `file`(含原始 filename);前端多选/拖拽/粘贴各文件**逐个** POST |
| 落点 | `workspace/uploads/<安全名>`(保留原扩展名——Agent 要按扩展名挑解析器,决策P4.6-3) |
| 格式 | **不限扩展名**(暂存区非源,PDF/docx/任意二进制皆可);**无** P4.1 投喂的 `.md`-only 准入 |
| 大小 | `MAX_UPLOAD_BYTES`(默认 50 MiB,> 投喂的 5 MiB;PDF 体量更大) |
| 安全名 | `_safe_workspace_target(root, "uploads", filename)`:剥目录 → NFKC+映射规范化(复用 P4.1 归一)→ slug **stem**、**保留原扩展名小写** → resolve 越界校验(须在 `workspace/uploads/` 内) |
| 同名 | **覆盖**(`_atomic_write_upload`,200)——暂存区 scratch 语义:content 由文件名定位,重复 attach 同一文件不该 409 卡住;**与投喂(写 `raw/` 源、默认不覆盖)刻意不同**(决策P4.6-3) |
| 并发/落盘 | 网络接收**不持锁**;收齐后把原子写**提交单写者 `JobQueue`**(`submit_and_wait`,job kind `upload`,与投喂/ingest/可写 turn 同写者串行,决策P4.6-3)。`workspace/uploads/` 不存在则先建 |
| 返回 | `{saved: "workspace/uploads/<安全名>", name, bytes, kind}`;`kind ∈ {image, text, binary}`(`_classify_upload`,供前端徽章配图与附件分流) |
| 错误 | `400` 空文件/超限/文件名规范化后为空/落点越界;`500` 落盘失败。**无 `409`**(同名覆盖) |
| 守卫 | **层③ 423**:可写 turn 活跃 → 拒(决策P4.5-10)——否则 agent shell `curl /api/upload` 的写作业排在 turn 的 `write_lock` 后空等、而 turn 在等这条响应,互相空等死锁 |

- **与投喂的差别**(决策P4.6-3):投喂收**粘贴文本**写 `raw/`(源、`.md`-only、文本准入拒控制字符、默认不覆盖);上传收**二进制文件**写 `workspace/uploads/`(暂存、多格式、无文本准入、同名覆盖)。两者都经单写者作业,但落点/准入/覆盖语义不同——上传**不是**投喂的替代,是其前段补充。
- **保留原扩展名**:`x.pdf` slug 化为 `x.pdf`(不强制 `.md`),否则 Agent 无法按扩展名选解析器。stem 走 P4.1 slug、扩展名小写归一。
- **`kind` 三分**:`image`(扩展名命中视觉白名单 `_IMAGE_EXT_TO_MIME`)/ `text`(扩展名命中文本表或前 64KiB 可 UTF-8 解码且无 NUL)/ `binary`(余者)。前端据此选徽章图标(🖼 缩略图 / 📄 / 📦)、并预判图像是否走视觉通道。

## 3. 聊天附件(`ChatBody.attachments`，已实现)

上传后文件可作 chat 附件随消息发给 Agent——**读路径,不写持久状态**(决策P4.6-7 / P4.6-8)。

- **请求**:`POST /api/chat/{id}` body 增 `attachments: list[str]`,每项是 `POST /api/upload` 回传的 `workspace/uploads/<名>`。
- **标签渲染**:`_augment_with_attachments` 把每个附件以 `<attachment uri="workspace/uploads/<安全名>" [mimetype="…"]/>` 标签追加进消息末尾(与正文空行分隔,逐字同构 agentao `_render_image_reference_fallback` 的降级标签)。非图不带 `mimetype`——Agent 凭只读工具自己读文本附件、二进制由其如实说明无法解析。
- **图像视觉通道**:扩展名命中 `_IMAGE_EXT_TO_MIME`、大小 ∈ (0, `MAX_IMAGE_BYTES`]、单轮前 `MAX_IMAGES_PER_TURN` 张者,额外读盘 base64 成 `{data, mimeType, _source}` 透传 `arun(images=)`;超限/超额者只留标签(与降级后的文本引用同形)。
- **非视觉模型降级归 agentao**:`_is_image_unsupported` 命中即自动以同格式标签重试,**宿主不做能力探测、不维护视觉模型表**。
- **base64 不落快照**(chahua 不变量):多模态 base64 只进本轮请求,会话快照经 `chat._lean_messages` 只留 text part(其中已含与 `_source` 同 uri 的 `<attachment>` 标签),防 `.agentao/sessions/` 膨胀。
- **路径校验**:`_safe_upload_file` 把 `rel` 限定在 `workspace/uploads/` 内(越界 400 / 缺失 404,含 TOCTOU 兜底)。
- **前端气泡**只显示原始 message + 文件徽章/缩略图(见 §7)。

## 4. 晋级端点 `POST /api/raw {source}`(宿主写 `raw/`，人确认;已实现)

把 P4.1 的 `POST /api/raw` 从「只收粘贴文本」扩成**判别式 body**(决策P4.6-4):

```jsonc
// content / source 二选一（互斥、必选其一）：
{ "name": "x", "content": "粘贴正文…", "overwrite": false }                                   // P4.1 投喂（原样保留）
{ "name": "x", "source": "workspace/parsed/x.md", "origin": "workspace/uploads/x.pdf",
  "overwrite": false }                                                                        // P4.6 晋级（origin 可选，仅 source 分支有意义）
```

- **`source` 校验**:① 路径包含——必须落在 `workspace/` 内(resolve + `relative_to(workspace)`,越界 `400`);② 必须 `.md`(`raw/` 是 `.md` 单格式源,与投喂同;非 `.md` `400`,提示「先在可写会话里解析成 .md」);③ 必须存在(`404`);④ **文本准入(关键)**——`uploads/` 允许任意二进制/50 MiB,但 `raw/` 源须是文本,故晋级**读 bytes 后按 UTF-8 解码**(失败 → `400`,非「decode 异常变 500」)、**复用投喂的 `MAX_RAW_BYTES`(5 MiB)上限**(超 → `400`)+ **P4.1 文本准入**(拒控制字符)。即 `source` 分支与投喂 `content` 过**同一道文本闸**,杜绝二进制/超大/控制字符 `.md` 混进 `raw/`。
- **晋级动作**:宿主**读 `source` 内容**(读发生在单写者作业内,与其它排队写原子隔离)→ 过文本准入(上)→ 按 provenance 规则**归一 frontmatter**(下,注入/补 `origin`)→ 经 `_safe_raw_target(root, name)` 定 `raw/<安全名>.md` → **复用 P4.1 `_atomic_write_raw`**(单写者作业、**默认不覆盖**、原子落盘)。`raw/` 拿一份独立副本(`workspace/parsed/` 原件留作草稿,workspace 是 scratch)。**注意:`raw/` 内容 ≠ 逐字等于 `source`**——**正文(frontmatter 之后的 body)一致**,但 frontmatter 经 provenance 归一可能新增/补全(故验收按「正文一致 + frontmatter 按规则归一」,非字节相等)。
- **provenance(可追溯,决策P4.6-10)**:出处由**前端/人显式提供**——宿主只拿到 `{name, source, overwrite, origin?}`,**无法自行推断 `parsed/x.md` 源自哪个上传原件**,故 `origin` 由前端预填(它编排了 upload→parse,记得谱系,如 `workspace/uploads/报告.pdf`)或留人手填(外部 URL/书目)。**空值语义(钉死)**:`origin` 先 `strip()`;**省略、或 strip 后为空(`""`/`"   "`)→ 一律回退为 `source` 路径**(`workspace/parsed/x.md`,workspace 内最小锚)——**绝不写空 provenance**,保「每条 `raw/` 可追溯」。写入规则(确定性、**复用 `pages.split_frontmatter`**,非裸插一行;注意 `split_frontmatter` 对**无 `---` 块**与**未闭合块**都返回「无块」、**二者不可区分**,均按①处理,不试图检测未闭合):① **无块** → 新建 `---origin: …---` 块前置、**原文逐字作 body**;② **有闭合块、`yaml.safe_load` 得 mapping 且缺 `origin`** → 往映射插入 `origin` 键、`yaml.safe_dump` 重序列化 + body;③ **有块且已有 `origin`** → **永远保留 parsed 自带值、忽略 body.origin**(`overwrite` 专表「覆盖同名 `raw/` 文件」,不重载它表达「覆盖 origin」;要改 origin 走 P4.5 修订回路再晋级);④ **有闭合块但 `yaml.safe_load` 不可解析 / 非 mapping(list/标量)→ `400`**「parsed frontmatter 损坏或非键值映射,请在可写会话修正后再晋级」(**不静默当 body、不试图插坏块**)。**YAML 安全序列化(关键)**:`origin` 作 YAML 标量经 `yaml.safe_dump`(或等价)写入,**绝不裸拼** `origin: <值>`——否则含 `:`、引号、`#`、换行、前后空白的出处(如 `https://…`、`A: B`、带引号书目)会生成坏 frontmatter。`origin` 是解析 `.md` 成源后回链原始 PDF/出处的唯一锚(决策P4.6-2 让 PDF 可弃),补原则 1「可追溯」缺口。
- **覆盖是显式人背书例外(决策P4.6-10);端点默认 `409`、不自动改名。** 分两层明确,避免实现成「端点自助改名」:**端点契约**——同名已存在且无 `overwrite` → `409`(与 P4.1 一致),**端点绝不自助改名/覆盖**。**前端**——收 `409` 默认**引导用户改名(开新源)**;仅当用户显式选「覆盖」(`overwrite:true`)才改写已有源,此时前端**告警**「引用此源的 wiki 页可能失真,建议覆盖后重 ingest + check」(`check` 只验源**存在**、不验内容漂移,失真需人主动复核)。
- **人确认是契约、不是 UI 礼貌**(决策P4.6-1):晋级**只由人点按钮触发**(前端审阅 `workspace/parsed/x.md` 后)。端点不在 Agent 工具集;它虽在 `127.0.0.1`、可写会话里 agent shell 能 `curl`,但被 **P4.5 层③(可写 turn 活跃期 `423` 拒,决策P4.5-10)** 掐死同步自助路径(残留派生后台进程待网络 sandbox)。在此前提下「源由人背书」钉死在架构层。
- **复用 P4.1 退出码分流**:`409`(同名已存在,给改名/覆盖)/ `500`(落盘失败)/ `200 {saved, bytes}`。
- **`content` 与 `source` 同时给或都不给** → `400`(判别式 body,互斥且必选其一)。

## 5. workspace 浏览/预览(只读宿主端点，path-contained;已实现)

- **`GET /api/workspace`**:列 `workspace/uploads/`(待解析)+ `workspace/parsed/`(待晋级),供人晋级前审阅。零 LLM、`to_thread`、无副作用(决策P4.6-5)。响应结构(每项字段为前端徽章/排序、`?path=` 预览/删除、`source` 晋级共用):

```json
{
  "uploads": [
    {"path": "workspace/uploads/报告.pdf", "name": "报告.pdf", "bytes": 1258291, "kind": "binary", "mtime": 1733846400.0}
  ],
  "parsed": [
    {"path": "workspace/parsed/报告.md", "name": "报告.md", "bytes": 3072, "kind": "text", "mtime": 1733846500.0}
  ]
}
```

  `path` = `workspace/<子目录>/<名>` 相对根,**直接用作** `GET`/`DELETE /api/workspace/file?path=` 与 `POST /api/raw {source}` 的实参(单一字符串、无需前端拼接);`kind` 复用 `_classify_upload`(`image`/`text`/`binary`);`bytes`/`mtime`(epoch 秒)供大小徽章与按时间排序。两组各自按 `mtime` 倒序。
- **`GET /api/workspace/file?path=`**:预览 `workspace/` 内 `.md`(resolve + `relative_to(workspace)`、越界 `409`、非 md/缺失 `404`、复用 `render_page`)。
- **`DELETE /api/workspace/file?path=`**(决策P4.6-11):删 scratch 文件——清 `uploads/`(纯当附件用过、不再晋级的堆积)、弃 `parsed/`(失败的解析草稿、合并后的原件、不打算晋级的派生物)。**目标须落在 `workspace/uploads/` 或 `workspace/parsed/` 内**(resolve 后 `relative_to(uploads)` **或** `relative_to(parsed)` 命中其一;**`workspace/` 根、其它子目录(未来的状态/会话目录)、越界 → `400`**;缺失 `404`)——**白名单两子目录、不删整树**,防将来 workspace/ 下出现非 scratch 状态文件被误删。删除是宿主写,故**经单写者 `JobQueue`(job kind `workspace-delete`)+ 层③ 423**(可写 turn 活跃期拒——防删到 turn 正读/写的 scratch),与上传/晋级/ingest 同写者串行。**绝不碰 `raw/`/`wiki/`**(`raw/` 删除仍不在 P4 范围)。返回 `{deleted}`。

## 6. 串联:解析(P4.5)与入库(P4)，本相位不改

- **解析**(②):人 `/mode workspace-write`(P4.5)→ chat「把 `workspace/uploads/x.pdf` 解析成 Markdown 放 `workspace/parsed/x.md`」→ Agent 读 `uploads/`(只读工具恒可;PDF/图还能直接走附件视觉通道,省掉 shell 解析器)、跑 shell `pdftotext`/`marker`、写 `workspace/parsed/`(P4.5 放行 `workspace/` 写)。**P4.6 不实现解析**,只提供其上游(上传/附件)与下游(晋级)。
- **入库**(④):晋级后 `raw/x.md` 就位,人点「立即 ingest」→ `POST /api/ingest`(P4,gated 子进程 + `raw/` 快照 + 强制 `check`)。**完全复用、不改**。
- **依赖关系**:headline 全流的②需要 P4.5;但①上传 + ③晋级是**姿态无关的宿主写**——即便没 P4.5,也能「上传一个 `.md` → 直接晋级 → ingest」(退化为「带文件版的投喂」)。此退化路径不只是端点能力:**前端为 `uploads/*.md` 提供条件式 `[晋级为源]`**(`source` 校验只要 `workspace/`+`.md`+存在,uploads/*.md 合格,见 §7.2 ①),故无 P4.5 也能在浏览器里走完。「当附件」则只需上传 + P4 chat,与晋级解耦。

### 6.1 审阅—修订—晋级回路(已实现 / 复用 P4.5，宿主零新增)

晋级不是「解析一次就定稿」,而是一个**人在环的草稿池迭代**:`workspace/parsed/` 是一个**工作池**,人预览池中
文件 → 不满意就让 Agent 修订(改/拆/合)→ 产物**仍落回池里继续审阅** → **满意才把池中某篇晋级**。修订
**没有新机制、没有新端点**——它就是 §6 的「解析」同一条路:

```
人下指令（composer，workspace-write）          Agent 写 workspace/parsed/（产物留池）
"把 parsed/x.md 拆成算法篇和范例篇"  ──────▶  parsed/x-算法.md + parsed/x-范例.md  （都留池，继续审阅/再修订，不自动晋级）
"把 parsed/a.md b.md 合并成一篇"     ──────▶  parsed/<agent 建议名>.md            （名字 agent 提议、人最终定；原件可删）
"删掉 parsed/x.md 第3节、补个小结"   ──────▶  parsed/x.md（就地改写）
                                              └────────▶ 池中文件满意后，逐篇 [晋级为源]（晋级 = commit 点）
```

- **拆分 / 合并 / 精修都是同一动作**:人下自然语言指令 → Agent 在 `workspace/parsed/` 增/改 `.md`(读 `parsed/` 草稿与 `uploads/` 原件都用只读工具,恒可)。**这正是 P4.5 放行的 `workspace/` 写**——非新写边界,Agent **仍绝不写 `raw/`**(决策P4.6-1 / 决策P4.6-9)。
- **拆分后留池、不直接晋级**:拆出的每片回到 `parsed/` 池,可再预览/再拆/再合;**晋级是与修订解耦的、人对单篇的显式 commit**,不在拆分动作里隐式发生。这也避免「先晋级再拆」留下陈旧重复源(评审项)——**先在池里定稿,再晋级**。
- **合并:Agent 提议名、人最终定**:合并产物落 `parsed/<agent 建议名>.md`,前端把 agent 建议名填进可编辑输入框,**人确认/改名后**才落(决策P4.6-9);原件人可删(§5 / 决策P4.6-11)。
- **拆分恰是「解析 Markdown 当源」模型的红利**(决策P4.6-2):一份 PDF 可先在 `workspace/` 里拆成**按主题的干净小源**再逐篇晋级,每篇成独立可被引用的 `raw/` 源——这是「PDF 原件当源」做不到的。
- **晋级按池中文件逐篇 commit**:`POST /api/raw {source}` 仍是「一个 `.md` → 一个 `raw/` 源」。晋级**默认改名不覆盖**;晋级后还想改 → 再修订 `parsed/` → 重晋级**默认开新源(改名)**,覆盖是需显式确认的人背书例外(且告警 wiki 引用可能失真,建议重 ingest+check,决策P4.6-10)。
- **修订本身宿主零新增**:浏览器只需在一轮修订**turn 收尾**(流式 `done` 或 `stopped` 帧任一,**非 EventSource**)后**重新拉 `GET /api/workspace`** 刷新池;修订的写全由 P4.5 可写会话承担。本相位仅另加一个轻量 `DELETE /api/workspace/file` 删 scratch(§5 / 决策P4.6-11),不加其他写端点/守卫。

### 7.2 「暂存区」弹层(已实现)

- **① 待解析**(`GET /api/workspace` 的 `uploads/`):每项 `[问 agent 解析]` → 复用 composer,填入「把 `uploads/报告.pdf` 解析成 markdown 写到 `workspace/parsed/`」(若当前 read-only,顺带提示 `/mode workspace-write`);`🗑` 删该 scratch(`DELETE /api/workspace/file`,清纯当附件用过的堆积)。**`uploads/*.md`(`kind=text` 且扩展名 `.md`)额外给 `[晋级为源]`**——这是退化路径「上传 `.md` → 直接晋级」的 UI 入口(§6 依赖关系);非 `.md`(PDF 等)无此按钮,须先 `[问 agent 解析]` 成 `.md`。
- **② 待晋级**(`GET /api/workspace` 的 `parsed/`):每项 `[预览]`(`GET /api/workspace/file` → `render_page`,只读)+ `[让 agent 修订]` + `[晋级为源 →]`(`POST /api/raw {source}`,复用投喂 slug/409 交互)+ `🗑`(弃这篇草稿)。

- **各项 textContent 防注入**(同投喂/heal 列表);整个晋级/ingest 都走 `#overlay` + `pollJob`,不引新组件。

### 7.3 姿态守卫在 UI 上的体现(防误解)

- 暂存区里**没有**「让 agent 自动晋级」的按钮——晋级永远是人点(决策P4.6-1,人为「解析忠实」背书)。
- 可写 turn 跑动期间(`#mode-badge` = workspace-write、停止按钮亮),`[晋级为源]`/`[立即 ingest]`/`🗑` 等**宿主写动作**应**置灰**并提示「会话写作业进行中…」——对应后端层③ 423(上传/晋级/删除同口径),避免人点了撞 423 一脸懵。

## 8. 决策记录

- **\[决策P4.6-1] Agent 永不写 `raw/`;派生物经「人审阅 + 宿主」晋级为源。** 原则 1 的可追溯要求源不被 Agent 改写/伪造。三写者各守其界:宿主上传写 `workspace/uploads/`、Agent 写 `workspace/parsed/`(P4.5 硬拦 `raw/`)、宿主晋级写 `raw/`(人点按钮触发)、ingest 子进程写 `wiki/`。晋级端点是宿主 HTTP 面、**不在 Agent 工具集**。注意它在 `127.0.0.1`、可写会话里 agent 的 shell 能 `curl` 打到(非「工具集外即不可达」)——故「源由人背书」不靠「工具不可达」单点,而靠 **P4.5 层③:可写 turn 活跃期间 `/api/raw` 等写端点一律 423**(决策P4.5-10),把 agent 经 shell 自助晋级的同步路径掐死;残留(`curl … &` 派生后台进程在 turn 结束后才打)待网络 sandbox。在此前提下「源由人背书」成立。沿用 P4.1 决策P4.1-1(投喂=人投喂源、非 gated agent 写)。
- **\[决策P4.6-2] 采「解析 Markdown 当源」模型,不采「PDF 原件当源」。** 解析出的 `.md` 晋级进 `raw/` 作源,复用现有 `.md` 单格式 ingest **不动**;PDF 原件留 `workspace/uploads/` 作暂存/可弃。否决 `AGENTAO.full.md` 提到的「PDF 留 raw/、解析物落 workspace/」——那要 ingest 读二进制/多格式,改动大、属 P5。代价:多一步「人确认晋级」(这步本就是原则 1 要的人背书,非额外负担)。
- **\[决策P4.6-3] 上传端点多格式、不限扩展名、保留原扩展名、同名覆盖、单写者落盘。** `workspace/uploads/` 是暂存区非源,故无投喂的 `.md`-only 准入、许二进制;slug stem 但**保留原扩展名**(Agent 按扩展名挑解析器)。**同名覆盖**(scratch 语义:content 由文件名定位,重复 attach 同一文件不该 409 卡住;与投喂写 `raw/` 默认不覆盖刻意不同)。网络接收不持锁、收齐后原子写提交单写者 `JobQueue`(job kind `upload`,与投喂/ingest/可写 turn 同写者串行)。`MAX_UPLOAD_BYTES` 默认 50 MiB。返回 `{saved, name, bytes, kind}`,`kind ∈ {image, text, binary}` 供前端徽章。
- **\[决策P4.6-4] 晋级 = 扩 `POST /api/raw` 判别式 body(`content` XOR `source`),复用 P4.1 落盘原语。** `source` 指 `workspace/` 内 `.md`,校验路径包含 + `.md` + 存在,读内容经 `_safe_raw_target` + `_atomic_write_raw` 单写者晋级为 `raw/<安全名>.md`、默认不覆盖。所有「人投喂源」走同一端点(投喂粘贴 / 晋级 workspace 文件),`raw/` 写入口单一。
- **\[决策P4.6-5] `workspace/` 浏览/预览是只读宿主端点,path-contained。** `GET /api/workspace` 列 `uploads/`+`parsed/`;`GET /api/workspace/file?path=` 预览 `workspace/` 内 `.md`(resolve + `relative_to(workspace)`、越界 409、复用 `render_page`),供人晋级前审阅。零 LLM、`to_thread`。
- **\[决策P4.6-6] 解析能力不在本相位。** PDF→Markdown 由 P4.5 可写会话 Agent 跑 shell / 读附件完成;多格式自动解析管线属 P5。P4.6 只做上传(前段)+ 晋级(后段)的宿主管线,**不引入任何解析器依赖**。
- **\[决策P4.6-7] 上传文件双用途:聊天附件 ∥ 晋级为源;上传端点/落点共用,分叉在「用途」。** ①**当附件**(composer 📎 → `ChatBody.attachments` → `<attachment>` 标签 + 图像视觉通道):读路径、不写持久状态、不成源,适合问答/摘要。②**晋级为源**(暂存区审阅 → `POST /api/raw {source}`):成永久可追溯源。二者**不互替**——附件读 ≠ 晋级落源;若误以为「agent 读了附件就能直接写 wiki」会击穿 `raw/` 可追溯(wiki 引用 `raw/` 里不存在的源),正是决策P4.6-1/2 要防的。
- **\[决策P4.6-8] 附件机制复用 agentao 视觉约定,宿主不做能力探测,base64 不入快照。** `<attachment uri=… [mimetype=…]/>` 标签与 agentao 降级标签逐字同构(uri == `images=` 的 `_source`);图像走 `arun(images=)`,非图只发标签(agent 自读);超限/超额图像降级为文本引用;非视觉模型由 agentao `_is_image_unsupported` 自动同格式标签重试(宿主不维护视觉模型表);多模态 base64 仅进本轮、经 `_lean_messages` 不落会话快照(chahua 不变量)。
- **\[决策P4.6-9] 待晋级文件可由人驱动 Agent 修订(含拆分/合并),修订复用 P4.5、非新写边界;`parsed/` 是工作池,拆分后留池、不直接晋级。** `workspace/parsed/` 是人在环迭代的**池**:预览 → 让 Agent 改/拆/合(产物回池)→ 重新审阅 → **满意才把池中某篇晋级**。拆分(一篇→多篇)、合并(多篇→一篇)、精修都是同一动作——人下自然语言指令、Agent 在 `workspace/parsed/` 增删改 `.md`,即 **P4.5 放行的 `workspace/` 写**,Agent **仍绝不写 `raw/`**(决策P4.6-1 不破)。**晋级与修订解耦**:拆出的片回池继续审阅、**不在拆分动作里隐式晋级**(避免「先晋级再拆」留陈旧重复源——先在池里定稿再 commit)。**合并由 Agent 提议文件名、人最终定名**(前端预填可编辑)。修订本身宿主零新增,浏览器只在每轮修订 turn 收尾(`done`/`stopped`)后重拉 `GET /api/workspace` 刷新池;晋级仍「一 `.md` → 一 `raw/` 源」逐文件由人 commit。拆分恰是「解析 Markdown 当源」(决策P4.6-2)相对「PDF 原件当源」的红利:可先在 workspace 把一份大 PDF 拆成按主题的干净小源再逐篇晋级。
- **\[决策P4.6-10] 晋级是 commit 点:端点默认 `409`(不自动改名/覆盖)、覆盖为前端显式人背书例外、`origin` provenance 由前端/人显式提供并按确定规则写入。** **端点契约**:`raw/` 默认不可变——同名已存在且无 `overwrite` → `409`(与 P4.1 一致),**端点绝不自助改名/覆盖**(两层职责分清,防实现成端点自动改名)。**前端**:收 `409` 默认引导改名(开新源);显式选覆盖(`overwrite:true`)才改写已有源——它超出 P4.1 决策P4.1-1「人可**追加**源」、是「人可**改写**源」,故必须告警「引用此源的 wiki 页可能失真,建议重 ingest + check」(`check` 只验源存在、不验内容漂移)。**provenance**:body 增可选 `origin`——**宿主拿不到 parsed↔upload 谱系**,故出处由前端预填/人手填;**省略、或 strip 后为空 → 回退 `source` 路径(绝不写空 provenance)**。写入规则(确定性、复用 `pages.split_frontmatter`,非裸插一行):无块(含未闭合,二者不可区分)→新建 `---`块、原文作 body;有闭合块得 mapping 缺 `origin`→插入键 + `safe_dump`;**已有 `origin`→永远保留、忽略 body.origin**(不重载 `overwrite`——它专表「覆盖同名 `raw/` 文件」;改 origin 走 P4.5 修订回路);**有闭合块但不可解析 / 非 mapping → `400`**(不静默当 body、不插坏块)。`origin` 作 YAML 标量经 **`yaml.safe_dump`(或等价)** 写入、**绝不裸拼**(含 `:`/引号/换行/`#` 的出处否则坏 frontmatter)。`origin` 是解析 `.md` 成源后回链原始 PDF/出处的唯一锚(决策P4.6-2 让 PDF 可弃),补原则 1「可追溯」缺口。
- **\[决策P4.6-11] `workspace/` scratch 可删(`uploads/` 与 `parsed/` 皆开放):`DELETE /api/workspace/file`,宿主写经单写者 + 层③ 423、path-contained,绝不碰 `raw/`/`wiki/`。** `workspace/uploads/` 纯当附件用过的文件会永久堆积(scratch 却无人清),`parsed/` 也会积失败草稿/合并后原件/不晋级的派生物——故加一个轻量删除端点,**白名单 `workspace/uploads/` 与 `workspace/parsed/` 两子目录**(目标须 `relative_to(uploads)` **或** `relative_to(parsed)` 命中其一;**不删 `workspace/` 根及未来可能出现的状态/会话目录**——防 workspace/ 下日后出现非 scratch 文件被整树删误伤)。删除是宿主写,纳入单写者 `JobQueue`(job kind `workspace-delete`)+ 层③ 423(可写 turn 活跃期拒,防删到 turn 正读/写的 scratch),与上传/晋级/ingest 串行。落点白名单外/越界 `400`、缺失 `404`。**绝不碰 `raw/`/`wiki/`**:`raw/` 删除仍不在 P4 范围(原则 1),`wiki/` 由 Agent 自治。
