## 2. 功能块

### 2.1 待解析：确定性解析 + 进度窗口

- **引擎**：宿主直调 `convert_to_markdown(src, stem, backend, cwd=root)`（P5.2 内核，已返回 `ConvertResult`）。
  **不再走 P4.5 Agent 解析**（决策P4.6.1-1）。仍是 **P5.2 口径的「宿主零-LLM」**——guanlan 不持 LLM，只 shell out
  外部转换器；backend 是否用 LLM 属用户环境选择（沿决策P5.2-4）。
- **落点 = `workspace/parsed/`，非 `raw/`**（决策P4.6.1-2）：新落盘 shell（区别于 CLI `run_convert` 落 `raw/`），
  写 `parsed/<slug>.md` + 图片落 `parsed/images/<slug>/<slug>-N.ext`，引用重写为 `images/<slug>/…`。
  `convert._images_dir(target)` 锚 `target.parent/"images"/<stem>` **已天然泛化**到 `parsed/`，引用在 parsed/ 与
  晋级后 raw/ **两处同形**（决策P4.6.1-2）；**晋级一律按 target stem 归一**（决策P4.6.1-13，见 §2.2）——canonical 同名时零变化、
  改名或非 canonical 时真重写。
- **作业化 + 进度窗口**（决策P4.6.1-7）：解析慢（marker/mineru 分钟级），作为 **JobQueue 新作业种类**（与
  ingest/heal/backfill/raw-write 同单写者 FIFO 串行）。前端「解析进度窗口」显示**解析阶段 + backend 分级
  回退日志**（MinerU→marker→pypdf 哪级在跑/失败）。
  - **扩展 JobQueue 支持「运行中增量 output」**（Q1 已定，**现机制不流式，需真扩展**）：现状 `jobs.py` worker 只在
    作业结束、`state="done"` 时一次性 `job.output = buf.getvalue()`（`jobs.py:217`），running 期 `job.output` 为空、
    `pollJob` 只渲染 running 徽标（`app.js:471`）→ 分钟级解析期间**无 backend 日志可见**。本阶段改：
    ① **新增 `emit` 进度 sink，改 `enqueue(kind, fn)` 的 `fn()` 为 `fn(emit)`**（决策P4.6.1-11）：现 `fn()` 无参、
      worker `finally` 又 `job.output = buf.getvalue()` **覆盖**一切（`jobs.py:197/217`）→「回调写 job.output」按现结构
      落不了地、必被 finally 盖掉。故 worker 构造 `emit(line)` 闭包（持 `buf`+`job`）：**追加进 `buf` 并整串赋值
      `job.output = buf.getvalue()`**；传给 `fn(emit)`。`finally` 的终态 `job.output = buf.getvalue()` 保留（与最后一次
      emit 幂等、作权威终态）。**签名改动须统一到 `_register` 归口（决策P4.6.1-11）**：`jobs.py:150` 的 `_register` 同时
      服务 `enqueue`（异步）**与** `submit_and_wait`（同步，raw_write/upload/delete 走它）；只改 enqueue 会漏掉同步作业。
      故把 `Callable[[], object]` 统一升为 `Callable[[Emit], object]`、worker 一律 `fn(emit)`，**`enqueue`/`submit_and_wait`
      两个入口的所有 thunk 全改签**（ingest/heal/backfill **以及** raw_write/upload/delete）。既有 thunk **忽略 emit**
      （仍靠 `print()` 经 redirect 进 buf，行为不变）；**仅 parse thunk 用 emit** 把内核回调的每行实时推上去。
      （备选：`_register` 内做 `lambda emit: fn()` 适配包装、保留零参 thunk；本 doc 采**统一升签**，因其类型显式、不藏隐式分叉。）
    - **emit 须线程安全（决策P4.6.1-14）**：内核 `progress=` 走 `Popen` + **两条管道并发 drain**，通常**从 drain 线程**调用
      `progress→emit`（可能两线程同时）；而 emit 写 **`StringIO buf`（非线程安全）+ `job.output`**，worker `finally` 也写
      `job.output`。现有 JobQueue 此前只有单 worker 写 output、无并发。故给每个 `Job` 一把 `threading.Lock`，emit 的
      `buf.write(line)` + `getvalue()` + 整串赋值 `job.output` **全在锁内**；worker `finally` 的 `job.output = buf.getvalue()`
      **取同一把锁**。parse 的进度**全程经 emit**（不旁路 `print()`/`redirect_stdout` 并发写 buf），故无锁外写者。
      非流式作业仍单线程 `print()` 经 redirect、无并发、不受影响。
    ② `/api/jobs/{id}` 已回传 `job.output`、自然透出，无需改形；
    ③ `pollJob` 在 **running 态也渲染 `job.output`**（不只徽标），done 后渲染终态。
    增量 output 是 **best-effort 进度**，权威终态仍以 `state="done"` 后的完整 output 为准；GIL 保证整串赋值原子，
    读者读到的是某个一致快照（不会撕裂）。此扩展在 JobQueue 层、**ingest/heal/backfill 同样受益**（顺带获得实时日志），
    但本阶段只为 parse 立验收。不新开 SSE 通道。
  - **转换内核须真流式（扩 JobQueue 不够，决策P4.6.1-10）**：现 `convert.py:210` 用 `subprocess.run(capture_output=True)`
    阻塞到结束、backend 日志在 **stderr 一次性捕获**、stdout 只留产物路径 → 即便 JobQueue 支持增量 output，**解析期间
    也无真实进度可喂**。故须给转换内核补进度契约：`convert_to_markdown(..., progress: Callable[[str], None] | None = None)`——
    `progress` 给定时改用 **`Popen` 边读 stderr 行边回调**（marker/mineru 分级回退日志即在 stderr），stdout 仍按末行取
    产物路径；**两条管道都要并发 drain**（线程/`select`）避免 pipe 满死锁。**CLI `convert` 传 `progress=None`**，走原
    `subprocess.run` 路径、**行为字节不变**（决策P5.2-4 向后兼容）；Web parse 传一个把行追加进 `job.output` 的回调。
- **按钮改名**：暂存区「让 agent 解析」类入口 → 改名「**开始解析**」，点击即建解析作业、弹进度窗。
- **全 backend 失败不回落 Agent 解析**（决策P4.6.1-9，Q4 已定）：MinerU→marker→pypdf 全失败即报错收场
  （沿 `convert` 的 `EXIT_USAGE`/错误回执），**不在本界面留 Agent 手解析兜底**——用户可外部转换后再上传，
  或在 Agent 对话里交互式转格式，不必塞进暂存区。

> **解析与「修订」分工（Q2 已定，维持 P4.5）**：解析=确定性宿主；**修订（拆分/合并/精修）仍由 P4.5 Agent 改
> `parsed/` 文本**（非新写边界）。Agent 改文本后图片可能断链 → 由 2.3/2.4 的**确定性重整**补救（人点按钮触发、
> 修复本身零-LLM、宿主做）。**不把拆分/合并升格成宿主确定性整文操作**——文本编辑留给 Agent，宿主只管图片重整。

### 2.2 待晋级：晋级为源（连图一起）

- 晋级 `parsed/<source_stem>.md` → `raw/<target_stem>.md` 时（`target_stem` = `body.name`，可与 `source_stem` 不同，见
  决策P4.6.1-13），**按 source md 的实际图片引用收集、连图一起搬**到 `raw/images/<target_stem>/`（决策P4.6.1-3/15）。
- **引用驱动收集，目录非判据（决策P4.6.1-15）**：搬图的输入**锚在 source md 里的 `![](url)` 引用**、**不是
  `parsed/images/<source_stem>/` 目录存在**。原因：拆分/合并后 md 可能仍引 `images/other/…`（别的 slug 目录），或
  source 目录里有**未被引用**的残图——按目录搬会**漏搬**（跨目录引用）、**多搬**（未引用残图）、甚至让 raw md 断图。故
  promotion **扫 source md 引用 → 经 `workspace/parsed/` root `_admit_image_ref` 准入（解相对 md 父目录、挡越界/symlink）
  → 按引用收集 → 归一编号到 `raw/images/<target_stem>/`**。**任一引用未准入（越界/非白名单）或悬空（文件缺失）→ 400**，
  提示「先 relocalize 修复断链再晋级」（§2.3）。目录 `parsed/images/<source_stem>/` 只是常见布局，**不作搬运判据**。
- 复用 P5.2.1 落盘原语：抽出的 `_stage_and_swap_images`/`_rollback_images`/`_images_dir` 走「图先换、md 末步
  提交、失败回滚」（决策P5.2.1-9），保证晋级成功的 `raw/<target_stem>.md` **永不指向缺图**。
- md 仍走 `rawio.atomic_write_raw`（**`rawio.py` 一字不改**，沿决策P5.2.1-1）；图片落盘是 convert/晋级侧专属。
- **晋级总按 target stem 归一，不据 name-equality 走 skip 捷径（决策P4.6.1-13，方案 b）**：`/api/raw {name, source}` 允许
  晋级改名（且 409 同名后前端默认引导改名）；`parsed/x.md → raw/y.md` 时正文仍引 `images/x/…`。promotion job 在写临界区里
  **一律按 target stem 归一**——`collect-rewrite` keyed on target stem：图按 md 首现序改名编号 `<target>-N.ext` 落
  `raw/images/<target>/`、md 引用重写为 `images/<target>/…`。这样 **`raw/images/<slug==raw stem>` 不变式（决策P5.2.1-2）
  恒成立**，与是否改名无关。
  - **「同名零重写」只是结果、非捷径**：当 parsed/ 图引用**已是 canonical `<stem>-N.ext`** 且同名时，归一输出与原引用逐字
    一致（实质 no-op）；但 **promotion 不靠 `name==stem` 判断跳过 rewrite**——parsed/ 图名可能因 relocalize 重编号、手工
    调整、缺号而**非 canonical**，此时即便同名也须归一。即：**无条件跑归一**（幂等、canonical-in 时零变化），不做易错的 skip 优化。
  - **不采** (a) 禁止改名（会废掉 409→改名的既有 UX）；**不采** (c) 图目录跟 source stem（破坏 `raw/images/<raw stem>`
    布局、与 P5.2.1-2 冲突）。
- **晋级须改成专用 promotion job，图与 md 在同一写临界区原子提交（决策P4.6.1-12）**：现状 `/api/raw {source}` 在
  **入队前** `_prepare_promotion` 把 source 读成 `content`，worker thunk 仅 `_atomic_write_raw(target, content, overwrite)`
  （`app.py:560/577-582`）——**临界区里已丢 source 路径 + `parsed/images/<source_stem>/` 上下文**。若沿这形状加搬图，必然实现成
  「文本晋级、图没搬」或把搬图放在**队列外**（破坏单写者原子性）。故 source 分支改为**专用 promotion job**：thunk 携
  `(target, 已 provenance 且已重写引用的 content, 收集到的图片字节集, 指纹, overwrite)`。**校验放入队前**（扩 `_prepare_promotion`，
  与其文本准入 4xx 同位，决策P4.6.1-12/15）：**扫 source md 引用 → 经 `workspace/parsed/` root 准入 → 检在 → 收集字节 + 记指纹**；
  **未准入/悬空 → 400**（worker 只回退出码、无 400 路径，故校验必在入队前）。thunk 在**同一 JobQueue 写临界区内**依次做
  ⓿ **指纹复检（决策P4.6.1-16）**：对 source md + 各收集图片**复算 SHA256（内容哈希，权威）**，**与入队前快照不符即
  fail-closed**（→ 409「source 已变，请重新审阅」，不提交旧快照）→
  ① 图按 `target_stem` 归一（决策P4.6.1-13）staging→swap 到 `raw/images/<target_stem>/`（`_stage_and_swap_images`）→
  ② md 末步 `atomic_write_raw` 提交 →
  ③ 任一步失败 `_rollback_images` + 清 `.bak`（沿决策P5.2.1-9 落盘顺序）。文本准入/坏 frontmatter 仍可留**入队前**转 4xx
  （纯文本变换、不碰盘），但**图搬 + md 提交 + 回滚是一个不可分单元、必须在写锁内**。投喂分支（`content`）不变。

### 2.3 拆分后图片断链：检查 + 修复按钮

- **场景**：人/Agent 把 `parsed/x.md` 拆成 `parsed/x-1.md` + `parsed/x-2.md`，文本随之分家，但图片**文件**仍躺
  `parsed/images/x/`，新文件引用 `images/x/x-3.png` → 既「不在本文件名下的 image 目录」、又可能彼此越界。
- **断链检查**（确定性、零-LLM）：扫 `parsed/<file>.md` 的 `![](…)` 引用，标出（a）解析不到的悬空引用、
  （b）指向**非本文件 slug** image 目录的「错位」引用。
- **修复按钮**（决策P4.6.1-5）：确定性 **re-localize**——把该文件引用到的图，落到「该文件名下 image 目录」
  `parsed/images/<file>/`，按 md 内**首次出现序改名/编号** `<file>-N.ext`，并重写引用。
  - **全局引用扫描，默认 copy、仅全局无引用才删（Q3 已定，安全边界收紧）**：**关键——P4.6 工作池允许拆分后原件
    `parsed/x.md` 及其它草稿仍留池中**，故判「某图能否删」**绝不能只看当前文件**，否则把 `images/x/foo.png` 移给
    `x-1/` 会断掉仍存在的 `parsed/x.md`/其它草稿的引用。规则：
    - re-localize **前后都对整个 `workspace/parsed/*.md` 做全局引用扫描**（建「图 realpath → 引用它的 parsed 文件集」映射）；
    - 目标目录**默认 copy**（`images/<file>/<file>-N.ext` 各留一份拷贝，接受这点冗余，Q3）；
    - **只有确认全局已无任何 `parsed/*.md` 引用**的原图才删除，绝不在修一个文件时破坏池里其它文件。
    即：copy-first，原图回收只发生在「全局零引用」时——把「move」安全地降为「copy + 全局 GC」。

### 2.4 合并：重整图片

- **场景**：合并 `parsed/a.md` + `parsed/b.md` → `parsed/c.md`，图来自 `images/a/` 与 `images/b/`。
- **重整**（决策P4.6.1-6）：与 2.3 **同一个确定性 re-localize 引擎**（N→1）——把 c.md 引用到的图**复制到**
  `parsed/images/c/`、按 c.md 内首次出现序统一改名/编号、重写引用；原图按 2.3 的**全局零引用才 GC** 规则回收。

## 3. 模块与归口

- **抽 P5.2.1 图片原语成 transport-neutral 归口**（决策P4.6.1-4，镜像 P5.2-6 抽 `rawio.py`）：把 `convert.py` 的
  `_collect_and_rewrite_images` / `_stage_and_swap_images` / `_rollback_images` / `_images_dir` / `_admit_image_ref`
  / 容量闸常量抽到共用模块（如 `guanlan/imageio.py`），供 **① CLI `convert`（landing→raw/）② Web 解析（landing→
  parsed/）③ 拆分/合并 re-localize** 三处共用、消漂移。**`rawio.py` 仍不碰。**
- **转换内核 `progress=` 契约（决策P4.6.1-10）**：`convert_to_markdown` 增 `progress: Callable[[str],None] | None = None`
  归口参数——`None`（CLI convert）走现 `subprocess.run` 路径、字节行为不变；给定（Web parse）走 `Popen` 边读 stderr 回调、
  两管道并发 drain。这是对**共用内核**的加法契约，CLI/Web 同口径。
- **新增 re-localize 原语**：P5.2.1 的收集是「从 temp 树 `read_bytes()` 入内存」；re-localize 是「**对磁盘上既有
  `parsed/<file>.md` + 既有图文件做 copy 到目标目录 + 改名 + 重写引用 + 全局零引用后 GC 原图**」。两者共用「按 md
  首现序编号 + `_admit_image_ref` 准入 + 引用重写」逻辑，落盘语义不同（一个写新字节、一个 **copy 既有文件 + 条件 GC**——
  绝不无条件 move，否则断池里其它草稿引用，见 §2.3）。
- **安全/容量闸沿 P5.2.1**：`_admit_image_ref` 五条 AND 的 root **从 `tmp_root` 换成 `workspace/parsed/`**（挡越界/
  symlink 逃逸）；扩展名白名单 `IMAGE_EXTS`、容量三闸（单图 20 MiB / 累计 200 MiB / 500 张）原样（决策P5.2.1-5/11）。
- **JobQueue API 改动（决策P4.6.1-11，统一到 `_register`）**：把 thunk 类型 `Callable[[],object]` 升为 `Callable[[Emit],object]`、
  worker 一律 `fn(emit)`。**因 `_register`（`jobs.py:150`）同时服务 `enqueue`+`submit_and_wait`，两入口的所有 thunk 必须全改签**——
  异步 ingest/heal/backfill **与**同步 raw_write/upload/delete（易漏！）。既有 thunk 忽略 emit、行为不变。**晋级 thunk（决策
  P4.6.1-12/15/16）携 `(target, 已重写 content, 收集到的图片字节集, SHA256 指纹, overwrite)`**——**引用驱动收集、绝不携图片
  目录按目录搬**（决策P4.6.1-15）；写锁内 ⓿ SHA256 复检 + staging-swap + md 提交 + 回滚，不再是 `lambda: _atomic_write_raw(...)`。
- **新端点（草案）**：`POST /api/parse {upload}`（建解析作业，进度经 `/api/jobs/{id}` 轮询；**需先扩展 JobQueue
  worker 运行中增量写 `job.output` + `pollJob` running 态渲染 output**，决策P4.6.1-7/Q1，不新开 SSE）；
  `GET /api/workspace/image-lint?file=`（断链检查）；
  `POST /api/workspace/relocalize {file}`（修复/重整）。均单写者 + 层③ 423 + path-contained。**reader 不注册。**

## 4. 决策（草案，沿仓库 `决策PX-N` 体例）

- **决策P4.6.1-1（解析改确定性宿主内核，替换 P4.5 Agent 解析）**：Web 暂存区解析直调 `convert_to_markdown`；
  Agent 解析退役为修订（拆分/合并/精修）。升格 P5.2-9 的「非阶段」判定。
- **决策P4.6.1-2（落 `parsed/`、不落 `raw/`；parsed/ 镜像 raw/ 图片布局）**：保「人审后晋级」人在环；引用两处同形，
  晋级按 target stem 归一（决策P4.6.1-13）。
- **决策P4.6.1-3（晋级连图一起搬）**：复用 P5.2.1 staging-swap，晋级后 raw 自洽。
- **决策P4.6.1-4（抽图片原语成共用归口 `imageio.py`，`rawio.py` 不碰）**：CLI convert / Web 解析 / re-localize 三处共用。
- **决策P4.6.1-5（拆分断链：确定性 re-localize，落各文件名下 image 目录 + 改名编号 + 重写引用）**：人点修复按钮触发、
  零-LLM。**安全边界：全局扫描 `workspace/parsed/*.md` + 目标默认 copy + 仅全局无引用才删原图**（Q3）——P4.6 池里拆分后
  原件/草稿仍在，判删绝不能只看当前文件，否则破坏池里其它引用；「move」安全降为「copy + 全局 GC」。
- **决策P4.6.1-6（合并：同 re-localize 引擎，N→1）**。
- **决策P4.6.1-7（解析作业化 + 进度窗口；扩展 JobQueue 运行中增量 output）**：JobQueue 新作业种类、单写者 FIFO。
  **现机制不流式（done 时一次性写 output），需真扩展**：worker 边读边累加 `job.output`、`pollJob` running 态渲染 output；
  best-effort 进度、终态以 done 为准、GIL 整串赋值原子。JobQueue 层改动，ingest/heal/backfill 顺带受益。不新开 SSE。
- **决策P4.6.1-8（安全/容量闸沿 P5.2.1，准入 root 换 `workspace/parsed/`）**。
- **决策P4.6.1-9（全 backend 失败不回落 Agent 解析）**（Q4）：报错收场；用户外部转换或 Agent 交互式转格式，不留兜底 UI。
- **决策P4.6.1-10（转换内核补 `progress=` 流式契约）**：现 `convert.py` `subprocess.run(capture_output=True)` 不流式
  → 给 `convert_to_markdown` 加 `progress: Callable[[str],None] | None`：给定时改 `Popen` 边读 stderr 回调（两管道并发
  drain 防死锁），CLI 传 `None` 走原路径行为不变。**仅扩 JobQueue 不足以让解析进度可见，二者缺一不可。**
- **决策P4.6.1-11（JobQueue 新增 `emit` 进度 sink，签名统一到 `_register`）**：现 `fn()` 无参 + worker `finally`
  覆盖 `job.output`，回调写 output 落不了地 → worker 构造 `emit(line)`（追加 buf + 整串赋值 `job.output`）传给 `fn(emit)`；
  终态 `getvalue()` 保留作权威值。**签名升 `Callable[[Emit],object]` 须统一到 `_register`**——它同服务 `enqueue`+
  `submit_and_wait`，**两入口 thunk 全改签**（含同步 raw_write/upload/delete，易漏）；既有 thunk 忽略 emit、行为不变，仅 parse 用之。
  备选 `_register` 适配包装零参 thunk，本 doc 采统一升签。
- **决策P4.6.1-12（晋级改专用 promotion job，图 + md 同写临界区原子提交）**：现晋级入队前读 content、thunk 仅写 md，已丢
  source 路径/图片引用上下文 → 改专用 job。**入队前**（扩 `_prepare_promotion`）扫引用收集字节 + 归一重写（决策P4.6.1-15，
  未准入/悬空 → 400）+ 记 SHA256 指纹（决策P4.6.1-16）；thunk 携 `(target, 已重写 content, 收集的图片字节集, SHA256 指纹, overwrite)`，
  **写锁内**做 ⓿ SHA256 复检（不符 → 409，决策P4.6.1-16）+ 图 staging-swap 到 `raw/images/<target_stem>/`（决策P4.6.1-13）+ md
  末步提交 + 失败回滚（沿决策P5.2.1-9）；**绝不把搬图放队列外**。投喂分支不变。
- **决策P4.6.1-13（晋级一律按 target stem 归一，不据 name-equality 走 skip，方案 b）**：`/api/raw {name, source}` 允许改名
  （409 后前端引导改名）。promotion job 在写锁内**无条件**按 target stem 归一（collect-rewrite keyed on target stem）：图改名
  编号 `<target>-N.ext` 落 `raw/images/<target>/`、md 引用重写 `images/<target>/…`。保 `raw/images/<raw stem>` 不变式
  （决策P5.2.1-2）。**「同名零重写」是结果非捷径**——不靠 `name==stem` 跳过 rewrite，因 parsed/ 图名可能 relocalize 重编号/
  手工调整/缺号而非 canonical；无条件跑归一（幂等、canonical-in 时零变化）。**不采** (a) 禁改名（废 409→改名 UX）/ (c) 图跟
  source stem（破坏布局）。
- **决策P4.6.1-14（emit 线程安全）**：内核 `progress=` 的 `Popen` 两管道**并发 drain 线程**会并发调 `emit`，而 emit 写
  非线程安全 `StringIO buf` + `job.output`、与 worker `finally` 竞态。给每 `Job` 一把 `threading.Lock`，emit 的
  `buf.write`+`getvalue`+`job.output` 赋值与 `finally` 的 `getvalue` 全在同锁内；parse 进度全程经 emit、无锁外写者。
- **决策P4.6.1-15（晋级搬图引用驱动、目录非判据）**：搬图输入锚在 **source md 的 `![](url)` 引用**、**非
  `parsed/images/<source_stem>/` 目录存在**——拆分/合并后 md 可能引 `images/other/…`、目录可能有未引用残图，按目录搬会
  漏搬/多搬/断图。promotion **入队前**扫 source md 引用 → 经 `workspace/parsed/` root `_admit_image_ref` 准入（解相对 md
  父目录、挡越界/symlink）→ 检在 → 读字节入内存收集 + 按 `target_stem` 重写引用；**任一未准入/悬空 → 400**（提示先
  relocalize，§2.3）。thunk 只在锁内 stage-swap 内存字节集 + 提交 md。**未被引用的残图不搬**。
- **决策P4.6.1-16（晋级 TOCTOU：写锁内指纹复检、fail-closed）**：扫描/收集在入队前、提交在后续写锁内——若前面有长作业排队、
  或之间可写 turn 改了 `workspace/parsed/`，会提交**点击时旧快照**而非提交时的当前状态（P4.6 文本晋级已有此窗口，P4.6.1 把
  图片一致性也绑入、风险更显）。**采策略 (a)**：入队前对 source md + 各收集图片记 **SHA256（内容哈希，权威）**，随字节集带进
  thunk；thunk 在写锁内**复读 + 复算 SHA256 复检**，任一不符即 **fail-closed → 409「source 已变，请重新审阅」**、绝不提交旧快照。
  - **钉死 SHA256、不留「必要时」自由度**：mtime+size 会漏变化（mtime 粒度、保时间戳写入 `os.utime`、同尺寸原位替换），
    安全边界须用内容哈希。`st_mtime_ns`+`size` 至多作**廉价 fast-path 预筛**（先比 stat、变了直接 fail；未变再 SHA256 兜底），
    但 **SHA256 是唯一权威判据**。源 md 内容与图片字节入队前已在内存，算哈希零额外读盘；写锁内仅多一次复读复算。
  - **不采策略 (b)**（把扫描/收集整体移进 job 并让 job 结果表达 400）——更重、且要新建 job-result→400 通道；(a) 保住入队前 400
    快反馈、复检失败天然是 409（worker 可表达）。此复检也**顺带收紧 P4.6 文本晋级**的同一窗口（md 哈希）。

## 5. 边界（明确不做）

- **不改 ingest / `raw/` 只读快照 / 退出码 / 既有 `POST /api/raw {content}` 投喂**；
- **不动 `rawio.py`**（决策P5.2.1-1）；图片处理全在 `imageio.py` 归口；
- **MCP 仍不做 convert/解析**（写 `raw/`/`workspace/` 与只读相悖，沿决策P5.2.1 / P4.10）；
- **不重编码/缩放图片**——逐字搬运、源非派生物（沿决策P5.2.1-8）；
- **reader 模式不注册** parse/relocalize 写端点（沿 P4.9）；
- **不在暂存区留 Agent 解析兜底**——全 backend 失败即报错（决策P4.6.1-9）。

## 6. 测试要点（草案）

- 确定性解析落 `parsed/<slug>.md` + `parsed/images/<slug>/`、引用 `images/<slug>/…`、与 raw/ 同形；
- 晋级连图：按 source md 引用收集 `→ raw/images/<target_stem>/`，**图搬 + md 提交在同一写临界区**（promotion job thunk），
  staging-swap 失败一致性（md 落盘 ⟹ 图就位）；**搬图不在队列外发生**；
- **晋级引用驱动（非目录）**：① md 跨目录引 `images/other/foo.png`（仍在 parsed/ 内）→ 正确收集归一、不漏搬；
  ② source 目录有未引用残图 → **不搬**；③ 悬空引用（文件缺失）/ 未准入（越界/symlink/非白名单）→ **400** 提示先 relocalize；
- **晋级 TOCTOU 指纹复检（SHA256）**：入队后、提交前 source md 或某收集图被改 → thunk SHA256 复检不符、fail-closed
  **409「请重新审阅」**、不提交旧快照；**同尺寸/保 mtime 的原位替换也须被 SHA256 抓到**（不靠 mtime/size）；未变 → 正常提交；
- **JobQueue emit sink**：解析作业 running 期 `emit` 推上的行不被 `finally` 覆盖、终态完整；**`enqueue` 与
  `submit_and_wait` 两路** thunk 全改签（ingest/heal/backfill **及同步** raw_write/upload/delete）后输出字节不变（回归硬门）；
- **改名晋级归一**：`parsed/x.md → raw/y.md` 后 `raw/images/y/y-N.ext` + md 引用 `images/y/…`、`raw/images/x/` 不残留；
- **晋级归一无 skip 捷径**：① canonical + 同名 → 输出逐字不变（实质 no-op）；② **非 canonical（缺号/手工改名）+ 同名 →
  仍归一重写**为 `<stem>-N.ext`（不因 `name==stem` 跳过）；
- **emit 线程安全**：内核 `progress=` 两 drain 线程并发 emit + worker `finally` 并发写 `job.output` 时无竞态/不丢行/不撕裂
  （锁保护 buf+output）；
- 拆分断链检查命中悬空/错位引用；修复后图落 `parsed/images/<file>/`、改名编号、引用重写；
- **re-localize 安全边界**：原件 `parsed/x.md` 仍在池时，修 `x-1.md` **不删**仍被 x.md 引用的原图（copy + 全局 GC）；
  仅当全局 `parsed/*.md` 零引用时才回收原图；
- 合并 N→1 重整；
- **运行中增量 output**：解析作业 running 态 `/api/jobs/{id}` 即可见已累加的 backend 日志（非 done 才出）；终态完整；
- **内核 `progress=` 流式**：`convert_to_markdown(progress=cb)` 走 `Popen`、stderr 行实时回调（CLI `progress=None` 路径
  与现 `subprocess.run` 字节行为不变）；大输出不 pipe 死锁；
- **二者缺一**：只扩 JobQueue 而内核仍 `subprocess.run` → running output 空；只改内核而 JobQueue 不增量 → 前端看不到——两测都要覆盖；
- 安全闸：准入 root=`workspace/parsed/` 挡越界/symlink；容量三闸超限 → 4xx，不静默丢图；
- 进度窗口：解析作业 backend 分级日志流式可见；
- **`rawio.py` 零行为变更 / P4.6 投喂晋级回归**（硬门，`tests/test_web.py` 逐条仍绿）；
- reader 不注册写端点（404/405）。

## 7. 开放问题裁定（已定）

- **Q1｜进度流** → **扩展 JobQueue 支持运行中增量 output**（现机制不流式，需真扩展：worker 边读边累加 output +
  pollJob running 态渲染），不新开 SSE（决策P4.6.1-7）。
- **Q2｜修订/拆分/合并谁改文本** → **维持 P4.5**：Agent 改 `parsed/` 文本，宿主只做确定性图片重整（决策P4.6.1-1 分工）。
- **Q3｜多文件共引同一图** → **接受拷贝 + 全局 GC**：全局扫描 `parsed/*.md`，目标默认 copy，仅全局无引用才删原图（决策P4.6.1-5）。
- **Q4｜全 backend 失败兜底** → **不保留 Agent 解析兜底**：报错收场，用户外部转换或 Agent 交互式转格式（决策P4.6.1-9）。
