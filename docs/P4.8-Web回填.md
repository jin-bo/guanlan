# P4.8 — Web 端问答回填（`query --backfill`）落地小设计

> 本文落地 [`P4-Web宿主.md`](P4-Web宿主.md) §10「**推后项**」里点名的那一项：**`query --backfill`（gated 写
> synthesis）** —— 把一次值得沉淀的好问答经写门禁回填成 `wiki/syntheses/<slug>.md`，全程不离开浏览器。
> 高层理念与原则以 [`DESIGN.md`](DESIGN.md)（§4.7 写门禁、§5.2 读写分线、原则 2「人不直接写 wiki、Agent
> 全权拥有 wiki 层」、原则「`query` 默认只读，仅 `--backfill` 才写 `syntheses/`」）为准；`--backfill` 的 CLI
> 契约见 [`P2-最小闭环.md`](P2-最小闭环.md) §8（`run_query(question, *, backfill, …)`、与 ingest **完全一致**
> 的失败语义与门禁）。本文只把 P4.8 的「怎么做」钉死到端点、写作业、前端 UX 与测试。文中标注的
> **\[决策P4.8-N]** 是对 P4 / P2 既有契约的具体化或受控偏离，集中记录在 §6。
>
> **不是新相位。** P5（多格式与自动化）保持路线图原义；本项是 P4 Web 宿主之上、**复用 `query.py` 既有
> `run_query`/`run_guarded_write` 的半相位增强**（承 P4.1…P4.7 之后，故编号 P4.8），**不写一行新 LLM 集成
> 代码、不新增退出码、不动门禁、不加 SSE**。请先读 [`P4-Web宿主.md`](P4-Web宿主.md)（尤其 §4.1 读写分线、
> §4.2 单写者作业、§5 HTTP 契约、§8 安全姿态、§10 推后项）与 [`P4.3-Web-heal.md`](P4.3-Web-heal.md)（最近的
> 同构写作业半相位：`enqueue` + 轮询 + 退出码徽标）。

## 1. 目标与范围

**P4.8 = 把 `guanlan query --backfill "…"` 放进浏览器：提一个值得沉淀的问题 → 一键经写门禁让 Agent 答出
带 `[[页]]` 引用的好答案并回填 `wiki/syntheses/<slug>.md` → 轮询看确定性门禁回执 + 右栏即时点开新综合
页 —— 全程不离开浏览器、不旁路任何 P2 门禁。**

现状（P4 + P4.1…P4.7）：浏览器能浏览 wiki、只读多轮问答、从 `raw/` 选篇 ingest、投喂源、补全缺页（heal）、
上传与晋级、双语切换，**唯独缺 `query --backfill`** —— 想把一次好问答沉淀成综合页，仍须回命令行跑
`guanlan query --backfill`。这是 P4 §10 明列的最后一项 Web 写入口缺口。P4.8 用**一个写端点 + 两个前端入口**
补齐它：

- **`POST /api/backfill {question, model?}`**：把 backfill 作为**新写作业 kind** 入 P4 既有单写者 FIFO
  `JobQueue`（提交方式**对标 ingest 的 `enqueue`**，即时返回 `job_id`、前端轮询 `/api/jobs/{id}`）。作业内
  **整体复用** `run_query(question, backfill=True, …)` —— 它本就走 `run_guarded_write`（P2 子进程 + `raw/`
  快照门禁 + 有界自愈，与 ingest **同一条写路径**），答案与门禁回执由 `report_outcome` 打印到 stdout、被
  worker 的 `redirect_stdout` 捕获进 `job.output`（与 ingest 同口径）。
- **两个前端入口**（§5）：① 顶栏「回填」动作 → 浮层输入问题 → 提交；② 每条只读问答气泡下的「沉淀」小按钮
  → 把**该轮的问题**预填进回填浮层（**不**复用只读答案原文，见决策P4.8-3）→ 同一端点提交。

backfill 在 Web 端**仍是写操作**：它经 `run_guarded_write` 走子进程 + `raw/` 快照门禁 + 有界自愈，故归到
**写线** —— 单 worker FIFO 串行、轮询拿结果，与 ingest / heal 端点**同构**。Web 层**零业务智能**：答案、
门禁裁决、退出码全来自被复用的 `query.py`/`gate.py` core，宿主只「收 HTTP → 调 `run_query` → 转发」。

### 纳入 P4.8

- `POST /api/backfill {question, model?}` → `{job_id}`（入单写者队列 `enqueue`，与 ingest/heal/投喂同 worker
  串行；`question` 必填非空，否则 422）。
- 作业线程**整体调** `run_query(body.question, root=root, backfill=True, model=body.model or model,
  runner=runner)` —— **返回退出码 `int`**（与 ingest 的 `_job` 同形），答案 + 门禁回执经 stdout 捕获进
  `job.output`。**无需 `Job.result` 结构化承载**（区别于 heal，见决策P4.8-2）。
- `on_job_done` 把 `"backfill"` 并入「写了 wiki/ → bump wiki 代际」的 kind 集（`ingest`/`heal`/`backfill`），
  使可写会话缓存的 check 基线失效、下轮重拍（P4.5-4 缓存基线，决策P4.8-5）。
- **前端**：顶栏「回填」按钮 + 浮层（问题文本域 → 「沉淀」提交 → 复用既有 `pollJob` 默认渲染：退出码徽标
  + `job.output`）；每条 assistant 气泡的「沉淀」小按钮（预填该轮问题、打开同一浮层）。完成后 `loadPages()`
  刷新 wiki 列表，新综合页在右栏可点开。
- 新增界面 chrome 词条（zh/en 双语对齐，P4.7 守恒，§5/§7）+ 一枚 `#i-backfill` 图标。

### 不纳入 P4.8（仍按路线图推后或刻意不做）

- **复用只读会话的答案原文直接落盘**（「把这条 bot 气泡一字不改存成 synthesis」）→ **不做**（决策P4.8-3）：
  只读问答走**嵌入只读 Agentao**（P4 决策P4-8），其答案是**预览/试水**、未过任何写门禁；backfill 是**另起一次
  gated 写**，由子进程 Agent **重新**读 wiki、综合、写 frontmatter 齐全的 synthesis（与 CLI `--backfill` 逐字节
  同义）。前端入口②只**预填问题**、不搬答案 —— 跨「只读嵌入」与「子进程写」两条线搬运 transcript 既破读写
  分线、又会把未审定文本钉成权威页。
- **零-LLM 预览**（仿 heal 的 `--dry-run`）→ **无**（决策P4.8-4）：backfill 写什么 synthesis 取决于 LLM，无法
  确定性预算。其「审阅」发生在**事后** —— `job.output` 给出答案 + 门禁回执，`loadPages()` 后新 `syntheses/`
  页在右栏可即时点开核对。
- **backfill 作业的 token 级流式（SSE）** → 不做：与写作业 ingest/heal **同口径只轮询**（决策P4-6：飞行中无
  流可推、轮询对单用户足够）。答案随门禁回执一并落 `job.output`。
- **多轮上下文 / 附件参与 backfill** → 不做：CLI `--backfill` 是**单问句 → 单 synthesis**，无会话上下文、无
  附件；Web 端保持同义（决策P4.8-3）。
- **`query --backfill` 的 `--no-backfill` 只读问答** → 已在 P4 `POST /api/chat` 覆盖（只读嵌入会话），不在本
  相位重复。

## 2. 端点契约（`guanlan/web/app.py`）

| 方法 路径 | 入参 | 复用 | 返回 | 错误 |
|------|------|------|------|------|
| `POST /api/backfill` | `BackfillBody {question:str(strip 后非空), model:str\|None=None}` | `enqueue` + `run_query(backfill=True)` | `{job_id}`（**即时返回**，前端轮询；唯一写入口之一，与 ingest/heal 并列） | `422` 体非法（`question` 经 strip 后为空——含纯空白） |
| `GET /api/jobs/{id}` | — | 作业表 | `{id,kind,state,exit_code,output, result}`——backfill 作业 `result` 恒为 `null`（与 ingest/投喂同，**无结构化回执**，决策P4.8-2）；答案 + 门禁回执在 `output` | `404` 未知 id（沿用 P4） |

```python
class BackfillBody(BaseModel):
    """`POST /api/backfill` 请求体（P4.8）。`question` 必填、**strip 后非空**（含纯空白 `"   "`
    → 422，对齐 CLI 必传问句、杜绝空白问句白触发一次 gated LLM 写）。`model` **省略或显式 `null`
    均回落 app 级 `model`**（端点 `body.model or model`，与 ingest/heal 同口径，决策P4.8-6）；二者皆无
    时**解析出的** `None` 直透子进程 runner 也合法（backfill 走 run_guarded_write 子进程、非嵌入会话、
    无 P4-8 模型坑）。"""

    question: str
    model: str | None = None

    @field_validator("question")
    @classmethod
    def _question_nonblank(cls, v: str) -> str:
        # `min_length=1` 挡不住纯空白（`"   "` 长度非零、会过校验后入队白触发一次 gated 写）。
        # 故 strip 后判空 → 抛 ValueError（Pydantic 归一为 422）；并**返回 strip 后的值**，使
        # 发给 Agent 的问句不带前后空白噪声（决策P4.8-8）。
        s = v.strip()
        if not s:
            raise ValueError("question 不能为空（含纯空白）。")
        return s
```

```python
# 写（唯一写入口之一 = backfill）：即时入队、立刻返回 job_id；前端轮询 /api/jobs/{id}（无 SSE）。
# 与 ingest 端点逐行同构 —— 整体复用 run_query(backfill=True)，答案 + 门禁回执经 stdout 捕获进 job.output。
@app.post("/api/backfill")
async def backfill(body: BackfillBody) -> dict:
    _reject_if_writable_active()  # 层③：可写 turn 活跃 → 423（决策P4.5-10 / P4.8-5）

    def _job() -> int:  # 返回退出码 int（与 ingest 同形）→ worker 鸭子分流走 int 分支、result 恒 None
        return run_query(
            body.question,
            root=root,
            backfill=True,
            model=body.model or model,
            runner=runner,
        )

    return {"job_id": jobs.enqueue("backfill", _job)}
```

- **比 ingest 还薄**：ingest 的 `_job` 调 `run_ingest`，backfill 的 `_job` 调 `run_query(backfill=True)` ——
  二者都**返回退出码、把人读文本打到 stdout**，故 worker 走既有 `int` 鸭子分流分支（决策P4.3-1），`Job.result`
  恒 `null`、`jobs.py` 一行不改。**无 core 重构**（区别于 heal 当初要抽 `run_heal_result`）：`run_query` 已是
  「答案 + 回填 + 门禁」一体的现成写入口，直接拿来用。
- **`question` 必填、strip 后非空**：`field_validator` strip 后判空 → 422（决策P4.8-8）——`min_length=1`
  挡不住纯空白 `"   "`（长度非零会过校验，入队白触发一次 gated LLM 写）；validator 还**返回 strip 后的
  值**，使发给 Agent 的问句不带前后空白。需把 `field_validator` 加进 app.py 既有
  `from pydantic import BaseModel, Field` 一行。其余校验（库根齐全、`raw/` 不变性、frontmatter 合规）全由
  `run_query` 内部的 `require_kb_root(writable=True)` + `run_guarded_write` 兜底，Web 不旁路、不预判。
- **退出码透传进 `job.exit_code`**：backfill 走 `run_guarded_write` 的写路径四值
  `EXIT_OK(0)`/`EXIT_RAW_MUTATED(4)`/`EXIT_CHECK_FAILED(3)`/`EXIT_AGENT_ERROR(5)`（P2 §8，与 ingest 完全一致），
  前端据此显示徽标；**P4.8 不新增退出码**（同 P4）。
- **`result` 恒 `null`**：backfill 的产物是**散文**（带 `[[页]]` 引用的答案 + 「✓ 门禁通过」回执），`job.output`
  捕获即够 —— 与 ingest 同（决策P4-7）。无 heal 那种「逐目标 resolved/still_broken」结构化回执，故**不动
  `Job.result`、不加序列化器**（决策P4.8-2）。

## 3. 复用 `run_query`：无 core 重构、无结构化结果——本设计的核心　**\[决策P4.8-2]**

heal 当初要把 `run_heal` 抽出不打印的 `run_heal_result -> HealRun`，是因为它的价值在**结构化回执**（逐目标
resolved/still_broken、批次级 `unexpected_writes`/`changed_paths`），不能从 stdout 文本反推（决策P3.2-10）。
**backfill 不同**：它和 ingest 一样，产物是**一段散文** —— 答案正文（已含 `[[页]]` 站内引用）+ 「✓ 门禁通过
（frontmatter + sources + raw/ 快照）」一行回执，外加自愈轮的 `↻` 进度（走 stderr）。这些**就是要给人读的
文本**，`job.output` 原样捕获即是最终展示形态，无任何「机器判定」需结构化承载。故：

- **`query.py` 一行不改**：现成的 `run_query(question, *, root, backfill, model, runner) -> int` 直接作作业
  thunk。`_run_backfill → run_guarded_write → report_outcome` 的打印路径**原样**被 worker 的
  `redirect_stdout`/`redirect_stderr` 捕获（与 ingest `run_ingest` 同口径，决策P4-7）。
- **`jobs.py` 一行不改**：thunk 返回 `int`，worker 走既有 `isinstance(outcome, int)` 分支存 `exit_code`、
  `result` 留 `None`。`Job.result` 是 heal 专属、对 backfill 向后兼容（前端忽略 `null`）。
- **`/api/jobs/{id}` 一行不改**：backfill 作业 `job.result` 为 `None` → 端点既有
  `heal_result_dict(...) if isinstance(job.result, HealRun) else None` 自然回 `null`。

> **为何不抽「`run_backfill_result` 结构化 core」**（被否的备选）：backfill 无 heal 那样必须程序化消费的逐项
> 回执；强抽一层只会凭空造一个 `result` 又没人读，违背 P4.3「`result` 单一职责 = 机器回执」的克制。ingest
> 至今也只回 `int` + `job.output`，backfill 与它同形即可 —— **这是本相位「最薄」的根因**：它是 P4 唯一一个
> **既有写入口函数能原样入队、零适配**的半相位。

## 4. 读写分线与单写者串行（复用 P4 §4.1/§4.2，不引新基础设施）

backfill 在 P4 的读写分线里归**写**（它过写门禁、改 `wiki/syntheses/` + `index.md` + `log.md`），故落点完全
沿用 ingest 的写路径设施，**零新写基础设施**（`JobQueue` 是通用的 `enqueue(kind, thunk)`）：

| | backfill（`POST /api/backfill`） |
|--|--|
| 性质 | LLM 写（过 P2 门禁，写 `wiki/syntheses/`） |
| 集成 | `enqueue` 入单写者 FIFO worker，`run_query(backfill=True)` |
| `raw/` 快照 | 由 `run_guarded_write` 内部取（与 ingest 同；`workspace-write` 姿态不护 `raw/`，靠快照硬保证，P2 §5.1/§10 决策4） |
| 并发 | 单 worker FIFO 串行（与 ingest / heal / 投喂 / 上传同 worker） |
| 拿结果 | 即时返回 `job_id` → 轮询 `/api/jobs/{id}` |

- **单写者假设不破**：backfill 与 ingest、heal、投喂在**同一条 worker** 上 FIFO 串行 —— 写作业永不并发，
  `raw/` 前后快照不被并发写互踩（决策P4-5）。
- **排在前序写作业之后**：若提交时正有 ingest/heal 在飞，backfill 作业 pending 到它跑完才轮到 —— 与
  heal 同款排队语义，用 `enqueue`（异步轮询）而非 `submit_and_wait`，因为 backfill 自身慢（LLM 子进程，
  同 ingest），轮询体验恰当。
- **走子进程 runner、无嵌入四坑**：backfill 经 `run_query(backfill=True) → run_guarded_write → agentao run`
  子进程（与 ingest 同），**不是**只读嵌入会话（§4.4 那条）。故 P4-8 的「`model=None` 盖掉 `.env` 模型触发
  ValueError」嵌入坑**不适用** —— backfill 经 `body.model or model` 解析出最终 model（省略/`null` 均回落
  app 级，可能为 `None`）直透子进程 runner，与 ingest 一致；解析出的 `None` 在子进程路径合法。
  `runner=app.state.runner` 供测试注入 fake（不打真实 LLM），同 ingest。
- **安全姿态沿用 P4 §8**：backfill 只写 `wiki/`（`syntheses/<slug>.md` + `index.md`/`log.md` catalog 追加），
  **永不碰 `raw/`** —— 由 P2 `raw/` 快照门禁硬兜底（`workspace-write` 允许写 `<wd>` 内任意路径含 `raw/`，故
  权限姿态不保护 `raw/`，必须靠快照，P2 §10 决策4）。端点无 `path`/`name` 入参（只 `question`/`model`），故
  **无路径穿越面**。
- **层③ 时序互斥**：backfill 是宿主写端点 → 进 `_reject_if_writable_active()`，可写 turn 活跃期间一律
  `423 Locked`，兜「shell `curl POST /api/backfill` 让宿主替 agent 起 gated 写」的旁路（决策P4.5-10，与
  `/api/raw`/`ingest`/`heal` 同口径，决策P4.8-5）。

## 5. 前端（`guanlan/web/static/`）

backfill 的 UX 双入口、单端点 —— **复用既有 overlay + `pollJob` + 退出码徽标骨架**，不引新组件（决策P4-3
克制）。

### 5.1 入口①：顶栏「回填」动作

- `index.html` 顶栏 `actions` 在「补全（heal）」后新增
  `<button id="backfill-btn" class="btn" data-i18n-title="tip.backfill" title="…">…回填</button>`（带 `#i-backfill`
  图标 + `data-i18n="btn.backfill"` 标签），紧邻 heal —— 三个 LLM 写入口（入库 · 补全 · 回填）成组。
- `backfill-btn` → `openBackfill(prefillQuestion = "")`：`showOverlay("backfill", …)` → 浮层渲染：
  - 一个**问题文本域**（`placeholder = t("backfill.placeholder")`，可预填 `prefillQuestion`）；
  - **「沉淀」提交按钮**（`t("backfill.submit")`，问题为空时禁用）→ `triggerBackfill(question)`；
  - 一行说明 `t("backfill.hint")`：「Agent 将重新综合带 `[[页]]` 引用的答案并回填 `wiki/syntheses/`，走写
    门禁（同 ingest）。」
  - `overlayRepaint = () => renderBackfillForm($("#backfill-question")?.value ?? prefillQuestion)`：语言切换
    纯重渲染时**从当前文本域读值**作新 prefill（而非吃最初的 `prefillQuestion`），故用户已输入但未提交的
    内容跨语言切换**不丢**；文本域尚未挂载（理论边界）时回退最初 prefill（决策P4.8-9）。

```js
async function triggerBackfill(question) {
  showOverlay("backfill", `<p class="muted">${escapeHtml(t("backfill.submitted"))}</p>`);
  try {
    // 不用 postJSON（它只 throw `url → status`、丢 detail，无法分流 423）：用裸 fetch 显式
    // 特判 423（可写 turn 活跃）→ 本地化 backfill.busy，与既有 staging.writableRetry 同款
    // （app.js 现有 `if (res.status === 423)` 模式）。其余非 2xx → common.submitFail（带状态码）。
    const res = await fetch("/api/backfill", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ question }),
    });
    if (res.status === 423) {
      $("#overlay-body").innerHTML = `<p class="report-bad">${escapeHtml(t("backfill.busy"))}</p>`;
      return;
    }
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    const { job_id } = await res.json();
    await pollJob(job_id, "backfill"); // 复用默认渲染：退出码徽标 + job.output（答案 + 门禁回执）
  } catch (e) {
    $("#overlay-body").innerHTML = `<p class="report-bad">${escapeHtml(t("common.submitFail", e.message))}</p>`;
  }
}
```

- 完成分支**直接复用 `pollJob` 的默认渲染**（无 `renderDone`）：退出码徽标（`✓ 门禁通过` / `退出码 N`）+
  `<pre>job.output</pre>`（答案正文 + 门禁回执，按字面显示 —— 答案是**内容**、不译，P4.7 决策P4.7-1）。
  `pollJob` 末尾 `loadPages()` 刷新 wiki 搜索列表 —— 新 `syntheses/` 页随即可在右栏搜索·点开核对。

### 5.2 入口②：每条问答气泡的「沉淀」小按钮

## 6. 决策记录

- **\[决策P4.8-1] backfill 在 Web 端是 gated 写、入单写者 FIFO `JobQueue`，用 `enqueue`（异步轮询）。**
  它经 `run_query(backfill=True) → run_guarded_write` 走 P2 子进程 + `raw/` 快照门禁 + 有界自愈（与 ingest
  同一条写路径），故归写线、与 ingest/heal/投喂同 worker 串行（单写者假设不破，`raw/` 快照不被并发写互踩）。
  自身慢（LLM 子进程）→ 用 `enqueue` 入队即返回 `job_id` + 轮询，非投喂的 `submit_and_wait`。**零新写
  基础设施**（`JobQueue` 本就是通用 `enqueue(kind, thunk)`，backfill 只是新 job kind）。
- **\[决策P4.8-2] backfill **不抽结构化 core、不动 `Job.result`** —— 与 ingest 同形（`int` + `job.output`）。**
  backfill 的产物是散文（带 `[[页]]` 引用的答案 + 「✓ 门禁通过」回执 + `↻` 自愈进度），`job.output` 原样
  捕获即最终展示形态，无须程序化消费的逐项回执（区别于 heal 的 `HealResult`）。故 `query.py`/`jobs.py`/
  `/api/jobs/{id}` **一行不改**，`run_query` 现成函数直接作作业 thunk —— 本相位是 P4 唯一「既有写入口零适配
  入队」的半相位。否决「强抽 `run_backfill_result`」：凭空造没人读的 `result`，违 P4.3「`result` 单一职责 =
  机器回执」的克制。
- **\[决策P4.8-3] 前端入口②只**预填问题**、绝不搬只读答案原文落盘；backfill 是另起一次 gated 写。**
  只读问答走嵌入只读 Agentao（决策P4-8），其答案是**未过写门禁的预览/试水**；backfill 由**子进程 Agent
  重新**读 wiki、综合、写齐 frontmatter 的 synthesis（与 CLI `--backfill` 逐字节同义）。搬 transcript 跨「只读
  嵌入」与「子进程写」两条线，既破读写分线、又会把未审定文本钉成权威页。按钮只作「这个问题值得沉淀」
  的快捷起点，hint 点明 synthesis 不承诺与气泡逐字相同。同理：**无多轮上下文、无附件参与 backfill**（CLI
  `--backfill` 是单问句 → 单 synthesis，Web 保持同义）。
- **\[决策P4.8-4] 无零-LLM 预览（区别于 heal 的 `--dry-run`）。** backfill 写什么 synthesis 取决于 LLM、无法
  确定性预算，故无 `GET /api/backfill/preview`。审阅在**事后**：`job.output` 出答案 + 门禁回执，`loadPages()`
  后新 `syntheses/` 页在右栏即时可点开核对。这与 heal（worklist 可零-LLM 预算）刻意不同。
- **\[决策P4.8-5] backfill 是宿主写端点 → `_reject_if_writable_active()`（层③ 423）+ `on_job_done` bump
  wiki 代际。** 进锁前按层③拒（兜「shell `curl POST /api/backfill`」旁路，与 `/api/raw`/`ingest`/`heal` 同
  口径，决策P4.5-10）；作业写了 `wiki/` → 把 `"backfill"` 并入 `on_job_done` 的 bump 集
  （`ingest`/`heal`/`backfill`），使可写会话缓存的 check 基线失效、下轮重拍（P4.5-4 缓存基线）。
- **\[决策P4.8-6] 走子进程 runner、`model` 用 `body.model or model`（省略≡`null`，均回落 app 级）；嵌入
  四坑（决策P4-8）不适用。** 与 ingest/heal **逐字同口径**：请求体 `model` 省略**或显式 `null`** 都回落
  app 级 `model`（`body.model or model`）——刻意**不**用 `model_fields_set` 区分「缺省」与「显式 null」（CLI
  无此区分、Web 不引新语义）。最终解析出的 model 可能为 `None`（app 级也未设），直透子进程 runner 合法
  （非只读嵌入会话，故 P4-8「`model=None` 盖掉 `.env` 触发 ValueError」坑不适用）。backfill 经
  `run_guarded_write → agentao run` 子进程，`runner` 可注入 fake 供测试；门禁/自愈/`REPAIR_PROMPT`/`raw/`
  快照全继承 `gate.py` core，Web 端零新增。
- **\[决策P4.8-8] `question` 用 `field_validator` strip 后判空（422），不用 `Field(min_length=1)`。**
  `min_length=1` 只看长度、挡不住纯空白 `"   "`（长度非零会过校验，入队白触发一次慢且 gated 的 LLM 写）。
  故用 `@field_validator("question")` strip 后判空 → 抛 `ValueError`（Pydantic 归一为 422），且**返回
  strip 后的值**使发给 Agent 的问句不带前后空白噪声。前端 `triggerBackfill` 另在按钮层禁用空输入（双重
  兜底，非依赖）。
- **\[决策P4.8-9] 语言切换的 `overlayRepaint` 从当前文本域读值、不回放最初 prefill。** 回填浮层是少数
  带**用户可编辑输入**的动态面（区别于报告/heal 那种纯只读重渲染），故 `overlayRepaint` 必须
  `renderBackfillForm($("#backfill-question")?.value ?? prefillQuestion)` —— 读回实时输入作新 prefill，使
  「切语言不丢已输入文本」成立（P4.7 决策P4.7-1「切换纯重渲染、不 reload/refetch」在此具体化为「从 DOM
  读回易失态再重渲染」）。文本域未挂载的边界回退最初 prefill。
- **\[决策P4.8-7] 不新增退出码、不做 backfill SSE。** 沿用 P4 退出码面（进程 `EXIT_OK`/`EXIT_USAGE`，写路径
  四值 `0/4/3/5` 只进 `job.exit_code`，与 ingest 一致）；写作业只轮询、不流式（决策P4-6）；backfill 仍**显式
  按需触发**（点按钮才跑），不内联进 query、不定时。

## 7. 测试要点（`tests/test_web.py` + `tests/test_web_i18n.py`，不打真实 LLM）

`run_query(backfill=True)` 的薄 adapter。**Web 测试只聚焦 adapter 本身**：入队、退出码透传、`result` 恒 null、
旧 job 兼容、FIFO 一个烟测、i18n 键奇偶。

`test_web.py`（adapter，`web` extra 缺失 → 整组 `importorskip` 跳过）：

| 测试 | 覆盖 |
|------|------|
| POST 入队 + 答案落 output | fake runner 写合规 synthesis → `POST /api/backfill {question}` **即时**返回 `job_id`（非阻塞，区别于投喂 `submit_and_wait`）→ **轮询** `/api/jobs/{id}` 至 `done`：`exit_code==0`、`output` 含答案 + 「✓ 门禁通过」、`result==null`（决策P4.8-2）。仅此一条 happy-path 用 fake runner 跑通端到端 |
| 退出码透传（纯 adapter，不碰 gate） | **monkeypatch `run_query` 返回一个选定退出码**（如 `EXIT_CHECK_FAILED`）→ 断言它原样进 `job.exit_code`、`result` 仍 `null`。**不**用 fake runner 制造坏 frontmatter 去触发真 gate 判定——那会在 `test_web.py` 重证 P2 gate 行为，违背本节测试分工（gate 四值已由 `test_query.py` 证过）。adapter 只需证「core 给什么退出码，端点就透传什么」 |
| 空/空白问句 422 | `POST /api/backfill {question:""}`、`{question:"   "}`（纯空白）、缺 `question` 三例 → 均 `422`（`field_validator` strip 后判空，决策P4.8-8）；**未触 runner / 不入队**（断言 fake runner 零调用）。另验合法问句被 strip：**monkeypatch `run_query`** 截获 `question` 实参，`{question:"  问题  "}` 入队后断言端点传入的 `question == "问题"`（前后空白已剥）——直接断 endpoint→core 的入参，**不**断 runner 收到的文本（runner 拿到的是 `run_query` 组装后的完整 prompt、非原始 question） |
