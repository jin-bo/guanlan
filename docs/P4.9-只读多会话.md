# P4.9 — 只读多会话（无用户管理的多用户隔离）落地小设计

> 本文落地一类**最轻**的多用户诉求：**多用户 / 每用户单 Session / 全只读 / 无用户管理**。它把
> [`DESIGN.md`](DESIGN.md) §5.2 的**读写分线**从「进程内」抬到「部署姿态」—— 用一个**只读部署**承接所有
> 用户的并发问答，会话隔离不靠账号、不靠 owner 表，而靠**前端已在用的能力 UUID**（`?c=<conversation_id>`，
> 见 `static/app.js:1160`）。高层理念以 [`DESIGN.md`](DESIGN.md)（§5.2 读写分线、原则「`query` 默认只读」）与
> [`P4-Web宿主.md`](P4-Web宿主.md)（§4.1 读写分线、§4.2 单写者作业、§8 安全姿态、决策P4-4「仅 127.0.0.1」、
> 决策P4-8「嵌入只读会话内存上限 `MAX_CONVERSATIONS`」）为准；会话落盘契约见 [`P4.2-会话落盘.md`](P4.2-会话落盘.md)。
> 本文只把 P4.9 的「怎么做」钉死到部署开关、端点裁剪、隔离模型、回收策略与测试。文中标注的
> **\[决策P4.9-N]** 是对 P4 既有契约的具体化或受控偏离，集中记录在 §6。
>
> **不是新相位，是 E2 前哨。** 路线图（[`DESIGN.md`](DESIGN.md) §7）的 **E2 — 多租户权限** 方向是
> 「**source 级作用域** + 信任边界」，那是**内容分租**（不同用户看不同 source），要动数据模型，归企业版、排在
> P 轨跑通之后。**本相位刻意只做 E2 的会话分租子集**：KB 内容对所有用户**全共享**（只读可见），隔离的只是
> **各自的会话历史**。它落在 P4 Web 宿主演进线上（承 P4.1…P4.8，故编号 P4.9），**不写一行身份/鉴权代码、
> 不新增退出码、不动门禁、不加 SSE、不碰数据模型**。`source 级作用域`显式列为 non-goal（§1、决策P4.9-8）。

## 1. 目标与范围

**P4.9 = 把现有 P4 Web 宿主以「只读部署」对多个用户开放：每个浏览器持有自己的会话 UUID（`?c=`）各聊各的、
互不可见；对 KB 零字节写入、无账号、内存会话不落盘。** 改动**少量但集中**（横跨 `cli.py`/`server.py`/`app.py`/
`chat.py`/`static/`，见 §8 diff 面）—— 前端的能力 UUID 已现成（`static/app.js`），后端主要是 ①一个 `--reader`
部署开关裁掉写端点与会话枚举端点 + 三态日志旗标 + 写权限早失败、②一个内存会话 idle 回收防表涨爆、③`?c=`
恢复改走按-id 探针。

> **「零落盘」是对 KB 而言**，非进程级绝对零写：首次构造会话时 `ensure_skill_available`（`chat.py:296`）若
> skill 未装会**幂等写全局** `~/.agentao/skills/`（`skill.py:97`，**KB 外、一次性**）。要进程级零落盘，reader
> 部署**预装 skill**（`guanlan install-skill`）即可——届时不触发自动安装（决策P4.9-14）。

现状（P4 + P4.1…P4.8）：宿主是**单用户**设计 —— 写端点无条件注册、`GET /api/conversations` 列出**全部**会话、
内存会话受全局 `MAX_CONVERSATIONS=100` 硬上限（`chat.py:71`）。直接对多用户开放会漏两件事：任何人能
`POST /api/ingest` 写共享 KB（破只读）、能 `GET /api/conversations` **枚举并续聊别人的会话**（破隔离）。P4.9
用**一个部署模式**堵死这两口，把「多用户只读问答」压到最简。

**核心洞察**：只读 + 无用户管理两约束一加，**没有任何共享可变状态要保护**（无写 → 无写锁、无 `raw/` 快照、
无跨进程协调），「会话隔离」退化为「**持有猜不到的 128-bit UUID = 能访问**」的能力模型（capability URL，形如
「知道链接即可读」）。这个 UUID 前端**已经**在 SSE `start` 帧拿到并镜像进 `?c=`（`app.js:1160/1167/1960`），
P4.9 要做的只是**别让它可被枚举**。

### 纳入 P4.9

- **`--reader` 部署模式**（`cli.py` 加旗标 → `create_app(reader=True)`，决策P4.9-2）：`reader=True` 时
  **不注册**全部写路由（`POST /api/raw`·`/api/upload`·`/api/ingest`·`/api/heal`·`/api/backfill`、
  `DELETE /api/workspace/file`·`/api/workspace/dir`、**`POST /api/chat/{id}/undo`**（写者，决策P4.9-17）、
  **`GET /graph` 重建**（写者，决策P4.9-11）），命中即 `404`（路由不存在，物理写不了 KB）。`GET /api/workspace*`
  读端点**保留**——workspace scratch 定义为共享可读（决策P4.9-16，纳入威胁模型 §3）。
- **关会话枚举**（决策P4.9-3）：`reader=True` 时**不注册** `GET /api/conversations`（`app.py:1399`，列全部会话）。
  按 id 寻址的 `GET /api/conversations/{id}/messages`（`:1404`）与 `DELETE /api/conversations/{id}`（`:1434`）
  **保留** —— 它们是**能力式**的（持 id 才能用），不泄露他人 id；这正是隔离的支点。
- **强制只读姿态**（决策P4.9-4）：`reader=True` 时新会话开局 `mode="read-only"`，且 `POST /api/chat/{id}/mode`
  拒绝切到 `workspace-write`（`409`/`403`）—— 否则可写工作会话能写 `workspace/`，破「全只读」。
- **每用户单 Session**：浏览器持一个 `?c=<id>`（`app.js:1160`），刷新经 URL 恢复同一会话。单 session 语义由
  前端持有「一个」UUID 表达，后端不强制（决策P4.9-5）。**但恢复路径要改**：现前端 `?c=` 恢复**先拉枚举**
  `GET /api/conversations`（`app.js:2174`）再在列表里找 id、找不到就抹掉 `?c=`（`app.js:2185`）——而 reader
  关了枚举（决策P4.9-3），照现状刷新会**丢会话、核心诉求塌掉**。故 reader 下 `?c=` 恢复**改走按-id 端点**
  （`GET /api/chat/{id}/info` 或 `GET /api/conversations/{id}/messages`）：命中→恢复，404→抹掉 `?c=` 干净起手
  （决策P4.9-12）。
- **内存会话 idle 回收**（决策P4.9-6）：`ConversationStore` 加 idle TTL，淘汰久无活动的会话 —— 缓解多并发
  用户把 `MAX_CONVERSATIONS=100` 顶满后新用户被「内存会话数已达上限」拒（`chat.py:752`）。**注意**：仅在有 idle
  slack 时解围、且淘汰会丢该用户会话（容量边界 + UX 悬崖见 §4）。

### 不纳入 P4.9（仍按路线图推后或刻意不做）

- **用户管理 / 鉴权 / owner 表** → **不做**（决策P4.9-1）：隔离靠能力 UUID，不靠账号。要真鉴权（连聊天记录也
  不许旁人凭链接读）→ 把身份交给**反向代理 / IdP**（注入可信 `X-Forwarded-User`），宿主仍 bind 127.0.0.1
  只读可信 header —— 那是更高一档需求，**不在本相位**。
- **source 级作用域（内容分租）** → **不做**（决策P4.9-8）：这是 **E2 proper** 的核心（不同用户看不同 wiki
  内容），要动 `pages.*` 可见性与数据模型。P4.9 的 KB 对所有用户**全共享只读**。显式 non-goal。
- **会话落盘 / 跨重启恢复**（P4.2 持久化）→ **不做**（决策P4.9-5）：sessions 保持**内存态**，进程重启即丢
  —— 对「轻量只读多会话」可接受，且省掉 `.agentao/sessions/` 共享目录的 per-user 隔离与 rotation 串味问题
  （P4.2 决策的 10-file 目录级轮转在多用户下会互挤）。`reader=True` 强制 `session_persist=False`、`agent_log`
  默认关（`--agent-log` 可开调试，开则经显式写探针保证只读挂载启动早失败）——默认对 KB 零字节写入
  （决策P4.9-13，落盘面钉死见 §4）。
- **对外网络暴露 / TLS / 限流** → **不做**：宿主仍 `workers=1` + **仅 127.0.0.1**（决策P4-4 红线不破）；多用户
  的对外暴露、TLS、限流全交**反向代理 / VPN / Tailscale**（运维层），本相位代码面不碰网络（决策P4.9-7）。
- **写共享 KB 的并发协调**（跨进程文件锁 / broker-worker）→ **不做**：本相位**全只读**，无写即无此问题。要让
  用户也能写，是另一条线（单写者拓扑），不在 P4.9。

## 2. 端点契约（`guanlan/web/app.py`，`reader=True` 下的增量）

| 方法 路径 | 默认（P4） | `reader=True` | 理由 |
|------|------|------|------|
| `POST /api/raw`·`/api/upload`·`/api/ingest`·`/api/heal`·`/api/backfill` | 注册（写） | **不注册 → 404** | 物理只读：写端点不存在（决策P4.9-2） |
| `DELETE /api/workspace/file`·`/api/workspace/dir` | 注册（写） | **不注册 → 404** | 同上 |
| `POST /api/chat/{id}/undo` | 注册（**写者**：持 `write_lock`、回放写日志改 `wiki/`+`SCHEMA.md`，`app.py:1364`） | **不注册 → 404** | 撤销是写能力。reader 无可写 turn → 本无 undo token，但保留写端点与「写端点不存在」的安全边界矛盾、给未来持久化/迁移/bug 留绕口（决策P4.9-17） |
| `GET /graph`（**重建**写 `graph/`） | 注册（**写者**，过 `write_lock`） | **不注册 → 404** | 它调 `build_and_write_graph` 写 `graph/graph.{json,html}`，是宿主写者（`app.py:1067`，串行于 ingest/heal）；reader 只看**预生成**图、不触发重建（决策P4.9-2，订正初稿误归只读） |

| `POST /api/chat`、`POST /api/chat/{id}/stop` | 问答/停止 | **保留** | 只读问答主路 + 停在飞 turn（`stop` 零写、幂等）；`undo` 已移裁剪组（决策P4.9-17） |
| `GET /api/chat/{conversation_id}/info` | 按 id 取会话姿态 | **保留** | **GET**（非 POST，`app.py:1315`）；reader 的 `?c=` 按-id 恢复探针（决策P4.9-12） |
| `GET /api/workspace`·`/api/workspace/file`·`/api/workspace/raw` | 浏览/预览 scratch（`app.py:985`） | **保留（共享可读）** | workspace scratch（`uploads/`+`parsed/`：暂存件、parsed markdown、图像缩略图）**定义为对所有 reader 用户共享可读**（决策P4.9-16）；写端点（`POST /api/upload`、`DELETE workspace/*`、promote）已裁 |
| `GET /api/pages`·`/api/page`·`/api/raw`·`/api/report/{name}`·`/graph/{filename}`·`/api/heal/preview` | 只读 | **保留** | 浏览/报告/**服务预生成图**/零-LLM 预览，本就只读 |

> **端点 vs UI**：上表是**端点**层（安全边界 = 路由是否注册）。`check`/`health`/`lint` 报告端点
> `GET /api/report/{name}` 与服务预生成图的 `GET /graph/{filename}` **只读、保留端点**，但其**顶栏按钮**在
> reader 下**隐藏**（维护/诊断对只读消费者无用，决策P4.9-10）——隐藏是 UX、非安全边界。
>
> `GET /graph`（**重建**，`app.py:1067`）是写者、reader 下不注册；`GET /graph/{filename}`（服务静态产物，
> `app.py:1100`）只读、保留。`GET /api/heal/preview`（`app.py:1129`，零-LLM worklist）只读保留，其配对写端点
> `POST /api/heal` 被裁；同理 `GET /api/raw`（读源）保留、`POST /api/raw`（投喂/晋级写）被裁。
>
> **404 vs 405（实现细节）**：被裁的写方法若其**路径**仍有一个保留的只读方法（`POST /api/raw` 与
> `GET /api/raw` 共路径、`DELETE /api/workspace/file` 与 `GET /api/workspace/file` 共路径），FastAPI 对
> 该路径的写方法返回 **405 Method Not Allowed**（路径在、方法 handler 未注册）而非 404；独占路径的写
> 端点（`/api/ingest`·`/api/heal`·`/api/backfill`·`/api/workspace/dir`·`GET /graph`·`/api/upload`·`/undo`）
> 返回 **404**。两者都=该写方法不可达、写不了 KB，安全属性一致（决策P4.9-2）。`GET /api/conversations`
> 独占路径、不注册 → 404。

## 3. 隔离模型：能力 UUID（`?c=`）— 本设计的核心　**\[决策P4.9-1]**

- **令牌**：`conversation_id` 是 `uuid.uuid4()`（128-bit，约 122 bit 熵）。前端在 SSE `start` 帧拿到
  （`app.js:1960`），唯一改写入口 `setConversation` 同步进 URL `?c=`（`app.js:1167`），刷新经 URL 恢复
  （`app.js:1160`，reader 下走按-id 端点恢复——决策P4.9-12）。**这就是「每用户单 Session」的载体** ——
  浏览器持一个 id、各请求带它、后端 `ConversationStore`
  按 id 命中内存会话（`chat.py:710`）。
- **隔离**：会话表是**进程内** dict（`chat.py`），按 `conversation_id` 寻址。只要 **`GET /api/conversations`
  枚举端点被关**（决策P4.9-3），用户**无法得知**他人的 id → 够不到他人会话。reader 保留的按-id 端点
  `/messages`、`/info`、`stop`、`DELETE`（**`undo` 在 reader 下已裁**，决策P4.9-17）全是 `{conversation_id}` 路径
  参数，**持 id 才能调**，天然不跨用户。这些端点对未知 id 返回 `404`、已知返回 200/204——**确实区分存在性**；
  隔离**不靠响应码统一**，而靠 122-bit UUID 不可枚举（不可猜则该差异无 oracle 价值）。
- **威胁模型（诚实边界）**：这是「**靠不可枚举的 id 隔离**」，**不是鉴权**。谁拿到那个 `?c=` URL（转发链接、
  浏览器历史、反代日志、Referer 头）就**持 id = 可读该会话历史、并能控制/删除它**（`stop`/
  `DELETE /api/conversations/{id}` 按 id 寻址、不校验持有者，§2；**`undo` 在 reader 下已裁、不在控制面**）——即能读
  会话、打断其进行中的回答、或删掉它。
  但**不影响 KB**（内容对所有用户共享只读、写已被物理裁掉）。在 P4.9 约束下暴露面 = **该会话的历史 + 控制权**，
  故风险被框得很小，对「无用户管理的只读多用户」是**合适取舍**。要连会话历史/控制权也保密 → 上反代身份层
  （§1 不纳入），非本相位。
- **workspace scratch = 共享可读（显式纳入暴露面）**（决策P4.9-16）：reader 保留 `GET /api/workspace*`，故
  `workspace/uploads|parsed` 的暂存件、parsed markdown、图像缩略图**对所有 reader 用户可读**（不分会话、无 id
  门槛——这是 wiki 之外的第二片共享可读面）。reader 下 `POST /api/upload` 已裁，scratch 只可能含 **writer/admin
  侧**预置的内容；把任何不宜公开的素材放进 `workspace/` 前要知道它对全体 reader 可见。要隐藏 → 改为 reader 下
  **不注册** `/api/workspace*`（本相位按你的裁决取「共享可读」，未采此路）。

## 4. `--reader` 部署模式与会话回收（`guanlan/cli.py` + `guanlan/web/{server.py,app.py,chat.py}`）

- **开关与 CLI 接线**（决策P4.9-15）：`--reader` 的 argparse 旗标加在 **`guanlan/cli.py` 的 web 子解析器**
  （现 `cli.py:243` 一带，**非** `server.py`——Web 参数全在 cli.py 解析），经 `_cmd_web`（`cli.py:107`）
  `serve(..., reader=args.reader)` → `create_app(..., reader=reader)`。`create_app` 内**按 `reader` 分支注册路由**
  （写路由 + `POST /api/chat/{id}/undo` + `GET /graph` 重建 + `GET /api/conversations` 包在 `if not reader:` 内；
  `GET /api/workspace*` **保留**——共享可读，决策P4.9-16）。零业务智能：纯路由裁剪 + 姿态钳制，不改任何 core。
- **`agent_log` 旗标改三态**（决策P4.9-15）：现仅有 `--no-agent-log`、`agent_log=not args.no_agent_log`
  （`cli.py:107/254`），**默认恒开** —— 照搬则 `--reader` 仍开日志、破「默认 KB 零字节写入」；若在 server 里
  对 reader 强制关，又没有正向旗标让用户显式 opt-in。故把日志旗标改成 `argparse.BooleanOptionalAction`
  （`--agent-log` / `--no-agent-log`，`default=None`=未指定），在 `_cmd_web` 里按 reader 解析：
  **非 reader** 未指定→开（沿用旧默认）；**reader** 未指定→关；显式 `--agent-log`/`--no-agent-log` 任意模式都
  覆盖默认。这样「非 reader 默认开 / reader 默认关 / `--reader --agent-log` 显式开」三态都可表达。
- **姿态钳制**：`reader=True` 时 app 级 `mode` 恒 `"read-only"`；`--reader --mode workspace-write` 同给 →
  **启动即 `EXIT_USAGE`**（互斥，**不**静默忽略——与 §7 测试一致）；运行时 `/api/chat/{id}/mode` 切
  `workspace-write` 一律拒。
- **KB 零字节写入钉死**（决策P4.9-13）：`--reader` 把现有启动写入面默认关死，否则「只读」名不副实——
  ① `create_app(reader=True)` **内部强制** `session_persist=False`（否则会话落 `<kb>/.agentao/sessions/`，
  `create_app` 默认 `True`，`app.py:812`）——钳制在 `create_app`，嵌入/测试直建也零写（决策P4.9-2）；
  ② `agent_log` **默认 `False`**（否则 chat 会话日志落 `<kb>/agentao.log`，`server.py:91`），但**可经 `--agent-log`
  显式打开、仅供调试**——非强制关。
  ①② + 写端点裁剪（决策P4.9-2）后，默认 reader 对 KB **零字节写入**。
- **写权限早失败（仅 `--agent-log`）**（决策P4.9-13）：注意 `require_kb_root`（`paths.py:23`）**只查必需文件/目录
  是否存在、不试写、不查权限** —— 故它**给不了**「只读挂载 → `EXIT_USAGE`」的保证（上轮 `writable=agent_log` 的
  机制是错的：`writable` 只控**哪些文件须存在**、与写权限无关）。reader 沿用既有 `require_kb_root(writable=True)`
  对**完整 Web KB 形状**——`_WRITABLE_REQUIRED` = `raw/` + `wiki/` + `AGENTAO.md` + `SCHEMA.md`（`paths.py:14`，
  **含 `raw/`**）——做**存在性**把关（reader 服务的是一个完整 KB，要求齐全合理；不复用则须另立存在集，徒增分支）。
  真正的写权限守卫是**显式**的，**主方案 = 打开目标日志文件并转译异常**：`--agent-log` 下实际 `open(<kb>/agentao.log, "a")`
  探一次、把 `OSError` 转成 `GuanlanError(EXIT_USAGE)`，**在启动处早失败**而非启动后写日志时崩。这能覆盖目录可写但
  **既有 `agentao.log` 不可写 / ACL / 挂载语义变化**等 `os.access(kb, os.W_OK)` 测不到的情形；`os.access` **至多作
  友好提示、不作唯一判据**。**关键**：探针**不能复用 `configure_agent_log` 的返回**——它命中 `_agent_log_paths`
  幂等缓存即早返回、**不重开目标文件**（`chat.py:233`），同进程重复起同一 KB / 测试中改权限后都证明不了当前可写。
  故探针要么是**独立**的 `open(..., "a")` 步骤（在 `configure_agent_log` 之外、之前），要么给 `configure_agent_log`
  加 `force_probe` 绕过缓存。默认 reader 不开日志、对 KB 零写，无需此守卫。
- **idle 回收**（决策P4.9-6）：`ConversationStore` 给每个 `Conversation` 记 `last_active`（每轮问答/控制刷新），
  在「新建会话」与「按 id 命中」的**锁内**惰性扫一遍，淘汰 `now - last_active > IDLE_TTL`（如 30 min）**且无在飞
  turn**的会话（连带其 `CancellationToken`/嵌入 agent）。**必须跳过有在飞 turn 的会话**——否则会把正在流式回答
  的会话清掉。时间源用 **`time.monotonic()`**（idle 时长应免受墙钟跳变影响），并**可注入**以便测试决定性
  （`chat.py` 现无时间源，此为新增）。
  - **容量边界（诚实）**：idle 回收只在有 idle slack 时解围。现 `new`/restore 满上限 `max_conversations`（默认
    100，`chat.py:71/752`）是**直接拒**（抛「已达上限」）；若达上限的会话都在 TTL 内活跃，无可回收 → 新用户照样被拒。
    故 `MAX_CONVERSATIONS` **改为可配**（`--max-conversations`，默认 100；决策P4.9-18）——真要多用户调高。「对并发
    活跃用户够用」仅在**活跃用户 ≤ 上限**时成立。
  - **UX 悬崖（诚实）**：`session_persist=False`（决策P4.9-5）+ idle 淘汰 → 用户离开 >`IDLE_TTL` 回来，`?c=` 已
    404、历史**永久丢失**（无盘可恢）。这是「每用户单 Session」耐久性的已知折中，reader 部署文案应提示「久置会
    新开会话」；要耐久则上 P4.2 落盘（本相位不做）。
- **不变量保持**：仍 `workers=1`（决策P4-2）、仍仅 `127.0.0.1`（决策P4-4）、读全走嵌入只读 Agentao（内存态、
  **不抢进程级 `write_lock`**）→ 多用户并发问答**不被写门禁串行**。读写分线（DESIGN §5.2）在此天然成立。
  （`ConversationStore._lock`（`chat.py:736`）仍在——`new`/restore/idle-扫短暂持它串行，是会话表自身的锁、
  非写门禁；O(n ≤ `max_conversations`) 持锁可忽略。故是「不抢 `write_lock`」而非「无锁」。）

## 6. 决策记录

- **\[决策P4.9-1] 隔离靠能力 UUID（`?c=`），非鉴权、无 owner 表。** 持 `conversation_id` = **可读该会话历史
  并控制/删除它**（`stop`/`DELETE /api/conversations/{id}` 按 id 寻址、不校验持有者，§2/§3；**`undo` 在 reader 下
  已裁，不在控制面**，决策P4.9-17）；暴露面 = **该会话的历史 + 控制权**，但**不影响 KB**（共享只读、写已裁）。
  对无用户管理的只读多用户是合适取舍。要真鉴权
  → 反代注入身份（不在本相位）。
- **\[决策P4.9-2] `--reader` 部署模式：`create_app(reader=True)` 不注册全部写路由（`raw`/`upload`/`ingest`/
  `heal`/`backfill`/`workspace 删`/`GET /graph 重建`/`chat undo`）。** 物理只读 —— 写端点不存在（404），强于
  运行时拒绝；不靠「隐藏按钮」。（`undo` 见决策P4.9-17、`/graph` 重建见决策P4.9-11。）
  - **钳制落点在 `create_app` 自身、非仅 `serve`/CLI**：`create_app` 现签名 `session_persist=True`、`mode="read-only"`
    默认（`app.py:812`），**无 `reader` 形参**。本相位加 `reader: bool=False`，且 **`reader=True` 时 `create_app`
    内部强制 `session_persist=False` + `mode="read-only"`（覆盖入参）**。这样**任何 caller**——CLI、嵌入、测试直
    `create_app(reader=True)`——都拿到「零写 + 只读姿态」，不会出现「测试/嵌入只关了路由却仍落 `.agentao/sessions/`」
    的漏口。`serve(--reader)` 只是把布尔透传下来，不是唯一钳制点。
- **\[决策P4.9-3] reader 模式关 `GET /api/conversations`（枚举）、保留 `/{id}/messages`·`DELETE /{id}`
  （能力寻址）。** 不可枚举 = 不可发现他人 id = 隔离成立；按 id 的端点持 id 才能用，不泄露。
- **\[决策P4.9-4] reader 强制 `mode=read-only`，拒切 `workspace-write`。** 否则可写工作会话（P4.5）能写
  `workspace/`，破「全只读」。
- **\[决策P4.9-5] 每用户单 Session 由前端 `?c=` 表达；sessions 内存态、不落盘（不启用 P4.2 持久化）。** 重启
  即丢，可接受；省掉 `.agentao/sessions/` 共享目录的 per-user 隔离与 10-file 目录级轮转串味问题。
- **\[决策P4.9-6] `ConversationStore` 加 idle TTL 回收，时间源 `time.monotonic()`（可注入）；**仅 reader 启用****。** 缓解并发用户顶满
  上限 `max_conversations`（默认 100）；惰性锁内扫淘汰，**跳过有在飞 turn 的会话**。用 `monotonic` 是为免受墙钟跳变、可注入
  是为测试决定性（`chat.py` 现无时间源，新增）。**仅 reader 启用（`create_app` 传 `idle_ttl=IDLE_TTL_SECONDS if reader else None`，评审 codex P2）**：reader=多用户共享、需缓解上限压力；非 reader=单用户无压力，且 workspace-write 会话被逐会丢 memory-only 的 undo 日志/姿态（用户拿着的 undo token 随即 404）——故非 reader 关回收，不动既有 P4/P4.5 行为。**按-id `get()` 刷新 `last_active`**（锁内、与 reclaim 串行）堵「端点取出 conv → 起 turn 前被并发 reclaim 误逐、turn 撞 closed agent」竞态；逐出会话 close() 在**锁外**做（与 `delete()` 同约定，慢 MCP 拆连不占 store 锁）。**边界**：① 仅在有 idle slack 时解围——满 `max_conversations` 全活跃仍直接拒
  （`chat.py:752`），故 `MAX_CONVERSATIONS` **改为可配**（`--max-conversations`，决策P4.9-18）；② `session_persist=False` 下淘汰=该用户会话**永久
  丢失**（UX 悬崖，文案提示「久置会新开会话」）。不叫「LRU」——是纯 idle-TTL，不在未满时主动逐出。

  问答全只读、单事件循环 async 即可，**不抢 `write_lock`**（会话表的 `_lock` 仍在、短暂串行 `new`/restore/idle-扫，

  现状：Web 参数解析全在 `cli.py` 的 web 子解析器（`cli.py:243`），日志只有 `--no-agent-log`、
  `agent_log=not args.no_agent_log`（`cli.py:107/254`）**默认恒开**。直接挂 `--reader` 会让默认仍开日志、破
  「默认 KB 零字节写入」；在 server 里对 reader 硬关又没正向旗标可 opt-in。故 `--reader` 加在 `cli.py` web 子
  解析器，日志旗标换 `argparse.BooleanOptionalAction`（`--agent-log`/`--no-agent-log`、`default=None`），
  `_cmd_web` 按 reader 解析三态后 `serve(reader=…, agent_log=…)`：**非 reader 未指定→开 / reader 未指定→关 /
  显式旗标任意模式覆盖**。`serve`/`create_app` 只收已解析布尔，不自己读 argv。
- **\[决策P4.9-16] workspace scratch 定义为「共享可读」，reader 保留 `GET /api/workspace*`。** reader 保留
  `GET /api/workspace`·`/api/workspace/file`·`/api/workspace/raw`（`app.py:985`）—— `workspace/uploads|parsed`
  的暂存件 / parsed markdown / 图像缩略图**对所有 reader 用户可读**，是 wiki 之外的**第二片共享可读面**，**显式
  写进威胁模型**（§3）。reader 下写端点（`POST /api/upload`、`DELETE workspace/*`、promote）已裁，scratch 只可能
  含 writer/admin 预置内容。**取舍**：评审给两路——(a) reader 不注册 `/api/workspace*`、(b) 定义为共享可读 + 写进
  威胁模型与测试；**本相位取 (b)**。代价：放进 `workspace/` 的素材对全体 reader 可见，部署须知。
- **\[决策P4.9-17] reader 不注册 `POST /api/chat/{id}/undo`（写者）。** `undo`（`app.py:1364`）持 `write_lock`、
  回放写日志改 `wiki/`+`SCHEMA.md`，**本质是写端点**。reader 无可写 turn → 本无 undo token 可用，但保留一个写
  能力端点与「写端点不存在 = 安全边界」自相矛盾、且给未来持久化/迁移/bug 留绕口。故 reader 下**不注册**（→404），
  与其它写路由同包进 `if not reader:`（决策P4.9-2 的写路由集**增列 `/undo`**）。
- **\[决策P4.9-18] `MAX_CONVERSATIONS` 改为可配（纳入 P4.9 范围，非"宜可配"空话）。** 现 `MAX_CONVERSATIONS=100`
  是模块常量、`ConversationStore` 直读（`chat.py:71/752/863`）；多用户下 100 活跃会话顶满即拒新用户。本相位把它
  做成参数：`ConversationStore(max_conversations=…)` 形参（默认 `100` 不回归）← `create_app(max_conversations=100)`
  ← `serve(...)` ← CLI `--max-conversations`（`cli.py` web 子解析器，`type=int default=100`）。env 兜底可选、不强求。
  非 reader 专属——所有 Web 部署通用。
  - **正整数校验（必做，钳制在 `create_app`）**：`type=int` 不拦 `0`/`-1`——传 `0` 会得到「服务能起、新会话全拒」
    的坏配置。**校验落点与 `reader` 不变量同口径——在 `create_app` 自身**（决策P4.9-2）：`create_app` 收到
    `max_conversations < 1` 即 `raise GuanlanError(EXIT_USAGE)`，**任何 caller**（CLI/`serve`/嵌入/测试直建
    `create_app(root, max_conversations=0)`）都早失败、堵掉「绕过 CLI 得到坏配置 app」的直建漏口。CLI `_cmd_web`/
    `serve` 可另作一道**友好早提示**（更早、错误信息更具体），但**权威校验在 `create_app`**、不下沉到
    `ConversationStore`（那时 app 已建、太晚）。
  - 若你判定**不**做可配，则把 §4 改成显式「out-of-scope / 100 是部署硬限」——但既然 P4.9 主打多用户，此处取**做**。
