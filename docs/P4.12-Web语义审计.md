## 1. 目标与范围

**P4.12 = 把 `guanlan audit`（source-drift 语义审计）放进浏览器**：浏览器已能投喂源（P4.1）、
`convert --overwrite` 之外用 Web 投喂覆盖一个 `raw/`、能 ingest / heal / backfill，却**唯独缺 audit**——
覆盖了源之后，要查「哪些 wiki 页因引用的 raw 已变而论断可能过期」仍须回命令行跑 `guanlan audit`。这正是
P4.12 要补的**闭环**：覆盖源（P4.6 投喂 / P5.2 `convert --overwrite`）→ **审计漂移** → LLM 复核 → 刷新指纹，
全留浏览器内。用一对端点补齐：

- **`GET /api/audit/preview?limit=`**：零-LLM 算漂移源**组** worklist（== `audit --dry-run`），返回
  「本批将复核哪些漂移源组（slug · source 摘要页 · raw 路径 · 组内被牵连的全部页）、因 `--limit` 推迟了哪些」。
  **纯读、不入队、不触 Agentao**。
- **`POST /api/audit {limit?, model?}`**：把 audit 作为**新写作业 kind** 入 P4 既有单写者 FIFO `JobQueue`
  （`enqueue`、对标 ingest/heal/backfill），即时返回 `job_id`，前端**轮询** `/api/jobs/{id}`。作业内调 audit
  的结构化 core `run_audit_result`（走 P2 写门禁 + P2.1 源不回退闸 + 有界自愈），完成后把 `AuditResult`
  经 `Job.result` 回传。

audit 在 Web 端**仍是写操作**：它经 `run_guarded_write_result(page_guard=True)` 走 P2 子进程 + `raw/`
快照门禁（与 ingest/heal/backfill 同一条写路径），故归**写线**——单 worker FIFO 串行、轮询拿结果。Web 层
**零业务智能**：漂移粗筛、门禁、回执的结论全来自被复用的 `audit.py` core，宿主只「收 HTTP → 调既有函数 →
序列化/转发」。

### 纳入 P4.12

- `GET /api/audit/preview?limit=` → `{groups, postponed}`（零 LLM，复用 `audit._compute` + `_build_groups`，`to_thread` 卸载；== `audit --dry-run --json` 体）。
- `POST /api/audit {limit?, model?}` → `{job_id}`（入单写者队列 `enqueue`，与 ingest/heal/backfill 同 worker 串行）。
- `GET /api/jobs/{id}` 对 audit 作业回 `result == audit_result_dict(run.result, run.postponed)`（四字段机器回执 `{refreshed, postponed, receipts, exit_code}`），ingest/投喂/backfill 仍为 `null`、heal 仍为 `heal_result_dict`。agent 散文一律走 `job.output`（决策P4.3-1，沿用）。
- **`on_job_done` 升版集加 `"audit"`**：audit 刷 source 页 `raw_digest`（provenance 写）+ LLM 可能重写正文 → 改 `wiki/` → 必须 bump 代际，使可写会话缓存的 check 基线失效（与 ingest/heal/backfill 同，决策P4.5-4）。
- **前端**：顶栏 `audit` 动作 → 浮层先拉 preview（可调 `--limit`）→「审计」按钮 → `POST /api/audit` → 轮询渲染回执（逐组 refreshed ✓ / incomplete · reviewed 成员 / 推迟 / 退出码徽标；agent 散文取自 `job.output`）；i18n `overlay.audit`。

### 不纳入 P4.12（按既有收窄推后）

- **slug 子集勾选过滤器**（不同于 P4.3 修订后的 `targets`）→ **不做**（决策P4.12-3）：audit 唯一旋钮是
  `--limit`（限漂移源**组**数，决策P3.7-8），`run_audit_result` 无 `slugs` 入参；加子集过滤要改 core，违背
  audit 一期收窄。整批按 `--limit` 截断即够，调 `--limit` 即控批量。子集过滤降后续项。
- **矛盾复检**（P3.7.1 高价值兄弟）→ core 能力、与 Web 无关，不在本相位（audit 一期只跑通 source-drift 一类，决策P3.7-2）。
- **audit 作业 token 级流式（SSE）** → 不做：与写作业 ingest/heal/backfill 同口径**只轮询**（决策P4-6）。散文与 `↻` 自愈进度由 worker 捕获进 `job.output`，回执经 `job.result`。
- **自动 / nightly audit** → 仍**用户显式按需触发**（点按钮才跑），不内联 ingest、不定时。「nightly 而非每次 ingest 跑」是 P3.7 给的**运营建议**（缓解指纹漂移≠语义过期的成本），非宿主该实现的定时器。

## 2. 端点契约（`guanlan/web/app.py`）

| 方法 路径 | 入参 | 复用 | 返回 | 错误 |
|------|------|------|------|------|
| `GET /api/audit/preview` | `limit: int=DEFAULT_LIMIT (ge=1)`（query） | `audit._compute` + `_build_groups` | `{groups:[{slug,source_page,raw_path,members}], postponed:[…]}`（== `audit --dry-run --json` 体；零 LLM、`to_thread` 卸载） | `422` 参数越界（FastAPI `Query(ge=1)`，对齐 CLI `positive_int`） |
| `POST /api/audit` | `AuditBody {limit:int=DEFAULT_LIMIT(ge=1), model:str\|None=None}` | `enqueue` + `audit.run_audit_result` | `{job_id}`（**即时返回**，前端轮询；唯一写入口之一，与 ingest/heal/backfill 并列） | `422` 体非法；`423` 可写 turn 活跃（决策P4.12-6） |
| `GET /api/jobs/{id}` | — | 作业表 | `{id,kind,state,exit_code,output,result}`——audit 作业 `result == audit_result_dict(run.result, run.postponed)`（四字段机器回执），heal 仍 `heal_result_dict`、其余 `null`；agent 散文在 `output` | `404` 未知 id（沿用 P4） |

- **默认值零漂移**：`limit` 默认**直接 import `audit.DEFAULT_LIMIT`**（与 CLI 同源、不另设 Web 专属值——audit 的批量天然以「漂移源组」计、组数通常远小于 heal 的缺页数，无需像 P4.3 那样压小默认）。
- **preview 是 read-only 读端点**：只算漂移源组（`_scan_drifted_sources` 读 `wiki/sources/*.md` 比指纹 + `iter_pages` 沿 `sources:` slug 图传播），**不取 `raw/` 快照、不入队、零 LLM**，与 `/api/heal/preview`·`/api/report/*` 同类——经 `anyio.to_thread.run_sync` 卸离事件循环（决策P4-2）。
- **`POST /api/audit` 是写端点**：入 §4 单写者 `JobQueue`、**轮询**（决策P4-5/P4-6）。退出码透传进 `job.exit_code`（写路径四值 `0/4/3/5`，门禁口径），前端据此显示徽标；**P4.12 不新增退出码**（同 P4）。
- **`result` 对既有作业向后兼容**：`/api/jobs/{id}` 的 `result` 键已存在（P4.3 落地），audit 只是**新增一个 `elif isinstance(job.result, AuditRun)` 分支**用 `audit_result_dict` 序列化；ingest/投喂/backfill 仍取 `null`、heal 仍取 `heal_result_dict`，**既有轮询逻辑零回归**。

## 3. 承载结构化结果：**复用既有设施，零 `audit.py` / 零 `jobs.py` 改动**　**\[决策P4.12-1]**

P4.3 当初要给 heal **抽**不打印 core（`run_heal_result`）、**把** `_result_dict` 改名公开、**给** `Job` 加
`result` 字段 + worker 鸭子分流——那是基础设施的**首次**铺设。**到 P4.12 这些已全部就位**，故本相位是 P4 写作业
里**最薄**的一片：

| 设施 | 状态 | P4.12 是否需改 |
|---|---|---|
| 不打印结构化 core `run_audit_result(*, root, limit, model, runner) -> AuditRun` | **已存在**（`audit.py`，P3.7 即仿 `run_heal_result` 抽好；`AuditRun` 带 `.result/.postponed/.had_batch/.final_text/.exit_code`） | **否** |
| wire 序列化器 `audit_result_dict(result, postponed) -> dict` | **已公开**（`audit.py` `__all__`，四字段 `{refreshed, postponed, receipts, exit_code}`，CLI `--json` 已用） | **否**（heal 当年需 `_result_dict`→公开改名，audit 不需要） |
| `Job.result` 字段 + worker 鸭子分流（`int`→exit_code / 否则→存 `result` 取 `.exit_code`，域无关不 import 类型） | **已存在**（`jobs.py`，P4.3 落地） | **否**——`AuditRun` 有 `.exit_code` property，直接走既有「非 int」分支 |
| `/api/jobs/{id}` 的 `result` 序列化 | heal 专支已在 | **加一个 `elif isinstance(job.result, AuditRun)` 分支**（唯一一处 jobs 相关改动，在 app.py） |

故 **P4.12 的后端改动全部集中在 `app.py`**：`AuditBody` + `GET /api/audit/preview` + `POST /api/audit` 端点
（含 `_reject_if_writable_active`/`@_writer_only`）+ `/api/jobs/{id}` 扩一个 `elif` 分支 + `on_job_done`
升版集加 `"audit"`。**不碰 `audit.py`、不碰 `jobs.py`**——这是 P4.12 比 P4.3 更薄的根因，也复证了 P3.7 当初
仿 heal 抽 core 的前瞻性（`AuditResult` docstring 即写「CLI `--json` / **未来 Web** 共用」）。

audit 作业 thunk（与 heal 端点逐行同构）：

```python
@_writer_only(app.post("/api/audit"))   # reader 下不注册（审计是 gated 写，决策P4.9-2）
async def audit(body: AuditBody) -> dict:
    _reject_if_writable_active()         # 层③：可写 turn 活跃 → 423（决策P4.5-10 / P4.12-6）
    def _job() -> AuditRun:              # 返回结构化结果（非 int）→ worker 走「非 int」分支存 job.result
        run = run_audit_result(root=root, limit=body.limit,
                               model=body.model or model, runner=runner)
        if run.final_text:               # agent 散文 → worker 的 redirect 捕获进 job.output（同 heal）
            print(run.final_text)
        return run                       # worker 取 run.exit_code、整体存 job.result（端点经 audit_result_dict 序列化）
    return {"job_id": jobs.enqueue("audit", _job)}

# /api/jobs/{id} 既有 result 序列化扩一个分支：
#   elif isinstance(job.result, AuditRun):
#       result = audit_result_dict(job.result.result, list(job.result.postponed))
```

- **职责分离不破（决策P4.3-1 沿用）**：`job.result`（`AuditRun`）→ 端点经 `audit_result_dict` 序列化为**机器回执**；`job.output` 承载**人读散文**（thunk 打印的 `final_text` + core 内 `run_guarded_write_result` 走 stderr 的 `↻` 自愈进度，被 worker 一并捕获）。两者各司其职，`result` 不掺展示文本。
- **`redirect_stdout` 红线不破**：`run_audit_result` core **本身不向 stdout 打印**（这正是 P3.7 抽 core 的目的）；散文由 thunk 在 worker redirect 块内显式 `print`，捕获仍只在单 worker 串行发生、无跨线程竞态（决策P4-5）。

## 4. 读写分线与单写者串行（复用 P4 §4.1/§4.2，零新写基础设施）

audit 在 P4 读写分线里归**写**（过写门禁、刷 `raw_digest`、LLM 改 `wiki/`），落点完全沿用 ingest/heal 的写路径：

| | preview（`GET /api/audit/preview`） | 审计（`POST /api/audit`） |
|--|--|--|
| 性质 | 零 LLM 读 | LLM 写（过 P2 门禁 + P2.1 源不回退闸） |
| 集成 | `_compute`+`_build_groups`，`to_thread` 卸载 | `enqueue` 入单写者 FIFO worker，`run_audit_result` |
| `raw/` 快照 | 不取（纯读） | 由 `run_guarded_write_result` 内部取（与 ingest/heal 同） |
| 并发 | 可并发（不写、无竞态） | 单 worker FIFO 串行（与 ingest/heal/backfill 同 worker） |
| 拿结果 | 同步返回 | 即时返回 `job_id` → 轮询 `/api/jobs/{id}` |

- **单写者假设不破**：audit 与 ingest/heal/backfill/投喂在**同一条 worker** 上 FIFO 串行——四类写作业永不并发，`raw/` 前后快照不被并发写互踩（决策P4-5）。
- **`enqueue`（异步轮询）非 `submit_and_wait`**：audit 是写、且自身慢（LLM 子进程，同 heal），故对标 ingest/heal 的「入队即返回 `job_id` + 轮询」（决策P4.12-2）。
- **走子进程 runner、无嵌入四坑（决策P4.12-5）**：audit 经 `run_guarded_write_result` 走 `agentao run` 子进程（与 ingest/heal 同），**非**只读嵌入会话。故 P4-8 的「`model=None` 盖掉 `.env` 模型触发 ValueError」嵌入坑**不适用**——audit 把 `model`（含 `None`）直接透传子进程 runner。`runner=app.state.runner` 供测试注入 fake（不打真实 LLM），同 ingest/heal。
- **`page_guard=True` 由 core 内显式开（决策P3.7-4）**：audit 的写门禁带 P2.1「源不回退」闸（`sources` 只增不减），Web 端**不放宽、不旁路**——`run_audit_result` 内部已写死，端点不传该参。
- **安全姿态沿用 P4 §8**：audit 经 LLM 复核可写 `wiki/`（重综合过期论断 + provenance 刷 `raw_digest`），**永不碰 `raw/`**——由 P2 快照门禁双重兜底（core 既有，Web 不放宽）。端点无 `path`/`name`/`slug` 入参（只 `limit`/`model`），故**无路径穿越面**。reader 下 `@_writer_only` 不注册端点（决策P4.9-2，写姿态一致）。

## 5. 前端（`guanlan/web/static/`）

- 顶栏 `actions` 新增 `<button id="audit-btn" data-i18n-title="...">` `audit`（紧邻 `heal`/`backfill`）。
- `audit-btn` → `openAuditPreview()`：`showOverlay("audit", …)`（`#overlay-title` 打 `data-i18n="overlay.audit"`，沿用既有 i18n 白名单自动重绘，决策P4.7-8）→ `GET /api/audit/preview` →

  - 否则列出本批漂移源组（每组 `slug` · `source_page` · `raw_path` · 组内 `members` 引用页清单，文件名 `textContent` 防注入）+ 推迟项（`postponed`）。**顶部一行**：左侧 **「审计 N 个源」** 按钮（N = 本批组数）→ `triggerAudit()`，右侧 `--limit` 微调输入 + 「预览」按钮（默认即常量）。

- **复用既有交互骨架（决策P4-3 克制）**：浮层 `#overlay`、`pollJob` 轮询、退出码徽标全沿用 heal/ingest 那套，不引新组件、不改两栏布局。
- **i18n（决策P4.7 沿用）**：`static/i18n.js` 的 `zh`/`en` 各加 `"overlay.audit": "审计"` / `"Audit"`（与 `overlay.ingest/heal/backfill` 并列），按钮 `title` 走 `data-i18n-title`；`tests/test_web_i18n.py` 的 zh/en key 平价校验自动覆盖。

## 6. 决策记录

- **\[决策P4.12-1] audit 的结构化结果复用既有设施回传，零 `audit.py` / 零 `jobs.py` 改动。** P3.7 实现时已仿 `run_heal_result` 抽好不打印 core `run_audit_result -> AuditRun`、并把序列化器 `audit_result_dict` 公开入 `__all__`（其 `AuditResult` docstring 已写「未来 Web 共用」）；`Job.result` + worker 鸭子分流（只碰 `.exit_code`、不 import 类型）在 P4.3 已落地。故 P4.12 后端改动**全集中在 `app.py`**：两个端点 + `/api/jobs/{id}` 扩一个 `elif isinstance(_, AuditRun)` 分支 + `on_job_done` 升版集加 `"audit"`。`result` 单一职责 = 机器回执（四字段），散文走 `job.output`（决策P4.3-1 沿用）。这是 P4.12 比 P4.3（heal）更薄的根因。
- **\[决策P4.12-2] audit 入单写者 FIFO `JobQueue`、用 `enqueue`（异步轮询）非 `submit_and_wait`。** audit 是写、自身慢（LLM 子进程，同 heal），对标 ingest/heal/backfill 的「入队即返回 `job_id` + 轮询」。与四类写作业同 worker 串行 → 单写者假设不破、`raw/` 快照不被并发写互踩（决策P4-5）。`JobQueue` 本就是通用 `enqueue(kind, thunk)`，audit 只是新 job kind，**零新写基础设施**。
- **\[决策P4.12-3] 一期不做 slug 子集勾选过滤器（不同于 P4.3 修订后的 `targets`）。** audit 唯一旋钮是 `--limit`（限漂移源**组**数，决策P3.7-8），`run_audit_result` 无 `slugs` 入参；加子集过滤需改 core，违背 audit 一期收窄。整批按 `--limit` 截断即够，preview 调 `--limit` 即可控批量。子集过滤降后续项（与 P4.3 `targets` 的取舍各按自身命令形态定，不强行对齐）。
- **\[决策P4.12-4] preview 是零-LLM 只读端点（== `audit --dry-run`），`to_thread`、不入队、不取 `raw/` 快照。** 复用 `_compute`+`_build_groups`，返回 `{groups, postponed}`（与 dry-run `--json` 体同形），绝不写、绝不触 Agentao。POST 才是写、才入队、才过门禁。读写两端口分明，呼应 P4 §4.1 读写分线，与 `/api/heal/preview` 同类。
- **\[决策P4.12-5] audit 走子进程 runner，复用 P2 门禁 + P2.1 源不回退闸；嵌入四坑（决策P4-8）不适用。** audit 经 `run_guarded_write_result(page_guard=True)` → `agentao run` 子进程（与 ingest/heal 同），非只读嵌入会话；故 `model`（含 `None`）直接透传子进程、无 `.env` 覆盖坑，`runner` 可注入 fake 供测试。门禁/自愈/`page_guard`/写集审计全继承 core，Web 端零新增。
- **\[决策P4.12-6] reader 下不注册 + 可写 turn 活跃 → 423 + `on_job_done` 加 `"audit"`，跟最新写作业 backfill 同口径。** audit 是 gated 写：`@_writer_only` 使 reader 挂载下端点根本不存在（决策P4.9-2）；`_reject_if_writable_active()` 防可写 turn 期并发写自死锁（决策P4.5-10）；audit 刷 `raw_digest`（provenance 写）+ LLM 可能重写正文 → 改 `wiki/` → 必须 bump 代际使可写会话 check 基线失效（决策P4.5-4）。
- **\[决策P4.12-7] 不新增退出码、不做 audit SSE、不内联自动化。** 沿用 P4 退出码面（进程 `EXIT_OK`/`EXIT_USAGE`，作业退出码只进 `job.exit_code`）；写作业只轮询、不流式（决策P4-6）；audit 仍**显式按需触发**（点按钮才跑），不入 ingest、不定时（P3.7 的「nightly 跑」是运营建议、非宿主定时器）。
