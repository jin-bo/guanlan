## 1. 目标与范围

**P4.1 = 给 Web 宿主补上唯一缺口：在浏览器里把一篇新素材存进 `raw/`，使「投喂 → ingest → 浏览 → 问答」整条环不再需要离开浏览器回到文件系统。**

现状（P4）：浏览器能 `从 raw/ 选一篇 ingest`，却不能**把素材放进 `raw/`**——那步必须在文件系统里手动完成。P4.1 用一个 `POST /api/raw`（粘贴/上传存盘）补齐它。它是**人投喂源**（普通文件写），**不是** gated 的 agent 写。

### 纳入 P4.1

- `POST /api/raw {name, content, overwrite?}` —— 把 `content` 存为 `<kb>/raw/<安全名>.md`，返回 `{saved, bytes}`。
- **经 §4.2 单写者队列串行**：杜绝与在飞 `ingest` 的 `raw/` 快照窗口竞态（§3，**本设计的核心**）；端点**同步等待**该写作业完成后返回（不轮询）。
- **文件名安全 + 默认不覆盖**：slug 化 + 强制 `.md` + 路径穿越校验 + 大小上限；目标已存在默认 `409`，须显式 `overwrite=true` 才覆盖。
- **前端**：顶栏 `投喂` 动作 → 浮层（`<textarea>` 正文 + 文件名输入）→ `POST /api/raw` → 成功后刷新 `/api/raw` 列表，并就地提供「立即 ingest」一键续跑（复用既有 ingest 流）。

### 不纳入 P4.1（仍按 P4 §10 / 路线图推后）

- **多格式上传（docx/pdf/web clip）** → P5；P4.1 只收 `.md` 文本（与 CLI `ingest` 同口径）。
- **二进制 / 多文件批量上传** → 推后；v1 一次一篇文本。
- **Web 端编辑/删除已存在的 `raw/` 源** → 推后（删源会让已 ingest 的 wiki 失去出处，需级联策略）。v1 只「新增」与「显式覆盖」。
- **把投喂直接接到 ingest 一步到位（存盘即自动 ingest）** → 不做：存与消化是两步、两种失败模式（写盘 vs 门禁），分开更可控；前端用「存盘成功 → 一键 ingest」串起体验即可。

## 2. 端点契约（`guanlan/web/app.py`）

| 方法 路径 | 入参（pydantic `RawBody`） | 返回 | 错误 |
|------|------|------|------|
| `POST /api/raw` | `{name: str, content: str, overwrite: bool=false}` | `{saved: "raw/<name>.md", bytes: N}`（同步：作业完成后返回；**会等待队列前序写作业**，如在飞 ingest 完成后才轮到，见 §3.1） | `400` 用法（空名/空正文/sanitize 后非法/含 NUL 或控制字符/超 `MAX_RAW_BYTES`）· `409` 已存在且未 `overwrite`（或并发同名抢占）· `500` 写盘失败 |

- `name`：用户给的文件名或标题。服务端 **slug 化**为安全 basename（见 §4），强制 `.md` 后缀，解析后**必须**落在 `<kb>/raw/` 内（`resolve()` + `relative_to(raw)`，越界 `400`），与 `_safe_wiki_file` 同口径但归到 `raw/`（新增 `_safe_raw_target`）。
- `content`：素材正文，按 **UTF-8 原样**写盘（不做 markdown 渲染、不做 `[[wikilink]]` 重写——`raw/` 是未加工源）。超过 `MAX_RAW_BYTES`（默认 5 MiB）→ `400`。**文本准入**（决策P4.1-5）：含 NUL（`\x00`）或 `\t\n\r` 以外的 C0 控制字符 → `400`（任意字节的二进制更早被 JSON/UTF-8 传输层挡掉，根本到不了本端点）。
- `overwrite`：默认 `false`。目标已存在 → `409`（见决策P4.1-3）；`true` 时原子覆盖。
- **`/api/raw` 仍保留 `GET`（只列）**；P4.1 只新增 `POST`，不动既有读端点。
- **P4.1 不新增退出码 / 不动 CLI**：投喂是纯 Web 能力（CLI 把 `.md` 放进 `raw/` 用文件系统即可）。HTTP 状态码即对外契约。

## 3. 与 ingest 快照窗口的竞态——本设计的核心　**\[决策P4.1-2]**

P2 写门禁 `run_guarded_write`（`gate.py`）在 agent 调用**前后**各取一次 `snapshot_raw(root)`，任意条目增/删/改即判 `EXIT_RAW_MUTATED(4)`。若 `POST /api/raw` 把一篇新文件写进 `raw/` **正落在某个 ingest 作业的快照窗口里**，会同时造成两个错误：

1. **ingest 作业被冤判失败**：after 快照多出一个 `added` 条目 → 门禁 `raw_mutated` → 该 ingest 以退出码 4 失败，并在面板里**错误地指控 agent 改了 `raw/`**。
2. **投喂结果语义不明**：用户以为只是存了篇源，却让一个无关的 ingest 挂掉。

**解法：把 `POST /api/raw` 的写盘动作也送进 §4.2 的单写者 `JobQueue`。** 该队列本就是「唯一写者」的归口（决策P4-5：FIFO 串行、`raw/` 快照不被并发写互踩）。投喂作业与 ingest 作业在**同一条 worker 线程**上 FIFO 串行，于是投喂**永不**落在某个 ingest 的 before/after 之间——零新锁、零新并发原语，直接复用既有单写者假设。

> 这正是 P4 §10 把本项「推后」时点名的那处复杂度（"与 `raw/` 快照窗口的并发"）。P4.1 的回答是**不引锁、复用单写者队列**——与决策P4-5「`workers=1` + 单 worker 串行」一脉相承。

### 3.1 同步返回，而非轮询　**\[决策P4.1-2]**

ingest 用「入队 → 返回 `job_id` → 轮询 `/api/jobs/{id}`」，因为它慢（LLM 子进程）。投喂作业**自身极快**（一次文件写），单独为它再起一套轮询体验割裂。故 `POST /api/raw` **入队后在服务端等待该作业完成再返回结果**。

> ⚠️ **「快」指作业本身，不指响应延迟。** 投喂与 ingest 共用同一条 FIFO worker，故投喂会**排在队列前序写作业之后**——若此刻正有一个 ingest 在飞（LLM 子进程，可能数十秒），`POST /api/raw` 会一直 pending 到它跑完才轮到投喂落盘返回。这是有意取舍(决策P4.1-2)：换来「投喂永不落进 ingest 快照窗口」的零锁保证。**契约上必须写明会等待前序写作业**；前端 `存盘` 期间显示「保存中…」并禁用按钮,不可让用户以为卡死。（若日后排队等待成为体感问题,再退回 ingest 同款「入队即返回 job_id + 轮询」——但 v1 不做。）

```python
# jobs.py：Job 增一个完成信号；enqueue 不变，新增 submit_and_wait（封装「入队 → 等完成」，仍走同一 worker、同一 FIFO）
@dataclass
class Job:
    id: str; kind: str; state: JobState = "queued"
    exit_code: int | None = None
    output: str = ""
    done_event: threading.Event = field(default_factory=threading.Event)  # 新增：worker 收尾时 set

def _run(self):
    while True:
        job, fn = self._queue.get()
        ...
        finally:
            # 顺序要紧：先 output、再 state，最后 set()——等待者醒来即见完整 output/exit_code/state。
            job.output = buf.getvalue(); job.state = "done"
            job.done_event.set()          # 新增：唤醒同步等待者

def _register(self, kind, fn) -> Job:              # 新增：建表 + 入队的内部归口，返回 Job 本体（非 Optional）
    with self._lock:
        self._counter += 1
        job = Job(id=str(self._counter), kind=kind)
        self._jobs[job.id] = job
    self._queue.put((job, fn)); return job

def enqueue(self, kind, fn) -> str:                # 公开：异步作业（ingest）仍返回 job_id（薄封装 _register）
    return self._register(kind, fn).id

def submit_and_wait(self, kind, fn) -> Job:        # 公开：同步作业（投喂）——直接持 Job，无 get_job 的 Optional
    job = self._register(kind, fn)
    job.done_event.wait()                          # 阻塞当前线程；端点经 to_thread 调用，不堵事件循环
    return job

# app.py：投喂端点——校验（同步、立即 4xx）→ 入队写盘作业 → to_thread 等完成 → 按退出码分流
@app.post("/api/raw")
async def write_raw(body: RawBody):
    if not body.content.strip():                               # 空正文
        raise HTTPException(400, detail="正文不能为空。")
    payload = body.content.encode("utf-8")
    if len(payload) > MAX_RAW_BYTES:
        raise HTTPException(400, detail=f"正文超过 {MAX_RAW_BYTES} 字节上限。")
    if "\x00" in body.content or any(c < " " and c not in "\t\n\r" for c in body.content):
        raise HTTPException(400, detail="raw/ 只收文本素材；检测到 NUL/控制字符。")
    target = _safe_raw_target(root, body.name)                 # 400/越界即抛，无需进队列
    if target.exists() and not body.overwrite:
        raise HTTPException(409, detail=f"raw/{target.name} 已存在；改名或传 overwrite=true。")
    job = await anyio.to_thread.run_sync(                       # 入队 + 等完成，不堵事件循环
        jobs.submit_and_wait, "raw_write",
        lambda: _atomic_write_raw(target, body.content, body.overwrite),
    )
    if job.exit_code == EXIT_USAGE:                            # worker 内复检：并发同名抢占
        raise HTTPException(409, detail=job.output or "raw/ 同名文件已存在。")
    if job.exit_code != EXIT_OK:                               # 其余（落盘 IO 失败 → EXIT_AGENT_ERROR 等）
        raise HTTPException(500, detail=job.output or "写盘失败。")
    return {"saved": f"raw/{target.name}", "bytes": len(payload)}
```

- **校验在端点（快 4xx）、写盘在 worker（串行）**：空正文/大小/文本准入/路径/存在性的快速否决在端点同步给出；真正落盘进队列串行，杜绝与 ingest 快照窗口重叠。
- **退出码→HTTP 分流要对齐 §2**：`EXIT_USAGE`（worker 复检撞同名）→ `409`；其余非 `EXIT_OK`（写盘异常被 `_run` 归一为 `EXIT_AGENT_ERROR`）→ `500`。**不可一律塌缩成 `409`**——否则「磁盘满」会被误报成「文件已存在」。
- **worker 内复检存在性 + 原子写**：两个并发投喂同名新文件时，端点的 existence 预检可能都放过；故 `_atomic_write_raw` 在**串行的** worker turn 内再查一次 `overwrite` 语义并**原子落盘**（写临时文件 `os.replace` 同目录换名），把覆盖保护落在真正串行的临界区里。第二个同名作业返回 `EXIT_USAGE` → 端点转 `409`。
- **`redirect_stdout` 红线不破**：worker 仍只对一个作业串行捕获输出；投喂作业不打印业务文本，`output` 只承载失败原因，不与 ingest 抢 stdout。

## 4. 文件名 slug 与安全　**\[决策P4.1-4]**

`_safe_raw_target(root, name) -> Path`（新增于 `app.py`，与 `_safe_wiki_file` 并列）：

1. 取 `Path(name).name`（**剥掉任何目录成分**），杜绝 `../`、绝对路径、子目录穿越。
2. **规范化混淆字符（先于过滤跑）**：文件名/标题常由大模型生成，会夹带 ASCII 之外、肉眼难辨的「同形/排版」字符。直接交给步骤 3 的「其余→`-`」会把它们劣化成 `-` 串（如 `"深度学习"导论` → `-深度学习-导论`）。故先收敛：
   - **`unicodedata.normalize("NFKC", name)`**：折叠全角与兼容字符（`Ａ１！` → `A1!`、全角括号/分号等），一步处理大类。
   - 再过一张**显式混淆映射表**（NFKC 不覆盖的排版符号）：
     - 各类引号 → **删除**（不留 `-`，避免 `"X"` 变 `-X-`）：直/弯双引号 `" " " „ « »`、直/弯单引号 `' ' ' ‚ ‹ ›`、中文 `「」『』`、反引号 `` ` ``。
     - 各类破折号 / 连接号 / 波浪号 → **单个 `-`**：em/en dash `— –`、horizontal bar / figure dash `― ‒`、中文破折号 `——`、全/半角波浪 `～ ~`、下划线连写不动（`_` 已在保留集）。
     - 省略号 `…` → 删除；连续点 `。。` 不特殊处理（交步骤 3）。
     - 各类空白 → 普通空格：NBSP ` `、全角空格 `　`、零宽 `​`/`﻿`。
3. slug 化：保留 CJK、字母、数字、`-`、`_`、`.`；**空格转 `-`**（连同其余非保留字符一并替换为 `-`，故步骤 2 折出的普通空格在此收敛成 `-`，文件名内无空白）；折叠连续 `-`；去首尾 `-`/空白。空结果 → `400`。
4. 强制 `.md`（**判定顺序须钉死**：先剥目录（步骤 1）→ NFKC + 映射（步骤 2）→ slug（步骤 3）→ **本步基于已规范化的 basename 取 `suffix`**）：已是 `.md` 后缀则保留；**否则一律补 `.md`**——含小数点/版本号的标题（`GPT-4.5 笔记`、`Llama 3.1 速记`、`notes.v2`）视作普通标题，落为 `GPT-4.5-笔记.md`。**唯一例外**：规范化后 basename 的 `suffix.lower()` 命中**已知非-md 扩展名黑名单常量**（见下）→ 直接 `400`，提示「投喂只收 `.md`」（挡住多格式误投，与 P5 边界一致）。
   - **拒绝列表写成模块级常量,不用「等」**(测试与实现共一份真相):
     ```python
     _RAW_REJECT_EXTENSIONS = frozenset({".txt", ".pdf", ".docx", ".doc", ".html", ".htm", ".rtf", ".epub"})
     ```
     判定 `Path(basename).suffix.lower() in _RAW_REJECT_EXTENSIONS`。新增格式时改这一个常量。
   - **必须用规范化后的 basename 判 suffix**:否则全角点 `．`(U+FF0E)等会先漏过——`x．PDF` 若在 NFKC 前判,`Path(...).suffix == ""`(无 ASCII 点)放行,随后 NFKC 把 `．`→`.` 又叠补成 `x.PDF.md`(多格式漏网)。规范化在前即根除:`x．PDF` → NFKC → `x.PDF` → `suffix==".pdf"` 命中 → `400`。
   - **大小写不敏感**:`suffix.lower()` 比对(`X.PDF`/`foo.Docx` 同样 `400`);`.md` 本身也大小写不敏感(`X.MD` 视作已是 `.md`、不叠补,且**后缀归一为小写 `.md` 落盘** → `X.md`,免得盘上混入 `.MD`)。
   - ⚠️ 不可用「`Path(name).suffix != ".md"` 即 400」的朴素判定——它会把 `GPT-4.5 笔记`（`suffix==".5 笔记"`）、`v1.2` 等合法标题误杀。只对**拒绝列表内**的已知扩展名拒绝。
5. 解析校验：`(root/"raw"/safe).resolve()` 必须 `relative_to((root/"raw").resolve())`，否则 **`400`** 越界（与 §2 一致——越界是入参非法，非资源冲突；纵深防御，理论上第 1 步已挡住）。

- **规范化只为产出干净 basename，不改正文**：步骤 2 只作用于 `name`（文件名）；`content`（素材正文）**永远按 UTF-8 原样写盘**（§2），不做任何字符替换——`raw/` 是未加工源，正文里的弯引号/破折号必须保真。
- **映射表用模块级常量**（`_RAW_NAME_FOLDMAP: dict[str, str]`）+ `str.translate`，零正则、确定性、可单测；新增 `import unicodedata`。
- **不执行、不渲染**：`raw/` 内容是未加工源，`/api/page`（仅 `wiki/`）不会渲染它，故投喂正文**不经** XSS 渲染路径——但仍不放行写到 `raw/` 之外。`render.py` 的安全闸只作用于 `wiki/` 浏览与对话答案，与 `raw/` 写无关。
- **大小上限** `MAX_RAW_BYTES`（默认 5 MiB）：防误粘巨量文本；可后续按需调。

## 5. 前端（`guanlan/web/static/`）

- 顶栏新增 `投喂` 按钮 → 复用既有 `#overlay` 浮层：一个 `<input>`（文件名/标题）+ 一个 `<textarea>`（正文）+ `存盘` 按钮。
- `存盘` → `POST /api/raw`。**请求期间按钮置「保存中…」并禁用**（响应会等待队列前序写作业，如在飞 ingest，可能数十秒——不可让用户以为卡死，见 §3.1）。`409`（已存在）时浮层就地给出「改名 / 覆盖」二选项（覆盖 = 带 `overwrite=true` 重发）。
- 成功 → 刷新 `/api/raw` 列表（写后 `loadRaw()`），并显示「✓ 已存 `raw/x.md`」+ 一颗**「立即 ingest」**按钮（点了直接走既有 `triggerIngest('raw/x.md')` → 入队轮询）。
- 与既有 ingest/报告浮层同一交互骨架，不引新组件、不改两栏布局（决策P4-3 克制）。

## 6. 决策记录

- **\[决策P4.1-1] `POST /api/raw` 是人投喂源、非 gated agent 写；不破坏 `raw/` 不可变不变量。** DESIGN 的「`raw/` 只读不可变」约束的是**agent 不得改 `raw/`**（由 P2 快照门禁兜底）；**人新增源**始终是知识进入库的正途（CLI 下就是把 `.md` 放进 `raw/`）。故投喂**不取快照、不跑 check、不走门禁自愈**——它压根不是 agent 写。与「人不直接写 wiki」（原则 2）也不冲突：写的是 `raw/`（源），不是 `wiki/`（agent 全权的生成层）。
- **\[决策P4.1-2] 投喂经 §4.2 单写者队列串行、同步等待完成返回。** 唯一真实风险是与 ingest 的 `raw/` 快照窗口竞态（§3）；复用既有单写者 worker FIFO 串行即彻底消除，**不引文件锁**（与决策P4-5 同源）。因投喂极快，端点入队后**同步等作业完成**（`done_event` + `to_thread.wait`）再返回，省去轮询割裂感；这是与 ingest「入队即返回 job_id」的有意分流。
- **\[决策P4.1-3] 默认拒绝覆盖（`409`），须显式 `overwrite=true`。** 静默覆盖一篇**已被 ingest 的**源，会让 `raw/` 的事实记录变了而 `wiki/` 未重新消化、两者失同步。默认 `409` 逼用户要么改名（加新源）、要么显式确认覆盖（知道自己在替换源）。覆盖语义的最终判定落在串行 worker 的原子写里（§3.1）。
- **\[决策P4.1-4] 文件名 slug 化（含混淆字符规范化）+ 强制 `.md` + 大小上限 + 路径穿越校验。** 投喂只收 `.md` 文本（多格式属 P5），与 `_resolve_raw_target` 的 `.md` 边界一致；slug 先 `Path(name).name` 剥目录再过滤，纵深防御路径穿越（§8 安全姿态延续）。**过滤前先规范化**（NFKC + 显式映射表）大模型常误产的同形/排版字符——各类弯引号、中文破折号、波浪号、全角符号、特殊空白——把它们收敛到 ASCII（引号删除、破折号→`-`），而非劣化成 `-` 串；规范化**只作用于文件名**，`content` 正文按 UTF-8 原样保真（决策P4.1-1 的「未加工源」一脉相承）。`.md` 强制用**已知非-md 扩展名拒绝列表**（`_RAW_REJECT_EXTENSIONS`）判定（命中 `.txt/.pdf/.docx…` 才 `400`），不用朴素 `suffix!=".md"`，以免误杀 `GPT-4.5 笔记`、`v1.2` 等带点标题。
- **\[决策P4.1-5] `content` 文本准入：拒 NUL / C0 控制字符（`400`）。** `raw/` 是文本源；含 NUL（`\x00`）或 `\t\n\r` 以外的 C0 控制字符即判二进制/垃圾，端点同步 `400`，**不落盘**。真·二进制（任意字节、magic number）更早在 JSON/UTF-8 传输层被框架挡掉，到不了端点；多格式（docx/pdf）属 P5。准入只作用于 `content`，是一行确定性检测（零-LLM、零正则），不与规范化映射表混为一谈。`raw/` 存后**永不被 Web 宿主解析/渲染**（仅 `wiki/` 经 `/api/page` 渲染），故「不解析」是现状不变量,本条只把好入口。

| 测试 | 覆盖 |
|------|------|
| 投喂正常路 | `POST /api/raw {name,content}` → `{saved:"raw/<slug>.md", bytes}`；盘上 `raw/<slug>.md` 内容**字节等于** `content`（UTF-8 原样）；随后 `GET /api/raw` 列出该文件 |

| 文本准入 | `content` 含 `\x00` 或 C0 控制字符（非 `\t\n\r`）→ `400` 且**盘上不留文件**；纯文本（含 `\t\n\r`）正常通过 |
| 非-md 扩展拒绝列表 | 带点标题 `GPT-4.5 笔记` → `GPT-4.5-笔记.md`、`v1.2` → `v1.2.md`（**不**被 `suffix` 误判 `400`）；`x.txt` / `y.pdf` → `400`（命中 `_RAW_REJECT_EXTENSIONS`）；**大小写不敏感**：`X.PDF` / `foo.Docx` → `400`，`X.MD` → 落 `X.md`（后缀归一小写、不叠补）；**全角点不漏**：`x．PDF`（全角 `．`）→ 规范化后判 suffix → `400`（**不**漏成 `x.PDF.md`） |
| 退出码→HTTP | worker 复检撞同名（`EXIT_USAGE`）→ `409`；worker 落盘抛错（`EXIT_AGENT_ERROR`）→ `500`（**非** `409`，断言两者分流，对齐 §2） |
| 覆盖语义 | 已存在且无 `overwrite` → `409` 且**盘上原内容不变**；带 `overwrite=true` → 覆盖成功、内容更新 |
| **串行不撞 ingest 快照** | fake runner 下：先入队一个会「卡住」的 ingest（worker 占用中），再 `POST /api/raw`；断言投喂作业在 ingest **之后**完成（FIFO），且该 ingest 的 `exit_code` **不为** `EXIT_RAW_MUTATED`（投喂没落进它的快照窗口）；两者皆 `done` |
| 同步返回 | `POST /api/raw` 的响应**已含** `saved`（无需再轮询 `/api/jobs`）；worker 失败（monkeypatch 写盘抛错）→ `500`/`409` 且响应带原因 |
| 形态/范围 | 仍**无** `DELETE /api/raw`、**无**多文件/二进制端点；`web` extra 缺失整组 `importorskip` 跳过 |
