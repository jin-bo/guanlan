## 1. 目标与范围

**P4.1 = 给 Web 宿主补上唯一缺口：在浏览器里把一篇新素材存进 `raw/`，使「投喂 → ingest → 浏览 → 问答」整条环不再需要离开浏览器回到文件系统。**

现状（P4）：浏览器能 `从 raw/ 选一篇 ingest`，却不能**把素材放进 `raw/`**——那步必须在文件系统里手动完成。P4.1 用一个 `POST /api/raw`（粘贴/上传存盘）补齐它。它是**人投喂源**（普通文件写），**不是** gated 的 agent 写。

### 纳入 P4.1

- `POST /api/raw {name, content, overwrite?}` —— 把 `content` 存为 `<kb>/raw/<安全名>.md`，返回 `{saved, bytes}`。
- **经 §4.2 单写者队列串行**：杜绝与在飞 `ingest` 的 `raw/` 快照窗口竞态（§3，**本设计的核心**）；端点**同步等待**该写作业完成后返回（不轮询）。
- **文件名安全 + 默认不覆盖**：slug 化 + 强制 `.md` + 路径穿越校验 + 大小上限；目标已存在默认 `409`，须显式 `overwrite=true` 才覆盖。
- **前端**：顶栏 `投喂` 动作 → 浮层（`<textarea>` 正文 + 文件名输入）→ `POST /api/raw` → 成功后刷新 `/api/raw` 列表，并就地提供「立即 ingest」一键续跑（复用既有 ingest 流）。

### 不纳入 P4.1（仍按 P4 §10 / 路线图推后）

- **多格式上传（docx/pdf/web clip）** → P5；P4.1 只收 `.md` 文本（与 CLI `ingest` 同口径）。
- **二进制 / 多文件批量上传** → 推后；v1 一次一篇文本。
- **Web 端编辑/删除已存在的 `raw/` 源** → 推后（删源会让已 ingest 的 wiki 失去出处，需级联策略）。v1 只「新增」与「显式覆盖」。
- **把投喂直接接到 ingest 一步到位（存盘即自动 ingest）** → 不做：存与消化是两步、两种失败模式（写盘 vs 门禁），分开更可控；前端用「存盘成功 → 一键 ingest」串起体验即可。

## 2. 端点契约（`guanlan/web/app.py`）

| 方法 路径 | 入参（pydantic `RawBody`） | 返回 | 错误 |
|------|------|------|------|
| `POST /api/raw` | `{name: str, content: str, overwrite: bool=false}` | `{saved: "raw/<name>.md", bytes: N}`（写作业完成后同步返回） | `400` 用法（空名/空正文/sanitize 后非法/超 `MAX_RAW_BYTES`）· `409` 已存在且未 `overwrite` · `500` 写盘失败 |

- `name`：用户给的文件名或标题。服务端 **slug 化**为安全 basename（见 §4），强制 `.md` 后缀，解析后**必须**落在 `<kb>/raw/` 内（`resolve()` + `relative_to(raw)`，越界 `400`），与 `_safe_wiki_file` 同口径但归到 `raw/`（新增 `_safe_raw_target`）。
- `content`：素材正文，按 **UTF-8 原样**写盘（不做 markdown 渲染、不做 `[[wikilink]]` 重写——`raw/` 是未加工源）。超过 `MAX_RAW_BYTES`（默认 5 MiB）→ `400`。
- `overwrite`：默认 `false`。目标已存在 → `409`（见决策P4.1-3）；`true` 时原子覆盖。
- **`/api/raw` 仍保留 `GET`（只列）**；P4.1 只新增 `POST`，不动既有读端点。
- **P4.1 不新增退出码 / 不动 CLI**：投喂是纯 Web 能力（CLI 把 `.md` 放进 `raw/` 用文件系统即可）。HTTP 状态码即对外契约。

## 3. 与 ingest 快照窗口的竞态——本设计的核心　**\[决策P4.1-2]**

P2 写门禁 `run_guarded_write`（`gate.py`）在 agent 调用**前后**各取一次 `snapshot_raw(root)`，任意条目增/删/改即判 `EXIT_RAW_MUTATED(4)`。若 `POST /api/raw` 把一篇新文件写进 `raw/` **正落在某个 ingest 作业的快照窗口里**，会同时造成两个错误：

1. **ingest 作业被冤判失败**：after 快照多出一个 `added` 条目 → 门禁 `raw_mutated` → 该 ingest 以退出码 4 失败，并在面板里**错误地指控 agent 改了 `raw/`**。
2. **投喂结果语义不明**：用户以为只是存了篇源，却让一个无关的 ingest 挂掉。

**解法：把 `POST /api/raw` 的写盘动作也送进 §4.2 的单写者 `JobQueue`。** 该队列本就是「唯一写者」的归口（决策P4-5：FIFO 串行、`raw/` 快照不被并发写互踩）。投喂作业与 ingest 作业在**同一条 worker 线程**上 FIFO 串行，于是投喂**永不**落在某个 ingest 的 before/after 之间——零新锁、零新并发原语，直接复用既有单写者假设。

> 这正是 P4 §10 把本项「推后」时点名的那处复杂度（"与 `raw/` 快照窗口的并发"）。P4.1 的回答是**不引锁、复用单写者队列**——与决策P4-5「`workers=1` + 单 worker 串行」一脉相承。

### 3.1 同步返回，而非轮询　**\[决策P4.1-2]**

ingest 用「入队 → 返回 `job_id` → 轮询 `/api/jobs/{id}`」，因为它慢（LLM 子进程）。投喂作业**极快**（一次文件写），轮询体验割裂。故 `POST /api/raw` **入队后在服务端等待该作业完成再返回结果**：

```python
# jobs.py：Job 增一个完成信号；enqueue 不变，新增 submit_and_wait（仍走同一 worker、同一 FIFO）
@dataclass
class Job:
    id: str; kind: str; state: JobState = "queued"
    exit_code: int | None = None
    output: str = ""
    done_event: threading.Event = field(default_factory=threading.Event)  # 新增：worker 收尾时 set

def _run(self):
    while True:
        job, fn = self._queue.get()
        ...
        finally:
            job.output = buf.getvalue(); job.state = "done"
            job.done_event.set()          # 新增：唤醒同步等待者

# app.py：投喂端点——校验（同步、立即 4xx）→ 入队写盘作业 → to_thread 等完成 → 返回
@app.post("/api/raw")
async def write_raw(body: RawBody):
    target = _safe_raw_target(root, body.name)                 # 400/越界即抛，无需进队列
    if target.exists() and not body.overwrite:
        raise HTTPException(409, detail=f"raw/{target.name} 已存在；改名或传 overwrite=true。")
    if len(body.content.encode("utf-8")) > MAX_RAW_BYTES:
        raise HTTPException(400, detail=f"正文超过 {MAX_RAW_BYTES} 字节上限。")
    job_id = jobs.enqueue("raw_write", lambda: _atomic_write_raw(target, body.content, body.overwrite))
    job = jobs.get_job(job_id)
    await anyio.to_thread.run_sync(job.done_event.wait)        # 不堵事件循环
    if job.exit_code != EXIT_OK:                               # worker 内复检失败（如并发抢同名）
        raise HTTPException(409, detail=job.output or "写入失败")
    return {"saved": f"raw/{target.name}", "bytes": len(body.content.encode("utf-8"))}
```

- **校验在端点（快 4xx）、写盘在 worker（串行）**：路径/大小/存在性的快速否决在端点同步给出；真正落盘进队列串行，杜绝与 ingest 快照窗口重叠。
- **worker 内复检存在性 + 原子写**：两个并发投喂同名新文件时，端点的 existence 预检可能都放过；故 `_atomic_write_raw` 在**串行的** worker turn 内再查一次 `overwrite` 语义并**原子落盘**（写临时文件 `os.replace` 同目录换名），把覆盖保护落在真正串行的临界区里。第二个同名作业返回 `EXIT_USAGE` → 端点转 `409`。
- **`redirect_stdout` 红线不破**：worker 仍只对一个作业串行捕获输出；投喂作业不打印业务文本，`output` 只承载失败原因，不与 ingest 抢 stdout。

## 4. 文件名 slug 与安全　**\[决策P4.1-4]**

`_safe_raw_target(root, name) -> Path`（新增于 `app.py`，与 `_safe_wiki_file` 并列）：

1. 取 `Path(name).name`（**剥掉任何目录成分**），杜绝 `../`、绝对路径、子目录穿越。
2. slug 化：保留 CJK、字母、数字、`-`、`_`、空格、`.`；其余字符替换为 `-`；折叠连续 `-`；去首尾 `-`/空白。空结果 → `400`。
3. 强制 `.md`：无 `.md` 后缀则补；已有非 `.md` 后缀（如 `x.txt`）→ 仍存为 `x.txt.md`？**否**——直接 `400`，提示「投喂只收 `.md`」（与 `_resolve_raw_target` 的 `.md` 约束、P5 多格式边界一致）。
4. 解析校验：`(root/"raw"/safe).resolve()` 必须 `relative_to((root/"raw").resolve())`，否则 `409` 越界（纵深防御，理论上第 1 步已挡住）。

- **不执行、不渲染**：`raw/` 内容是未加工源，`/api/page`（仅 `wiki/`）不会渲染它，故投喂正文**不经** XSS 渲染路径——但仍不放行写到 `raw/` 之外。`render.py` 的安全闸只作用于 `wiki/` 浏览与对话答案，与 `raw/` 写无关。
- **大小上限** `MAX_RAW_BYTES`（默认 5 MiB）：防误粘巨量文本；可后续按需调。

## 5. 前端（`guanlan/web/static/`）

- 顶栏新增 `投喂` 按钮 → 复用既有 `#overlay` 浮层：一个 `<input>`（文件名/标题）+ 一个 `<textarea>`（正文）+ `存盘` 按钮。
- `存盘` → `POST /api/raw`。`409`（已存在）时浮层就地给出「改名 / 覆盖」二选项（覆盖 = 带 `overwrite=true` 重发）。
- 成功 → 刷新 `/api/raw` 列表（写后 `loadRaw()`），并显示「✓ 已存 `raw/x.md`」+ 一颗**「立即 ingest」**按钮（点了直接走既有 `triggerIngest('raw/x.md')` → 入队轮询）。
- 与既有 ingest/报告浮层同一交互骨架，不引新组件、不改两栏布局（决策P4-3 克制）。

## 6. 决策记录

- **\[决策P4.1-1] `POST /api/raw` 是人投喂源、非 gated agent 写；不破坏 `raw/` 不可变不变量。** DESIGN 的「`raw/` 只读不可变」约束的是**agent 不得改 `raw/`**（由 P2 快照门禁兜底）；**人新增源**始终是知识进入库的正途（CLI 下就是把 `.md` 放进 `raw/`）。故投喂**不取快照、不跑 check、不走门禁自愈**——它压根不是 agent 写。与「人不直接写 wiki」（原则 2）也不冲突：写的是 `raw/`（源），不是 `wiki/`（agent 全权的生成层）。
- **\[决策P4.1-2] 投喂经 §4.2 单写者队列串行、同步等待完成返回。** 唯一真实风险是与 ingest 的 `raw/` 快照窗口竞态（§3）；复用既有单写者 worker FIFO 串行即彻底消除，**不引文件锁**（与决策P4-5 同源）。因投喂极快，端点入队后**同步等作业完成**（`done_event` + `to_thread.wait`）再返回，省去轮询割裂感；这是与 ingest「入队即返回 job_id」的有意分流。
- **\[决策P4.1-3] 默认拒绝覆盖（`409`），须显式 `overwrite=true`。** 静默覆盖一篇**已被 ingest 的**源，会让 `raw/` 的事实记录变了而 `wiki/` 未重新消化、两者失同步。默认 `409` 逼用户要么改名（加新源）、要么显式确认覆盖（知道自己在替换源）。覆盖语义的最终判定落在串行 worker 的原子写里（§3.1）。
- **\[决策P4.1-4] 文件名 slug 化 + 强制 `.md` + 大小上限 + 路径穿越校验。** 投喂只收 `.md` 文本（多格式属 P5），与 `_resolve_raw_target` 的 `.md` 边界一致；slug 先 `Path(name).name` 剥目录再过滤，纵深防御路径穿越（§8 安全姿态延续）。

## 7. 测试要点（`tests/test_web.py`，不打真实 LLM）

| 测试 | 覆盖 |
|------|------|
| 投喂正常路 | `POST /api/raw {name,content}` → `{saved:"raw/<slug>.md", bytes}`；盘上 `raw/<slug>.md` 内容**字节等于** `content`（UTF-8 原样）；随后 `GET /api/raw` 列出该文件 |
| slug / 校验 | `name="../../etc/passwd"`、绝对路径、含 `/` → 剥成 basename 落 `raw/` 内、绝不越界（断言落点 `relative_to(raw)`）；空名/空正文 → `400`；非 `.md` 名 → `400`；超 `MAX_RAW_BYTES` → `400` |
| 覆盖语义 | 已存在且无 `overwrite` → `409` 且**盘上原内容不变**；带 `overwrite=true` → 覆盖成功、内容更新 |
| **串行不撞 ingest 快照** | fake runner 下：先入队一个会「卡住」的 ingest（worker 占用中），再 `POST /api/raw`；断言投喂作业在 ingest **之后**完成（FIFO），且该 ingest 的 `exit_code` **不为** `EXIT_RAW_MUTATED`（投喂没落进它的快照窗口）；两者皆 `done` |
| 同步返回 | `POST /api/raw` 的响应**已含** `saved`（无需再轮询 `/api/jobs`）；worker 失败（monkeypatch 写盘抛错）→ `500`/`409` 且响应带原因 |
| 形态/范围 | 仍**无** `DELETE /api/raw`、**无**多文件/二进制端点；`web` extra 缺失整组 `importorskip` 跳过 |

**验收（P4.1 Done）**：上述测试全绿；浏览器里 `投喂` → 粘贴正文 + 命名 → 存盘 → 列表刷新 → 「立即 ingest」一键消化，全程不离开浏览器；并发的 ingest 作业**不被投喂冤判** `raw_mutated`；`raw/` 之外的任何路径都写不进去。
