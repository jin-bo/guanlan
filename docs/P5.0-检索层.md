# P5.0 检索层 `guanlan search`（零 LLM，确定性 BM25 + CJK 2-gram）

> 半阶段增强。给 query / skill / 未来 web·mcp 一个**确定性、零 LLM、零基建**的整页正文召回原语，
> 取代"只读 `index.md` 一句话摘要 + 2-gram"这一条召回路径在语料变大后的退化。**不是** E1 的存储/检索
> 升级（向量 / 重排 / 多租户隔离仍属 E1，见 §6 与 DESIGN §8 路线表）。

## 0. 定位：为什么是 P5.0，而不是 E1

DESIGN §8 把"更强检索"挂在 **E1（存储与检索升级，"index-only 撑不住时再引入，选型另文"）**，并立"**不预先加机制，
确有实证后再评估**"的纪律（`docs/backlog/notes/cjk-retrieval-enhancements.md`）。本阶段之所以现在落、且落在 **P5.0**，
有三条实证与一条边界判断：

- **实证①**：Karpathy 原文与本仓 DESIGN 都承认 `index.md` 目录召回"约数百页封顶"；
- **实证②**：三个对照实现（nashsu/llm_wiki、garrytan/gbrain、llm-wiki-agent）**无一例外**都在检索上投入（混合检索 /
  图信号 / 向量），是该模式做大后价值的共同迁移方向；
- **实证③**：**P5 主体 = 多格式摄入**，源一多，`index.md` 一句话摘要的召回面会被正文体量迅速淹没——多格式上线**之前**
  先把召回原语备好，正是 P5.0 该做的开篇。
- **边界判断**：真正属于 E1、需"另文选型"的是**向量 / 重排 / 持久化索引基础设施**。而 BM25 + CJK 2-gram 是
  **纯 Python、零依赖、零 LLM、查询时即时计算**的确定性算术——它落在**现有零 LLM 脚本层**内（与
  `check`/`health`/`lint`/`graph`/`reindex` 同性质），不引入任何 E1 才需评估的基础设施。故拆出来先做，名正言顺。

> 与 backlog 的关系：CJK 笔记把增强分三级——①别名（**已在 P3.1 落地**）②同义词表 ③轻量/向量检索。P5.0 = 第③级里
> **确定性、零基建的那一半**（轻量检索）；向量那一半仍按笔记留给 E1。分词**依旧不做**（决策 P5.0-7，理由同笔记）。

## 1. 命令契约

```bash
guanlan -C <kb> search "<query>"            # 确定性 BM25+2-gram 召回，按分数降序打印 top-N 页（默认 10）
guanlan -C <kb> search "<query>" --limit 20 # 调召回条数（须 ≥ 1）
guanlan -C <kb> search "<query>" --json     # 结构化回执，供 agent / web / mcp 消费
```

- **纯读、零 LLM、零写盘**：只读 `wiki/` 下 content 页，**不**起 Agentao、**不**取 raw 快照、**不**写任何派生文件
  （连 `graph/` 那样的派生产物都不落——查询时即时计算，见决策 P5.0-1）。`require_kb_root(writable=False)` 即可。
- **无新退出码**（沿用半阶段纪律）：成功恒 `EXIT_OK(0)`；非知识库根 / 空 query / 用法错 `EXIT_USAGE(1)`。
- 与 `query` 的分工：`search` 只做**确定性召回**（给候选页 + 片段），**不综合、不引 LLM**；`query` 仍走 Agentao
  做带 `[[引用]]` 的综合（§5）。二者正交：search 是 query 的召回前端，不替代它。

## 2. 模块与归口

新增 `guanlan/search.py`，复用 `pages.py` 的 `iter_pages / page_type / split_frontmatter`。**注意：不复用
`load_page`**——它会 `yaml.safe_load` 每页 frontmatter，是全库扫描的隐藏瓶颈；search 只要 body（split_frontmatter
纯文本切，零 YAML）+ title/aliases（轻量行扫，§3.6 决策 P5.0-13）。对外：

```python
@dataclass(frozen=True)
class SearchHit:
    page: str      # 相对**库根**的 posix 路径，如 wiki/entities/DeFi.md（与 graph.Node.path 同口径，决策 P5.0-22）
    title: str     # 轻量扫取，缺/坏退化 stem（§3.6）
    type: str      # page_type 取值（type 字段简单，行扫即可）
    score: float   # BM25 分数，定点舍入后参与排序与输出（决策 P5.0-5）
    snippet: str   # 确定性正文片段（§3.4）

@dataclass(frozen=True)
class SearchResult:
    hits: list[SearchHit]; pages_searched: int; query: str   # 详见上版（决策 P5.0-10）

def tokenize(text: str) -> list[str]: ...
    # 分词**单一归口**：CJK 2-gram + 非 CJK 词，对齐 DESIGN §4.5「2-gram 滑窗」。文档与 query 同函数。

# —— 昂贵部分（建语料）与廉价部分（打分）分立，让 memo 只缓存昂贵的那半 ——
@dataclass(frozen=True)
class DocBag:
    page: str; title: str; type: str
    tf: dict[str, int]        # 增强词频：body_cnt + 3×title_cnt + 2×alias_cnt（§3.3 决策 P5.0-17）
    dl: int                   # 文档长度 = len(body_tokens)，**仅 body**，boost 不计入（决策 P5.0-17）
    body: str                 # 原始正文，供 snippet 原文切窗（§3.4）
    mtime_ns: int; size: int  # memo 失效键（§3.6）

def build_doc(path: Path, *, root: Path) -> DocBag: ...      # 单页：split_frontmatter 取 body + 轻量扫 title/aliases + tokenize
    # page 字段 = path.relative_to(root).as_posix()——root 由调用方传入，**不**用 cwd/绝对路径（决策 P5.0-22），
    # 与 graph.Node.path 同口径（`wiki/entities/DeFi.md`）。
def build_corpus(wiki: Path) -> list[DocBag]: ...           # iter_pages → build_doc(p, root=wiki.parent) 每页（**昂贵**：读盘+分词）
def score(docs: list[DocBag], query: str, *, limit: int) -> SearchResult: ...  # **廉价**：BM25+排序+snippet
    # 内核契约：limit < 1 → raise ValueError（决策 P5.0-15）。web/mcp 直接复用，不能靠 CLI 的 positive_int 兜底。

def search_pages(wiki: Path, query: str, *, limit: int = 10) -> SearchResult:
    # 无状态冷算：CLI 路径，**恒为等价权威**。limit 校验在 score（决策 P5.0-15），此处只透传。
    return score(build_corpus(wiki), query, limit=limit)

class CorpusCache:
    """长驻进程（web/mcp）专用**进程内** memo：按 (mtime_ns,size) 复用已建 DocBag、只重建变更页。
    **不落盘**、可幂等重建、与冷算 search_pages 字节等价（§3.6 决策 P5.0-14）。CLI 不用（新进程吃不到）。"""
    def corpus(self, wiki: Path) -> list[DocBag]: ...        # 命中缓存跳过 build_doc；变更/新增页才重建

def run_search(query, *, root, limit, json_output) -> int: ...   # 打印壳 + 退出码（走 search_pages 冷算）
def search_entrypoint(root_dir, *, query, limit, json_output) -> int: ...
def main(argv=None) -> int: ...                                  # python -m guanlan.search
```

打印壳与未来 web/mcp 端点都从同一个 `SearchResult` 出（决策 P5.0-10）；web/mcp 另持一个 `CorpusCache`、调
`score(cache.corpus(wiki), …)` 省去每查重建（决策 P5.0-14）。「建语料 / 打分」分立同时也呼应「内核与打印壳分立」
纪律（决策 P4-7 / P3.2）。CLI 接线在 `cli.py` 加 `p_search`（`query` 位置参、`--limit`、`--json`），与 health/lint/reindex 同形。

## 3. 行为细则

### 3.1 语料 = 仅 content 页（与 graph/lint 同口径）

`iter_pages(wiki)` 的非 config 页（排除 `index.md`/`log.md`/`overview.md`，`SCHEMA.md` 天然在库根不被扫）。**理由**：
config 页是导航/派生层（`index.md` 的内容已是各页一句话摘要的二手转述），纳入会与正文重复计权、污染排序——
与"config 不进 index/graph/lint 扫描"的既有不变式一致（决策 P5.0-2）。每页读取**必须**沿用 `load_page` 的容错口径
`path.read_text(encoding="utf-8", errors="replace")`（坏字符 → `�`），**不**因为绕开 `load_page` 而退回裸 `read_text()`
——否则一张 GBK/Latin-1 页会让 `build_doc` 崩在 `UnicodeDecodeError`，丢掉原有"绝不抛"承诺（决策 P5.0-16）。读入后用
`split_frontmatter`（纯文本切、**零 YAML**）取 body + 轻量行扫取 title/aliases（§3.6），坏/缺 frontmatter 照常按 body +
stem 索引、**绝不抛**（决策 P3-8 一脉）——而**不**走 `load_page`（避开其 PyYAML 解析，§3.6 决策 P5.0-13）。

### 3.2 分词（确定性，CJK 2-gram + 非 CJK 词）

`tokenize(text)`：把文本切成"CJK 连续段"与"非 CJK 段"，分别处理（决策 P5.0-3）：

- **CJK 码点谓词（钉死，决策 P5.0-18）**：归口必须穷尽定义"什么算 CJK"，否则"确定性单一归口"立在未定义的谓词上。
  采**最小够用集**——统一表意文字基本区 `一–鿿` + 扩展 A `㐀–䶿` + 兼容表意 `豈–﫿`
  （扩展 B 及以上的星文字按需，本稿不纳入，命中极罕见）。**不含**假名（`぀–ヿ`）、CJK 标点/全角符号、Bopomofo
  ——它们与非 CJK 段同走"取词/丢弃"逻辑（标点天然落在 `[a-z0-9]+` 之外被丢）。日文若成需求另行评估，不在本稿。
- **非 CJK 段**：小写化后取 `[a-z0-9]+` 词（`DeFi`→`defi`，`GPT-4`→`gpt`,`4`）。**重音拉丁落在 `[a-z0-9]` 之外**：
  `café`→`caf`（`é` 被丢），即 `café` 与 `cafe` 不同 token——CJK 库可接受，真要正规化是 E1 的事，本稿不做（决策 P5.0-18）；
- **CJK 段**：2-gram 滑窗（`去中心化`→`去中`,`中心`,`心化`；`去中心化金融`才会出现`化金`,`金融`）；**段长为 1** 时退化为该单字 1-gram
  （保「李」这类单字仍可召回）；
- query 与文档**同函数**，故 `[[别名]]`、混排（`L2 扩容`）两侧切法一致。

**不上分词器**：词典外实体词（OOV）最易被分词器漏标错切，结果随词典版本漂移、引入词典依赖、违背"脚本零 LLM"；
真要补精度，向量检索优于词典分词——这是 E1 的事（决策 P5.0-7，理由同 backlog 笔记「分词为何不优先」）。

### 3.3 BM25 + 字段轻量加权

标准 BM25（`k1=1.5`、`b=0.75`，非负 idf `ln(1+(N-df+0.5)/(df+0.5))`），但词频做**字段增强**——
title / alias 命中按 boost 倍数计入 `tf`：

```
tf(t, D) = body_cnt(t) + TITLE_BOOST × title_cnt(t) + ALIAS_BOOST × alias_cnt(t)   # 默认 ×3 / ×2
dl(D)    = len(body_tokens)        # 文档长度 = 仅 body，boost 不计入（BM25F-lite）
```

**关键：boost 只抬 `tf`、绝不进 `dl`/`avgdl`（决策 P5.0-17）。** 若像朴素做法那样把 title token ×3、alias ×2
**重复灌进同一词袋并以袋长为 `dl`**，会撑大 `dl/avgdl`、经 `b=0.75` 的长度归一化**反向压低本页 body 词的得分**——
标题长 / 别名多的核心实体页越容易被边缘页压下去，与"字段加权抬核心页"的初衷正相反（这正是 BM25F 分字段饱和、
而非拼接重复的动机）。故此处 `dl` 只取 body token 长度，boost 仅作为命中该 term 的额外词频——**boost 不进 `dl`/`avgdl`、
不触发长度归一化副作用**（它仍参与得分与召回面，见决策 P5.0-23，**不是**"只影响排序"）。`df`/`avgdl` 同口径：`df` 按
"该 term 是否出现在增强 `tf` 中"计、`avgdl` 按 body 长度均值计。

title / aliases 取自该页 frontmatter 的**轻量行扫**（§3.6，不经 `load_page`/PyYAML，也不经 `alias_index`——那是解析
`[[别名]]` 用的全库映射，此处只要本页自己的别名串进匹配面）。**只做字段加权，不做向量 / 不做重排**——那是 E1
（决策 P5.0-4）。**字段进入匹配面、参与 BM25 得分**——纯标题 / 纯别名命中（query 不在正文）的页也会得分 >0 被
召回，故字段影响的是**召回面**、不止排序（决策 P5.0-23）；但 **snippet 只从 body 取**，字段-only 命中时 snippet
退化首窗（§3.4）。至于轻量行扫（§3.6）的**近似**——漏一个字段至多略降该页召回、非正确性问题，frontmatter
正确性归口仍是 `check`。

**空语料 / 零除守卫（决策 P5.0-12，必须先于任何除法）**：新建库或早期库常见——

- **N = 0**（`iter_pages` 无 content 页）：直接返回 `SearchResult(hits=[], pages_searched=0, query=…)`，`ok:true`。
  无任何 doc → 无 idf、无 corpus，自然短路。
- **`avgdl = 0`（body 全空，但 title/aliases 可能有 token）：不短路、不丢字段召回（决策 P5.0-12 / P5.0-23 边界归口）**。
  此情形下 `dl(D)=len(body_tokens)=0`，而 `tf` 可能因字段 boost 非空——按 P5.0-23 这类页**应当可被字段命中**。故
  **不**把 `avgdl=0` 当空语料短路，而是把长度归一化比值定义为 `dl/avgdl ⟶ 0 当 avgdl=0`，即归一化因子
  `(1 − b + b·0) = 1 − b = 0.25` 这一**稳定有限值**——BM25 照常用 `tf`+`idf` 打分，字段-only 命中得分 >0、被召回，
  **绝不除零**。若此时连字段也无任何 token（全库 `tf` 皆空），则无 term 有 postings、query 命中空，自然回 `hits=[]`
  （非短路、是真无命中），`pages_searched = len(content_pages)` 仍如实计数。
- **query 分词后无 token**（纯标点/空白）：见 §1，`run_search` 层即 `EXIT_USAGE`；`search_pages` 若被直接调用则回空 hits。

即**唯一硬短路是 N=0 与零-query-token**；`avgdl=0` 不短路而以"比值定义为 0"消除除零（保字段召回）。`df`+0.5 / N 在
N≥1 时天然非零。`pages_searched` 始终如实计数。

### 3.4 片段（确定性）

从**原始正文**取片段，保证同 query 同 wiki 字节稳定：

- **命中定位直接在原文上做、大小写不敏感**（决策 P5.0-11）：**不**对整段正文 `casefold()` 后复用下标——只要真实命中
  之前出现 `ß→ss`/`İ` 等改长字符，casefold 副本与原文的下标就会错位、切错窗。改为**逐 token 在原文上定位**：
  - 非 CJK token：**带词边界**地匹配、`re.IGNORECASE`——`re.search(rf"(?<![a-z0-9]){re.escape(token)}(?![a-z0-9])",
    body, re.IGNORECASE)`，直接拿**原文**下标。词边界**与 tokenizer 的 `[a-z0-9]+` 整词口径对齐**（决策 P5.0-11）：否则
    query token `ai` 会命中正文 `said`/`chain` 内部，把"最小下标"窗口拽到无关位置——而该词根本不是 `ai` 这个 token；
  - CJK token：表面 = 那 2 字（中文无大小写、2-gram 本就允许跨词重叠），`body.find(surface)` 直接拿原文下标。
  这样每个 token 的下标天然就是原文坐标，不存在 casefold 长度漂移、也不存在非 CJK 词内误命中。
- 命中位置 = 各 query token 在原文中**最小**的命中下标（多 token 命中取下标最小者，确定性）。这是**刻意从简**的
  确定性选择，非召回正确性的一部分——它可能把窗口落在最常见(信息量最低)的 token 上、漏掉更显著的稀有词;"最优窗口"
  (近邻含最多 distinct token)留待未来,不在本稿;
- 取该下标起约 140 字符的窗口，折叠连续空白为单空格，首尾去空白；
- 分数 >0 **未必 body 有表面命中**——字段-only 命中（query 仅中标题/别名、正文无该表面串）时 body 定位不到，
  **退化取正文首窗**（决策 P5.0-23）；body 有命中则按上条最小下标切窗。**snippet 永远只从 body 取，绝不回填
  标题/别名**——故 **body 为空时（`avgdl=0` 那类页）首窗即 `""`，是合法结果**，不是错误、也不拿字段串去填（决策 P5.0-24）。
- **不**做高亮标记（CLI 纯文本 / JSON 原样；高亮交给未来 web 前端）。

### 3.5 确定性与幂等

无**盘上**持久化、无时间戳进结果、无随机：同 query 同 wiki **每次输出字节一致**。排序键 `(-round(score, 6), page)`
——分数定点舍入后降序，**path 升序兜底 tie-break**（与 graph 稳定排序同纪律，决策 P5.0-5）。**分数口径钉死
（决策 P5.0-20）**：排序与所有呈现都先 `round(score,6)`，但**两侧形态不同**——
- **CLI 文本**：固定 `f"{round(score,6):.6f}"`，**6 位定点补零**（`8.4`→`8.400000`），人读对齐、字节稳定；
- **JSON `score`**：保持 **number**，序列化 `round(score,6)`。**注意 `json.dumps(round(8.4,6))` 输出 `8.4`，不补零**
  ——故 JSON 只承诺"**数值 round 到 6 位**"，**不承诺** `8.400000` 这种零填充字节形态（下游应按数值比较，勿按字符串比）。
  这仍是字节稳定的：同一 float 值经 `json.dumps` 的 repr 恒定，只是不带尾零。要尾零字节形态请读 CLI 文本，不读 JSON number。

`CorpusCache`（§3.6）是**进程内** memo、不落盘，且按 (mtime_ns,size) 失效后产物与冷算 `search_pages` **字节等价**，不破坏本承诺。

### 3.6 性能路径：轻量 frontmatter 解析 + 进程内 mtime memo

全库每查重扫的两大成本是 **frontmatter 的 PyYAML 解析**（隐藏瓶颈）与**分词**。两条优化进基线：

**(a) 轻量 frontmatter 行扫，绕开 PyYAML（决策 P5.0-13，普惠 CLI/web/mcp）**

- **body**：`split_frontmatter`（纯文本切 `--- … ---`，**零 YAML**）即得——这是 BM25 的主信号，付零 YAML 成本。
- **title / aliases**（仅字段加权用，次要）：在 frontmatter 块上**行扫**——`title: …` 取冒号后标量（剥引号）；`aliases:`
  取其后的块列表 `- x` 或单行 flow `[a, b]`。**最大努力、可近似**：罕见花式 YAML（多行折叠标量、锚点、嵌套）扫不全
  就当该字段缺失退化（title→stem、aliases→空）。**正当性**：字段虽进召回面（决策 P5.0-23），但行扫漏一个别名至多
  **略降该页召回**（不影响其它页、不动 body 主信号、不改正确性），**非正确性问题**；frontmatter 正确性归口仍是
  `check`（决策 P3-8 一脉）。**不**因此把这套行扫塞进 `pages.load_page` 归口——它是 search 专用的 recall 近似路径，
  与 `check`/`graph` 用的严格/容错档**有意分立**，不冒充同口径。
- 收益：去掉每页一次 `yaml.safe_load`，把全库扫描的隐藏大头砍掉。

**(b) 长驻进程内 mtime memo（决策 P5.0-14，仅 web/mcp）**

- `CorpusCache` 按 `(path, mtime_ns, size)` 缓存已建 `DocBag`；每查 `iter_pages` + `stat` 各页，键未变就复用、**只对
  变更/新增页重跑 `build_doc`**（读盘+分词），删除页移除。**进程内、不落盘**——仍无盘上派生、仍可幂等重建。
- **线程安全（决策 P5.0-19）**：web 虽 `workers=1`，FastAPI 同步端点跑在 threadpool，两个 `/api/search` 可并发进
  `corpus()` 改同一个 memo dict → 竞态 / 双重 build。故 `corpus()` 的"读 stat → 比键 → 重建 → 写回 dict"全程**持一把
  `threading.Lock`**（或上层把 search 串行化 / `asyncio.to_thread` + 锁）。锁只护 cache 字典，**不**改确定性：命中与否
  产物都与冷算 `search_pages` 字节等价。CLI 单进程单线程，天然无此问题、不加锁。
- **CLI 不受益**（每次新进程），但有 OS 文件缓存兜底；**web/mcp** 长驻则把每查的 `build_doc`（读盘+分词）降为只对增量页
  重跑。注意**地板仍在**：每查的 `iter_pages` 树遍历 + 逐页 `stat` 是 O(N)、不省——省的是读盘与分词那块大头，不是遍历本身。
- **不破坏确定性**：memo 命中与否，产物与冷算 `search_pages` 字节等价（§3.5）。失效键用 `mtime_ns + size`：正常编辑
  必动其一；理论上"同 mtime tick 内改写且 size 不变"会漏更新——极端罕见，且**冷算 `search_pages`（CLI / 不带 cache 的
  调用）恒为等价权威**，可随时校验；真要消除可加 `--no-cache` 走冷算。**绝不**为此引入盘上索引（那是 §3.5/决策 P5.0-1
  的红线，留给 E1）。

## 4. 决策（沿用仓库 `决策PX-N` 体例）

- **决策P5.0-1（无状态即时计算 vs 持久化倒排索引）**：默认**查询时即时计算**，不落任何派生文件。最忠于"markdown 是
  唯一事实来源、派生可幂等重建"——索引根本不存在就无所谓陈旧/重建/纳入 .gitignore。**性能预期**：数百~数千页全量读 +
  分词 + 打分**预期**在交互级延迟内够用；**但本稿未做基准**——实现时补一个**非门禁** benchmark（如对 N=10²/10³/10⁴
  合成页测端到端延迟），实测后再决定"持久化倒排索引 + SHA256 增量缓存（llm-wiki-agent 的做法）"是否、何时作 `--build`
  选项 / E1 引入。在拿到实测前**不预先加**持久化机制，也不在文档里把"够用到上万页"当既成结论。
- **决策P5.0-2（仅索引 content 页）**：见 §3.1，排除 config，与 index/graph/lint 扫描口径一致。
- **决策P5.0-3（CJK 2-gram + 非 CJK 词，单一归口 `tokenize`）**：见 §3.2，对齐 DESIGN §4.5 既定「2-gram 粗召回」，
  query 与文档同函数。
- **决策P5.0-4（BM25 + 字段加权，不上向量/重排）**：见 §3.3。向量/重排是 E1「另文选型」，本阶段守零依赖。
- **决策P5.0-5（确定性输出：定点舍入 + path tie-break）**：见 §3.5，浮点求和顺序差异由舍入 + path 兜底消除，输出字节稳定。
- **决策P5.0-6（无新退出码）**：成功 `EXIT_OK`；空 query / 非库根 / 用法错 `EXIT_USAGE`。沿用半阶段不引新码纪律。
- **决策P5.0-7（不上分词、不上同义词表）**：见 §3.2 / §6，逐级增强次序与 backlog 笔记一致。
- **决策P5.0-8（search 是召回前端，不改 query 的综合路径）**：`query` 仍由 Agentao 走带 `[[引用]]` 的综合；P5.0 只把
  "召回"这一步从 Agent 在 prompt 里自行 2-gram，升级为可 shell out 的确定性 `guanlan search`（§5）。综合、引用、回填门禁
  **一律不动**。
- **决策P5.0-9（不写 log.md / 不碰 raw）**：纯确定性只读检索，无 LLM 写作、无 raw 风险，不留痕 `log.md`——与 `graph` /
  `reindex` 一致（heal 因走 LLM 写门禁才记 log，语境不同）。
- **决策P5.0-10（内核返回 `SearchResult`，含 `pages_searched`）**：`search_pages` 返回 `SearchResult(hits, pages_searched,
  query)` 而非裸 `list[SearchHit]`——`pages_searched` 是 JSON 契约字段（§7），若内核只回 hits，则打印壳和未来 web/mcp
  端点得各自二次扫一遍页数。让内核一次算清、所有消费侧从同一个 `SearchResult` 生成各自输出，杜绝元数据二次扫描与口径漂移。
- **决策P5.0-11（snippet 逐 token 在原文上大小写不敏感定位，不复用整段 casefold 下标）**：§3.2 把非 CJK token 小写化
  （`DeFi`→`defi`），故 §3.4 找表面串须大小写不敏感。**但不对整段正文 casefold 后复用下标**——`ß→ss`/`İ` 等改长字符
  若出现在命中前会致下标错位、切错窗。改为非 CJK token 用**带词边界**的 `re.search(rf"(?<![a-z0-9]){re.escape(tok)}
  (?![a-z0-9])", body, re.IGNORECASE)`、CJK token 用 `body.find(surface)`，**直接拿原文坐标**，无长度漂移。词边界**与
  tokenizer `[a-z0-9]+` 整词口径对齐**——否则 token `ai` 误命中 `said`/`chain` 内部、把窗口拽偏到非该 token 的词上。
- **决策P5.0-12（零除守卫；硬短路只有 N=0 与零-query-token，`avgdl=0` 不短路而保字段召回）**：见 §3.3。**硬短路两种**：
  `N=0`（无 content 页 → 无 corpus）返回 `hits=[], pages_searched=0`；query 无 token 回空。**`avgdl=0`（body 全空）刻意
  不短路**——否则与决策 P5.0-23"字段进召回面"冲突（body 空但 title/aliases 有 token 的页本应可字段命中）。改以把长度
  归一化比值定义为 `dl/avgdl ⟶ 0 当 avgdl=0`（归一化因子退化为稳定有限值 `1−b`），BM25 照常用 `tf`+`idf` 打分、绝不除零；
  连字段也无 token 时自然回空(真无命中、非短路)。`pages_searched` 始终按 content 页如实计数（N=0 时为 0）。新建/早期库常见，必须不崩。
- **决策P5.0-13（轻量 frontmatter 行扫，绕开 PyYAML；不冒充 `load_page` 归口）**：见 §3.6(a)。body 走 `split_frontmatter`
  零 YAML；title/aliases 行扫、最大努力可近似（漏则退化）。砍掉全库扫描的 PyYAML 隐藏大头。**字段进召回面但 snippet
  只取 body**（决策 P5.0-23），行扫近似至多略改召回、非正确性问题（正确性归口仍 `check`），故**有意**与 `check`/`graph`
  的严格/容错档分立、search 专用。
- **决策P5.0-14（长驻进程内 mtime memo；进程内不落盘，冷算恒权威）**：见 §3.6(b)。`CorpusCache` 按 `(mtime_ns,size)` 只
  重建变更页，仅 web/mcp 受益、CLI 不用。**进程内、不落盘**——不破"无盘上派生"红线（决策 P5.0-1）；与冷算 `search_pages`
  字节等价（§3.5），不破确定性；罕见同-tick-同-size 漏更新由"冷算/`--no-cache` 恒为等价权威"兜底。**绝不**升级为盘上索引
  （那留给 E1）。
- **决策P5.0-15（`limit ≥ 1` 校验归口在内核 `score`，非仅 CLI）**：`score`（及透传的 `search_pages`）对 `limit < 1`
  **raise `ValueError`**——web/mcp 直接复用内核，不能靠 CLI 的 `positive_int` 兜底，否则 `limit=0`/负数落到切片会静默给空
  结果或 `[:-1]` 类错。CLI 的 `positive_int`（argparse）是**第一道门**、`run_search`/`search_entrypoint` 捕获 `ValueError`
  → `EXIT_USAGE`、web/mcp 端点捕获 → 422（与 P4.9-18 的"权威校验在核心 / CLI 友好早提示"同纪律）。
- **决策P5.0-16（页面读取沿用 `errors="replace"` 容错，绕开 `load_page` 不丢容错）**：见 §3.1。`build_doc` 必须
  `read_text(encoding="utf-8", errors="replace")`——绕开 `load_page` 只为避开 PyYAML，**不**连它的非 UTF-8 容错一起丢；
  退回裸 `read_text()` 会让 GBK/Latin-1 页崩在 `UnicodeDecodeError`，破"绝不抛"。
- **决策P5.0-17（字段 boost 只抬 `tf`、不进 `dl`/`avgdl`；BM25F-lite，非词袋重复）**：见 §3.3。朴素的"title token ×3、
  alias ×2 重复灌进词袋并以袋长为 `dl`"会撑大 `dl/avgdl`，经 `b=0.75` 长度归一化**反向压低本页 body 词得分**——别名多的
  核心实体页反被边缘页压下,与初衷相反(BM25F 分字段饱和的动机)。故 `tf = body_cnt + 3·title_cnt + 2·alias_cnt`,而
  `dl = len(body_tokens)` 仅 body、`avgdl` 按 body 长度均值——boost 抬 `tf`（参与得分、扩召回面，决策 P5.0-23）但不进
  `dl`、无长度归一化副作用。
- **决策P5.0-18(钉死 CJK 码点谓词;归口必须穷尽)**:见 §3.2。"确定性单一归口"不能立在未定义的"什么算 CJK"上。
  采**最小够用集**:统一表意基本区 `一–鿿` + 扩展 A `㐀–䶿` + 兼容表意 `豈–﫿`;**不含**假名/全角/Bopomofo,扩展 B+
  星文字按需不纳入(命中极罕见)。非 CJK 段 `[a-z0-9]+` 不含重音拉丁(`café`→`caf`),正规化属 E1、本稿不做。
- **决策P5.0-19(`CorpusCache` 线程安全)**:见 §3.6(b)。web 同步端点跑 threadpool,并发 `corpus()` 改同一 memo dict 会
  竞态/双重 build;`corpus()` 全程持 `threading.Lock`(或上层串行化)。锁只护字典,不改确定性。CLI 单线程无此问题。
- **决策P5.0-20(分数口径钉死;CLI 补零、JSON 只承诺数值)**:见 §3.5。排序与呈现都先 `round(score,6)`,但**两侧形态有别**:
  **CLI 文本**固定 `f"{…:.6f}"` 6 位定点补零(`8.4`→`8.400000`);**JSON `score` 保持 number**、序列化 `round(…,6)`,
  但 `json.dumps(round(8.4,6))` **输出 `8.4` 不补零**——故 JSON **只承诺"数值 round 到 6 位"**,**不**承诺 `8.400000`
  零填充字节(下游按数值比较、勿按字符串比);要尾零字节形态读 CLI 文本。两侧都字节稳定(同 float 经 repr 恒定)。
- **决策P5.0-21(`--json` 出错也吐 `{"ok":false,"error":…}`)**:见 §7。带 `--json` 时错误返回结构化 `{"ok":false,...}`
  + `EXIT_USAGE`,不静默、不混吐半个对象;无 `--json` 走 stderr 纯文本 + 同退出码。下游一律先看 `ok` 再取字段。
- **决策P5.0-22(`page` 字段口径 = 相对库根 posix,`root` 由调用方传入)**:见 §2。`build_doc(path, *, root)` 以
  `path.relative_to(root).as_posix()` 生成 `page`,`build_corpus(wiki)` 传 `root=wiki.parent`——**不**用 cwd/绝对路径。
  否则同一页在不同工作目录下 `page` 串会漂移、与 `graph.Node.path`(`wiki/entities/DeFi.md`)口径不一致,污染 JSON/下游 join。
- **决策P5.0-23(字段进召回面、参与 BM25 得分,不止排序;snippet 只取 body)**:title/aliases 的 token 计入 `tf`(决策 P5.0-17),
  故"纯标题/纯别名命中、正文不含 query"的页**得分 >0、会被召回**——字段影响的是**召回面**、非仅排序(早先"只影响排序"的
  表述不准,已据此订正 §3.3/§3.6/决策 P5.0-13/17)。但 **snippet 仅从 body 取**:字段-only 命中时 body 无表面串、退化取
  正文首窗(§3.4)。轻量行扫(决策 P5.0-13)的近似性独立于此——它只是"漏字段→略降该页召回"的容错,不改本条召回语义。
- **决策P5.0-24(snippet 只取 body;body 空时 `""` 是合法 snippet,不回填字段)**:见 §3.4。snippet **永远只从 body 切**,
  **绝不**用标题/别名串回填。故字段-only 命中且 **body 非空**→正文首窗;**body 为空**(`avgdl=0` 那类页)→首窗即 `""`,
  这是**合法结果**而非错误。测试**不得断言 snippet"非空"**(与 body-空场景自相矛盾),只断言不报错、snippet 内不含标题/别名串。

## 5. 与 query / skill 的集成（加法、可优雅降级）

把"召回"从"Agent 在 prompt 里凭 2-gram 扫 index.md"升级为"先跑确定性 `guanlan search`"：

- **`skills/guanlan-wiki/SKILL.md`** query 工作流 step 1 增补：
  > 1. **先 `guanlan search "<关键词>"` 拿 top-N 候选页路径**（确定性整页召回，CJK 走 2-gram、别名已纳入匹配面）；
  >    读这些候选页 + `wiki/index.md` 综合。`search` 空手而回（生僻/跨义）时，**退回**原路径：扫 index.md /
  >    相关目录 / 现有页 `aliases`，或请用户补关键词（graceful fallback，原行为保留）。
- **`guanlan/query.py`** `QUERY_PROMPT` 同步：把"先读 `index.md`…2-gram 粗召回"改述为"先用 `guanlan search` 召回候选页，
  不中再扫目录/补词"。**仅改召回措辞**，综合 / `[[引用]]` / `--backfill` 门禁逐字不动（决策 P5.0-8）。
- **下游复用（本阶段不实现，仅预留）**：未来 `web` 的 `/api/search` 端点、`mcp` 的 `search` 工具直接复用
  `search_pages()` 内核（仿 web 复用 `build_and_write_graph`）——故内核与打印壳必须分立（§2）。

## 6. 边界（明确不做）

- **不做向量 / 嵌入 / 重排**——E1「另文选型」（决策 P5.0-4）；
- **不做盘上持久化 / 增量索引**——默认无状态即时计算；`CorpusCache` 仅**进程内** memo、不落盘（决策 P5.0-1 / P5.0-14）；
- **不用 PyYAML 解析正文/字段**——body 走 `split_frontmatter`、字段走轻量行扫（决策 P5.0-13）；frontmatter 正确性归口仍 `check`；
- **不上分词器 / 同义词表**——逐级增强次序见 backlog（决策 P5.0-7）；
- **不做多租户 / 权限 / source 隔离**——E1（DESIGN §8）；
- **不改 query 的 Agentao 综合、引用与回填门禁**——search 只是召回前端（决策 P5.0-8）；
- **不索引 config 页、不写盘、不碰 `raw/`/`log.md`、不起 LLM**（§3.1 / 决策 P5.0-9）。

## 7. JSON 契约 / 退出码

打印壳与 JSON 都从 `search_pages` 返回的同一个 `SearchResult` 渲染：`query`/`pages_searched` 取自 `SearchResult`
字段、`results` 由 `hits` 序列化（决策 P5.0-10，无二次扫描）。

```jsonc
{ "ok": true,
  "query": "去中心化金融",       // SearchResult.query 回显
  "pages_searched": 795,        // SearchResult.pages_searched（参与打分的 content 页数）
  "results": [                  // SearchResult.hits 序列化
    { "page": "wiki/entities/DeFi.md", "title": "DeFi", "type": "entity",
      "score": 8.4213, "snippet": "DeFi（去中心化金融）指建立在公链上的…" }   // number；值 = round(…,6)，JSON 不补尾零（决策 P5.0-20；CLI 文本才是 8.421300）
  ] }            // 无命中时 results: []、pages_searched 仍如实计数，ok 仍 true
```

**错误形态（决策 P5.0-21）**：`--json` 下**出错也吐 JSON**——`{"ok": false, "error": "<原因>"}`（空/纯空白 query、
非库根、IO/用法错），配 `EXIT_USAGE(1)`；不静默、不混吐半个对象。无 `--json` 时错误走 stderr 纯文本 + 同退出码。
成功恒 `{"ok": true, …}` + `EXIT_OK(0)`。下游 agent/web/mcp 可一律先看 `ok` 再取字段。

退出码：成功 `EXIT_OK(0)`；空/纯空白 query、非知识库根、IO/用法错 `EXIT_USAGE(1)`。**不引入新码。**

## 8. 测试要点（`tests/test_search.py`）

- **召回正确性**：正文含「去中心化金融」的页能被 query「去中心化」「金融」「DeFi」召回，并排在前列。
- **CJK 2-gram**：中文 query 走 2-gram 命中正文；单字 query（如「李」）退化 1-gram 仍召回。
- **字段加权**：标题/别名命中的页排序高于仅正文边缘提及的页（TITLE_BOOST/ALIAS_BOOST 生效）。
- **字段加权不被长度归一化反噬**（决策 P5.0-17）：构造一页**标题/别名很长**（boost token 远多于 body）但正文确含 query
  词——它**不应**因 `dl` 被撑大而排到一个标题短、正文同样命中的页之后；即 `dl` 只按 body 算（验证 boost 进 `tf` 不进 `dl`）。
- **CJK 谓词边界**（决策 P5.0-18）：基本区/扩展 A/兼容表意字按 2-gram 切；假名/全角符号**不**进 CJK 段（按非 CJK 逻辑走，
  标点被丢）；`café` 切出 `caf`（重音字符不入 `[a-z0-9]`）。tokenize 对同一输入跨调用字节一致。
- **别名匹配面**：frontmatter `aliases: [大模型, LLM]` 的页能被「大模型」召回。
- **字段进召回面、snippet 只取 body**（决策 P5.0-23 / P5.0-24）：构造一页 query 词**仅在标题/别名、正文完全不含**——该页
  仍被召回（得分 >0），`snippet` 退化为**正文首窗**：body **非空**时是正文开头窗、body **为空**时是 `""`——两种都合法，
  断言**不报错、且 snippet 里不出现标题/别名串**（**不**断言"非空"）；正文也含 query 的页 snippet 才切到命中窗。
- **`page` 字段口径**（决策 P5.0-22）：从不同 cwd 跑同一库，`results[].page` 恒为相对库根 posix（`wiki/entities/DeFi.md`），
  与 `graph.json` 的 `Node.path` 逐字一致；`build_doc(path, root=wiki.parent)` 不受绝对路径/工作目录影响。
- **确定性 / 幂等**：同 query 连跑两次输出字节一致；近似分数靠 path tie-break 稳定排序。
- **容错**：坏/缺 frontmatter 页照常按 body + stem 索引、title 退化用 stem，不抛；**非 UTF-8 页（GBK/Latin-1）用
  `errors="replace"` 兜底、不崩 `UnicodeDecodeError`**（决策 P5.0-16：`build_doc` 读取沿用容错口径，绕开 `load_page` 不丢）。
- **轻量 frontmatter 解析**（决策 P5.0-13）：常见块标量 `title:` 与块/flow 列表 `aliases:` 行扫结果与 `load_page` 的
  meta **在常见样式上一致**；花式 YAML（多行标量/锚点）扫不全则退化（title→stem、aliases→空），**不抛、不串其它字段**；
  body 不经 YAML（验证 `search.build_doc` 不调 `yaml.safe_load`，例如注入坏 YAML frontmatter 仍正常按 body 索引）。
- **进程内 mtime memo**（决策 P5.0-14）：`CorpusCache` 命中 `(mtime_ns,size)` 复用、改一页后该页重建且**输出与冷算
  `search_pages` 字节等价**；删/增页正确反映；CLI 冷算路径与带 cache 路径同 query 结果一致。
- **config 排除**：`index.md`/`log.md`/`overview.md` 永不出现在 results。
- **边界**：空/纯空白/纯标点 query → `EXIT_USAGE`；有词但无命中 → `ok:true, results:[], EXIT_OK`；`--limit` 截断且须 ≥1。
- **`limit` 内核校验**（决策 P5.0-15）：直接调 `score`/`search_pages` 传 `limit=0`/负数 → `ValueError`（不静默给空或 `[:-1]`）；
  CLI `--limit 0` 经 `positive_int` 即 `EXIT_USAGE`（第一道门）；`run_search`/web 端点把 `ValueError` 映射成 `EXIT_USAGE`/422。
- **非 CJK 词边界**（决策 P5.0-11）：query `ai` **不**因正文有 `said`/`chain` 而把 snippet 窗口拽到那些词上（带词边界匹配，
  与 tokenizer 整词口径一致）；`DeFi` 仍能由 `defi` 整词命中。
- **片段确定性**：snippet 取最小命中下标窗口、折叠空白；纯字段命中退化取正文首窗。
- **片段大小写不敏感且下标不漂移**（决策 P5.0-11）：正文写 `DeFi`、query `defi`（或反之）时 snippet 切到含 `DeFi` 的
  真实窗口，**不**误退化成首窗；**在真实命中前放一个改长字符**（如 `ß`/`İ`）仍切对窗（验证不复用整段 casefold 下标）。
- **空语料 / 零除不崩**（决策 P5.0-12）：无 content 页 → `results:[], pages_searched:0, ok:true`。`avgdl=0`（body 全空）
  分两路验证：(i) **body 全空且无字段 token** → `results:[], pages_searched=页数, ok:true`、无除零；(ii) **body 全空但某页
  `aliases` 含 query token**（`avgdl=0` 仍成立）→ 该页**被字段命中召回**（得分 >0；body 空故 snippet 合法地为 `""`、
  不报错、不回填别名串，决策 P5.0-24），**不**被误短路成空（守 P5.0-23 / P5.0-12 / P5.0-24 的边界一致）。
- **内核元数据**（决策 P5.0-10）：`search_pages` 返回 `SearchResult`，`pages_searched` == 实际参与打分的 content 页数；
  打印壳/JSON 不二次扫描即得该值。
- **分数口径**（决策 P5.0-20）：值均 `round(score,6)`。**CLI 文本**6 位定点补零（断言 `8.4`→`8.400000`）；**JSON `score`**
  是 number 且**不补零**（断言 `json.dumps` 出 `8.4`、**非** `"8.400000"` 字符串、**非** `8.400000` 字面）。同 query 两次各自字节一致。
- **`--json` 错误形态**（决策 P5.0-21）：`--json` + 空 query → stdout 是 `{"ok":false,"error":…}`、退出 `EXIT_USAGE`、非半个对象；
  无 `--json` 的同错误走 stderr 文本、stdout 不被污染。
- **`CorpusCache` 并发**（决策 P5.0-19）：多线程并发 `corpus()` 不损坏 memo、不双重 build，结果与单线程冷算字节等价
  （可用线程池打同一 cache 实例验证）。
- **（非门禁）benchmark**（决策 P5.0-1）：对 N≈10²/10³/10⁴ 合成页测端到端延迟，作为"是否上持久化索引"的实测依据；
  仅记录、不作 CI 门禁。
- **只读保证**：跑完 `raw/` 与 `wiki/` 字节零变动、无新派生文件（无状态）。
