## 1. 目标与范围

**P4.2 = 让只读问答会话跨进程重启存活：每轮落盘，重启后能在 UI 里看到历史会话、点开续聊。** 把 P4 决策P4-8 刻意留的「仅内存、进程退出即清」升级为「内存 + 落盘、懒恢复」。

落盘**不需要自造序列化**：agentao 已有文档化的会话持久化面 `save_session` / `load_session` / `list_sessions` / `delete_session`（`agentao.embedding`），落 `<kb>/.agentao/sessions/*.json`（已 `.gitignore`、扫描只看 `*.md` 故不污染库）。P4.2 只把现有 `Conversation` / `ConversationStore`（`chat.py`）接到这组面上。

### 纳入 P4.2

- **每轮落盘 + best-effort prune**：`Conversation.turn` 成功收尾后，**先 best-effort prune 本会话旧快照、再 `save_session(..., session_id=conv.id, project_root=kb)`**（off-loop，`to_thread`）。prune 是**纯文档化的文件 I/O**（按 `session_id` 删本会话旧 `*.json`），只为「每会话留一份快照」的卫生，使一个长会话不至于用满 10 文件槽把别人挤掉——**不**是硬保证「恰好 10 个唯一会话」，也**不**依赖 agentao 私有轮转细节（§2.1 决策P4.2-1）。**prune-在-前是为不误伤他会话**：先删净旧份，save 写新份时目录不会瞬时顶到 11、触发 `_rotate_sessions` 把别的会话挤出（save-后-prune 在已满 10 时每次更新都误淘汰别人、更糟，回应 codex 轮转评审）。**失败语义从简**：prune 删不掉就跳过、照常 save；落盘任何异常只记日志、**绝不**翻掉已成功的本轮答案（off-loop、失败不毁答案）。
- **作用域过滤（库级目录可能含非 Web 会话）**：`.agentao/sessions/` 是 agentao **项目级**目录，同库被 `agentao` 交互 CLI 直接用时也会落 session（`cli/session.py:55`）。故 catalog 与 restore **只接受 `active_skills` 含 `SKILL_NAME` 的会话**、恢复时**只激活 `SKILL_NAME`**（不回放任意 `active_skills`），免得把别处的 agentao 会话捞进 Web 历史或恢复成非只读姿态（决策P4.2-6）。
- **会话 id = agentao UUID**：`Conversation.id` 由进程内自增计数器改为 agentao 的稳定 `session_id`（UUID，写进文件内容），使**前端持有的 id == 持久 `session_id` == 跨重启稳定**（决策P4.2-2）；对外**一律用 `session_id`、不用 `list_sessions` 的 `id`/`path.stem`**。
- **懒恢复（无启动缓存）**：**不**在启动期缓存 catalog（那会随 save/delete 立刻陈旧）；`GET /api/conversations` **每次即时** `list_sessions(kb)`（≤十几份文件、成本可忽略）、按 `session_id` 去重、过滤 `SKILL_NAME`。用户**打开**某历史会话时才 rebuild——`build_from_environment`（同 §4.4 只读构造）→ `agent.messages = loaded` → 激活 `SKILL_NAME` → 注册进内存（决策P4.2-3/4）。
- **`GET /api/conversations` 合并视图**：内存现存会话 ∪ 盘上即时 catalog（**按 `session_id` 去重**，内存态优先），带 `title`/`updated_at`/`live` 标记（冷条目另带 `messages`=消息数，非轮次；见决策P4.2-3）。
- **`DELETE` 级联删盘**：丢内存对象**且** `delete_session(id, kb)`。
- **开关**：默认开；`guanlan web --no-session-persist` 关（隐私/临时场景，回退到 P4 的纯内存行为）。

### 不纳入 P4.2（仍按 P4 §10 推后）

- **可写的多轮「工作会话」（逐轮 gated 写 `wiki/`）** → 推后：P4.2 持久化的仍是**只读**会话；恢复出的会话同样重走只读两点置位（决策P4.2-4），绝不因落盘而变可写。
- **LRU 淘汰 / 自定义淘汰策略 / 硬保证 N 个唯一会话** → 不做：agentao `save_session` 自带轮转（保最新 `_MAX_SESSIONS=10` **个文件**）；P4.2 靠 best-effort prune 让每会话一般只占一份文件，**尽量**少丢会话，但**不**承诺「恰好 10 个唯一会话」、**不**为此引入全局锁/私有轮转细节依赖（§2.1）。若将来必须硬保证 N 会话，再单独做持久层或推上游 API。
- **跨知识库 / 全局会话列表** → 不做：会话按 `project_root=kb` 隔离在各库的 `.agentao/sessions/` 下，天然随库走。
- **会话内容搜索 / 导出** → 推后；v1 只「列 / 开 / 删」。
- **把写作业（ingest）也落盘恢复** → 不做：ingest 无状态、一次性，无「会话」可言（决策P4-1）。

## 2. 复用的 agentao 会话面（`agentao.embedding`）

已核对 `agentao/embedding/sessions.py`，全部是**纯磁盘 I/O、文档化导出**（非内部 API）：

| 函数 | 签名（要点） | P4.2 用法 |
|------|------|------|
| `save_session` | `(messages, model, active_skills=None, session_id=None, project_root=None) -> (path, session_id)` | 每轮收尾保存；传 `session_id=conv.id`（我们的 UUID，写进文件内容）、`project_root=kb` 落本库。**每次调用新建一份 `<时间戳>.json` 快照、非就地更新**（`sessions.py:142`，见 §2.1） |
| `load_session` | `(session_id=None, project_root=None) -> (messages, model, active_skills)`；无则 `FileNotFoundError` | 懒恢复。**⚠️ 内部是前缀匹配**：`session_id.startswith(input)` 取最新，**且无匹配时 fallback 到时间戳文件名前缀**（`sessions.py:193-203`）——故**绝不**把外部裸 id 直接喂它，必先校验为规范全 UUID（§2.2） |
| `list_sessions` | `(project_root=None) -> List[Dict]`（newest first；每条 `id`=文件名时间戳 stem、**`session_id`=UUID**、`message_count`/`updated_at`/`active_skills`…） | 取 catalog（不构造 agent）；**对外 id 取 `session_id`、按它去重**（`sessions.py:242`，**不可**用 `id`/`path.stem`，见 §2.1）；亦用作 restore/delete 前的**精确匹配源**（§2.2） |
| `delete_session` | `(session_id, project_root=None) -> bool` | `DELETE` 端点级联删盘。**⚠️ 内部前缀匹配**：删掉**所有** `session_id.startswith(input)` 的快照文件（`sessions.py:275`）——短前缀会**跨会话误删**，故同样必先校验全 UUID + 作用域（§2.2） |

- **存储位置**：`<kb>/.agentao/sessions/<时间戳>.json`，每文件含 `{session_id, title, created_at, updated_at, model, active_skills, messages}`。`.agentao/` 已在 `.gitignore`（与 `agentao.log` 同列），扫描只看 `*.md`，**不污染库、不进 git**。
- **`created_at` 在本设计下退化为 `updated_at`（已知、不依赖）**：`save_session` 靠 `_find_created_at` 扫**同 UUID 的已存文件**取最早 `created_at`（`sessions.py:139`）；prune-在-前先删了本会话旧快照，故每次更新都找不到旧份 → `created_at` 被重置为 `now`。P4.2 catalog 只用 `updated_at`/`title`/`message_count`（§3.2），**不读 `created_at`**，故无功能影响；此处记录以免误以为它是会话创建时刻。
- **不与 `raw/` 快照交叉，且无需任何写串行**：会话落在 `.agentao/`、**不在 `raw/`** 下，`snapshot_raw` 只遍历 `raw/`，故会话落盘**不需要** P4.1 那种 `raw/` 快照串行（投喂经 `JobQueue` 串行是为护 `run_guarded_write` 的快照窗口——这里没有该窗口）。**也不需要自建写锁**：prune 仅是 best-effort 卫生、不追求「文件数=会话数」的硬不变量，故并发 `_save` 最坏只是偶尔多留/少留一份快照、罕见地丢某个会话的持久副本——与「直接不 prune」同等的可接受 blast radius（§2.1）。这是与 P4.1 的本质区别：投喂要重串行（快照窗口 + 单写者 + FIFO + worker），会话写 `.agentao/` **零串行**。

### 2.1 「快照文件」语义与小适配层　**\[决策P4.2-1]**

agentao 的会话面是**快照文件**语义，不是「会话记录」语义——P4.2 围着它做**两小步**薄适配（不引 DB、不引作业模型、不引锁），否则历史列表会重影、长会话会挤掉别人：

1. **`save_session` 每轮新建一份文件**（`sessions.py:142-143`：文件名 = 当前时间戳；`session_id` 只写进**文件内容**）。一个 10 轮会话 → 10 份 `*.json`，内容里 `session_id` 同一个 UUID。**不做处理则历史列表会出现同一会话的多份快照。**
2. **`_rotate_sessions` 按文件数保 10**（`sessions.py:306-310`：`_MAX_SESSIONS=10` 是**文件**上限、**非唯一会话**上限）。**不做处理则一个长会话能占满全部 10 槽、把别的会话挤掉**（这是只 dedup-on-read、不 prune 的真实缺陷：10+ 轮的单会话会把别人冲出盘，重启后只剩它）。
3. **`list_sessions` 的 `id` 是文件名 stem（时间戳）**，稳定 UUID 在 `session_id` 字段（`sessions.py:242-244`）。**不能**把 `entry["id"]` 当会话 id 暴露给前端。

**适配层（两小步）：**

- **读侧按 `session_id` 去重**：`list_sessions(kb)` 已是 newest-first，遍历时每个 `session_id` **只取首见（即最新）一条**，丢弃其余快照（并过滤 `active_skills` 不含 `SKILL_NAME` 的非 Web 会话，§3.2）。对外 `public_id = entry["session_id"]`（绝不暴露 `entry["id"]`）。这单独就消解 finding 1（重影）与 finding 3（id 混淆）。
- **写侧每轮 save *前* best-effort prune**：`_save` 先 `_prune_old_snapshots` 删掉本 `session_id` 的旧快照、再 `save_session`（`_prune_old_snapshots` 在前，§3.1）。目的**仅是卫生**——让每会话一般只占一份文件，从而缓解 finding 2（长会话不至于把别人冲出 10 文件槽）。prune **是 best-effort**：读坏 JSON 或删不掉都**跳过、照常 save**，不抛、不放弃本轮。**放在 save *前* 是为不误伤他会话**（回应 codex 轮转评审）：先删净本会话旧份，`save_session` 写新份时目录不会因「旧+新」瞬时顶到 11、触发 `_rotate_sessions` 把**别的**会话挤出 10 文件槽（save-后-prune 在已满 10 时**每次更新**都误淘汰别人、更频繁更糟）。代价是单一窗口：删旧份后若 save 失败且进程在下一轮前崩溃，丢**本**会话最近副本（仍 live 在内存、下一轮即补）——比 save-后-prune 的「频繁误伤他会话」窄得多。

**这套适配刻意不追求硬保证**：不承诺「恰好 10 个唯一会话」，不引全局锁，不依赖 `_rotate_sessions` 的私有排序/上限细节来推导不变量（那与 `chat.py`「只用文档化嵌入面、不碰内部 API」相冲突）。**代价（已知、可接受）**：① prune 留一个小窗口——更新已有会话时若在「删旧份后、写新份成功前」save 失败/**进程崩溃**，丢**该会话**最近持久副本（进程仍活则下一轮重存、不误伤他会话）；② 并发 `_save` 或单会话保留多份的边角情形下，盘上文件可能短暂 >10、触发 `_rotate_sessions` 偶尔多淘汰一个会话。二者 blast radius 都≤「丢某个会话的持久副本」，与「干脆不 prune」同量级，本地单用户下风险低。**注**：prune 顺序是 save-前 vs save-后的取舍——save-前丢的是「本会话 + 罕见崩溃窗口」，save-后丢的是「他会话 + 每次满额更新」，故取 save-前。要更强的保留保证须另上持久层（§1 不纳入）。

### 2.2 精确 id 契约：HTTP id 必须是规范全 UUID，restore/delete 前精确匹配　**\[决策P4.2-7]**

`load_session`/`delete_session` 内部是**前缀匹配**（§2 ⚠️）：`load_session` 还会在 UUID 无匹配时 fallback 到时间戳 stem 前缀（`sessions.py:200-203`），`delete_session` 删**所有** `startswith(input)` 的文件（`sessions.py:275`）。若把外部裸 `{id}` 直接转给它们，短前缀可能**恢复错会话**、**跨会话删除**，甚至删到 `active_skills` 不含 `SKILL_NAME` 的**非 Web 会话**，破坏决策P4.2-6 的作用域边界。故钉死两道闸：

1. **HTTP id 必须是规范全 UUID**：端点收到的 `{id}` 先过 `_is_canonical_uuid`（`str(uuid.UUID(id)) == id`，拒短前缀 / 时间戳 stem / 非规范写法），否则当未知 id 走 `404`。规范全 UUID 下「前缀匹配」收敛为「精确匹配」（36 字符 UUID 不会是另一个等长 UUID 的真前缀，也不会前缀命中 21 字符时间戳 stem）。
2. **restore/delete 前用即时 catalog 精确校验**：`_disk_session(id)` 扫 `list_sessions(kb)` 找 `entry["session_id"] == id` **且** `SKILL_NAME in active_skills` 的条目，命中才放行 `load_session`/`delete_session`，否则 `404`。这把「身份精确」与「作用域归属」一次性夹死，不依赖底层前缀语义恰好收敛。

二者叠加：① UUID 规范化堵住前缀/时间戳歧义，② catalog 精确匹配 + `SKILL_NAME` 堵住跨会话与越作用域。`Conversation.id` 本就是 `uuid4()`（决策P4.2-2），前端只回传它，正常路径零摩擦。

## 3. 落盘与恢复（`guanlan/web/chat.py`）

### 3.1 每轮落盘 + UUID 作 id　**\[决策P4.2-2/5]**

```python
# Conversation：id 改为 UUID（构造期生成，首个 done 即回传稳定 id）；turn 成功后落盘
class Conversation:
    def __init__(self, cid: str, kb: Path, model: str | None, *, persist: bool):
        self.id = cid                       # cid 现在是 UUID 字符串（由 Store 生成），非自增计数
        self._kb = kb; self._persist = persist
        self._model = model                 # _save 要用（save_session 的 model 参数）；恢复时 = load 出的 model
        ...                                 # 其余构造同 §4.4（只读两点置位 + 激活 skill）

    async def turn(self, msg, emit) -> str:
        async with self.lock:
            ...
            answer = await self.agent.arun(msg)
            if self._persist:               # 仅成功轮落盘；off-loop 不堵事件循环
                try:                        # 落盘任何异常（prune / save_session）只记日志，绝不让它冒泡
                    await anyio.to_thread.run_sync(self._save)
                except Exception:           # 把已成功的 arun 答案翻成 error（失败不毁答案，同 §4.4 降级精神）
                    _logger.warning("会话 %s 落盘失败，本轮未持久化", self.id, exc_info=True)  # 复用 §4.4 的 _logger
            return answer                   # 无论落盘成败，成功答案照常返回

    def _save(self):
        active = list(self.agent.skill_manager.get_active_skills().keys())   # 镜像 cli/session.py
        # 先 prune 本会话旧份、再 save。prune-在-前是为不误伤他会话：先删净旧份，save 写新份时
        # 目录不会因「旧+新」瞬时顶到 11、触发 _rotate_sessions 把别的会话挤出（save-后-prune 在
        # 已满 10 时每次更新都误淘汰别人，更糟，回应 codex 轮转评审，§2.1）。prune 失败不阻断 save。
        _prune_old_snapshots(self._kb, self.id)
        save_session(self.agent.messages, self._model, active_skills=active,
                     session_id=self.id, project_root=self._kb)

# 小适配层：best-effort 删掉本 session_id 现存的旧快照，使每会话一般只留一份文件（save 前调）。
def _prune_old_snapshots(kb: Path, session_id: str) -> None:
    """纯卫生、**全程 best-effort**：读坏 JSON 跳过、删不掉也跳过（不抛、不阻断后续 save）。
    它只为「长会话不挤占别人」的保留质量，不是正确性前提——故不引「删不掉就别保存」那类硬语义。"""
    sessions = kb / ".agentao" / "sessions"
    if not sessions.exists():
        return
    for p in sessions.glob("*.json"):
        try:
            with open(p, encoding="utf-8") as f:
                sid = json.load(f).get("session_id")
            if sid == session_id:
                p.unlink()
        except (OSError, json.JSONDecodeError):
            continue                       # 读坏 / 删失败：跳过，best-effort
```

- **id 为何改 UUID**：P4 用进程内自增计数器（`"1"`,`"2"`…）当 `Conversation.id`——重启后计数归零、与盘上 UUID 对不上，无法续。改用 agentao 的 UUID：`ConversationStore.create` 生成 `uuid4()` 当 id，构造期即定、首个 `done` 回传给前端。`save_session(session_id=conv.id)` 把这个 UUID 写进**每份快照**的 `session_id` 字段（**每轮新建一份快照、非就地覆写**，§2.1）；`load_session`/`delete_session` 都按该 UUID 前缀匹配，故所有读/删/续都归口到同一稳定 id。前端体验不变（仍是一个不透明 id 串）。
- **轮转上限（best-effort，非硬保证）**：`_rotate_sessions` 保最新 **10 个文件**（`_MAX_SESSIONS`，**按文件数、非按会话数**，§2.1 finding 2）。`_save` 的 best-effort prune 让每会话一般只占一份文件，故**通常**能保留约 10 个唯一会话——但**不**承诺恰好 10 个：并发/读坏/删失败的边角下盘上文件可短暂 >10、`_rotate_sessions` 偶尔多淘汰一个会话。内存上限 `MAX_CONVERSATIONS=100` 仍独立管在线会话（决策P4.2-1）。
- **内存 100 vs 盘 ~10 的悬崖（已知、v1 接受）**：两上限差 10×，故 `live=true` **不保证**可恢复——在线会话 >10 时，rotate 会挤掉最旧会话的盘文件，那个会话仍 live 但已无持久副本、重启即丢。v1 不调和此差（不抬 `_MAX_SESSIONS`、不压 `MAX_CONVERSATIONS`）、前端不就此做特殊提示。
- **save 失败不毁答案**：`_save` 的调用点（`turn`）包 `try/except`（同 §4.4「渲染失败仅降级」精神）——prune 或 `save_session` 的任何异常都只记日志，不把已成功的 `arun` 答案翻成 `error`。prune 内部本就 best-effort（删不掉跳过、不抛）；若 `save_session` 自身失败，本轮无持久副本，**进程仍活时**下一轮成功 save 即补上。真正跨重启丢失只在「prune 删旧份后、`save_session` 写新份成功前，save 失败/进程崩溃、内存随之消失」的极小窗口（§2.1 代价①，取 save-前-prune 以不误伤他会话）。

### 3.2 懒恢复　**\[决策P4.2-3/4]**

启动**不**批量重建 agent（每个都要构造 LLM client，10 个 = 启动期长阻塞且多半用不上）。改为：

- **catalog 即时算、不缓存**：**不**在启动期构建 `dict[session_id→entry]`（save/delete/rotate 后立刻陈旧、又无同步规约）。改为 `GET /api/conversations` **每次**直接 `list_sessions(kb)`（≤十几份文件、可忽略），现场**按 `session_id` 去重**（newest-first 取首见=最新）、**过滤 `active_skills` 不含 `SKILL_NAME` 的会话**（库级目录可能含 `agentao` CLI 留下的非 Web 会话，§1/决策P4.2-6）；**不**构造任何 agent、**不**暴露 `entry["id"]`（时间戳文件名，§2.1）。每条对外 `{id=session_id, title, updated_at, messages=message_count}`——**冷条目报 `messages`（消息总数）不报 `turns`**（`list_sessions` 给的是 `message_count`，非轮次；真实轮次只在 live/恢复后由 user 消息数算出，§4/决策P4.2-3）。
- **按需 rebuild**：当 `POST /api/chat` 带一个**内存里没有、但盘上有**的 id（或前端显式 `打开`）时，触发 `ConversationStore.restore(id)`。**restore 是慢路径**（`load_session` + 构造 agent），端点**必须**像 `create` 一样 `await anyio.to_thread.run_sync(...)` 卸到线程池（决策P4.2-3）——否则在 async route 里同步调用会**阻塞整个 ASGI 事件循环**（恢复一个冷会话期间所有请求卡住）。restore 内部复用 `create` 的并发骨架（持 `self._lock`、查 `MAX_CONVERSATIONS`、锁内 double-check `_convs`），并在 load 前过 §2.2 的**两道精确闸**：
  ```python
  import uuid as _uuid
  def _is_canonical_uuid(s) -> bool:                       # §2.2 闸①：拒短前缀 / 时间戳 stem / 非规范写法
      try:
          return isinstance(s, str) and str(_uuid.UUID(s)) == s
      except (ValueError, AttributeError, TypeError):
          return False

  class ConversationStore:
      def _disk_session(self, cid: str) -> dict | None:    # §2.2 闸②：即时 catalog 精确匹配源
          for e in list_sessions(self._kb):                # newest-first；本地单用户 ≤十几份、可忽略
              if e.get("session_id") == cid and SKILL_NAME in (e.get("active_skills") or []):
                  return e                                 # session_id 全等 + 属 Web 只读会话，才认
          return None

      def restore(self, cid: str) -> Conversation | None:  # 端点经 anyio.to_thread.run_sync 调用（off-loop，慢路径）
          if not self._persist:                            # 持久化关时不读盘（破坏「--no-session-persist 等价纯内存」，§4）
              return None
          if not _is_canonical_uuid(cid):                  # 闸①：使底层前缀匹配收敛为精确，杜绝前缀/stem 误命中
              return None                                  # 非规范 id → 当未知 id（404），绝不喂 load_session
          if self._disk_session(cid) is None:              # 闸②：盘上无此**精确** session_id 的 Web 会话 → 404
              return None                                  #   （含 active_skills 不含 SKILL_NAME 的非 Web 会话也拦在外）
          try:                                             # catalog 与文件间有竞态：命中后文件可能刚被删/轮转/读坏
              messages, model, _ = load_session(cid, project_root=self._kb)  # cid 全 UUID + 已确认存在，前缀==精确
          except (FileNotFoundError, OSError, json.JSONDecodeError):
              return None                                  # 竞态/坏文件 → 当未知 id（404），**绝不**冒泡成流式 error
          with self._lock:                                 # 同 create：构造慢但本地单用户罕见，换无并发绕过
              existing = self._convs.get(cid)              # double-check：并发已 rebuild 同一 id → 复用、丢弃本次
              if existing is not None:
                  return existing
              if len(self._convs) >= MAX_CONVERSATIONS:    # 恢复同样受内存硬上限约束（否则可经 restore 越界）
                  raise RuntimeError(f"内存会话数已达上限 {MAX_CONVERSATIONS}，请先 DELETE 一些会话。")
              conv = Conversation(cid, self._kb, self._default_model, persist=self._persist)  # 不恢复盘上 model（见下）；§4.4 只读构造，已激活 SKILL_NAME
              conv.agent.messages = messages               # 镜像 cli/commands/sessions.py:121
              # 只认构造已激活的 SKILL_NAME，**不**回放盘上任意 active_skills（扩大姿态，决策P4.2-4/6）
              conv.turns = len([m for m in messages if m.get("role") == "user"])        # 还原轮次显示
              conv.title = (next((m.get("content") for m in messages                    # 回填标题：否则 live 视图（内存优先合并）
                                  if m.get("role") == "user"), "") or "")[:50] or "（空）"  #   会把刚续聊的历史会话标题显示成空
              self._convs[cid] = conv
              return conv
  ```
  - **精确 id（§2.2/决策P4.2-7）**：`_is_canonical_uuid` + `_disk_session` 两闸先行，杜绝 `load_session` 前缀/时间戳 fallback 误命中错会话；`SKILL_NAME` 校验同时把非 Web 会话拦在恢复之外（作用域边界，决策P4.2-6）。
  - **off-loop（决策P4.2-3）**：`restore` 内含 `load_session`（磁盘）+ `Conversation` 构造（LLM client），端点务必 `to_thread`，不可在 async route 同步调。
  - **catalog/文件竞态归 None（→404）**：`_disk_session` 命中与 `load_session` 之间，目标文件可能刚被 `DELETE`/`_rotate_sessions` 删掉或写坏（catalog 是即时快照、非锁定）。`load_session` 的 `FileNotFoundError`/`OSError`/`json.JSONDecodeError` 一律吞为 `None`，让端点据此回 `404`（未知会话），**绝不**让磁盘异常冒泡成流式 `error`——恢复出错与「id 不存在」对用户是同一回事。
  - **不恢复盘上 model（镜像 agentao PR #81 "don't restore persisted model on resume"）**：快照只存 model **名**、不存其 provider（`api_key`/`base_url` 从不落盘）。把这个名字重新绑到当前进程恰好在用的 provider 上，会得到不一致的 `(provider, model)` 对——在 provider A 存的模型在当前 provider B 上可能根本不存在，只在下次 LLM 调用时才炸。故 `restore` **一律用与新建同一**的当前进程模型（`self._default_model`，= `--model` / 环境发现），盘上的 `_model` 仅作参考、不回绑。这顺带免掉了早先「`load_session` 缺失 model 返回 `""` 会被当真值塞进 `opts["model"]=""`」的坑（`self._default_model` 本就是 `None` 或合法 model 串，不会是 `""`）。
  - **`conv.title` 回填**：§4 的 `GET /api/conversations` 是「内存态优先」合并，恢复后会话进内存 → `list()` 取内存对象的 `title`；不回填则刚点开续聊的历史会话标题变空。
- **`POST /api/chat` 的 id 处理升级**：内存命中 → 直接用；内存未命中 → `await anyio.to_thread.run_sync(conversations.restore, id)`（off-loop）：返回 conv 则续聊，返回 `None`（非规范 id / 盘上无 / 持久化关）→ `404`；`restore` 抛 `RuntimeError`（内存满）→ `503`（同 `create`）。新建（无 id）路径不变。
- **恢复 = 同一构造路径**：rebuild 走 `Conversation.__init__` 的**同一**只读构造（`permission_engine.set_mode(READ_ONLY)` **且** `tool_runner.set_readonly_mode(True)` + 激活 skill），只是多注入 `agent.messages`。**绝不**因恢复而绕过只读两点置位——否则会「恢复出一个可写会话」，破坏 §8 安全姿态。为此把 §4.4 的「构造只读 agent」抽成 `Conversation` 内一个私有 helper，新建与恢复共用，杜绝两处漂移。

### 3.3 删除级联　**\[决策P4.2-5]**

`DELETE /api/conversations/{id}` **按内存命中与否分两支**（关键：勿把 `_disk_session` 闸加到内存命中支上，否则一个 live 但盘上已被 rotate 掉的会话会删不掉、`新会话` 的 best-effort DELETE 也会失效）：

- **内存命中**（`get(id)` 有对象）：先取会话锁待在飞 turn 收尾（同 §4.4）→ 丢内存对象 → **best-effort** `delete_session(id, kb)` 删盘文件（删不到/无盘文件都不报错——盘上副本可能早被 rotate 掉，§3.1 悬崖）。此支**只隐含要求 id 是规范 UUID**（内存 key 本就是 `uuid4()`，能命中即规范，决策P4.2-2），**不**过 `_disk_session`——它属 Web 会话已无疑。返回 `{deleted:id}`。
- **仅在盘上**（内存无对象）：必须 `_is_canonical_uuid(id)` **且** `_disk_session(id)` 命中（精确 `session_id==id` + 含 `SKILL_NAME`）才 `delete_session(id, kb)`，否则 `404`——这才是防 `delete_session` 前缀误删一串/删到 `agentao` CLI 非 Web 会话的闸（§2.2，`sessions.py:275`）。

**`persist=false`（`--no-session-persist`）时盘整个不碰**，两支都收窄（与 §4「关时 DELETE 不碰盘」对齐，勿只看本节写出不一致行为）：内存命中支**只删内存、不调 `delete_session`**；disk-only 支**一律 `404`、不读 catalog（`_disk_session`）、不碰盘**（关时启动就没读 catalog、盘上即便有遗留快照也不归本进程管，等价 P4 纯内存）。

前端 `新会话` 删的是当前 live 会话（走内存命中支），故即便其盘文件已被 rotate 掉（或 `persist=false` 根本没落盘）也能清掉内存对象、不留孤儿。

## 4. 端点契约增量（`guanlan/web/app.py`）

| 方法 路径 | 变化 | 返回 |
|------|------|------|
| `GET /api/conversations` | **即时** `list_sessions(kb)`（不缓存），**合并**内存 ∪ 盘上（**按 `session_id` 去重**、**过滤 `active_skills` 含 `SKILL_NAME`**，内存态优先） | `{conversations:[{id,title,updated_at,live, turns?|messages?}]}`（`id`=`session_id` UUID、**非** `path.stem`；`live`=是否已在内存；live 条目带真实 `turns`，冷条目带 `messages`=消息数，§3.2/决策P4.2-3） |
| `POST /api/chat` | 未知 id 时 `await anyio.to_thread.run_sync(restore, id)`（**off-loop**，§3.2/决策P4.2-3）：返回 conv 续聊；`None`（**非规范 UUID** / 盘上无此 Web 会话 / 持久化关 / **catalog-文件竞态读不出**）→ `404`；`RuntimeError`（内存满）→ `503`（同 `create`） | 流式不变（§4.4）；恢复成功后续聊、`done.conversation_id` 仍为该 id |
| `DELETE /api/conversations/{id}` | **内存命中**：丢对象 + best-effort `delete_session`（不强求盘上仍有副本——live 会话盘文件可能已 rotate 掉，§3.1）；**仅在盘上**：须规范全 UUID + `_disk_session` 精确/作用域校验，否则 `404`（§3.3/§2.2/决策P4.2-7） | `{deleted:id}`（仅在盘上且非规范/不命中 → `404`） |

- **无新退出码 / 无新流式契约**：P4.2 不动 SSE（仍 `token`/`done`/`error`），不动 ingest/报告端点。
- **`--no-session-persist`**：CLI 新增（同 `--no-agent-log` 风格），透传给 `ConversationStore(persist=...)`；关时 `create` 仍给 UUID（前端契约一致），但 `turn` 不落盘、启动不读 catalog、`DELETE` 不碰盘——完全等价于 P4 的纯内存行为。

## 6. 决策记录

- **\[决策P4.2-1] 复用 agentao 文档化会话面 + 一层**薄而 best-effort** 的适配（按 `session_id` 去重 + 留一份快照），不自造序列化、不引 DB、不引锁、不依赖私有轮转细节。** `save_session`/`load_session`/`list_sessions`/`delete_session` 是 `agentao.embedding` 的纯磁盘 I/O 导出（非内部 API，与 §4.4「只碰文档化嵌入面」一致），落 `<kb>/.agentao/sessions/`（已 gitignore、不扫描）。它是**快照文件**语义而非「会话记录」语义（§2.1：每轮新建文件、`list_sessions` 的 `id` 是时间戳）。P4.2 只加**两小步**适配——**读侧按 `session_id` 去重并以 `session_id` 为对外 id**（消解重影/混淆），**写侧每轮 save 前 best-effort prune 本会话旧快照**（卫生：让长会话不至于用满 10 文件槽把别人挤掉；prune 在 save 之前是为不误伤他会话——先删净旧份，save 时目录不会瞬时顶到 11 触发轮转挤掉别人，回应 codex 轮转评审）。**刻意不追求硬保证**：不承诺「恰好 N 个唯一会话」，不引全局锁、不靠 `_rotate_sessions` 的私有排序/上限来论证不变量（那与「只用文档化面、不碰内部 API」相冲突，也是上一稿过度设计之处）。代价是边角情形偶尔多/少留一份快照，与「干脆不 prune」同量级、本地单用户可接受（§2.1）。内存上限 100 仍独立管在线会话；不另写 LRU（§1 不纳入）。
- **\[决策P4.2-2] 会话 id 改用 agentao UUID（=文件内 `session_id`），替换进程内自增计数器；对外只暴露它、绝不暴露 `path.stem`。** 计数器重启归零、与盘上对不上，无法续。UUID 在 `create` 时生成、构造期即定、首个 `done` 回传；`save_session(session_id=conv.id)` 把它写进每份快照的 `session_id`，`load_session`/`delete_session` 按它前缀匹配（取最新 / 删全部快照），故读/删/续全归口同一稳定 id。**注意 `list_sessions` 另有一个 `id`=文件名 stem——那是快照标识、非会话标识，对外一律用 `session_id`**（§2.1 finding 3）。这是落盘恢复能成立的前提，故列为独立决策。
- **\[决策P4.2-3] 懒恢复 + catalog 即时算（不启动缓存）+ restore off-loop：列表每次 `list_sessions`，打开会话时才 rebuild agent，且恢复走 `to_thread`。** 批量重建 ≤10 个 agent 会拖慢启动且多半用不上；改为 `GET /api/conversations` **每次**即时 `list_sessions(kb)`（≤十几份文件、可忽略），`POST /api/chat` 命中盘上 id 时才 `restore`（构造 + 注入 `messages`）。**不**在启动期缓存一张 catalog——那会随 save/delete/rotate 立刻陈旧、又无同步规约。`restore` 含 `load_session`（磁盘）+ agent 构造（LLM client）两个慢操作，端点**必须** `await anyio.to_thread.run_sync(restore, id)` 卸到线程池（与 `create` 同，`app.py:241`），**不可**在 async route 同步调用——否则恢复一个冷会话会阻塞整个 ASGI 事件循环。**冷条目报 `messages`（`message_count`）不报 `turns`**：`list_sessions` 给的是消息总数、非轮次，真实轮次只在 live/恢复后由 user 消息数算出（§4 finding 4）。轻启动、按需付费、无陈旧缓存、不阻塞事件循环。
- **\[决策P4.2-4] 恢复走与新建同一只读构造路径，只激活 `SKILL_NAME` + 两点只读置位。** 恢复出的会话**必须**重走 `permission_engine.set_mode(READ_ONLY)` **且** `tool_runner.set_readonly_mode(True)`（§8/决策P4-8 的「两点同步」），否则会恢复出可写会话、破坏只读纵深防御。把「构造只读 agent」抽成共用 helper，新建/恢复零漂移。`agent.messages = loaded` 镜像 agentao `cli/commands/sessions.py` 的 resume 路径；但 skill **只认共用 helper 已激活的 `SKILL_NAME`、不回放盘上任意 `active_skills`**——回放他处会话的 skill 会扩大只读问答的姿态，与作用域过滤（决策P4.2-6）同一意图。
- **\[决策P4.2-5] 每轮成功后落盘、`DELETE` 级联删盘；默认开、`--no-session-persist` 关。** 每轮落盘 off-loop、**失败不毁答案**（prune/save 任何异常只记日志）；**prune 在 save 前**（先删本会话旧份再写新份），以免 save 时目录瞬时顶到 11 触发 `_rotate_sessions` 把别的会话挤出（save-后-prune 更频繁误伤他会话，回应 codex 轮转评审，§2.1）。prune 本身 best-effort（删不掉跳过、不阻断 save）；真正丢失只在「删旧份后、写新份成功前 save 失败/崩溃」的极小窗口，丢**本**会话副本、不误伤他会话（§2.1 代价①）。删除同时清内存与盘，避免孤儿会话占满轮转上限。默认开（与 CLI 持久会话一致、本地单用户自己的机器）；`--no-session-persist` 给隐私/临时场景一个等价于 P4 纯内存的退路。**会话落 `.agentao/`、不碰 `raw/`，故既无需 P4.1 的 `raw/` 快照串行（`JobQueue`）、也无需任何自建写锁**（prune 是 best-effort 卫生，不追求需串行护持的硬不变量，§2.1）。
- **\[决策P4.2-6] catalog 与 restore 限定 `active_skills` 含 `SKILL_NAME`，恢复只激活 `SKILL_NAME`。** `.agentao/sessions/` 是 agentao **项目级**目录：同一 `kb` 被 `agentao` 交互 CLI 直接用时也会落 session（`cli/session.py:55` 的退出落盘），它们 `active_skills` 不含 `guanlan-wiki`。若不过滤，`GET /api/conversations` 会把这些非 Web 会话列进历史、`restore` 会按其 `active_skills` 把无关 skill 回放进只读问答（既越界又可能恢复成非预期姿态）。故**读侧 catalog 过滤掉 `active_skills` 不含 `SKILL_NAME` 的条目**、**恢复侧只认构造已激活的 `SKILL_NAME`、不回放盘上任意 `active_skills`**（与决策P4.2-4 的「只读姿态零漂移」同一意图）。这是 P4.2 把库级共享目录收窄到「只管自己写的 Web 只读会话」的作用域边界。
- **\[决策P4.2-7] HTTP id 必须是规范全 UUID，restore/delete 前用即时 catalog 精确匹配（不直接把裸 id 交给前缀匹配的 `load_session`/`delete_session`）。** agentao 的 `load_session`（`sessions.py:193-203`）按 `session_id.startswith(input)` 匹配、无果再 fallback 到时间戳 stem 前缀；`delete_session`（`sessions.py:275`）删**所有** `startswith(input)` 的文件。直接转外部裸 `{id}` 会让短前缀**恢复错会话 / 跨会话删除 / 删到非 Web 会话**，破坏决策P4.2-6 边界。故两道闸：① 端点先 `_is_canonical_uuid`（`str(uuid.UUID(id))==id`）拒短前缀/时间戳 stem/非规范写法，使前缀匹配收敛为精确；② **盘上读/删前** `_disk_session(id)` 扫即时 `list_sessions` 精确匹配 `session_id==id` 且含 `SKILL_NAME` 才放行，否则 `404`。**唯一例外是 DELETE 的内存命中支**：live 对象已确属 Web 会话、其 id 即自身 `uuid4()`，仅需「能命中内存（即规范）+ best-effort 删盘」，**不**过 `_disk_session`——否则盘文件已 rotate 掉的 live 会话会删不掉（§3.3 finding）。前端只回传 `Conversation.id`（本就是 `uuid4()`，决策P4.2-2），正常路径零摩擦（§2.2）。
