# P5.2 多格式摄入 `guanlan convert`（小而硬的 CLI 补洞 · 脚本零 LLM · 解析 Markdown 当源 · 复用 pdf-to-markdown skill）

> **定位收窄（反向评审结论）**：P5.2 是一个**小而硬的 CLI 补洞阶段**——「多格式转 Markdown 并落 `raw/` 的最小
> adapter」，**不是** P5「另一半正题」式的新主流程，**也不**推翻或替代 [`P4.6-Web上传与晋级.md`](P4.6-Web上传与晋级.md)。
> P4.6 已把「上传 → 解析 → 人审 → 晋级为源 → 立即 ingest」的 Web 链路**端到端跑通**（`workspace/{uploads,parsed}/`、
> `POST /api/raw {source}`、预览/修订/晋级），其浏览器内人审在严格性上还**强于** CLI 侧（`workspace/parsed/` 池内预览、拆分、合并、修订后再晋级）。
> P5.2 的**唯一不可替代点**就是把这条能力补到**命令行**：当前 `ingest` 仍硬拒非 `.md`（`ingest.py:46` 错误文案
> 「多格式摄入属 P5」），CLI 用户没有一条官方的 `PDF/DOCX → raw/*.md → ingest` 路径，只能手动跑 skill / 外部工具
> 再自己落 `raw/`。`guanlan convert` 正补这个洞——**别无其它新价值主张**。
>
> 做法：把 PDF/DOCX/PPTX/XLSX/HTML/图片… 经**既有 `pdf-to-markdown` skill 的 `convert.py`**（MinerU→marker→pypdf
> 分层兜底，已随 wheel 全局安装）转成 markdown、落成 `raw/` 下的 `.md` 源，**复用既有 `.md` 单格式 ingest 不动**，
> **复用 P4.6 既有的 raw 写入规则**（§2.2 把它抽成 transport-neutral `rawio` 归口）。这把 `pdf-to-markdown` skill 从
> 「P4.6 Agent-only 路径」升格为 **Agent（P4.6）与 CLI（P5.2）共用的多格式→md 单一引擎归口**。
>
> **无新运行依赖、无新 extra**（决策P5.2-5）：复用 skill 既有的分层后端与 graceful degrade。**「脚本零 LLM」不变量
> 不受影响**：guanlan 自身不内嵌 LLM 客户端/密钥，只 shell out 到外部转换进程；该进程（如 marker）是否用 LLM 增强、
> 用谁的 key，由**用户环境**决定，与 guanlan 正交（决策P5.2-4）。宿主侧只需一条 **temp 暂存**约束防污染用户目录（§3.2）。
>
> **不是** E1（存储/检索升级）、**不是** E3（nightly 自动富化）。多格式**自动解析管线 / 高保真 OCR 多
> backend / nightly** 仍按 DESIGN §8 留后。**不单列 P5.3**——Web 转换接入是 P4.6 的**机会性优化**（需要时一个
> `POST /api/convert` 直调内核即可、替掉 Agent shell 解析），**MCP 明确不做 convert**（写 `raw/` 与 P4.10 只读姿态
> 相悖）；内核/命令壳分立只作零成本期权，不承诺独立阶段（决策P5.2-9）。文中 **\[决策P5.2-N]** 集中记录在 §4。
>
> **补漏半阶段 → [`P5.2.1-图片落盘.md`](P5.2.1-图片落盘.md)**：本文的 `convert_to_markdown` 内核**只取产物 `.md`
> 文本、把转换器抽出的图片连同 temp 树一并丢弃**，落进 `raw/<slug>.md` 的 `![](…)` 引用因此全部悬空。P5.2.1 让图片
> **随转换文件一并落 `raw/images/<slug>/<slug>-N.ext`**（按 md 内出现序编号）、并**重写 markdown 图片引用**，使
> `raw/<slug>.md` 自洽——引擎无关（统一按「相对产物 md 父目录解析引用」处理 mineru/marker 两种布局），仍零 LLM、仍
> 只写 `raw/`、`rawio.py` 一字不改。

## 0. 定位：补的是 CLI 那个洞，不是开新主流程

DESIGN §7 P5 交付栏白纸黑字：「多格式转换 adapter（markitdown 兜底，评估更高质量 PDF backend）；docx/web clip
等摄入」。但**这条 adapter 的 Web 那半其实已被 P4.6 吃掉**——浏览器里「上传 → 解析 → 人审 → 晋级为源 → ingest」
已端到端可用。所以 P5.2 **不是**「P5 另一半正题」，而是把这条 adapter **唯一还缺的 CLI 入口**补齐。

**P5.2 的真实不可替代点只有一个——命令行**：当前 `ingest._resolve_raw_target`（`ingest.py:46`）对非-`.md` 硬抛
「MVP ingest 只吃 `.md`（多格式摄入属 P5）」，CLI 用户**没有**一条官方的 `PDF/DOCX → raw/*.md → ingest` 路径，
只能手动跑 skill / 外部工具、再自己把 `.md` 落进 `raw/`（slug/准入/provenance 全靠手抄、易漂移）。`guanlan convert`
正补这个洞，**别无其它新价值主张**。它**不**取代 P4.6 的 Web 路径——后者更强（`workspace/parsed/` 池内预览、拆分、
合并、修订后再晋级），P5.2 也**不**是 Web 路径的前置依赖（决策P5.2-9）。

> DESIGN 当初设想的兜底后端是 markitdown；P5.2 **改为复用 P4.6 已落地的 `pdf-to-markdown` skill**（MinerU→marker→
> pypdf，本就随 wheel 全局安装、已内建分层 graceful degrade），转换逻辑单份、无新依赖、无第二实现漂移（决策P5.2-5）。

落在 **P5.2** 而非别处，有三条依据 + 一条边界判断：

- **依据①（转换引擎与 raw 写入规则都已存在，CLI 只接线，非从零起）**：P4.6 已把端到端流跑通在浏览器里——
  `pdf-to-markdown` skill（随 wheel 全局安装，MinerU→marker→pypdf 分层兜底、已内建 graceful degrade，
  `skills/pdf-to-markdown/scripts/convert.py`）+ `workspace/{uploads,parsed}/` + 「解析 Markdown 当源」模型
  （决策P4.6-2）+ raw-write 归口（slug/文本准入/`origin` provenance/原子写，`web/rawfeed.py`）。P5.2 **既不重造
  转换器、也不重写 raw 写入规则**：CLI 直接复用 skill 的 `convert.py` 当后端引擎、复用 P4.6 的 raw 写入规则（§2.2
  抽成 `rawio` 归口），自己只做「定位 skill → 子进程调用（temp 暂存、透传环境）→ 经 `rawio` 落 `raw/`」的薄宿主壳。
- **依据②（唯一拦截点单一明确）**：当前 `ingest._resolve_raw_target` 对非-`.md` 抛「MVP ingest 只吃 `.md`
  （多格式摄入属 P5）」——这是**唯一**挡 CLI 多格式的门。P5.2 不松这道门（ingest 仍只吃 `.md`），而是**在它前面加
  一步确定性转换**，把多格式变成它能吃的 `.md`。
- **依据③（与 P5.0→P5.1 同构，但不照搬其分量）**：检索那半先 CLI 内核（P5.0 `search_pages`）后 Web 接入（P5.1）。
  摄入那半同样先 CLI 转换内核（P5.2 `convert_to_markdown`）；但**与检索不同的是，Web 接入这步 P4.6 已先行交付**，
  故**不单列 P5.3**：Web 侧只剩「用确定性内核替掉 Agent shell 解析」的机会性优化（需要时再做），MCP 侧因 convert
  写 `raw/` 与只读姿态相悖而不做（决策P5.2-9）。内核与命令壳分立仍保留，纯为这个零成本期权留口。
- **边界判断**：真正属 E 轨的是**向量/重排/持久化索引**（E1）与 **nightly 自动富化**（E3）。而「把一个文件
  转成 markdown 落进 `raw/`」是**零 LLM、查询无关、一次性人背书的源创建**——落在现有零 LLM 脚本层内（与
  `init`/`check`/`search` 同性质），**且转换引擎已存在**（复用 `pdf-to-markdown` skill，无新依赖、无新 extra，
  graceful degrade 复用 skill 既有分层兜底）。故作为小补洞拆出来先做，名正言顺。

> 与 P4.6 的关系（诚实划界）：P4.6 解决「在浏览器里、Agent 不写 `raw/`」的多格式前段，且已**实现**该前段（决策
> P4.6-6 当时把「多格式自动解析管线」标为「属 P5」）。P5.2 **只补 CLI 那一段**——把同样的转换能力以**宿主侧确定性
> CLI 命令**形态提供给命令行用户，复用同一 skill 引擎与同一 raw 写入规则。它**不**接管、**不**替代、**也不**前置
> 于 P4.6 的 Web 路径；两条路并存，各服务各自的用户（浏览器 vs 终端）。

## 1. 命令契约（新增 `guanlan convert`，无新退出码）

```bash
guanlan -C <kb> convert path/to/报告.pdf            # 转 markdown → 写 raw/报告.md（含 origin frontmatter）
guanlan -C <kb> convert 报告.pdf --dry-run          # 只把转换结果打到 stdout，raw/ 零写（人审预览）
guanlan -C <kb> convert 报告.pdf --overwrite        # 同名 raw/ 已存在时显式人背书覆盖（默认不覆盖）
guanlan -C <kb> convert 报告.pdf --origin "https://…"  # 显式出处（默认 = 转换前原始 src 路径，见 §3.3/决策P5.2-11）
guanlan -C <kb> convert 报告.pdf --name 季度报告      # 覆盖目标 slug（默认取原件 stem）
guanlan -C <kb> convert 报告.pdf --ingest           # 便利：转换成功后串联 `ingest raw/<slug>.md`（默认关）
```

- **两步默认、不自动 ingest**（决策P5.2-1/8）：`convert` 只做**转换 + 落源**（零 LLM、宿主写 `raw/`）；建页那步
  仍由独立的 `guanlan ingest raw/<slug>.md`（走 Agentao + gate）完成。`--ingest` 是可选便利串联，默认关——保留
  「人审转换产物再入库」的 checkpoint（接续决策P4.6-1）。
- **纯宿主写 `raw/`、零 LLM、不起 Agentao、不取 raw 快照**（决策P5.2-3）：`convert` 是「人投喂源」（与 P4.1 投喂
  同性质，决策P4.1-1），**不**走 ingest 的 `run_guarded_write` 子进程 + 快照门禁——那是 ingest 的事。`require_kb_root(writable=True)`。
- **无新依赖、复用 skill 分层兜底**（决策P5.2-5）：转换后端 = skill `convert.py` 的 MinerU→marker→pypdf；后端全
  不可用（skill「all backends exhausted」）→ 命令打印 skill 的 stderr + 安装提示（`pip install pypdf`，或 mineru/marker
  求更高保真）并 `EXIT_USAGE`（不崩、不静默）。**核心 `guanlan` 不背任何转换依赖、不加 `[convert]` extra。**
- **无新退出码**（沿用半阶段纪律）：成功 `EXIT_OK(0)`；非库根 / 输入不存在 / skill 不支持的格式 / 全后端耗尽 / 文本
  准入不过 / 同名已存在且无 `--overwrite` / **raw 写盘 IO 失败（CLI 壳 `except OSError` 转码、不 traceback，§2.2）**
  → `EXIT_USAGE(1)`。

## 2. 模块与归口

### 2.1 新增 `guanlan/convert.py`（转换内核 + 命令壳分立，后端 = 复用 skill）

```python
def _skill_convert_script() -> Path:
    """定位 pdf-to-markdown skill 的 convert.py：复用 skill.py 的归口
    `bundled_skill_dir("pdf-to-markdown")`（安装态 guanlan/_skill/ / 开发期仓库根 skills/），
    取其 `scripts/convert.py`。找不到 → raise ConvertError（命令层 EXIT_USAGE，提示重装 skill）。"""

def convert_to_markdown(src: Path, *, backend: str = "auto", cwd: Path | None = None) -> str:
    """**转换内核**：把一个多格式文件转成 markdown 文本字符串，零 LLM。复用 skill `convert.py` 当引擎。
    `cwd` = 子进程工作目录 / skill `.env` 发现锚点（run_convert 传 **KB root**；None → 回退 `Path.cwd()`）：
    1. **temp 暂存**：把 src 复制进一个 tempfile.TemporaryDirectory、得**绝对路径** `staged`（convert.py 把产物写在
       *输入文件同目录*、无 --output_dir——`staged` 在 tmpdir 内，故产物落 tmpdir、不污染用户目录，决策P5.2-10）；
    2. **子进程调用**：`subprocess.run([sys.executable, convert_py, staged, "--backend", backend],
       cwd=cwd or Path.cwd(), capture_output=True, text=True)`——**cwd 取 KB root、绝不取 tmpdir**（决策P5.2-12）：
       skill 的 `find_dotenv()` 从 cwd 向上找 `.env`（`convert.py:82`），cwd=tmpdir 会让它**找不到 KB/项目的 `.env`**、
       marker 无法按 §3.2 经 `.env` 启用 Gemini。`staged` 是绝对路径、产物目录来自 `input.parent`=tmpdir、**与 cwd 无关**，
       故 cwd≠tmpdir **不**重新引入污染（决策P5.2-10/12）。透传调用方环境（marker 是否用 Gemini 由用户环境/`.env` 决定，
       guanlan 不干预、不携带密钥，决策P5.2-4）；
    3. **解析产物**：成功（exit 0）→ stdout 末行即产物 `.md` 路径（convert.py 把 backend 日志全导 stderr、
       stdout 只留路径行）→ `read_text(encoding="utf-8", errors="replace")`（容错口径，P5.0-16）返回文本；
    4. exit≠0（全后端耗尽 / skill 不支持的格式 / 文件不存在）→ raise ConvertError(带 skill stderr)（§7）。
    - **格式白名单不在此重复**：交给 skill `convert.py` 的 `SUPPORTED_EXTENSIONS` 归口（含图片 OCR），不支持
      它自己 exit 1，本层映射 EXIT_USAGE——消除白名单漂移（决策P5.2-2/5）。
    - **不**承诺字节确定性（决策P5.2-4）：输出随 mineru/marker/pypdf 版本、输入结构、及外部后端是否启用 LLM 增强
      而变——它是**源**、非可重建派生物。"""

def run_convert(src, *, root, name=None, origin=None, overwrite=False,
                dry_run=False, do_ingest=False, backend="auto") -> int:
    """命令壳 + 退出码：解析输入 → `convert_to_markdown(src, backend=…, cwd=root)`（cwd=KB root 保 skill `.env`
    发现，决策P5.2-12）→ 文本准入 → provenance → 写 raw/（或 --dry-run 打印）→（可选）串联 ingest。所有错误
    （skill 缺失 / 不支持 / 全后端耗尽 / 准入 / 同名 / 越界）→ EXIT_USAGE。
    **落盘 IO 失败**：`atomic_write_raw` 上抛的 `OSError` 由本壳 `except OSError` 捕获 → 打印「写 raw/ 失败」、返回
    `EXIT_USAGE`（镜像 `graph_entrypoint`，不外抛 traceback，§2.2/决策P5.2-6）。"""

def convert_entrypoint(root_dir, *, src, ...) -> int: ...
def main(argv=None) -> int: ...     # python -m guanlan.convert
```

CLI 接线在 `cli.py` 加 `p_convert`（位置参 `src`、`--name`/`--origin`/`--overwrite`/`--dry-run`/`--ingest`/
`--backend`），与 `ingest`/`search` 同形。`--backend` 透传给 skill `convert.py`（`auto`/`mineru`/`marker`/
`python`），默认 `auto`。**`convert` 不设 `--model`**（决策P5.2-4）：转换那步由 guanlan 视作零 LLM，外部后端是否用
LLM 增强只由 `--backend` + 用户环境决定，guanlan 不代为指定 model——故**不**把任何 model 透传给 skill `convert.py`
（虽然 skill 本身支持 `--model`），以免暗示「guanlan 控制转换的 LLM」；`--ingest` 串联的建页那步用 `ingest` 自身的
默认 model，需指定特定 model 时单独跑 `guanlan ingest raw/<slug>.md --model X`（§3.4）。内核 `convert_to_markdown`
与命令壳 `run_convert` 分立——供将来 Web 机会性优化直接复用
内核（仿 P5.0 `search_pages` 被 P5.1 复用，但不单列阶段，决策P5.2-9），呼应「内核与打印壳分立」纪律（决策P4-7 /
P3.2 / P5.0-10）。

### 2.2 raw-write 归口抽出（transport-neutral 核心，消 web↔cli 漂移）

P4.6 的 raw-write 原语（`_raw_slug`/`_normalize_basename`/`_safe_raw_target`/`_check_text_admission`/`_apply_origin`/
`_atomic_write_raw`）现都在 **`web/rawfeed.py`**（web-coupled）：其中**校验/变换 helper 抛 `HTTPException`**，
**原子写 helper `_atomic_write_raw(...) -> int` 返回退出码**（与现有实现一致）。CLI `convert` 与 web 投喂/晋级
必须**同一套 slug/准入/provenance 规则**，否则两处漂移（安全敏感 + provenance 不一致）。故（决策P5.2-6，镜像
P4.10 把 `report_json` 抽成 `report_dict` 归口、P5.0-15「校验在内核 raise ValueError」）：

抽出 transport-neutral 核心模块 **`guanlan/rawio.py`**，但**两类契约必须分明、不可混为一谈**（决策P5.2-6，**关键**）：

- **校验/变换纯函数 → `raise ValueError`**（非 HTTPException）：`raw_slug` / `normalize_basename` / `safe_raw_target`
  （NFKC→slug→强制 `.md`→越界）/ `check_text_admission`（空 / 超 `MAX_RAW_BYTES` / 控制字符）/ `apply_origin`
  （provenance 四分支，逐字保留 P4.6 决策P4.6-10 语义）。它们都在**入队之前**跑（web 端点的 `to_thread` 内、cli 命令壳
  里）——web 薄壳包成 `HTTPException(400)`、cli 映射 `EXIT_USAGE`，**不经 JobQueue worker 的退出码归一**。
- **`atomic_write_raw(target, content, overwrite) -> int` → 仍返回退出码、绝不抛 `ValueError`**（决策P5.2-6 关键修正，
  否则砸掉 web 的 409）：它在**串行 worker turn 内**复检覆盖语义——同名已存在且无 `overwrite` → 返回 `EXIT_USAGE`
  （决策P4.1-2/3）；落盘 IO 异常**继续向上抛**（由 worker 归一为 `EXIT_AGENT_ERROR`）。**为何独不 raise**：web 把它当
  JobQueue 作业 thunk 跑（`lambda: atomic_write_raw(...)`，`app.py:497`），worker 只认**返回的 `exit_code`**——
  `EXIT_USAGE`→端点转 **409**「已存在」、`EXIT_AGENT_ERROR`→**500**「写盘失败」（注释明言「不可一律塌缩成 409，
  否则磁盘满被误报已存在」）。若按前一草案把它也改成 raise `ValueError`，worker 会把异常归一成 agent error →
  **409 退化成 500**，直接破 P4.6「零行为变更」硬门。
- `web/rawfeed.py` **保留薄壳**：`import` 核心；校验函数的 `ValueError` 包成 `HTTPException(400)`，`atomic_write_raw`
  的**返回码原样透传给 JobQueue**（worker `exit_code` → 409/500 分流逐字不变）——**字节行为不变**（P4.6 既有 web 测试
  全绿是硬回归门，§8）。
- `guanlan/convert.py` 与 `cli.py` 直接调核心：校验 `ValueError` → `EXIT_USAGE`（同 P5.0-15 的 CLI↔web 分流）；
  `atomic_write_raw` 的**返回码直接用作命令退出码**（`EXIT_USAGE` = 同名已存在且无 `--overwrite`）。**落盘 IO 失败的
  CLI↔web 分流不同**：`atomic_write_raw` 契约不变（IO 异常 `OSError` 一律上抛），但 **web 让它上抛给 JobQueue worker →
  归一 `EXIT_AGENT_ERROR` → 500**，而 **CLI `run_convert` 自己包一层 `except OSError`**——打印「写 raw/ 失败：{exc}」、
  返回 `EXIT_USAGE`（**镜像 `graph_entrypoint`，`graph.py:431`**：guanlan 顶层不统一兜 `OSError`，故每个写盘命令壳自捕、
  不外抛 traceback）。这只补 **CLI 壳的错误映射**，不改 `atomic_write_raw` 契约。

> 这是一次**零行为变更的抽出重构** + P5.2 新增，不是改写。验收以「P4.6 web 投喂/晋级测试逐条仍绿 + 新 convert
> 与 web 晋级走同一核心产出字节一致」双向钉死（§8）。

## 3. 行为细则

### 3.1 「转换 Markdown 当源」+ 命令行用户即背书人（决策P5.2-1/2/3）

完全沿用 P4.6 决策P4.6-2 的模型，只是背书人从「Web 里点按钮的人」变成「敲 `guanlan convert` 的人」。

> **诚实口径（反向评审）**：`convert` 写 `raw/` 是一次**显式落源动作**——「这名用户主动把这份转换产物声明为源」，
> **不是**「这份内容已被审阅」的自动证明。CLI 不强制人逐字读过产物（不像 P4.6 Web 有 `workspace/parsed/` 池内
> 预览/修订这道更强的人审）。`convert` 默认**两步、不自动 ingest**，正是为给这道（可选的）人审留 checkpoint：
> `--dry-run` 预览、落源后 `ingest` 前自审 `raw/<slug>.md`。把「敲命令」当成「背书」成立的边界是「人主动发起、为
> 后果负责」，而非「内容已校对」——后者要不要做、做多严，由人自己决定（§3.3 的 `--dry-run` 是为此留的口）。

```
报告.pdf ──convert（零 LLM 转换）──▶ raw/报告.md（origin: 报告.pdf）── 人审 ──▶ guanlan ingest raw/报告.md ──▶ wiki/
  原件（留原处/库外，          宿主确定性写、非 gated         （读 raw/ 文本，         既有 .md ingest，gated 子进程
   不入 raw/、仅 provenance）      agent 写、非 LLM）            改坏就删 raw/ 重转）        + raw/ 快照，逐字不动
```

- **原件不入 `raw/`、仅作 `origin` provenance**（决策P4.6-2 一脉）：`raw/` 收**转换出的 `.md`**作源，原 PDF/DOCX
  留在原处（库外或用户目录）、可弃；`origin` frontmatter 是回链原件/出处的唯一锚。这让 `raw/` 维持 **`.md` 单格式**、
  ingest 不必读二进制/多格式——**ingest 逐字不动**（决策P5.2-1）。
- **命令行用户是背书人**：P4.6 的核心约束是「**Agent** 永不写 `raw/`，由**人**审阅后晋级」（决策P4.6-1）——防 Agent
  伪造源击穿引用信任。CLI `convert` 由**人主动敲**，人即背书人：写 `raw/` 是人投喂源（决策P4.1-1），**不**经
  Agent、**不**经 workspace 中转。转换产物落 `raw/` 后，人在 `ingest`（真正建页的 LLM 步）**之前**自行审阅
  `raw/<slug>.md`；不满意就**删掉 `raw/` 那个文件**重转（人删 `raw/` 合法；只有 Agent 不可碰 `raw/`）。`--dry-run`
  让人**落源前**先 `convert … --dry-run | less` 预览（§3.3）。
- **不取 raw 快照、不起 Agentao**（决策P5.2-3）：`convert` 不是 gated agent 写，是确定性宿主写（与 P4.1 投喂同
  路径），故复用 `rawio` 原子写、**不**走 `run_guarded_write`；不写 `log.md`（与 `graph`/`search`/投喂一致，
  非 LLM 写作无需留痕）。

### 3.2 转换：复用 skill 引擎、graceful degrade、透传环境，但**非字节确定性**（决策P5.2-4/5）

- **后端 = skill `convert.py`（MinerU→marker→pypdf 分层）**：CLI 不重造转换器，子进程调既有 skill 引擎，`--backend`
  透传（默认 `auto`）。**graceful degrade 复用 skill 既有逻辑**：某层后端未装/失败自动降到下一层，pypdf 是纯 Python
  最后兜底（仅 PDF）；全层耗尽 → skill exit 1，CLI 映射 `EXIT_USAGE` + 安装提示（§1/§7）。**核心 `guanlan` 不背
  mineru/marker/pypdf 依赖**——它们是外部可选、与 skill 既有「best available wins」哲学一致（决策P5.2-5）。
- **「脚本零 LLM」不变量 = guanlan 不内嵌 LLM 客户端/密钥，与外部后端是否用 LLM 正交**（决策P5.2-4，**关键澄清**）：
  该不变量约束的是 **guanlan 自己的代码**（不自带 litellm/密钥、不把 LLM 反塞进脚本）。`guanlan convert` 只是 shell
  out 到一个**外部转换进程**（skill `convert.py` → mineru/marker/pypdf），**不携带任何 LLM 客户端或密钥**——不变量
  天然成立。skill 的 marker 层在用户环境有 `GEMINI_API_KEY`（或 `.env`）时会启用 Gemini 增强，但那是**用户对外部
  工具的配置选择**（marker 会在 stderr 明示「LLM enhancement on」），**不是 guanlan「carry 一个 LLM」**——故**不做
  env-scrub**：强行剔除用户密钥反而违背其意愿、压低转换质量。需要时用户可经 `--backend python`/marker 自身开关控制，
  guanlan 不代为决定。
- **temp 暂存避免污染用户目录**（决策P5.2-10）：skill `convert.py` 把产物 `.md` 写在**输入文件同目录**（无
  `--output_dir`、且嵌套 `<stem>/…/<stem>.md`）。CLI **先把 src 复制进 `TemporaryDirectory`（得绝对 `staged`）再跑、
  读出产物文本、连 temp 一并丢弃**——用户的工作目录零残留。这是「直接复用 skill」唯一需在宿主侧补的约束。
- **但子进程 `cwd` 取 KB root、绝不取 tmpdir——否则砸掉 skill 的 `.env` 发现**（决策P5.2-12，**关键修正**）：skill
  `find_dotenv()` 从 **cwd 向上**找 `.env`（`convert.py:82`），若把 cwd 设成 tmpdir，就找不到 KB/项目目录里的 `.env`，
  上一条说的「marker 经 `.env` 启用 Gemini」会**静默失效**。故 `cwd=root`（KB 根，向上仍能命中项目/家目录 `.env`）。
  污染防护**不靠 cwd**：产物目录来自 `staged.parent`=tmpdir（`staged` 是绝对路径），与 cwd 无关——cwd 改回 KB root
  **不**重新引入残留（决策P5.2-10）。两条正交：temp 暂存管「产物落哪」，cwd 管「`.env` 从哪找」。
- **转换不承诺字节确定性——与 graph/search 的强确定性明确区分**（决策P5.2-4，本阶段关键诚实点）：graph/search 是
  **派生物**，必须「同输入字节稳定、可幂等重建」（核心原则 1）。但转换产物**一旦落 `raw/` 就是源、不是派生物**——它
  **不需要**「能从 PDF 字节级重建」。mineru/marker/pypdf 跨版本/对复杂版式/是否启用 LLM 增强，输出都会变，这**可
  接受**：源由人一次性背书固化，原件仅 provenance、不在事实链上。故 P5.2 **不冒充「字节稳定幂等重建」承诺**——这正是
  P4.6「解析 Markdown 当源」让人在环背书的根本原因（决策P4.6-1/2）。

### 3.3 落源：复用 raw-write 归口（slug / 文本准入 / provenance / 默认不覆盖）

转换得 markdown 文本后，**逐字复用 §2.2 抽出的 `rawio` 核心**（与 web 投喂/晋级同一道闸）：

- **目标名**：`--name` 给定则用之，否则取**原件 stem**；过 `rawio.safe_raw_target`（NFKC 归一 → slug → 强制 `.md` →
  越界校验），得 `raw/<安全名>.md`。
- **文本准入**：`rawio.check_text_admission`——空 / 超 `MAX_RAW_BYTES`(5 MiB) / 含控制字符 → `ValueError` → `EXIT_USAGE`
  （与投喂/晋级同闸；转换出的超大/坏字符 `.md` 不混进 `raw/`）。skill 产物为 UTF-8 文本，读取沿用 P5.0-16 `errors="replace"` 容错口径。
- **provenance**：`rawio.apply_origin`（P4.6 四分支逐字保留）。`origin` = `--origin`（strip 后）非空则用之，否则**回退
  默认 origin**（绝不写空 provenance，决策P4.6-10）。`origin` 经 `yaml.safe_dump` 写入、绝不裸拼。
- **默认 origin 口径钉死**（决策P5.2-10/11）：默认 origin **取转换前、用户传入的原始 `src` 路径**——必须在**复制进
  temp 暂存之前**于命令入口就捕获，**绝不**用 temp staged 路径、**也绝不**用 skill stdout 里那行产物 `.md` 路径
  （二者都是宿主内部的临时物、对用户无意义、且会随 temp 名变化）。具体口径（**二选一已钉**）：把 `src.expanduser().resolve()`
  得绝对路径，**若落在库根 `<kb>` 内**则存**相对库根的 posix 路径**（可移植、随库迁移仍有效），**否则**（原件在库外/
  用户目录，常态）存该 resolved 绝对路径字符串。`--origin` 显式给定时覆盖此默认（出处可为 URL/任意字符串）。
- **默认不覆盖**（决策P5.2-8）：`raw/<slug>.md` 已存在且无 `--overwrite` → `EXIT_USAGE`，提示「改名（`--name`）或
  `--overwrite` 覆盖」；`--overwrite` 是显式人背书覆盖例外（镜像 P4.6-10：覆盖已被引用的源可能让 wiki 页失真，建议
  覆盖后重 `ingest` + `check`，CLI 文案提示之）。
- **`--dry-run`**（决策P5.2-8）：转换 + 文本准入照跑，但**把归一后的 markdown（含 provenance frontmatter）打到 stdout、
  `raw/` 零写**、退 `EXIT_OK`——供人落源前预览。

### 3.4 `--ingest` 便利串联（默认关，决策P5.2-1/8）

`--ingest` 时，落源成功后**在同一进程内调 `run_ingest("raw/<slug>.md", root=…)`（用 ingest 自身默认 model，convert
不透传 model，见 §2.1/决策P5.2-4）**、透传其退出码。需指定特定 ingest model 时拆成两步（`convert` 落源后单独
`guanlan ingest raw/<slug>.md --model X`）。它只是省一次
手敲，**不改 ingest 任何契约**（仍 gated 子进程 + raw 快照 + 强制 `check`）。`--dry-run --ingest` 互斥（dry-run 没落源，
无可 ingest）→ `EXIT_USAGE`。默认关：保「人审转换产物再入库」checkpoint。

## 4. 决策（沿用仓库 `决策PX-N` 体例）

- **决策P5.2-1（单独 `guanlan convert` 命令，不并入 `ingest`；默认两步、不自动 ingest）**：转换（确定性宿主写、
  guanlan 侧无 LLM）与建页（LLM、gated）是两类操作，分立成两命令——保「人审转换产物」checkpoint、ingest 的 raw 快照
  门禁不被「转换刚写完 raw/ 又立刻被快照」搅浑、内核可被将来 Web 机会性优化复用。`--ingest` 仅便利串联、默认关。接续决策P4.6-1。
- **决策P5.2-2（采「转换 Markdown 当源」，复用 `.md` ingest 不动；原件不入 `raw/`、仅 `origin` provenance）**：逐字沿用
  P4.6 决策P4.6-2。`raw/` 维持 `.md` 单格式，ingest 不读二进制/多格式、逐字不动；唯一拦截点
  `ingest._resolve_raw_target` 不松，只把它前面那步多格式→`.md` 补上。**格式白名单归口在 skill**（`convert.py` 的
  `SUPPORTED_EXTENSIONS`），CLI 不重复维护；`.md` 输入由 CLI 早提示拒（已是 markdown、直接投喂/晋级、不必 convert）。
- **决策P5.2-3（convert 是人投喂源：宿主确定性写 `raw/`，不起 Agentao、不取快照、不写 log.md）**：与 P4.1 投喂同
  性质（决策P4.1-1），复用 `rawio` 原子写，**不**走 `run_guarded_write`（那是 ingest 的 gated 路径）。命令行用户即
  「解析忠实」背书人（P4.6-1 的 Agent 约束在 CLI 退化为「人自敲自背书」）。
- **决策P5.2-4（「脚本零 LLM」管 guanlan 自身、与外部后端是否用 LLM 正交；转换不承诺字节确定性）**：①「脚本零
  LLM、不自管密钥」约束的是 **guanlan 代码**——`guanlan convert` 只 shell out 到外部转换进程、**不内嵌 LLM 客户端/
  密钥**，不变量天然成立。skill marker 层在用户环境有 `GEMINI_API_KEY`/`.env` 时启用 Gemini，是**用户对外部工具的
  配置选择**（marker stderr 明示）、**非 guanlan carry LLM**——故 **不做 env-scrub**（强剔密钥违用户意愿、压低质量）。
  **③`convert` 不设 `--model`、不向 skill 透传 model**：转换的 LLM 用法由 `--backend` + 用户环境决定，guanlan 不代为
  指定；即便 skill `convert.py` 自身支持 `--model`，CLI 也**不**暴露/透传它，以免暗示「guanlan 控制转换的 LLM」而与
  ①矛盾。`--ingest` 串联的建页步用 `ingest` 自身默认 model（需指定特定 model 则拆两步），见 §2.1/§3.4。
  ②**明确区别于 graph/search 的强字节确定性**：转换产物落 `raw/` 即**源**（人一次性背书固化），不是必须可幂等重建的
  **派生物**，故后端跨版本/是否用 LLM 增强带来的输出漂移可接受、不破核心原则 1。这正是「解析 Markdown 当源 + 人在环
  背书」（P4.6-1/2）的根本理由。**修订记录**：先前草案误把「外部进程调 Gemini」当成破零 LLM 红线而设 env-scrub，已
  据用户更正移除——外部转换进程的 LLM 用法是用户环境的事，与 guanlan 不变量无关。
- **决策P5.2-5（复用 `pdf-to-markdown` skill 的 `convert.py` 当后端引擎，无新依赖 / 无 `[convert]` extra）**：CLI
  `convert` 不重造转换器、不引 markitdown——直接子进程调既有 skill 引擎（`bundled_skill_dir("pdf-to-markdown")/
  scripts/convert.py`，已随 wheel force-include + 全局安装）。**graceful degrade 复用 skill 既有分层**（MinerU→marker→
  pypdf，未装/失败自动降级，全耗尽→exit 1→`EXIT_USAGE` + 安装提示）；mineru/marker/pypdf 是外部可选、**不进 guanlan
  依赖**（与 skill「best available wins」哲学一致）。`--backend` 透传 skill。这把 `pdf-to-markdown` skill 从「P4.6
  Agent-only」**升格为 Agent 与 CLI 共用的多格式→md 单一引擎归口**——转换逻辑单份、无第二实现漂移。**取代**先前草案
  的 markitdown+`[convert]` extra 方案（用户决策：独立命令 + 直接复用 skill convert.py）。
- **决策P5.2-6（raw-write 归口抽出 `guanlan/rawio.py`：校验函数 raise ValueError、`atomic_write_raw` 仍返回退出码；
  web/rawfeed 保薄壳）**：消 web↔cli 的 slug/准入/provenance 漂移（安全 + provenance 一致性敏感）。**两类契约分明
  （§2.2）**：①**校验/变换函数**（`safe_raw_target`/`check_text_admission`/`apply_origin`/`raw_slug`/`normalize_basename`）
  入队前跑、**raise `ValueError`** → web 包 `HTTPException(400)`、CLI 映射 `EXIT_USAGE`（同 P5.0-15 分流）；②
  **`atomic_write_raw` 在串行 worker turn 内、仍返回 `EXIT_OK`/`EXIT_USAGE`**（同名复检→`EXIT_USAGE`、IO 异常上抛→
  worker `EXIT_AGENT_ERROR`），**绝不改成 raise**——否则 worker 把同名冲突归一成 agent error，web 的 **409 退化成 500**
  （`app.py:497` 靠 `exit_code` 分流 409/500，注释明言「不可一律塌缩成 409」），破零行为变更门。**修订记录**：先前草案误把
  `atomic_write_raw` 列进「纯函数 + raise ValueError」组，已据评审更正为「校验 raise / 原子写返回码」二分。**零行为变更
  抽出**——P4.6 web 测试逐条仍绿是硬回归门（§8）。镜像 P4.10 `report_json`→`report_dict` 归口抽出。
- **决策P5.2-7（无新退出码）**：成功 `EXIT_OK`；非库根 / 输入不存在 / 不支持格式 / 转换失败 / 文本准入不过 / 同名
  已存在无 `--overwrite` / 找不到 skill `convert.py` / **raw 写盘 IO 失败（CLI 壳 `except OSError`，§2.2/决策P5.2-6）**
  → `EXIT_USAGE`。
  沿用半阶段不引新码纪律。
- **决策P5.2-8（`--dry-run` 预览 / `--overwrite` 人背书覆盖 / `--origin`·`--name` 显式控制 / `--ingest` 便利且默认关）**：
  `--dry-run` 打 stdout、`raw/` 零写（人审）；`--overwrite` 显式覆盖（镜像 P4.6-10 失真告警文案）；`--origin` 默认回退
  默认 origin（决策P5.2-11 钉死口径：库内→相对库根 / 库外→resolved 绝对路径，取转换前原始 src）、绝不空；`--ingest`
  串联但不改 ingest 契约、`--dry-run --ingest` 互斥 → `EXIT_USAGE`。
- **决策P5.2-9（不单列 P5.3：Web 接入降为 P4.6 机会性优化、MCP 明确不做 convert）**：本阶段只 CLI 内核 + 命令。
  **与 P5.0→P5.1 的关键差别**：检索的 Web 接入此前不存在、P5.1 是真正的新能力；但摄入的 Web 路径 **P4.6 已交付且
  可用**（Agent 在可写会话里跑 shell 解析），故**不再单列「P5.3 Web/MCP 接入」阶段**，拆成两半各自定性：
  - **Web 半 = P4.6 的机会性优化，非阶段**：唯一剩余改进是「把那步 Agent shell 解析替换为直接调确定性
    `convert_to_markdown` 内核」（需要时一个 `POST /api/convert` 落 `workspace/parsed/` 即可）——去掉一层 Agent 中转、
    确定性更强，但**是优化不是缺口**，Web 用户现在已能用 P4.6 走完全程；何时做、做不做，与 P5.2 验收无关。
  - **MCP 半 = 明确不做**：`convert` 的本质是**写 `raw/`**，而 P4.10 MCP 是只读 server（零字节 KB 写）——convert
    工具会击穿其只读不变量；唯一不矛盾的形态「只读 convert 预览」（≈`--dry-run` 返回文本、不落盘）价值极低（MCP
    客户端手里本就有文件）。故 MCP **不暴露 convert**，原「P5.3 Web/MCP 接入」里的 MCP 半段作废。
  内核 `convert_to_markdown` 与命令壳分立仍保留，但**仅作零成本期权**（为上面的 Web 优化留口），不承诺独立阶段。
- **决策P5.2-10（复用 skill 引擎需在宿主侧补一条约束：temp 暂存防污染）**：见 §3.2。skill `convert.py` 把产物写在
  **输入同目录**（无 `--output_dir`、嵌套 `<stem>/…/<stem>.md`），故 CLI **把 src 复制进 `TemporaryDirectory` 再跑、
  读产物、丢 temp**，用户目录零残留。这是「直接复用 skill」唯一需在宿主侧补的边界，**不改 skill 本身**（skill 仍服务
  P4.6 Agent 路径）。子进程**透传调用方环境**、不做 env 改写（marker 是否用 Gemini 由用户环境决定，决策P5.2-4）。
- **决策P5.2-11（默认 origin 取转换前的用户原始 `src`，绝不取 temp staged / skill 产物路径）**：因实现先把 src 复制
  进 temp 再跑转换（决策P5.2-10），provenance 必须**在 staging 之前于命令入口捕获原始 `src`**，**绝不**写 temp staged
  路径或 skill stdout 那行产物 `.md` 路径（宿主内部临时物、对用户无意义、随 temp 名漂移）。**口径二选一已钉**：`src`
  resolve 为绝对路径后，**库内→相对库根 posix 路径**（可移植）、**库外（常态）→resolved 绝对路径**；`--origin` 显式给定
  时覆盖。见 §3.3。**回归断言**：mock 转换使 temp/产物路径与原始 src 显著不同，核验写出的 `origin` == 原始 src 口径、
  绝不含 temp 路径片段（§8）。
- **决策P5.2-12（子进程 `cwd` 取 KB root、绝不取 tmpdir——保 skill `.env` 发现）**：与决策P5.2-10 正交。skill
  `find_dotenv()` 从 **cwd 向上**找 `.env`（`convert.py:82`）；若沿用「cwd=tmpdir」就**找不到 KB/项目 `.env`**，
  决策P5.2-4 说的「marker 经 `.env` 启用 Gemini」会静默失效。故 `convert_to_markdown(..., cwd=root)`（run_convert 传
  KB 根）。**污染防护不靠 cwd**：产物目录来自 `staged.parent`=tmpdir（`staged` 绝对路径）、与 cwd 无关，cwd 改回 KB
  root **不**重新引入残留。**修订记录**：先前草案 §2.1 子进程示例误写 `cwd=tmpdir`，会砸掉 skill 的 `.env` 口径，已据
  评审更正为 `cwd=KB root`。**回归断言**：mock `subprocess.run` 核验传入 `cwd==root`（非 tmpdir），且产物仍落 tmpdir、
  用户目录零残留（§8）。

## 7. JSON 契约 / 退出码

`convert` 是宿主写命令、**无 stdout JSON 契约**（与 `graph` 同：写盘为主）。`--dry-run` 的 stdout 是**纯 markdown
文本**（含 provenance frontmatter），供 `| less`/重定向预览，非 JSON。落源回执走 stderr/stdout 人读文案
（`已写 raw/报告.md（origin: 报告.pdf，N 字节）`）。

退出码（决策P5.2-7，**不引新码**）：

| 情形 | 退出码 |
|---|---|
| 转换 + 落源成功（或 `--dry-run` 打印成功 / `--ingest` 串联 ingest 返回 0） | `EXIT_OK(0)` |
| 非知识库根 / 输入文件不存在 / skill 不支持的格式 / 找不到 skill `convert.py` | `EXIT_USAGE(1)` |
| skill 全后端耗尽（mineru/marker/pypdf 均未装或失败，skill exit 1） | `EXIT_USAGE(1)` |
| 文本准入不过（空 / 超 `MAX_RAW_BYTES` / 控制字符） | `EXIT_USAGE(1)` |
| 同名 `raw/` 已存在且无 `--overwrite` | `EXIT_USAGE(1)` |
| raw 写盘 IO 失败（`OSError`，磁盘满/权限等）→ CLI 壳 `except OSError` 转码、**不 traceback**（镜像 graph） | `EXIT_USAGE(1)` |
| `--dry-run --ingest` 互斥 | `EXIT_USAGE(1)` |
| `--ingest` 时 ingest 自身的退出码 | 透传（见 errors.py） |
