### 0.1 评审收窄：为什么不照搬 `search_pages`，也不照搬 Levenshtein

初稿曾提「`search_pages(wiki, T, limit=1)` 找最近页」+「测 `[[Atention]] → attention.md`」，评审指出三处会出错，**均采纳**：

1. **BM25-over-正文会顶出引用页，而非候选目标页。** 断链文本 `[[T]]` 本身存在于**引用它的源页正文**里，BM25 召回必然把这些源页排上来（`missing_entity` 多引用场景尤甚）——它们恰恰**不是**候选目标。→ **候选只在既有页的 stem / title 上做（aliases 留可选后续，见 §2 复审修订），不扫正文**，天然避免「把引用页建议成目标」。
2. **BM25 ≠ 拼写相似度。** ASCII token 是精确词项，`atention` 与 `attention` 无共享 token，`search` 召不回；英文 typo / 编辑距离不是 `search` 的能力。→ **删掉英文拼写纠错承诺与对应测试**；只做共享 token / CJK-2-gram overlap / 包含。
3. **逐断链调 `search_pages` 太重。** 它每次 `build_corpus` + `build_graph`（算 backlink 先验），`B` 个断链 = `B ×` 全库扫描 + `B ×` 建图。→ **一次 `run_lint` 内只建一次轻量候选索引**（见 §2），从 `run_lint` 已建好的那张图取 stem/title，**零额外扫描**（aliases 若要纳入须另扫一次，见 §2）。

## 2. 候选索引：stem/title 从已建图零额外取，aliases 可选后续（决策P3.11-2）

> **复审修订**：初稿称「aliases 也能从已建图零额外取得」**不成立**——`Graph` 只暴露 `nodes/edges/adjacency/broken`，`Node` 字段只有 `id/title/type/path`（`graph.py:62`，**无 aliases**）；`build_graph` 内的 `link_resolution_index` 是**局部变量**，不随 `Graph` 返回。`run_lint` 拿不到 aliases。据此分层如下。

`run_lint`（`lint.py:109`）开头已 `build_graph(wiki)` 一次（`lint.py:112`），`g.nodes` 现成可取：

- **stem**：`Node.id`（= `path.stem` 小写，`graph.py:63`）。
- **title**：`Node.title`（= `page_title(meta, …)`，`graph.py:118`）。

**v1 候选 = 每页的 stem 与 title 两组 token 集**——遍历 `g.nodes`（非 config 内容页节点），`tokenize` 成**按字段分开**的 token 集（stem 一组、title 一组，**不并袋**，决策P3.11-3a），**一次 `run_lint` 内建一次**。这是**真·零额外扫描**。索引存为**倒排** `token → [(页, 字段 token 集)]`（决策P3.11-4a），让每个断链 target 只比对与之共享 ≥1 token 的候选、而非全库——配合**按 target 记忆化**，成本从 O(断链数 × 页数) 降到 O(唯一 target 数 × 共享 token 候选数)。**不重建图、不建 corpus、不扫正文**。

**aliases 留作可选后续（决策P3.11-2a）**：若要把别名也纳入候选键，须**另跑一次** `alias_index(wiki)`（`pages.py:276`，frontmatter `aliases` → 拥有页 stem 的零-LLM 扫描）或读页 `meta["aliases"]` 原始字段——这是**一次额外轻量扫描，不是零**，文档不再宣称零扫描。**不可**拿 `link_resolution_index`（`pages.py:349`）当 aliases 源：它混了 stem / 真实别名 / 安全 fold variant 且**不标来源**，反推会把 fold variant 误当候选键，与「候选来自 stem/title/aliases」语义不符。**也不为此扩 `Graph`/`Node` 契约**（面会变大，决策P3.11-2 不取）。

> 注：断链 target **不会**精确等于某既有页的别名——若等于，P3.1 别名解析在**解析期**已命中、不会成断链。故 aliases 只为**模糊 overlap** 多供一组键，边际价值有限——**v1 略去合理**，待实证再补那一次 `alias_index` 扫描。

## 3. 匹配：共享 token / CJK-2-gram overlap / 包含，不做拼写纠错（决策P3.11-3）

复用 `search.tokenize`（`search.py:95`，CJK-2-gram + 非 CJK `[a-z0-9]+` 词，**分词单一归口**，决策P5.0-18）把断链 target `T` 转 token 集，对每页**逐字段（stem / title）分别算 Jaccard、取该页最大分**（决策P3.11-3a），再取全库最佳候选：

- **共享 token / overlap（Jaccard）**：`score = |T∩字段| / |T∪字段|`。CJK 走 2-gram，故 `[[多头注意力机制]]` 与页 `多头注意力` 大量 2-gram 重叠（4/6=0.67）→ 命中。
- **共享整词覆盖「包含」直觉**：`[[attention]]` ↔ `self-attention.md`（字段 token `{self,attention}` 含 `attention`，Jaccard 1/2=0.5）→ 命中。
- **分字段取最大、不并袋**：页 `multi-head-attention.md`（title `多头注意力`）对 `[[多头注意力机制]]`——title 字段单算 4/6=0.67 命中；若把英文 stem token 并进同一分母会跌到 4/9 被压掉（决策P3.11-3a，源自 Codex 复审 P2）。
- **不命中即不建议**：`[[Atention]]`（typo）与 `attention` 任一字段无共享 token → **不附建议**（宁缺勿误；编辑距离不在本阶段范围）。
- **排除引用页**（决策P3.11-4a）：候选**剔除引用该 target 的页**（含写有该断链的源页本身）——否则页 `多头注意力机制.md` 内的 `[[多头注意力]]` 会因自身 stem/title 高度重叠而被建议指向**它自己**（review CONFIRMED）；missing_entity 同理不把引用页当目标。

设一个 overlap 下限 `SUGGESTION_MIN_OVERLAP`（取 0.5），最佳候选低于它则不附建议。产物：在 finding detail **追加**一句「（疑似已有页 `wiki/concepts/Y.md`：若同义，宜并入其 aliases 而非新建页）」——措辞定位为**与「建页」并列的可选项**，避免与 missing_entity 的「建议建 entities/X.md」读成两条矛盾指令（review 质量项）；并在 JSON 契约里加**可选** `suggestion` 字段（无建议时**不出**该键）。

## 6. 测试要点（仿 `test_lint.py` 风格）
