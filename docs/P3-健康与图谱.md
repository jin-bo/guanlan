# P3 — 健康与图谱（health + 结构 lint + 确定性 graph）实现设计

> 本文是 [`DESIGN.md`](DESIGN.md) §4.4 / §4.6 / §7-P3 / §8 的**实现级细化**，是 P3 编码的落地依据。
> 高层理念与原则以 `DESIGN.md` 为准；本文只在 P3 范围内把"怎么做"钉死到模块、数据结构、契约与测试。
> P3 复用大量 P2 的确定性基础设施（页面遍历、frontmatter 解析、wikilink 解析），故请先读 [`P2-最小闭环.md`](P2-最小闭环.md)。
> 文中标注的 **\[决策P3-N]** 是对 `DESIGN.md` 的具体化或受控偏离（含 §4.2/§4.6 留下的"落点待定"），集中记录在 §9。

## 1. 目标与范围

**P3 = 在 P2 的"能稳定摄入 / 查询"之上，叠加三件按需的、零 LLM 的维护工具，让知识库自检与可视化。**

纳入 P3：

- `guanlan health` —— **文件级结构健康检查**：空页/桩页、`index.md` 与磁盘双向同步。零 LLM。
- `guanlan lint` —— **图感知的结构 lint**：孤儿页、断链、缺失实体页。零 LLM。
- `guanlan graph` —— **确定性 wikilink graph**：解析 `[[wikilink]]` → 有向边，输出 `graph/graph.json` + 自包含 `graph/graph.html`。零 LLM。

**P3 三者都是按需运行的审计/可视化命令，不进任何写路径门禁。** 写门禁恒为 P2 的 `check` + `raw/` 快照（**\[决策P3-3]**）。

**不**纳入 P3（推后，见 §10）：

- **语义 lint**（矛盾复检、过期论断、资料缺口——需 LLM）→ P3 之后（DESIGN §8）。
- **graph 增强**（LLM 推断边 `INFERRED`/`AMBIGUOUS`、Louvain 社区检测、图感知语义 lint、SHA256 增量缓存）→ P3 之后（DESIGN §8）。
- **多格式摄入** → P5；**Web UI** → P4；**nightly 自动跑 health/lint** → P5（P3 只交付可被定时器调用的命令本身）。

## 2. 数据流总览

```
guanlan health           require_kb_root(root, writable=False)  # 仅需 wiki/
  └─ scan_pages(wiki) → 逐页结构检查（桩页/index 同步，均零 LLM）
     → HealthReport（findings: list[Finding]） → 渲染（人读 / --json）
     → EXIT_OK(0)；--strict 且有 findings → EXIT_LINT_FINDINGS(6)

guanlan lint             require_kb_root(root, writable=False)  # 仅需 wiki/
  └─ g = build_graph(wiki)                    # 复用 graph.py 的邻接表
     → 孤儿（入度 0）/ 断链（未解析边）/ 缺失实体（断链按频次聚合）
     → LintReport（findings） → 渲染 → EXIT_OK(0)；--strict 有 findings → EXIT_LINT_FINDINGS(6)

guanlan graph [--json-only]   require_kb_root(root, writable=False)  # 读 wiki/，写派生 graph/
  └─ g = build_graph(wiki)
     → 写 <root>/graph/graph.json（稳定契约）
     → 写 <root>/graph/graph.html（自包含、内联数据的邻接列表静态视图、零网络；--json-only 跳过）
     → EXIT_OK(0)；IO 失败 → EXIT_USAGE(1)
```

三者皆零 LLM、确定性、可幂等重建（DESIGN 核心原则 1：派生物绝不反向成为权威）。

## 4. health 工作流细化（`guanlan/health.py`）

**扫描范围**与 `check` 一致：`wiki/` 下递归所有 `*.md`，排除 `wiki/` 顶层 config 页（`index.md`/`log.md`/`overview.md`）；`raw/` 不参与。config 页本身不被体检，但 `index.md` 作为 index 同步的**参照物**被读取。

`health` 关注"知识库作为记账产物是否内部自洽"，与 `check`（frontmatter/链接是否合法）正交（关系见 §5.3 矩阵）。两类检查：

### 4.1 桩页 / 空页（`health.stub_page`）

页面正文（剥 frontmatter 后）满足以下任一即记一条：

- 去空白后为空；或
- 仅含标题行（`#`/`##` 开头）与空行，无任何实质正文；或
- 实质正文字符数 `< STUB_MIN_CHARS`（默认 **30**，按 CJK 字符计；可读注释说明阈值由来，SCHEMA 覆盖列为 P3 之后）。

> `stub` 只针对"几乎没有内容"的建库残留空壳；"有内容但连接稀疏"不在 P3 体检范围（稀疏页 lint 推后，见 §10）。

### 4.2 index 与磁盘双向同步

- **磁盘有、index 无**（`health.index_missing_page`）：`wiki/` 下每张**非 config** 页都应出现在 `index.md` 的**任意一条** markdown 链接里。解析 `index.md` 正文的 markdown 链接目标集合（相对 `wiki/` 的 posix 路径），磁盘页不在其中 → 违规。
- **index 有、磁盘无**（`health.index_dangling`）：`index.md` 链接指向的页文件不存在 → 违规。

> **P3 只做双向存在性，不校验分区/type 匹配。** 即不解析 `## Sources`/`## Entities` 等章节结构、不验"source 页是否落在 Sources 区"。分区结构耦合 Agent 生成的 index 文案格式、易误报，且超出"同步与否"的核心诉求——分区正确性推后或不做（见 §10）。

> 解析的是 `index.md` 的 **markdown 链接**（`[文字](路径)`），不是 `[[wikilink]]`——index 用前者（见 conventions §index.md）。该解析器收口为 `pages.index_md_links`，与 wikilink 解析同落 `pages.py`，集中口径、避免与 graph 的 wikilink 解析混淆。

### 4.3 输出契约

```jsonc
// guanlan health --json
{ "ok": false, "pages_checked": 12,
  "findings": [
    {"page": "wiki/entities/Foo.md", "kind": "health.stub_page",          "detail": "正文仅 8 字，疑为桩页"},
    {"page": "wiki/concepts/Bar.md", "kind": "health.index_missing_page", "detail": "磁盘存在但未收录进 index.md"},
    {"page": "wiki/index.md",        "kind": "health.index_dangling",     "detail": "index 链接 sources/x.md 无对应文件"}
  ] }
```

无 `--json` 时打印人类可读逐行报告（同 `check` 风格）。`Finding` 复用 `check.Violation` 的字段形状（`page`/`kind`/`detail`），但语义是**建议**而非门禁违规——退出码见 §7。

## 5. lint 工作流细化（`guanlan/lint.py`）

**lint 是图感知的**：先 `g = graph.build_graph(wiki)` 拿到节点与有向边（含未解析边），再在邻接表上算三类 finding。**只做结构 lint，零 LLM**（语义 lint 属 P3 之后）。

### 5.1 孤儿页（`lint.orphan`）

入度为 0 的非 config 页——没有任何其他页用 `[[…]]` 链到它。

- index 用 markdown 链接而非 wikilink，且 index 是 config（被排除出 graph），故"只被 index 收录"**不**算有入链——这正是孤儿的定义（Karpathy/llm-wiki-agent 同义）。
- `source` 页天然入度常低（多是被引而非引人），易触发；故 orphan 是**建议级**，报告里按 type 分组、不视作错误。

### 5.2 断链（`lint.broken_link`）

未解析的 `[[…]]` 目标——与 `check` 的 `wikilink.broken` **同源同口径**（都走 `pages.link_stem` + 全页面 stem 解析集）。lint 把它作为**质量建议聚合**呈现（按源页分组），而**独立 `check`** 把它作为 violation（写门禁里仍只是警告、不阻断，决策8）。**同一事实、两种用途**：写门禁靠 `check`（决策8 里断链在写门禁是警告、独立 `check` 是 violation），人工审计靠 `lint`/`check`。不重复造解析逻辑。

### 5.3 缺失实体页（`lint.missing_entity`）

断链的**高价值子集**：同一未解析目标 stem 被 `≥ MISSING_ENTITY_MIN_REFS`（默认 **2**）张**不同**页引用，却无对应 `entities/`/`concepts/` 页 → 建议建页。这是"该有页却还没有"的信号（被反复提到的术语），优先级高于零散断链。

> **check / health / lint 职责矩阵**（避免三者重叠引发困惑）：

| 维度 | `check`（P2） | `health`（P3） | `lint`（P3） |
|------|------|------|------|
| 定位 | 写门禁 + 按需校验 | 按需文件级体检 | 按需图级质量 lint |
| LLM | 零 | 零 | 零 |
| 视角 | 单页正确性（frontmatter/sources/断链合法性） | 单页 housekeeping（桩页/index 同步） | 全图质量（孤儿/缺失实体/断链聚合） |
| 断链 | `wikilink.broken`（独立 check 报 violation；写门禁 warning 不阻断，决策8） | 不查 | `broken_link` + `missing_entity`（建议聚合） |
| 退出码 | 0 / 3（硬） | 0 /（`--strict`）6（建议） | 0 /（`--strict`）6（建议） |

### 5.4 输出契约

```jsonc
// guanlan lint --json
{ "ok": false, "pages_checked": 12,
  "findings": [
    {"page": "wiki/sources/x.md",   "kind": "lint.orphan",        "detail": "无任何入链（type=source）"},
    {"page": "wiki/concepts/Bar.md","kind": "lint.broken_link",   "detail": "[[Baz]] 无对应页面"},
    {"page": "",                    "kind": "lint.missing_entity","detail": "[[Baz]] 被 3 页引用却无页面，建议建 entities/Baz.md"}
  ] }
```

`missing_entity` 是跨页聚合、无单一归属页，`page` 留空串（消费侧据此识别全局 finding）。

## 6. graph 工作流细化（`guanlan/graph.py`）

确定性 wikilink graph，是 `lint` 的算法底座，也是 `graph.html` 可视化的数据源。**\[决策P3-5]**

### 6.1 建图

```python
@dataclass(frozen=True)
class Node:   id: str; title: str; type: str; path: str   # id = stem（小写，与解析键一致）
@dataclass(frozen=True)
class Edge:   source: str; target: str; resolved: bool     # resolved=False 即断链边
@dataclass
class Graph:  nodes: list[Node]; edges: list[Edge]; adjacency: dict[str, set[str]]; broken: list[Edge]

def build_graph(wiki: Path) -> Graph: ...
```

- **节点** = `wiki/` 下所有**非 config** 页（排除 `index/log/overview`）。`id` = `pages.link_stem(stem)`（小写），与 `check`/`lint` 的解析键完全一致——同一口径，不漂移。
- **节点 `title`/`type` 容错取自 frontmatter**（决策P3-8）：`load_page` 拿不到合法 meta（缺块/无法解析/缺键）时回退 `title = path.stem`、`type = "unknown"`，**照常建节点、照常建图**——页存在于磁盘即是内容，坏 frontmatter 不该让它从图里消失。`"unknown"` 不在 `_VALID_TYPES` 里，但 graph/lint **不做** frontmatter 校验（那是 `check` 的职责），只把它当一个展示用的类型标签。
- **解析集与 `check` 完全一致**：用 `wiki/` 下**所有**页面 stem（**含 config**，大小写不敏感）作链接解析集，同 `check.py`。每个 `[[…]]` 经 `link_stem` 归一后**三分类**：
  - ① 命中**非 config** 页 → `resolved=True` 边（进 `adjacency`）；
  - ② 命中 **config** 页（index/log/overview）→ **直接丢弃**：config 不是知识节点，既不建边、也**不算断链**——与 `check` 不报此类断链一致；
  - ③ 谁都没命中 → `broken`（`resolved=False`，不进邻接表）。
- 如此 **graph 的 `broken` 恒等于 `check.wikilink.broken`**，`lint.broken_link` 复用 graph 的 broken 边不会报出 `check` 不报的断链（决策P3-6）。
- 自环（页链自己）按出现保留为边但建图时去重 `(source,target)`，避免可视化重边。

### 6.2 `graph.json`（稳定契约）

```jsonc
// <root>/graph/graph.json
{ "generated_from": "wiki/",
  "stats": {"nodes": 12, "edges": 18, "broken": 2, "orphans": 3},
  "nodes": [{"id": "foo", "title": "Foo", "type": "entity", "path": "wiki/entities/Foo.md"}],
  "edges": [{"source": "foo", "target": "bar", "resolved": true},
            {"source": "foo", "target": "baz", "resolved": false}] }
```

排序稳定（节点按 `id`、边按 `(source,target)`），保证同一 wiki 幂等产出字节级一致的 json，便于 diff 与缓存判定。

### 6.3 `graph.html`（自包含，最小只读邻接列表）

> **\[决策P3-7] P3 的 html 只做最小只读的邻接列表静态视图，不做任何图形布局/交互。** P3 的硬承诺是 `graph.json` 稳定；`graph.html` 仅是"打开就能看个大概"的便利产物，避免把确定性 graph **数据**退化成前端**渲染项目**。SVG/canvas 绘图、力导向、拖拽 / 缩放等图形化展示整体留待 **P4 Web UI**。

- **单文件、零网络**：内联 `graph.json` 数据 + 一段极小的内联标记，不引 CDN/外链、不打包第三方库。打开即用，可随 wiki 拷走。
- **形态 = 邻接列表**：按 `type` 分组列出每张页面 → 其 resolved 外链邻接；孤儿（入度 0）与断链目标以文字标注。**无图形布局算法**——纯文本/列表结构在 Python 端按稳定排序生成，天然**字节稳定、可幂等重建**（与 `graph.json` 同性质，不含任何坐标计算或 JS 随机渲染）。
- `--json-only` 时跳过 html，只写 json（CI/nightly 取数据不需要可视化）。

### 6.4 产物落点与可变性

- 落 `<root>/graph/`（DESIGN §4.2：`graph/` 持有 graph.json + graph.html，P3）。目录不存在则创建。
- `graph/` 是**可幂等重建的派生物**，默认 `.gitignore`（DESIGN §4.7 / 核心原则 1）——绝不反向成为权威，删了再 `guanlan graph` 即重建。
- 写 `graph/` 失败（权限/IO）→ `EXIT_USAGE(1)`，不污染 wiki。`graph` 只读 `wiki/`、只写 `graph/`，**绝不**碰 `raw/`/`wiki/`。

## 7. 退出码与失败处理（`guanlan/errors.py`）

P3 新增一个码，其余复用 P2：

| 码 | 常量 | 含义 | 触发 |
|----|------|------|------|
| 0 | `EXIT_OK` | 成功 | 命令完成（health/lint 即便有 findings，默认仍 0——**建议非门禁**） |
| 1 | `EXIT_USAGE` | 用法/IO 错误 | 非知识库根；`graph` 写 `graph/` 失败 |
| 6 | `EXIT_LINT_FINDINGS` | health/lint 在 `--strict` 下有 findings | 供 CI/nightly 把"有待办建议"表达为非零退出 |

**\[决策P3-4]** health/lint 默认 `EXIT_OK`（即便满屏 findings），因为它们是**建议性审计**而非正确性门禁——把孤儿/桩页当 CI 失败会逼人过早造桩页、与 Karpathy 自然增长相悖（同 P2 决策8 精神）。需要在流水线里 gate 的用户显式加 `--strict` → `EXIT_LINT_FINDINGS(6)`。`graph` 无 findings 概念，成功恒 0。

> 三命令都**不回滚、不改 wiki**（health/lint 只读；graph 只写派生 graph/）；与 P2 写门禁的"失败留盘"语义无关——P3 不在写路径上。

## 8. CLI 契约（`guanlan/cli.py`）

P2 命令不变，新增三个（沿用全局 `-C/--dir`；health/lint 支持 `--json`，graph 支持 `--json-only`）：

```
guanlan health [--json] [--strict]
guanlan lint   [--json] [--strict]
guanlan graph  [--json-only]
```

- 前置校验复用 `require_kb_root(root, writable=False)`（三者都不写 `wiki/`；`graph` 写派生 `graph/` 不纳入 kb-root 校验项，仅需 `wiki/` 存在）。失败 → `EXIT_USAGE(1)` 并提示 `guanlan init`。
- `--json` / `--json-only` 走稳定机器契约（§4.3 / §5.4 / §6.2）；无之则人类可读逐行报告。
- 各命令的 `*_entrypoint` 单一实现，CLI 与 `python -m guanlan.<mod>`（如有）共用，避免错误流/退出码/报告格式漂移（同 P2 `check_entrypoint` 范式）。

## 9. 对 DESIGN.md 的偏离与决策记录

- **\[决策P3-1] health/lint/graph 实现落 `guanlan` 包内，而非 skill 的 `scripts/`。** DESIGN §4.6 把 `health.py`/`build_graph.py` 列为 skill `scripts/` 候选，§4.2/§4.6 明记其"落点待定"。**本文定为落包内**（`guanlan/health.py`/`lint.py`/`graph.py`，暴露为 `guanlan health/lint/graph`），延续 P2 决策1 的全部理由：单一可信源、可 import 可测、规避开发期（repo `skills/`）与安装期（`~/.agentao/skills/`）的双路径分叉。虽然 P3 三工具**不**像 `check` 那样被写门禁内调（决策P3-3），但它们与 `check` **共享同一批页面/链接解析原语**（决策P3-2），分置两地必然双实现、口径漂移——故同地同范式最省。skill 只指示 agent shell 调 `guanlan health/lint/graph`。
- **\[决策P3-2] 抽取 `guanlan/pages.py` 共享页面原语，frontmatter 读取分严格/容错两档。** P2 把 `_extract_frontmatter`/`_parse_frontmatter`/`_WIKILINK_RE`/`_link_stem`/`_CONFIG_PAGES`/页面遍历都私有化在 `check.py`。P3 的 health/lint/graph 都要"遍历非 config 页 + 解析 frontmatter + 解析 `[[…]]`"，若各自重写，**断链/节点的解析口径会在四个模块间漂移**（大小写、`|别名`、`#锚点`、`.md` 后缀剥离规则）。故把这些上提为公共 `pages.py`，`check.py` 改为 import。**但 frontmatter 解析必须保留两档 API、不可合一**：`check` 走**严格档**（`split_frontmatter`+`parse_frontmatter`，坏 frontmatter 硬报 violation，行为与 P2 完全一致，`test_check.py` 回归护栏）；health/lint/graph 走**容错档**（`load_page`，坏 meta 返回 `None`、绝不抛，决策P3-8）。**严禁让 `check` 改用 `load_page`**——否则 `guanlan check` 会静默吞掉 frontmatter 错误，写门禁失守。这是 P3 的前置重构。
- **\[决策P3-3] P3 不改写写门禁。** DESIGN §4.7 / P2 §5.3 的写门禁 = `check` + `raw/` 快照，二者足以兜底正确性与不变性。health/lint/graph 是**按需维护命令**，绝不塞进 ingest/`query --backfill` 的收尾门禁——否则每次写都被全库历史质量债（孤儿/缺失实体）卡住，重蹈决策8 否掉的覆辙。三者只读 `wiki/`（graph 另写派生 `graph/`），与写路径解耦。
- **\[决策P3-4] health/lint 的 findings 是建议性（advisory），默认 `EXIT_OK`，仅 `--strict` 下以新码 `EXIT_LINT_FINDINGS(6)` 失败。** 孤儿/桩页是建库期的**自然中间态**（前向引用尚未补齐、新页待充实），把它们当硬失败会逼人提前造桩页、删链接，违背 Karpathy 自然增长（同 P2 决策8）。区别于 `check`：`check` 的 frontmatter/sources 是硬正确性（退 3），lint 的同一断链事实只是建议（退 0）。CI/nightly 想 gate 时显式 `--strict`。
- **\[决策P3-5] graph 产物落 `<root>/graph/`，`graph.html` 自包含、零网络。** DESIGN §4.2 已为 `graph/` 预留位置并标注 P3。`graph.json` 排序稳定、幂等可重建（核心原则 1：派生物不反向成权威，默认 `.gitignore`）。`graph.html` 内联数据 + 最小只读邻接列表静态视图（不引 CDN/第三方库、零布局算法，详见决策P3-7），保证脱网可用、字节稳定、可随 wiki 拷走。`lint` 复用 `build_graph` 的邻接表算孤儿/缺失实体——建图逻辑单份。
- **\[决策P3-6] graph 的断链口径与 `check` 完全一致；指向 config 的链接丢弃而非记 broken。** graph 的**节点集**只含非 config 页，但**链接解析集沿用 `check` 的全页面 stem 集（含 config）**。指向 config 页（index/log/overview）的 `[[…]]` **直接丢弃**（既不建边、也不算断链），唯有谁都不命中的才记 `broken`。如此 **`graph.broken ≡ check.wikilink.broken`**，`lint.broken_link` 复用 graph 的 broken 边不会与 `check` 打架。若反过来把"指向 config 的链接"在 graph 里记为 broken（本文初稿如此），lint 会报出 `check` 不报的 config 断链，并与 §11"与 check 比对"的测试直接冲突——故否掉。
- **\[决策P3-7] graph.html 是 Python 端生成的最小只读邻接列表静态视图，不做任何图形布局/绘制/交互。** DESIGN 未规定 html 形态；P3 的硬承诺只是 `graph.json` 稳定。html 仅为"打开看个大概"的便利产物，**只交付"按 `type` 分组、每页 + 其 resolved 外链邻接、孤儿/断链文字标注"的静态列表**——纯列表结构在 Python 端稳定排序生成，天然字节稳定、零坐标计算、零第三方库、脱网可开。任何图形化绘制（SVG/canvas 布局、力导向、拖拽/缩放）整体留 **P4 Web UI**。否决"Python 端预计算静态 SVG 布局"草案：它等于自写一个确定性图布局算法，是 P3 最易过度投入、又最不属于硬承诺的部分，与"技术路线简明、不过度设计"相悖。
- **\[决策P3-8] P3 工具对坏 frontmatter 容错读取，不重复做 frontmatter 校验。** health/lint/graph 是**按需审计命令**，绝不能因为某页 frontmatter 缺块/无法解析/缺 `type` 就崩溃——而这些恰是 `check` 负责报的问题。故 `pages.load_page` 容错（坏 meta 返回 `None`、不抛）；graph 建节点时回退 `title=path.stem`、`type="unknown"` 照常建图；lint 的 orphan 报告里直接展示该 `type`（可能为 `"unknown"`）。**frontmatter 正确性单一归口 `check`**，P3 工具不复制这套校验、也不因坏数据中断审计（与决策P3-3"P3 不进写门禁"一脉：P3 只观察、不裁决正确性）。

## 10. 不纳入 P3（推后）

- **语义 lint**（矛盾复检 open→resolved 状态流转、过期论断、资料缺口）→ P3 之后，需 LLM、成本高，应可按需触发而非每次全量（DESIGN §4.5 / §8）。P3 health 不碰矛盾统计（只做桩页 + index 同步），矛盾复检整体归语义层。
- **graph 增强**：LLM 推断边（`INFERRED`/`AMBIGUOUS`）、Louvain 社区检测、图感知语义 lint、SHA256 增量缓存 → P3 之后，需可关闭 + 采样策略（DESIGN §8）。P3 只做确定性 wikilink graph。
- **稀疏页 lint（`lint.sparse_page`）** → 推后：信号与 `stub`（几乎空）、`orphan`（无入链）高度重叠，却独占"出度 + 字数"双阈值，边际价值低；待有实证再评估（或并入 `stub`）。
- **log 覆盖体检（`health.log_uncovered`）** → 推后：仅在"有 source 页却零 ingest 记录"时触发，经 `guanlan ingest` 正常建库几乎永不亮；逐页 log 对账依赖标题语义，归语义层（P3 之后）。P3 的 health 不读 `log.md`。
- **阈值的 SCHEMA 覆盖**：`STUB_MIN_CHARS` / `MISSING_ENTITY_MIN_REFS` 在 P3 是包内常量（附注由来）；让 `SCHEMA.md` 覆盖之留待有实证需求后再做，避免过早加配置面。
- **index 分区/type 匹配校验**（source 页是否落在 `## Sources` 区等）→ 推后或不做：耦合 Agent 生成的 index 文案结构、易误报；P3 的 health 只做"页是否被 index 收录"的双向存在性（§4.2）。
- **graph.html 交互**（拖拽/缩放/动态力导向）→ P4 Web UI；P3 的 html 是 Python 端确定性预计算的最小只读静态视图（决策P3-7）。
- **nightly 自动跑 health/lint** → P5（自动化）。P3 只交付可被定时器调用的命令本身。
- **多格式摄入** → P5；**Web UI / graph 嵌入 host** → P4。
