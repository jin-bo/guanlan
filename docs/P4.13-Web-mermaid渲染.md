# P4.13 — Web 端 mermaid 图表渲染落地小设计

> 本文给 [`P4-Web宿主.md`](P4-Web宿主.md) 的安全 markdown 渲染补一类内容呈现：把 wiki 页与对话答案里的
> ` ```mermaid ` 围栏代码块**在浏览器内渲染成图**（流程图 / 时序图 / 类图…），而非现状的字面源码。
> 高层理念以 [`DESIGN.md`](DESIGN.md)（原则 1「markdown 是唯一事实来源」、§5.2 读写分线、附录 A）为准；
> 本文只把「怎么做」钉到**前端拾取点 / 资产承载 / 安全闸 / 测试**。标注的 **\[决策P4.13-N]** 集中记于 §6。
>
> **不是新相位、不动事实来源。** 这是 P4 Web 宿主之上、**纯前端呈现层**的半相位增强——mermaid 源始终是
> 页面 markdown 的一部分（` ```mermaid ` 围栏块），渲染只发生在浏览器、不写盘、不入任何派生物、不改 `render.py`
> 的转义姿态。故编号 P4.13，承 P4.12 Web-audit 之后。**与 P5 `graph`/`graph.html` 无关**——那是 wikilink
> 关系图（确定性 Python 生成），本相位是**用户在页面正文里手写的 mermaid DSL**，两者数据来源与渲染路径都不同。
> 请先读 [`P4-Web宿主.md`](P4-Web宿主.md)（§6 安全 markdown 渲染 / §7 决策P4-3「无 npm/CDN/第三方运行时」/
> §8 安全姿态）与 [`render.py`](../guanlan/web/render.py) 顶注（`_EscapeHtmlExtension` 关原始 HTML 透传）。

## 1. 目标与范围

**P4.13 = 让 Web 宿主把页面里的 mermaid 围栏块画成图。** 现状：` ```mermaid\ngraph TD\n  A-->B\n``` ` 经
`render_markdown` 的 `fenced_code` 扩展渲染成

```html
<pre><code class="language-mermaid">graph TD
  A--&gt;B
</code></pre>
```

——浏览器里显示为**图的源码文本**，不是图。LLM 综合 wiki 页时常会画流程/时序/状态图来表达实体关系或论证链，
用户也会在源里手写；P4.13 把这层「最后一公里」补上：**同一段 markdown 在浏览器里成图，在 CLI / 纯文本回退里
仍是 honest 的源码**（markdown 仍是唯一事实来源，§DESIGN 原则 1 不破）。

落点是**纯前端**：服务端**一字不改**（`fenced_code` 已把图源完整保留为 `code.language-mermaid` 的转义文本，
见 §3），前端在每个 markdown 注入点之后扫这些 `<code>`、读其 `.textContent`（即图源 DSL）、经 **vendored 的
mermaid 运行时**渲染成 SVG 就地替换。**唯一的真实张力**是：mermaid 是个第三方 JS 运行时，撞 P4 决策P4-3
「前端无 npm/构建/CDN/第三方运行时」——§6 决策P4.13-1 正面处理（vendored + 懒加载 + 归类为「内容渲染器」而非
应用框架，与服务端已 vendored 的 `markdown` 库同位阶）。

### 纳入 P4.13

- **前端拾取 + 渲染**（`static/mermaid_enhance.js`，约 80 行 vanilla）：`enhanceMermaid(container)` 扫
  `container.querySelectorAll('pre > code.language-mermaid')`，**懒加载**（首次遇到才注入 `<script>` 加载 vendored 运行时）→
  `mermaid.render(id, code.textContent)` → 用返回 SVG 就地替换那个 `<pre>`。在**四类实际注入点的每次 `innerHTML`
  赋值之后**各调一次——**关键是挂在重绘函数内部、而非「首渲后调一次」**（决策P4.13-8）：
  - **wiki 单页**：`wiki.js` `paintPage()` 内 `view.innerHTML = …`（`wiki.js:196`）之后——它**既**是首渲路径**又**是
    语言切换 `repaintPageChrome()`（`i18n_apply.js:24` → `wiki.js:200` → `paintPage`）的重绘路径，挂在 `paintPage`
    内部即一并覆盖二者，避免切语言把已渲 SVG 打回源码；
  - **对话答案**：`chat.js` 流式收尾 `botEl.innerHTML = payload.answer_html`（~620）+ 历史重渲 `el.innerHTML = m.html`（~168）；
  - **raw 预览**：`jobs.js` `view.innerHTML = data.html`（`jobs.js:114`，复用 `render_page`）；
  - **workspace parsed 预览**：`staging.js` 暂存区 parsed 预览的 `paint()` 闭包内、markdown 模式分支 `view.innerHTML = data.html`
    （`staging.js:542`）之后——它含 md/源码切换，挂在 `paint()` 内即令每次切回 markdown 都重新增强（与 `paintPage` 同理）。
- **vendored 运行时**（`static/vendor/mermaid.min.js`，钉版 v11.15.0、随 `web` extra 静态资产打包，**非 CDN**）：
  **自包含 UMD 单文件**（加载后 `window.mermaid` 就绪、**零运行时动态 `import()`**，离线自洽），经注入 `<script>`
  懒加载、**零构建步**（与决策P4-3「无打包」相容，见决策P4.13-1）。**故意不用 ESM**：mermaid v11 的
  `mermaid.esm.min.mjs` 是代码分割构建、运行时动态 import 子 chunk，单文件不离线自洽；UMD 单文件才适合离线 vendored。
- **安全配置硬编码**（决策P4.13-3 信任铰链）：`mermaid.initialize({ startOnLoad:false, securityLevel:'strict',
  htmlLabels:false, theme:'base', themeVariables: <观澜昼澜调色> })`。`strict` 令 mermaid 用其内置
  DOMPurify 消毒标签、**禁用 `click`/script 指令**——这是「内容→客户端 HTML」首次出现时的纵深防御核心。
- **优雅降级 + 错误兜底**：JS 关 / 运行时加载失败 / 图语法错 → **`<pre>` 源码原样留存**（mermaid 渲染包在
  `try/catch`，失败不替换、附一枚 `⚠ 图渲染失败` 错误徽标，徽标文案 `textContent` 注入防注入）。永不空白。
- **观澜配色主题（最小，不预留切换）**：经 `themeVariables` 把 mermaid 节点/线/字色映射到昼澜调色板（`app.css`
  既有 CSS 变量取值），**不引 mermaid 自带 CSS 框架**、不污染全局样式。仅交付昼澜一档常量，**不预留深色档切换接口**
  （决策P4-9 推后的「夜澜」真落地时再加，不在本相位预实现，避免过度设计）。
- **i18n**：仅一个错误徽标文案 key 进 `i18n.js` 的 `zh`/`en`（`mermaid.renderFail`，zh/en 两条），
  `tests/test_web_i18n.py` 的 key 平价校验自动覆盖。**不做加载占位**（`mermaid.loading`）——懒加载首付一次、本地
  同源毫秒级，占位是过度设计。

### 不纳入 P4.13（按既有收窄推后）

- **服务端预渲染 / 导出 SVG**（决策P4.13-2 否决）：mermaid 需 JS+DOM，服务端要么背 headless 浏览器
  （puppeteer/mermaid-cli，重依赖、与零新重依赖相悖），要么调远程 `mermaid.ink`（**把库内容外发 + 破离线**，
  撞 §8 仅 `127.0.0.1`/隐私姿态）——两条都否。渲染只在浏览器、按需。
- **非 mermaid 图语言**（plantuml / graphviz / 别的 DSL）→ 不做：用户要的是 mermaid，多引一种 = 多一个第三方
  运行时 + 多一片安全面，无实证需求。
- **CLI / MCP 端渲染** → 不做：那俩是**文本通道**（终端 / JSON-RPC），图渲染是宿主表现层职责，与「核心只读、
  呈现归宿主」一致；CLI `query` 仍回字面源码（honest），MCP 回 markdown 原文由客户端自渲。
- **实时编辑器 / 输入即预览 / 图源编辑** → 不做：本相位只读渲染既有页里的图。Web 不是 mermaid live editor。
- **图内 `click`/超链接交互** → 不做（被 `securityLevel:'strict'` 关死，安全优先；wiki 站内导航仍走
  `[[wikilink]]`，不靠图内 click）。
- **自动给页面注入图** → 不做：画不画图是 LLM 综合 / 用户手写的内容决策，宿主只渲染既有 ` ```mermaid ` 块。

## 3. 拾取与渲染（`mermaid_enhance.js`）

围栏块经 `fenced_code` 落成稳定结构 `pre > code.language-mermaid`（§2 实测），前端据此拾取，**与
`render.py` 不耦合**（class 是 python-markdown `fenced_code` 的契约，非我们自造）：

```js
let _mermaidP = null;                                  // 懒加载单例：首个 mermaid 块出现才付加载代价
function loadMermaid() {                               // 注入 <script> 加载自包含 UMD → window.mermaid
  if (_mermaidP) return _mermaidP;
  _mermaidP = new Promise((resolve, reject) => {
    const s = document.createElement("script");
    s.src = "/static/vendor/mermaid.min.js";
    s.onload = () => {                                  // UMD footer 已挂 window.mermaid
      window.mermaid.initialize({
        startOnLoad: false, securityLevel: "strict",   // 信任铰链：见 §4 / 决策P4.13-3
        htmlLabels: false, theme: "base",
        themeVariables: GUANLAN_MERMAID_THEME,           // 昼澜调色（取自 app.css 变量，决策P4.13-5）
      });
      resolve(window.mermaid);
    };
    s.onerror = () => reject(new Error("加载失败"));
    document.head.appendChild(s);
  }).catch((e) => { _mermaidP = null; throw e; });     // 失败不缓存、下次可重试
  return _mermaidP;
}

async function enhanceMermaid(container) {               // 在每个 innerHTML 注入点之后调用
  const blocks = container.querySelectorAll("pre > code.language-mermaid");
  if (!blocks.length) return;                            // 无 mermaid → 一行早退，零加载（懒）
  const m = await loadMermaid().catch(() => null);
  if (!m) { blocks.forEach(markMermaidFailed); return; } // 运行时加载失败 → 源码留存 + 徽标
  for (const code of blocks) {
    const pre = code.parentElement, src = code.textContent;  // 图 DSL（浏览器已反转义），非 HTML
    try {
      await m.parse(src);                                // 先校验语法（不注入 DOM）→ 不留孤儿错误节点
      const { svg } = await m.render(uniqueId(), src);   // strict 模式产 SVG（内置 DOMPurify 已消毒）
      const fig = document.createElement("figure");
      fig.className = "mermaid-rendered";
      fig.innerHTML = svg;                               // 唯一 mermaid 产物注入点（信任 strict）
      pre.replaceWith(fig);
    } catch { markMermaidFailed(code); }                 // 语法错 → 保留 <pre> 源 + 错误徽标
  }
}
```

- **懒加载（决策P4.13-1）**：`enhanceMermaid` 先 `querySelectorAll` 早退，**无 mermaid 块的页面/答案零加载**
  vendored 运行时——避免每次浏览都吃 ~800KB(gzip) 解析代价；只有真含图的页才付一次、之后单例复用。
- **幂等可重入 + 重绘必重增强（决策P4.13-8）**：替换后原 `code.language-mermaid` 已不在树里，对**同一 DOM**
  重复调 `enhanceMermaid` 不会二次渲染。但 `paintPage()`（语言切换）/ staging `paint()`（md↔源码切换）会**重设
  整页 `innerHTML`**、把已渲 SVG 打回 `<pre>` 源码——故 `enhanceMermaid` 必须挂在**这些重绘函数内部**，每次重设
  HTML 后再增强一次，而非「首渲后只调一次」。这是把渲染当作「对当前 HTML 的纯增强」而非一次性副作用的直接结论。
- **`uniqueId()` 无随机依赖**：用单调自增计数器生成 `mermaid-<n>` 容器 id（mermaid 内部需唯一 id），
  不依赖 `Math.random`（与前端既有惯例一致、可测）。
- **拾取点清单**（各在既有 `innerHTML` 赋值后补一行；**重绘路径挂在重绘函数内**）：
  - `wiki.js` `paintPage()` 内（:196）→ `enhanceMermaid(view)`——覆盖首渲 **+** 语言切换 `repaintPageChrome→paintPage` 重绘；
  - `chat.js` 流式收尾 `botEl.innerHTML = payload.answer_html`（~620）、历史重渲 `el.innerHTML = m.html`（~168）→ 各
    `enhanceMermaid(botEl/el)`；
  - `jobs.js` raw 预览 `view.innerHTML = data.html`（:114）→ `enhanceMermaid(view)`；
  - `staging.js` workspace parsed 预览 `paint()` 闭包 markdown 分支 `view.innerHTML = data.html`（:542）→
    `enhanceMermaid(view)`——覆盖 md↔源码切换重绘。

## 4. 安全分析（**本相位的实质**）　**\[决策P4.13-3]**

P4 的 XSS 纵深防御**全靠服务端转义**（`_EscapeHtmlExtension` 关原始 HTML + `_SafeLinkExtension` 中和危险协议，
决策P4-4），**前端此前不持任何 HTML 生成/消毒**。mermaid 把「页面内容」变成「客户端生成并 `innerHTML` 注入的
SVG」——这是该不变量的**首次破例**，故安全分析是本相位的核心，不是附注。

**威胁模型**：被投喂的 `raw/` 可能夹带恶意载荷（P4-4 既有假设），经 ingest 进 wiki 页正文；攻击者写一段
` ```mermaid ` 块，企图让节点标签 / 图指令携带 `<script>` 或事件处理器，经 SVG `innerHTML` 注入后以**同源**
身份调本地写/读 API。

**为何挡得住**，逐层：

1. **图源永远是文本、不是 HTML。** 前端读 `code.textContent` 拿到的是图 DSL 字符串（浏览器对 `<code>` 内
   实体已反转义为字面字符），**从不把它当 HTML 解析**。即便 DSL 里写了 `<img onerror=…>`，它只是 mermaid
   解析器眼里的一段标签文本。
2. **`securityLevel:'strict'`（铰链）。** mermaid strict 模式：① 节点/边标签经其**内置 DOMPurify** 消毒后才
   进 SVG；② **禁用 `click`/`callback`/script 指令**（`loose` 才放行）；③ `htmlLabels:false` 进一步令标签走
   纯 SVG `<text>`、不渲染 foreignObject-HTML。故恶意标签里的 `<script>`/`onerror` 在产物 SVG 里**已被剥除**。
3. **产物 SVG 注入 DOM、非以独立文档交付。** 对照 `helpers.py:_raw_image`（决策：raw `.svg` 是活跃内容，以
   `attachment`+`CSP: default-src 'none'; sandbox`+`nosniff` 关死直导航执行）——那是**用户原始 SVG 文件**经
   URL 直发的场景。本相位的 SVG 是 **mermaid strict 现造**、经其消毒、注入当前页 DOM（`<img>`/独立文档的脚本
   执行语义都不适用，且已无 `<script>` 残留）。两者风险面不同，但**项目对「SVG=活跃内容」的审慎一以贯之**，
   strict 模式正是本场景的等价闸。
4. **失败封闭。** `render` 抛错 → `catch` 保留 `<pre>` 源、不注入任何东西；加载失败 → 同样留源。任何异常路径
   都**不**产生未消毒注入。

**残余风险与处置（保守表述，不承诺 mermaid 产物绝对可信）**：mermaid 历史有过 XSS CVE（多发于
`loose`/`htmlLabels:true` 或旧版本）。**本期不把 mermaid 输出当作可无条件信任**——边界明确收在这组闸：
① **钉版本**到无已知 strict-mode 绕过的发行版、`vendor/README.md` 记版本 + SHA256，升级走显式 PR 复核；
② 配置**硬编码** `strict` + `htmlLabels:false`，前端无任何放宽旋钮（用户/页面都改不动）；③ **失败保留源码**
（任何异常不注入）。即本期边界 = **`strict` + `htmlLabels:false` + 钉版 + 失败保留源码**。**CSP 是独立加固、
不扩本期范围**：现状主 app 无 CSP（§helpers 那条只在 raw-svg 端点），给主文档补 `script-src 'self'` 等收紧是
独立的全局议题，单开、不绑 P4.13。

## 5. 前端呈现与主题（`app.css` / `i18n.js`）

- **自适应宽**：`.mermaid-rendered svg{ max-width:100%; height:auto; }`——图随右栏/对话气泡宽缩放，不溢出、
  不强行横向滚动（超宽大图给 `figure` 一个 `overflow-x:auto` 兜底）。
- **昼澜主题（决策P4.13-5，最小常量、不预留切换）**：` GUANLAN_MERMAID_THEME `（`mermaid_enhance.js` 顶部常量）
  把 mermaid `themeVariables`（`primaryColor`/`lineColor`/`textColor`/`background`…）取值为 `app.css` 既有昼澜
  调色板，使图与界面同一水墨澜色调，**不引 mermaid 默认 CSS**、不落全局样式污染。**仅一档常量、不预留深色切换
  接口**——深色「夜澜」（决策P4-9 推后）真落地时另加，不在本相位预实现（避免过度设计）。
- **错误徽标**：`.mermaid-error` 一枚行内徽标贴在保留的 `<pre>` 上方，文案走 i18n `mermaid.renderFail`
  （`textContent` 注入，绝不 `innerHTML`，防徽标本身成注入点）。**不做加载占位**（懒加载本地同源毫秒级）。
- **i18n（决策P4.7 沿用）**：`i18n.js` 的 `zh`/`en` 各加**仅** `"mermaid.renderFail"`（无 `mermaid.loading`），
  `test_web_i18n.py` 的 zh/en key 平价 + `data-i18n` 白名单解析自动覆盖。图本身是页面内容；**语言切换时图经
  `paintPage`/staging `paint()` 重绘路径被重新增强**（决策P4.13-8），故切语言不会把图打回源码。

## 6. 决策记录

- **\[决策P4.13-1] mermaid 运行时 vendored（非 CDN）+ 懒加载 + 自包含 UMD 单文件零构建，归类为「内容渲染器」
  而非应用框架——这是对决策P4-3 的有意、有界破例。** 决策P4-3「前端无 npm/构建/CDN/第三方运行时」针对的是**应用
  工程化**（React/Vue/打包链）。mermaid 与服务端已作为 `web` extra vendored 的 `markdown` 库**同位阶**：都是
  「把一种标记渲染成富呈现」的**内容渲染器**，不是 UI 框架。处置守住决策P4-3 的三条实质底线：① **无 CDN**——
  运行时随 wheel 静态资产打包、`127.0.0.1` 自发，**离线可用**（§8 姿态不破）；② **无构建**——用**自包含 UMD
  单文件**（`mermaid.min.js`，经注入 `<script>` 加载读 `window.mermaid`），无 npm/打包链；**故意不用 ESM**——
  mermaid v11 的 `mermaid.esm.min.mjs` 是代码分割构建、运行时动态 import 子 chunk，单文件不离线自洽（实测 UMD
  build 零运行时动态 `import(`、核心图型全内联，故离线自洽）；③ **零额外加载代价于不含图的页**——懒加载，首个
  mermaid 块出现才付一次。代价是 `web` extra 静态资产增约 3.3MB（gzip ~800KB），与既有 `logo.png`(1.6MB) 同量级、
  可接受；`vendor/README.md` 钉版本 v11.15.0 + SHA256。**这是本相位唯一需签字的偏离**——若不接受第三方运行时，
  退路是「不渲染、保持现状字面源码」（决策P4.13-6 的 honest 回退本就常驻）。
- **\[决策P4.13-2] 否决服务端预渲染 / 远程渲染。** mermaid 需 JS+DOM：服务端预渲染要背 headless 浏览器
  （puppeteer/mermaid-cli，重依赖，撞零新重依赖）；远程 `mermaid.ink` 会**把库内容外发并破离线**（撞 §8 仅
  `127.0.0.1`/隐私姿态、撞 DESIGN「本地全权」）。故渲染**只在浏览器、按需**——也使服务端零改、`render.py` 字节稳定。
- **\[决策P4.13-3] 「内容→客户端 HTML」首次破例的本期安全边界 = `strict` + `htmlLabels:false` + 钉版 +
  失败保留源码这一组闸；不承诺 mermaid 产物绝对可信。** 见 §4。strict 用内置 DOMPurify 消毒标签、禁
  `click`/script 指令；`htmlLabels:false` 令标签走纯 SVG `<text>`；配置硬编码、无放宽旋钮；渲染包 try/catch、
  任何异常都退回保留 `<pre>` 源、零未消毒注入；残余 CVE 风险以钉版 + 显式升级 PR 复核处置。**措辞刻意保守**：
  不把信任「完全压在」mermaid 上、不让实现者据此接受任意 mermaid SVG 无审查注入。**全局 CSP 收紧（`script-src
  'self'` 等）是独立加固、不扩本期范围**——现状主 app 无 CSP，单开议题、不绑 P4.13。
- **\[决策P4.13-4] 服务端零改、转义姿态不破——纯靠 `fenced_code` 的 `code.language-mermaid` + 前端读
  `.textContent`。** `fenced_code` 已把图源完整保留为转义文本（实测 `A-->B`→`A--&gt;B`），前端读 `textContent`
  拿到的是反转义后的**图 DSL（文本，非 HTML）**。故 `_EscapeHtmlExtension` 关原始 HTML 透传**全程有效**，
  `render.py` 不动一行、`render_markdown` 其余调用方（CLI 无关，但同函数）零回归。不在服务端造 `<div class="mermaid">`
  （那要改转义姿态、走 stash、扩攻击面）——前端拾取 class 是更薄更安全的归口。
- **\[决策P4.13-5] 主题经 `themeVariables` 映射昼澜调色，最小常量、不预留切换接口。** 一个常量令图与界面同
  水墨澜色调，不引 mermaid 自带 CSS、不污染全局；**仅昼澜一档、不预留深色切换接口**——深色「夜澜」（决策P4-9
  推后）真落地时另加，本相位不预实现（避免过度设计，按评审收窄）。
- **\[决策P4.13-6] 优雅降级常驻：JS 关 / 运行时加载失败 / 语法错 → `<pre>` 源码留存 + 错误徽标，永不空白。**
  markdown 仍是唯一事实来源（DESIGN 原则 1）：渲染是**叠加增强**，剥掉它页面退回 honest 的图源文本，零信息损失。
  这也是决策P4.13-1 不接受第三方运行时时的天然退路。
- **\[决策P4.13-7] 渲染只在 Web 表现层；CLI/MCP 不渲染。** 图渲染是宿主呈现层职责，与「核心只读、呈现归宿主」
  一致：CLI `query` 回字面源码、MCP 回 markdown 原文由客户端自渲；不把 mermaid 依赖渗进核心或 CLI。
- **\[决策P4.13-8] `enhanceMermaid` 必须挂在重绘函数内部、随每次 `innerHTML` 重设而重增强，而非首渲调一次。**
  `wiki.js` `paintPage()`（语言切换 `repaintPageChrome` 复用它重绘整页）与 `staging.js` `paint()`（md↔源码切换）
  都会**重设 `innerHTML`、把已渲 SVG 打回 `<pre>` 源**。把增强钩子放进这两个重绘函数内（而非「打开页面时调一次」），
  才能保证切语言 / 切预览模式后图仍是图。本质：把 mermaid 渲染当作**对当前 HTML 的纯增强**（幂等、可重入），而非
  一次性副作用。这条直接修正了初稿「图不随 UI 语言切换重渲」的错误论断（语言切换确会重绘，故必须重增强）。
