## 0. 为什么是文档先验、不并入 tf

P5.0 已有 title/alias **字段加权**（BM25F-lite，决策P5.0-17：boost 只抬 `tf`、不进 `dl/avgdl`）。那是**词项信号**
（与 query 词相关）。**入链数是查询无关的文档级先验**（一页被越多页引用 → 越可能是该主题枢纽），**无法**塞进
`tf`（它不依赖 query 词）。故 backlink 作为 **post-BM25 乘性文档先验**施加——标准 IR 做法（类 PageRank prior），
**与 P5.0 字段加权正交**、决策P5.0-17 不破。

## 1. boost 公式（关键：零反链页不可被归零）

在 `score()`（`search.py:294-348`）算出某页 BM25 原始分 `s` 后，**收录门槛判在 boost 之前**——先用未加
boost 的 `round(s,6)` 决定是否召回，仅对**已收录**页施加反链重排：

```python
rounded = round(s, 6)                  # 收录门槛 = 未加 boost 的 BM25 原始分（决策P5.0-20）
if rounded <= 0:
    continue                           # 弱命中（<5e-7）/无命中：纯 BM25 即排除，boost 绝不捞回
if inlinks is not None:                # 缺省 None → 不重排，纯 BM25（决策P5.3-4）
    c = inlinks.get(d.page, 0)         # d.page = 相对库根 posix，与 graph node.path 同基
    if c:                              # c=0 → 因子 1.0，跳过（零反链页分数 == 纯 BM25、字节不变）
        rounded = round(s * (1.0 + BACKLINK_WEIGHT * math.log1p(c)), 6)
scored.append((d, rounded))
```

- **因子 `1 + W·ln(1+c)`**，**非** `ln(1+c)`（决策P5.3-2）：纯 `ln(1+c)` 在 `c=0` 时 = 0，会把**零反链页直接归零、挤出结果**——错。`1 + W·ln(1+c)` 保证 c=0 因子恰 1.0、零反链页 BM25 分**原样保留**，只让有反链页温和上浮。
- `BACKLINK_WEIGHT` 默认 **0.5**（决策P5.3-2，可调、字节稳定）：10 入链页 ×(1+0.5·ln11)≈×2.2、1 入链 ×1.35、0 入链 ×1.0——温和、不喧宾夺主。
- **不改收录门槛、绝不凭空召回**（决策P5.3-7，review 加固）：收录判定用**未加 boost** 的 `round(s,6)>0`，故 boost **只重排已召回页**——`s=0`（无命中）恒被滤、低于显示地板（<5e-7）的弱命中页也**不被 boost 捞回**（与纯 BM25 召回集逐页一致，只名次可变）。
- **确定性 / 字节稳定**：`round(…,6)` + sort `(-score, page)`（`search.py:337`）仍是量化点 / tie-break，boost 后字节稳定不破（决策P5.3-7）。`math.log1p(int)` 确定性，c=0 跳过乘法使零反链页分数与纯 BM25 **逐字节相同**。

## 2. 入链计数：复用 graph 解析口径（不是"已算好"，是"复用同一张解析表"）

**绝不**另起第二套链接解析。复用 `graph.build_graph`（`graph.py:83-152`）已解析的 resolved 边（已 `link_stem`/
别名归一、排 config、P3.8 后含 fold 变体），算每页入度。`compute_backlinks` **落在 `graph.py`、紧挨
`compute_orphans`**（`graph.py:155`）——二者本就是同一「入链集」语义（同样排自环 `source != target`、同样只数
resolved），同放一处使口径单一、改一处即见另一处，杜绝漂移（决策P5.3-3）：

```python
# guanlan/graph.py（compute_orphans 旁，加入 __all__）
def compute_backlinks(g: Graph) -> dict[str, int]:
    """每页入链数（resolved 边入度，排自环、排 broken）。键用 node.path（相对库根 posix），对齐 DocBag.page。
    与 compute_orphans 同一入链口径；alias/fold 已在 build_graph 解析期归一（决策P3-6/P3.1-5/P3.8-3）。"""
    counts = {n.id: 0 for n in g.nodes}
    for e in g.edges:                                  # broken 边 resolved=False，被排除
        if e.resolved and e.source != e.target:        # 自环不算入链（同 compute_orphans）
            counts[e.target] += 1
    return {n.path: counts[n.id] for n in g.nodes}      # 转 path 键，对齐 DocBag.page
```

- **键用 `node.path`**（相对库根 posix，如 `wiki/entities/Foo.md`）对齐 `DocBag.page`（决策P5.0-22 同基）——免 stem 往返、零歧义。
- 排自环、排 broken 边（断链无语义入链）。alias/fold 链已在 `build_graph` 解析期归到拥有页节点，故反链计数 = lint/check 同口径。
- **`score` 与 `DocBag` 不 import graph**：`compute_backlinks` / `build_graph` 只在 `search_pages` 与 `CorpusCache.backlinks` 两个**调用点函数级导入**（决策P5.3-3），保 `score` 这一廉价核与数据类的 import 面 graph-free，守 search.py「零基建纯检索核」体例。

## 3. 四处接入（boost 进 `score()` → CLI / Web /api/search / chat 工具 / MCP 同享名次）

`score()` 是 CLI(`search_pages`)/Web(`/api/search`+内嵌 `guanlan_search` 工具)/MCP(`tool_search`) 的公共底座
（`search.py:294` / `web/app.py:665` / `web/chat.py:214` / `mcp/tools.py:171`），boost 必须**住 `score()` 内**才四处同得：

```python
def score(docs, query, *, limit, inlinks=None):   # inlinks 缺省 None → 零 boost
    ...
```

- **决策P5.3-4（`inlinks=None` 向后兼容 + 单一入口收口脚枪）**：不传 → 纯 BM25（既有直调 `score`/无图测试**零扰动**；boost 是加法）。但「漏传 `inlinks` 即静默丢重排」是脚枪，故**三处长驻接入收进 `CorpusCache.search(wiki, query, *, limit)` 单一入口**（内部配好 corpus + backlinks + score、保证 docs/inlinks 取自同一次 corpus），调用方只给 `(wiki, query, limit)`、无从漏传。CLI 冷路用对称的 `search_pages`（同样内部配好）。两入口都返回 `SearchResult`、字节等价：

  | 入口 | 实现 |
  |---|---|
  | CLI `search_pages(wiki, query, limit=)`（冷算权威） | `build_corpus` + `compute_backlinks(build_graph)` + `score` |
  | Web `/api/search` | `cache.search(wiki, q, limit=)` |
  | 内嵌 chat `guanlan_search` 工具 | `cache.search(wiki, q, limit=)` |
  | MCP `tool_search` | `cache.search(wiki, q, limit=)` |

  `CorpusCache.search` 内部统一形（`backlinks` 由签名 memo、签名变才 `build_graph`，§4）：

  ```python
  def search(self, wiki, query, *, limit):
      docs = self.corpus(wiki)
      inlinks = self.backlinks(wiki, docs)        # docs 必来自上一行同一次 corpus（契约在方法内自洽）
      return score(docs, query, limit=limit, inlinks=inlinks)
  ```

## 4. 性能口径（核心：冷路代价 + 热路 memo 底线 + 最小 memo 设计）

反链是**额外一次全库只读扫描**（§0 ⚠），不能假装免费。分层处理：

### 4.1 实现口径（正确性）

反链计数**复用 `build_graph` 解析口径**（§2），不写第二套 wikilink 解析——保证反链数 ≡ graph 入度 ≡ lint/check
同口径，alias/fold-aware。代价是 `build_graph` 比 `build_corpus` 重（全库 `yaml.safe_load` + 别名索引），这是为「单一
解析口径、零漂移」付的确定代价，**不**为省微秒去手搓一套更轻的反链解析（那会引入第二套口径、必漂移）。

### 4.2 CLI 冷路（接受双全库扫描）

`search_pages` 现在 = `build_corpus`（一遍全库读+分词）+ `build_graph`（再一遍全库读+YAML+链接解析）+ `score`。
冷启大致**翻倍全库扫描**（建图那遍更重）。**CLI 接受**：它本就是一次性冷算权威、已做整库扫描、人手交互触发、无长驻
进程可缓存（新进程吃不到 cache）。性能重点不在这。

### 4.3 Web/MCP 热路（memo 是硬底线）

长驻进程**绝不**每次 `/api/search` 都 `build_graph`——否则在 P5.1 的热 corpus 缓存之外又做一次全库 YAML 解析，
P5.1 的缓存收益被吃光。故反链结果**必须 memo**，签名不变直接返回缓存、只在语料真变时才重算图。

### 4.4 `CorpusCache.backlinks(wiki, docs)` 最小 memo 设计（不升级成缓存系统）

挂在既有 `CorpusCache` 上（复用其**按库根分桶** + `threading.Lock` 并发体例），但**只做最小 memo**，反链是与 DocBag
**并列的独立产物**、占**独立 memo 槽**（不与 DocBag memo 纠缠，回应评审「反链是另一种产物」之虑）：

```python
class CorpusCache:
    def __init__(self):
        self._caches: dict[str, dict[str, DocBag]] = {}          # 既有 DocBag memo
        self._backlinks: dict[str, tuple[tuple, dict[str, int]]] = {}  # 新增：bucket → (sig, {path:count})
        self._lock = threading.Lock()

    def backlinks(self, wiki: Path, docs: list[DocBag]) -> dict[str, int]:
        # 契约：docs **必须来自同一次 `self.corpus(wiki)`**——它按 `iter_pages` 稳定排序返回（content 页、
        # 不含 config），故签名**无须再排序**。传入乱序 docs 至多触发一次多余重算（安全降级），绝不误命中致陈旧。
        wiki = Path(wiki).resolve()
        bucket = wiki.parent.as_posix()                          # 复用既有分桶键
        # 签名 = 复用调用方刚拿到的 docs（DocBag 自带 mtime_ns/size、已是 iter_pages 序），**零额外 stat
        # 扫描、零额外排序**。content 页面、不含 config——失效面说明见下「失效信号」要点：
        sig = tuple((d.page, d.mtime_ns, d.size) for d in docs)
        with self._lock:                                         # 短临界区：纯字典读
            cached = self._backlinks.get(bucket)
            if cached is not None and cached[0] == sig:
                return cached[1]                                 # 签名命中 → 返回 memo，不建图
        from .graph import build_graph, compute_backlinks        # 函数级导入（决策P5.3-3）
        inlinks = compute_backlinks(build_graph(wiki))           # 锁外重活：确定性、幂等、只读 wiki/
        with self._lock:                                         # 短临界区：纯字典写
            self._backlinks[bucket] = (sig, inlinks)
        return inlinks
```

设计要点（决策P5.3-5）：

- **签名零额外扫描、零排序**：热路里 `docs = cache.corpus(wiki)` 紧接 `cache.backlinks(wiki, docs)`——`DocBag` 已带
  `mtime_ns`/`size` 且已是 `iter_pages` 稳定序，签名直接由这份 docs **顺序聚合**得出（不再排序），**不再做第二遍
  `iter_pages`+stat 全库扫描**。这正是评审「同一个 corpus stat 视图下顺手生成签名」的落地：签名复用 corpus 这趟已付的
  stat 视图与排序。
- **失效信号 = content 页 `(mtime_ns,size)`（与 DocBag memo 同源、同级）**：签名取自 `docs`（= `iter_pages` 的
  **content 页**，**不含** config index/log/overview）。失效粒度是**整库 content 页**的聚合签名——反链是全局产物
  （任一 content 页改链 → 别 content 页入链数变），「任一 content 页变就全量重算」恰是其正确失效粒度，比假装能增量更省心。
- **config 页变更不作失效信号（near-lossless，决策P5.3-5）**：`build_graph` 的解析表（`link_resolution_index`→
  `page_stem_index`）确**含 config 页**，故签名覆盖面（content-only）窄于 `build_graph` 输入面。但在**全库 stem 唯一**这一
  根不变式下，config 页**永不是图节点**、指向 config 的链接**一律走 `build_graph` 的 `else 丢弃`分支**，故 config 改动
  **不能改变 content→content 入链计数**。唯一理论例外（config 改名 / 撞 stem / 别名撞名致 fold 解析漂移）本就是 `check`
  级违规或病态；其残留仅「热路某次检索某页入链数偏旧、至下次 content 页变更自愈」——advisory rerank + CLI 冷算恒权威、
  bounded/benign。反向若把 config 纳入签名，须对 config 页**另起一遍 stat**（config 不在 `iter_pages`），恰抵消上一条
  「零额外 stat 扫描」收益，不值当。
- **签名变 → `build_graph` 重算**：`build_graph` 的重活放锁外并发跑（同 corpus() 体例）。并发 miss 最坏两线程各建一次图、
  last-writer-wins——两份都确定性等价，benign（同 CorpusCache「最坏多算一次、绝不永久陈旧」容忍）。
- **TOCTOU benign**：签名取自 corpus 视图、反链 dict 由 `build_graph` 另读一遍 `iter_pages` 现算，二者间若有在飞改写，
  至多某次检索的某页入链数偏 ±1（乘性因子对此鲁棒），下次签名必变而自愈——advisory rerank，不影响正确性。
- **零写、reader/MCP 可用**：用 `build_graph`（只读 `wiki/`）**不是** `build_and_write_graph`（那会写 `graph/`）——
  反链零写 `graph/`、reader/MCP 零字节写盘可用（同 P5.1-6/7、P4.10 只读姿态）。
- **IO 抖动优雅降级（review 加固）**：`build_graph` 经 `load_page→read_text` 可在「遍历与读盘之间页被删/不可读」
  时抛 `OSError`。`backlinks()` **catch `OSError` → 返回 `{}`（无 boost、纯 BM25）且不 memo 失败、下次重试**——
  守 `corpus()` 同款 best-effort，长驻读路径（`/api/search`、MCP `search`）绝不因瞬时 IO 抖动 500。

## 5. 决策（沿用仓库 `决策PX-N` 体例）

- **决策P5.3-1（文档先验、post-BM25 乘性、不并 tf）**：见 §0；与 P5.0-17 字段加权正交。
- **决策P5.3-2（`1+W·ln(1+c)`、零反链不归零、W=0.5）**：见 §1。
- **决策P5.3-3（复用 graph 解析口径、`compute_backlinks` 居 graph.py 与 `compute_orphans` 同口径、path 键、search.py 函数级导入 graph）**：见 §2；零第二套解析、alias/fold-aware、`score`/`DocBag` import 面 graph-free。
- **决策P5.3-4（`inlinks=None` 默认零 boost）**：见 §3；向后兼容、boost 加法可关；但四处接入点都必须传 `inlinks` 以保名次一致。
- **决策P5.3-5（反链 memo = CorpusCache 上独立 memo 槽、整库 content 页签名复用 corpus 已付的 stat 视图+排序、签名变才 `build_graph` 重算；config 页变更不作热路失效信号）**：见 §4.4；签名取自同一次 `corpus(wiki)` 返回的 docs（content-only、canonical iter_pages 序、**不排序**），失效面 content-only 是 near-lossless（config 改动在全库 stem 唯一下不改 content 入链数）；不引第二套失效框架、不与 DocBag memo 纠缠、零额外 stat 扫描+零额外排序。
- **决策P5.3-6（性能底线分层：CLI 冷路接受双扫描、Web/MCP 热路必须 memo）**：见 §4；**明确不做**增量维护入链图 / 倒排边索引 / file-level link diff / 目录 mtime 优化 / 任何持久索引——整库签名→全量重算即够，多余皆过度设计（亦守 P5.0 stateless、非 E1）。
- **决策P5.3-7（字节稳定 + 字段契约不变 + 不改收录门槛）**：`round`/tie-break 仍是量化点 / 名次键；**收录门槛判在 boost 之前**（未加 boost 的 `round(s,6)>0` 决定召回，boost 只重排已召回页、绝不把弱命中页捞回——召回集与纯 BM25 逐页一致，review 加固）；`search_result_dict`（`search.py:356`）结构不变——`score` 数值变、`{page,title,type,score,snippet}` 字段同（CLI/Web/MCP 字段平价不破，反链数**不**进出参）。无新端点/SSE/退出码。
- **决策P5.3-8（编号复用）**：见 题首 ※。

## 6. 测试要点（仿 `test_search.py` 风格）

- 同 BM25 分两页，入链多者排前（log 缩放生效）。
- **零反链页不被挤出**：建一页无任何入链 + 命中 query → 仍在结果里、分数 = 纯 BM25（因子 1.0）。
- 字节稳定：同库同 query 连跑两次 JSON 全等（boost 确定性）。
- `inlinks=None`：`score` 退化为纯 BM25，与 P5.0 既有断言一致（向后兼容）。
- `compute_backlinks` = graph 入度：排自环、排 broken、alias/fold 链计入拥有页（复用 `build_graph` 口径）；与 `compute_orphans` 入链集互证。
- CLI/Web/MCP/chat 四处同 query 同库 → 名次一致（boost 在 `score` 内、四处都传 `inlinks`）。
- **`CorpusCache.backlinks` memo**：(a) 同库连调两次，第二次签名命中、**不再 `build_graph`**（可 monkeypatch 计数断言）；(b) 改一 **content** 页链接后签名变、`build_graph` 重算、入链数更新；(c) 签名由传入 docs 推出、不触发第二遍 stat 扫描、**不排序**；(d) 跨两库分桶不串、不误剪；(e) **config 页变更不触发反链重算**（content-only 失效信号，决策P5.3-5）；(f) 传入乱序 docs → 安全多余重算（签名不匹配重建），绝不误命中陈旧。
- **零额外 stat 扫描 / 零写**：热路 `backlinks(wiki, docs)` 不再 `iter_pages`+stat；`build_graph` 零写 `graph/`；reader/MCP `/api/search`、MCP `search` 带 boost 且整库零字节写。

## 7. 边界（明确不做）

- **不做增量/倒排/持久化**：不增量维护入链图、不建倒排边索引、不做 file-level link diff、不做目录 mtime 优化、不落任何持久索引/缓存派生物到盘（决策P5.3-6；整库签名→全量重算即够；向量/rerank/持久索引仍是 E1）。
- 不引入 RRF/多臂融合（gbrain 那套 hybrid 是 E1 量级）——只在确定性 BM25 上叠一道文档先验。
- 不动 P5.0 字段加权、`tokenize`、snippet 逻辑（决策P5.3-1 正交叠加）。
- 不碰 `raw/`、不起 LLM、不引依赖（`math` 标准库；`graph` 已是仓库内零-LLM 模块）。
- 与 P3.8 链接归一正交：P3.8 让更多链接 resolved → 反链计数更全，二者自然叠加、无耦合。
- **不优化**：corpus() 与 backlinks() 在热路各自的轻量遍历可进一步合并以省一次 docs 签名遍历，**明确推后**——API 耦合不值当，整库签名 + docs 复用已把热路代价压到「一次 O(N) 签名聚合 + 比对 + 一次 dict 取」，非瓶颈。
