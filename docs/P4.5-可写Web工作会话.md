## 1. 目标与范围

**P4.5 = 让 Web 会话可写——人经 `/mode workspace-write` 把当前会话切到可写姿态，驱动 Agent 写 `workspace/`、跑 shell、维护 `wiki/`；`raw/` 对结构化写（层①）与同步打写端点（层③）确定性只读、config 再加层②快照还原；`wiki/` 写后过 `check` 校验 + 可撤销本轮写；全程不破 P2/P4 既有不变量。三类残留（shell 直写 raw/、shell 写 kb 外、派生后台进程打端点）deferred 到网络 sandbox、不宣称「任何路径不可变」。**

P4.4 已落地只读斜杠命令 + 自省端点，`/mode` 仅报告。P4.5 把 `/mode` 升级为**运行时真切换**，并为可写姿态补齐 P4.4 §7 列出的全部前置守卫。**典型用途**（呼应 `AGENTAO.full.md` 愿景）：Web 上传/投喂 → `workspace/uploads/` 暂存 → Agent 跑 shell 解析（`pdftotext`/`marker` 把 PDF→文本）→ `workspace/parsed/` → 再 ingest 入库；或人驱动 Agent 直接增补 `wiki/` 页、修链、补 frontmatter。

### 威胁模型（先挑明，决定守卫强度，决策P4.5-11）

- **不可信的是谁、为什么**：嵌入 agent 是用户自己的助手，但它**会 ingest `raw/` 里的源文档**——一篇被注入指令的源即可让 agent 干非预期写（**prompt-injection 是唯一现实的「为何不信任」理由**，本相位据此定守卫，而非假想远程攻击者）。
- **能封到哪、封不到哪（诚实）**：确定性层挡得住 **结构化写**（层①，覆盖 `raw/`+`AGENTAO.md`）、**可写 turn 内同步打宿主写端点**（层③）、并对 **`AGENTAO.md` 一个文件**做快照还原（层②）。**挡不住**对抗性 shell 的三条路径——直接写盘 `raw/`、写 kb 根外、`curl … &` 派生后台进程绕过 turn 窗口——**这三类只有 shell 网络/文件 sandbox 能堵**，本相位 deferred（§1 不纳入）。
- **`SCHEMA.md` 刻意不在只读集**（决策P4.5-12）：它是 agent 的领域/路由**指引**（被 ingest/heal 消费、**不被 `check`/`lint` 门禁消费**，`pages.py:51`），skill 定为「人与 Agent 共同演进」。agent 改它不构成提权（`AGENTAO.md` 才是行为约束），最坏情况是路由变化，**可见、随 git、且经层① wrapper 进 per-turn 写日志→「撤销本轮」一键回退**（§3）。
- **由此的定位**：确定性层 = 「**阻意外 + 挡同步/结构化旁路 + 抬门槛**」，**不是**对抗性封堵。要真封堵，可写会话应在 sandbox 落地前视作「**受限信任**」模式。**故本相位刻意不为 `raw/` 树堆昂贵的 per-turn 全量快照/回滚**（决策P4.5-3）——那是为一个已承认封不住的旁路付 O(`raw/`)/turn 的代价，不成比例。

### 写边界一句话契约（§2 详述，分区而非一刀切）

| 区域 | 策略 | 守卫 |
|------|------|------|
| `raw/` | **确定性只读（非对抗性封堵）** | 层①（结构化写拒）+ 层③（同步打写端点拒）+ AGENTAO 硬约束；**shell 直接写盘 → 残留**（不 per-turn 检测/回滚，决策P4.5-3/c） |
| `AGENTAO.md`（行为硬约束/宪法） | **确定性只读 + 快照还原** | 层①（结构化写拒）+ 层③ + AGENTAO 自约束；**外加层②**：一个文件 per-turn 快照、被旁路改则**自动还原**（近免费）。agent 改它＝提权，绝不放行（决策P4.5-3/12） |
| `SCHEMA.md`（领域/页面类型/规则，agent 指引） | **agent 可写（人驱动）** | **不在只读集**——门禁不读它、skill 定「共同演进」；经层① wrapper 进 per-turn 写日志 → 写后 surface + **可撤销本轮**（§3）。非提权面（决策P4.5-12） |
| `wiki/`（含 `index.md`/`log.md`/`overview.md`） | **Agent 可写** | 放行 + **写后 `check` surface + 可撤销本轮写**（§3；原则 2，Agent 拥有 wiki 层） |
| `workspace/`、`graph/` 及其余 kb 根内 | **可写**（工作区/派生物） | PathPolicy contain 在 kb 根内；`graph/` 写经 `write_lock` 串行（§5） |
| kb 根**之外** | **结构化写不可写；shell 写可逃逸（残留）** | 结构化写经 PathPolicy + wrapper 防御性 kb 包含（§2.2）；shell 写 kb 外 → 残留、deferred 到 sandbox |

### 纳入 P4.5

- **`/mode` 运行时翻姿态**：`POST /api/chat/{id}/mode {mode}` 翻当前会话 agent 的两点置位（`read-only`↔`workspace-write`），不重建 agent；冷会话回 409（决策P4.5-5/7）。
- **C0 前置 spike：验证层①地基**（决策P4.5-2）：层① 是唯一真正「阻止写」的层，落地前**先验**：① `build_from_environment`/`Agentao(filesystem=…)` 能把 `PolicyFileSystem` 透传到 `agent.filesystem`；② 内建写工具（`write_file`/`replace` 及任何 patch/notebook 类）**全部**终经 `self.filesystem.write_text`、无旁路。**验不过则可写模式不落地**（退到「无层①」就只剩层③+约束，不足以撑「确定性只读」）。
- **`/mode` 运行时翻姿态 + 层①+层②+层③ 守卫**：① 构造期注入 `PolicyFileSystem` wrapper（包住 `agent.filesystem`，对结构化写 `write_file`/`replace` 做**叶解引用包含判定**，命中 immutable 集 `raw/`/`AGENTAO.md` 则拒；**`SCHEMA.md` 不在集内、放行**，决策P4.5-12）；② 每轮 workspace-write turn **仅对 `AGENTAO.md` 一个文件**套快照、被旁路改则**自动还原**（近免费；`raw/` 树**不** per-turn 扫，其 shell 直写并入残留，决策P4.5-3/c）；③ 可写 turn 活跃期间宿主写端点（`/api/raw`·`upload`·`ingest`·`heal`）一律 `423 Locked` 拒，兜 shell 经 localhost 打回宿主写面的旁路（决策P4.5-2/3/10）。
- **`wiki/` 写后 `check` 校验 + 可撤销本轮写**：可写 turn 收尾跑 `check.run_check(wiki)`，findings 经 SSE surface + 一键「让 Agent 修复」（复用 `REPAIR_PROMPT`）+ 一键「撤销本轮写」（`wiki/` + `SCHEMA.md`，复用层① wrapper 的 per-turn 写日志，O(被改文件)，§3）（决策P4.5-4）。
- **单写者并发**：可写 turn 与 ingest/heal/投喂/`GET /graph` 复用**同一 `write_lock`**（与 `JobQueue._lock` 分离），四树写永不并发（决策P4.5-6）。
- **`--mode` 进程默认 + 可写会话恢复**：`guanlan web --mode {read-only,workspace-write}`（默认 `read-only`）定新会话开局；`restore` 用当前进程默认姿态、不回放盘上姿态（决策P4.5-8）。
- **前端**：姿态徽标 + `/mode` 切换 + 可写 turn 的 check 回执/修复入口 + 撤销本轮（`wiki/`+`SCHEMA.md`）+ `immutable_mutated`(`AGENTAO.md`) 告警。

### 不纳入 P4.5（按路线图推后）

- **`full-access` / `plan` 姿态**：Web 永不用（决策P4.5-1）。
- **改 `working_directory` 到 `workspace/` 子目录做物理隔离**：会破坏 skill 发现 / `AGENTAO.md` 加载 / `.agentao/sessions` 归属（全部 key 于 `working_directory`=kb 根），故不改；写边界靠 `PolicyFileSystem` wrapper + 快照守，不靠 cwd 收窄（决策P4.5-3）。
- **`raw/` 树的 per-turn 全量快照/回滚**：刻意不做（决策P4.5-3/c）。`raw/` 可能数百 MB，为抓一个已承认封不住的 shell 旁路而每条可写消息前置 O(`raw/`) 哈希+拷贝，成本与价值不匹配；`raw/` 的 shell 直写并入残留（下）。`AGENTAO.md` 一个文件例外（近免费、仍做）。
- **macOS `sandbox-exec` 强隔离 + shell 网络/localhost 隔离**：唯一能堵下述残留的手段，可作纵深防御后续接（决策P4.5-3/10/11 备注）。**三类残留只有网络/文件 sandbox 能堵**：① shell **直接写盘 `raw/`**（层②不再扫，§2.3）；② shell 写 kb 根**之外**（§2.1）；③ 可写 turn 内 `curl … &`/`nohup` **派生后台进程**在 turn 结束后才打宿主写端点（层③兜不住，§2.4）。均 deferred 到此项；sandbox 落地前可写会话视作「**受限信任**」模式（§威胁模型）。
- **多用户/远程可写**：P4 起 Web 即 `127.0.0.1`-only 单用户，本相位不变。

## 2. 写边界守卫（核心，决策P4.5-2/3）

### 2.1 写边界的事实：PathPolicy contain 在 kb 根、不查 shell 参数

agentao 的 `security/path_policy.py`：`PathPolicy.contain_file(file_path)` 把文件写工具的目标 contain 在 **`working_directory`**（`is_relative_to(project_root)`），`Conversation` 用 **kb 根**作 `working_directory`（同 P4）。故裸 `workspace-write`：

- **能写 kb 根内任意文件**（`raw/`、`wiki/`、`AGENTAO.md`…）——不细分到 `workspace/` 子目录；
- **shell 参数不被 PathPolicy 检查**（其 docstring 明示「only the cwd is contained」），`echo x > raw/foo`、`mv`、`python -c` 等旁路结构化工具的路径门禁。

因此对「只读集」= `raw/` ∪ `{AGENTAO.md}`（**注意：`SCHEMA.md` 不在内**，决策P4.5-12）加**分区**确定性守卫（不一刀切）：

- **层①（wrapper，§2.2）守结构化文件写**——覆盖 `raw/` + `AGENTAO.md`，是唯一真正「阻止写」的层（故 C0 先验其地基）。
- **层③（时序互斥，§2.4）兜 shell 经 localhost 打宿主写端点**——覆盖 `raw/` + `AGENTAO.md`。
- **层②（快照+还原，§2.3）只守 `AGENTAO.md` 一个文件**——`raw/` 树**不** per-turn 扫（决策P4.5-3/c）。
- **`SCHEMA.md` 三层都不守**——它 agent 可写（人驱动），靠 wiki 级软门禁（写后 surface + 撤销本轮，§3）兜，理由见决策P4.5-12。

**shell 经 `shell` capability、不经 `agent.filesystem`，层① wrapper 够不着**（上游 [`host-fs-policy.md`](../../agentao/docs/design/host-fs-policy.md) 把 shell enforcement 列为最难、P2/可能永不落地的 (c)）。对 shell **直接写盘**：写 `AGENTAO.md` → 层②还原；**写 `raw/` → 残留**（不再扫、不检测、不回滚，威胁模型已挑明）；写 kb 外 → 残留；写 `SCHEMA.md` → 放行（agent 可写）。shell 改走 HTTP 写端点的另一路由由层③专兜。三类残留只 sandbox 能堵。

### 2.2 层①：注入 `PolicyFileSystem` wrapper（构造期一次、姿态无关）

agentao 把 `Agentao(filesystem=…)`（`agent.py`）的 `FileSystem` capability 经 `_bind_and_register`（`registry.py`）绑到**每个**工具（含 `extra_tools`）；内建 `write_file`/`replace` 最终都调 `self.filesystem.write_text(...)` 落盘。宿主因此可注入一个**只能收紧、不能放宽**的策略 wrapper，对一切经该 capability 的结构化写做**包含判定**——这正是上游 [`host-fs-policy.md`](../../agentao/docs/design/host-fs-policy.md) 的 **interim 配方**，且 P4.5 命中的「**cwd 内只读子集**」facet 是该文判定「**今天零 agentao 改动即可满足、原生 `write_file` 直接生效**」的那一个（`raw/`+config 在 kb cwd 内 → 过内建单根预检 → 落到 wrapper → wrapper 拒）。

```python
# chat.py：Conversation.__init__ 注入；immutable 集 = raw/ ∪ {AGENTAO.md}（SCHEMA.md 不在内、agent 可写），
# 姿态无关、构造期定一次（read-only/workspace-write 都是这个 immutable，故无需热切换）。
@dataclass(frozen=True)
class _Rule:
    immutable: tuple[Path, ...]                       # writable 留空：cwd 隐式可写（augment 语义）

def _effective_target(raw: str, base: Path) -> Path:
    """open() 真正会写的位置：父链 ..-safe resolve、叶 symlink 跟随后再判成员——
    这是唯一正确的判定基（不能用 contain_file：它在解引用叶 symlink *之前* 判包含，会 fail-open，
    放过 wiki/link→raw/secret 的软链 clobber，上游 host-fs-policy.md §Interim 已点名）。"""
    p = (base / raw).expanduser() if not Path(raw).is_absolute() else Path(raw).expanduser()
    t = p.parent.resolve(strict=False) / p.name
    return t.resolve(strict=False) if t.is_symlink() else t

class PolicyFileSystem:
    def __init__(self, inner, kb: Path, rule: _Rule):
        self._fs, self._kb = inner, kb.resolve()                     # kb 隐式可写
        # 把 immutable 集**规范化到与 _effective_target 同一坐标系**（同样 resolve(strict=False)）——
        # 否则 kb 为相对路径 / 经 symlink 进入 / raw/ 父链含 symlink 时，t 是 resolved、m 是原样，
        # `t == m or m in t.parents` 跨两套路径表示比较会 fail-open（评审 Medium）。
        self._immutable = tuple(m.resolve(strict=False) for m in rule.immutable)
    def _check(self, raw: str):
        t = _effective_target(str(raw), self._kb)
        if t != self._kb and self._kb not in t.parents:              # 防御：目标必须在 kb 根内
            raise PermissionError(f"FsPolicy: out-of-kb: {t}")       # 不把 kb-containment 委托上游
        if any(t == m or m in t.parents for m in self._immutable):   # 两侧均 resolved，同坐标系
            raise PermissionError(f"FsPolicy: immutable: {t}")        # raw/ 或 AGENTAO.md → 拒
    def write_text(self, path, data, *, append=False):
        self._check(path); return self._fs.write_text(path, data, append=append)
    def __getattr__(self, name): return getattr(self._fs, name)      # 其余方法透传内层
```

- **比 regex deny-rules 严格更安全**：包含/叶解引用语义，不依赖「知道精确 arg 名 `file_path`」（写错即静默 fail-open 是 P4.4 评审 #1 的教训），且挡住 `wiki/link → raw/secret` 的**软链 clobber**——这类正是逐路径 regex 漏拦的 fail-open 缺口。
- **防御性 kb 包含判定（只收紧不放宽）**：wrapper 在 immutable 判定**之前**先断「目标必须在 kb 根内」（`t == kb or kb in t.parents`，否则拒）。内建 `write_file` 虽已过 agentao `contain_file` 单根预检，但 wrapper 自我定位是 host-level policy、**不把 kb-containment 委托给上游**——任何未来经该 capability 直调 `write_text(Path("/tmp/x"))` 而绕过内建预检的工具，也被这层兜住（评审 P4.5 #3）。
- **immutable 集与目标同坐标系（防 fail-open，评审 Medium）**：`_check` 的两侧——`_effective_target` 算出的 `t` 与 immutable 成员 `m`——**必须都 `resolve(strict=False)`**。故 `__init__` 把 `rule.immutable` 一次性规范化成 `self._immutable`（见上），**绝不**拿原样路径去比 resolved 的 `t`：否则 kb 是相对路径、kb 经 symlink 进入、或 `raw/` 自身/父链含 symlink 时，`t == m or m in t.parents` 在比较两套路径表示、漏判即放过对 `raw/`/`AGENTAO.md` 的写。测试补「symlink kb / symlink `raw/` 目录 / 相对 kb」三类坐标系一致性用例。
- **姿态无关、装一次、无需热切换**：immutable 集（raw/+config）在 read-only 与 workspace-write 下**恒等**——read-only 由 `tool_runner` 全拦写、workspace-write 由 wrapper 守只读子集。故 wrapper 构造期注入一次即可，`/mode` 翻姿态**不动 wrapper**（不需要上游 (b) 的 `set_policy` 热切换，对应该文「(b) 仅在可写集需中途改变时才用」）。**这也顺带消掉了旧方案 `add_run_rules` 无 `remove/clear`、`set_mode` 不装卸的 hack**（决策P4.5-2）。
- **只覆盖经 `agent.filesystem` 的结构化写**：shell（走 `shell` capability）、MCP 写工具（`mcp_*` 不经该 capability）均够不着——shell 由层②快照兜（§2.3）；P4.5 嵌入 agent **不挂任何 MCP 写工具**，测试断死这一前提。

> **落地两点须验（升为 C0 前置闸、非脚注——评审 D）**：① **`build_from_environment` 须能透传 `filesystem=`**——若该工厂路径未转下去，改走 `Agentao(filesystem=…)` 直构或给工厂补透传口；并枚举内建写工具确认**全部**终经 `write_text`、无旁路。**C0 验不过则 P4.5a 不开工**（层① 是唯一「阻止写」的层）。② **`_effective_target` 是安全关键 resolve**，按上述配方叶解引用、勿图省事复用 `contain_file`。上游建议加一个 `PathPolicy.contain_any(raw, writable=[…], immutable=[…])` classmethod 把这段归口（host-fs-policy.md §"The agentao-side helper…"）——**值得顺手推上游**，推成后 wrapper 的 `_check`/`_effective_target` 即可换成一行调用；未推成则按配方自托管（自洽可用）。

### 2.3 层②：每轮 workspace-write turn 套 **`AGENTAO.md` 一个文件**前后快照 + 自动还原（决策P4.5-3/c）

层① 已阻 agent 经 `agent.filesystem` 的结构化写。层② 兜「shell 直接写盘绕过层①」这一路——但**只对 `AGENTAO.md` 这一个固定小文件**（行为硬约束/宪法，被改影响全局、最该自动复原）；`raw/` 树**不**做（其 shell 直写并入残留），`SCHEMA.md`**不**做（agent 可写，见决策P4.5-12）：

```python
# chat.py：仅 workspace-write turn、仅 AGENTAO.md（read-only turn / raw/ 树 / SCHEMA.md 均零开销）。
async def turn(self, msg, emit):
    async with self.lock:
        guarded = self._mode == "workspace-write"
        before = snapshot_agentao(self._kb) if guarded else None    # 一个文件：content_bytes | None
        try:
            answer = await self.agent.arun(msg, cancellation_token=token)  # 流式（可能 raise/cancel/超时）
        finally:
            # 还原必须在 finally：agent 在 shell 改完 AGENTAO.md 后报错/取消/超时也要复原，
            # 且必须早于 error SSE 收尾、active_writable_turns--、write_lock.release()（评审 High）。
            if guarded:
                changed = restore_agentao(self._kb, before)  # AGENTAO.md 被旁路改/删 → 还原原字节
                if changed:
                    self._last_violation = changed       # 进 done/error SSE 的 immutable_mutated，标注「已还原」
        ...
```

- **为何只守 `AGENTAO.md`、不守 `raw/`**（决策P4.5-3/c）：`AGENTAO.md` 是**一个固定小文件**，快照=读一次、还原=写一次，**近免费**，且它是**行为硬约束（宪法）**、被改＝提权，最该自动复原。`raw/` 是**树、可数百 MB**，为「还原」要 O(`raw/`) 哈希+拷贝/可写 turn，而 shell 直写 `raw/` 是**已承认封不住的残留**（sandbox 才堵）——为堵不住的旁路付每轮全量代价，不成比例（反向评审 A/B 结论）。故 `raw/` 树不 per-turn 扫。
- **还原是单文件、但形态判定不能省**（评审 High）：`snapshot_agentao` 存原字节（不存则记「不存在」）。`restore_agentao` **先 `lstat` 当前路径再决定怎么写**——shell 旁路不止 `echo`，还能 `ln -s /etc/x AGENTAO.md`、`rm … && mkdir AGENTAO.md`、`mkfifo AGENTAO.md` 把它换成 symlink/目录/fifo/special：① 命中非普通文件 → 先 `unlink`（目录则 `rmtree`）清掉替身，**绝不顺着 symlink `write_bytes`**（否则写穿到 symlink 指向的 kb 外/敏感目标）；② 再以普通文件**原子写回**原字节（临时文件 + `os.replace`）；③ 快照记「不存在」则确保最终也清成不存在。比 `raw/` 整树的 `gate.snapshot_raw` 对象模型简单（单文件、不遍历/指纹整树），但**inode 形态检测照做**——能 `echo` 的 shell 同样能换 inode 类型。
- **shell-proof（仅 `AGENTAO.md`）**：`echo x > AGENTAO.md` 及 `ln -s`/`mkdir`/`mkfifo` 形态替换这类层①挡不住的旁路，被前后快照捕获、**先清替身再原子写回**还原（上）。`done`/`error` SSE 带 `immutable_mutated:["AGENTAO.md"]` 标注「已还原」+ 告警（理论上层①+AGENTAO 约束已挡结构化写，触发即 shell 旁路的异常信号）。
- **read-only turn 不套快照**（零开销）；workspace-write turn 也只读一个文件，开销可忽略。

### 2.4 层③：可写 turn 活跃期间宿主写端点时序互斥（兜 shell 经 localhost 旁路，决策P4.5-10）

层①/层② 都假设 agent 的写要么经 `agent.filesystem`、要么经 shell 直接落盘。但 Web 宿主自身在 `127.0.0.1` 上暴露**写端点**（`/api/raw`/`upload` 投喂晋级、`/api/ingest`、`/api/heal`），它们走 `_atomic_write_raw`/`JobQueue` 的**宿主合法写**、**完全不经 `agent.filesystem`**（层① 够不着），且由 JobQueue worker 线程异步落盘——可写 turn 里 agent 一句 `curl http://127.0.0.1:PORT/api/raw -d …` 即可让宿主**替它**写 `raw/`：

- **绕过链**：`curl` 让宿主经 `_atomic_write_raw`/JobQueue 写 `raw/`——**层①够不着**（不经 `agent.filesystem`）、**层②本就不扫 `raw/` 树**（决策P4.5-3/c）。即使旧版每轮快照 `raw/`，作业也可排到 turn 后、错过窗口。无论哪种，`raw/` 被宿主替 agent 改且无检出——击穿 P4.6「源由人背书是架构保证」「Agent 永不写 `raw/`」（评审 P4.5 #1）。故这条**同步**路径必须由层③（时序互斥）专堵。
- **本相位对策（时序互斥）**：宿主维护一个 `active_writable_turns` 原子计数（可写 turn 起跑 +1、收尾 -1，**独立于写锁**，不与 §5 的 `write_lock`/`JobQueue._lock` 混用）；`/api/raw`/`upload`/`ingest`/`heal` 入口先查 `if active_writable_turns > 0: return 423`「可写会话进行中，请稍后」（**用 `423 Locked` 而非 `409`**：`/api/raw` 的 `409` 已表「默认不覆盖冲突」，同端点两个 `409` 前端无法区分，故时序互斥另用 `423`）。agent 的 shell 仅在可写 turn 内执行 → curl 在 turn 内到达即被 `423`、**根本不入队**，同步竞态被掐死。read-only turn 不计数（不影响宿主写端点）。
- **为何不用 token/CSRF**：同机 same-uid shell 能 `curl GET /` 抓页内 token、或 `-c jar` 重放 Set-Cookie——经典「攻击者读不到响应」假设在此不成立。真边界需 Jupyter 式带外启动 token（响应永不回显、后代 shell ptrace 不到祖先内存），本相位不引入（决策P4.5-10）。
- **残留（deferred 到网络 sandbox）**：可写 turn 内 `curl … &`/`nohup` **派生后台进程**在 turn 结束（计数归 0）后才发请求 → 端点会接受。此残留与「shell 写 kb 外」同类，只有 shell 网络/localhost 隔离能堵，列纵深防御后续（§1 不纳入 / 决策P4.5-10）。

## 3. `wiki/` 写后校验（决策P4.5-4）

`wiki/` 对可写会话放行（原则 2，Agent 拥有 wiki 层），但 chat 直写**不经 ingest 子进程**、故不享 ingest 的「强制 `check` + 有界自愈」。P4.5 给可写 turn 补一道**写后校验**，与 ingest 收尾同口径：

- **可写 turn 收尾默认无条件跑** `check.run_check(wiki)`——**触发既不靠 wrapper 写日志、也不靠 mtime/size 指纹**（评审 High/Medium）：wrapper 日志漏掉 shell 直写 `wiki/`；而 mtime+size 指纹**不是确定性覆盖**（同大小写入、时间戳粒度碰撞、shell 写后 `touch -r` 复刻时间戳都能让指纹不变），用它补 shell 漏检并不可靠。check 是零 LLM、本就遍历 `wiki/`，相对刚结束的一次 LLM turn 成本可忽略，故**每条可写 turn 收尾都跑**，shell 与结构化写一律覆盖。若日后 profiling 证明 check 偏贵、需跳过未变更 turn，**只能用逐文件内容哈希指纹、绝不用 mtime+size**。撤销仍只覆盖写日志内的结构化写——shell 直写 `wiki/` 是**已知的撤销缺口**（但 check 无条件跑、必 surface 它，二者解耦）；
- findings 经 SSE `done` 帧的 `check` 字段 surface（`{ok, violations:[…], total, resolved}`）。**呈现按「本轮新增」口径（修订，2026-06-10）**：可写 turn 起跑时（已持 `write_lock`）先拍一份 `run_check` **基线**，收尾再跑一次、`violations` 只装两者之差（本轮新增）、`ok` = 本轮未新增、库存量进 `total`、本轮消除数进 `resolved`、`repair_prompt` 只针对新增（修存量是 heal/lint 的活）。动因：存量问题（数百条断链很常见）若整批进 `violations`，每个可写轮收尾都「✗ 发现 403 条」刷屏、把本轮真正引入的问题彻底淹没（不可用反馈）。**守卫强度不变**：收尾 check 仍每轮全量跑、shell 直写引入的新问题照样落进差集；基线拍取失败仅退化为全量呈现（不毁本轮）。**基线缓存（性能）**：起跑基线按 `WriteGate.wiki_generation` 写代际缓存到会话——同会话连续对话、代际未变即复用上次见到的 violation 键集、**跳过重拍**（稳态每轮只跑 1 次 check＝收尾那次，省每轮 ~0.36s/810 文件；否则可写轮比只读轮明显卡）；首轮、或 ingest/heal 完工 bump 代际后下一轮才真拍。改了 violation 集的可写 turn 收尾也 bump 代际，使别的可写会话缓存失效。漏/多 bump 至多多查一次基线，绝不影响守卫；
- UI 给两个一键动作（人在环中、非自动循环，决策P4.5-4）：
  - **「让 Agent 修复」** → 以 `gate.REPAIR_PROMPT.format(violations=…)` 作为**下一轮**消息再驱动 Agent；
  - **「撤销本轮写」（`wiki/` + `SCHEMA.md`）** → 还原本 turn 经结构化工具写过的文件。**复用层① wrapper 的 per-turn 写日志**（评审反馈，纠正「wiki 欠保护」）：wrapper 本就拦截 agent 的每次结构化 `write_text`，可顺手在 turn 内记 **`(路径, 写前字节, 写后哈希)`**——`wiki/` 页与 `SCHEMA.md` 都经该 capability 落盘 → **都进同一份日志**；撤销=回放还原，**O(被改文件数)、不需全树快照**。这正是决策P4.5-12 里 `SCHEMA.md` 可写「可撤销本轮」安全论证的**落地点**（撤销范围不再只覆盖 `wiki/`，否则该论证落空——评审 Medium）。**撤销可用性 = 本轮写日志非空，独立于 check（评审 Medium）**：`done` 帧单独带 `undo:{available, token, paths}`（写日志非空即 `available:true`），UI **只凭它**展示撤销键——故 `SCHEMA.md` 单独改（check 照常无条件跑、只是无 `wiki/` violations）、或 `wiki/` 改但 check 通过（无 violations），撤销键**照样出现**；撤销键**不挂在 `check.violations` 分支下**。**缺口**：shell 直写 `wiki/`/`SCHEMA.md` 不经 wrapper、进不了日志（故 `undo.paths` 不含它）；该缺口的兜底（开 turn 前拍 `wiki/`+`SCHEMA.md` 快照，比 `raw/` 便宜得多）列 P4.5c 评估，**阻断式合并门禁单独成相位**。
- **撤销的并发语义（不能盲回放，否则吞掉后续写——评审 High，决策P4.5-13）**：撤销按钮在 UI 上**留存到下一次写**，期间 turn B / ingest / `/api/raw` 可能改同一文件；盲目回放「写前字节」会把这些后续写覆盖掉。故撤销必须：① **本身是一次写、须异步取 `write_lock`**（§5；不能在锁外回放）；② **乐观校验**——逐文件比对**当前内容哈希 == 本 turn 写后哈希**，相等才还原、否则该文件**跳过并报 `409`-类 conflict**「该文件已被后续写改动，未撤销」（SSE note + UI 标红）；③ **token 仅指向最近一个可写 turn 的日志**，**任一后续写（chat/ingest/raw/heal）落到某 path 即令该 path 的撤销项失效**（per-path 失效），下一个可写 turn 收尾替换为新日志。撤销是「把本 turn 这几处改回去」、**不是**全局时间旅行；据此 `SCHEMA.md` 的「可撤销」安全论证只在「撤销前该文件仍是本 turn 留下的状态」时成立，否则退化为「人工 git 回退」（仍可见、随 git，决策P4.5-12 caveat 已覆盖）。
- **不硬阻断**（区别于 P2 `check` 对 ingest 的退出码硬门禁）：交互写流式中途硬拦成本高且 UX 差；写后 surface + 人触发「修复 / 撤销」，是单用户本地台的恰当门禁强度（决策P4.5-4）。`wiki/` 质量是 check-surfaced 的软门禁 + 可撤销；`raw/`/config 的只读由 §2 守。

> 与 ingest 的关系：ingest 仍是「批量入库 + 子进程 gated 写 + 强制 check + 有界自愈」的主写线；P4.5 可写 chat 是「人即时驱动的增量维护」，写后 check surface 是其轻量等价物，二者互补、共享 `check`/`gate` 原语。

## 4. 姿态与 `/mode`（决策P4.5-1/5）

- **取值只接受 agentao `PermissionMode` 的 `read-only` / `workspace-write`**；真实枚举为 `read-only`/`workspace-write`/`full-access`/`plan`（`permissions.py:75`，**无 `full`**）。**`full-access`（绕全部 preset 规则）/`plan`（内部态）永不在 Web 出现**，端点 `422` 拒、测试断死（决策P4.5-1）。
- **`set_mode` 只翻两点置位、不重建 agent、不动 wrapper**（层① wrapper 姿态无关、构造期装一次，§2.2）：

```python
def set_mode(self, mode: str) -> None:
    """镜像 CLI _apply_mode；/mode 调用方持 self.lock（与在飞 turn 串行）。"""
    if mode == "read-only":
        self.agent.permission_engine.set_mode(PermissionMode.READ_ONLY)
        self.agent.tool_runner.set_readonly_mode(True)
    elif mode == "workspace-write":
        self.agent.permission_engine.set_mode(PermissionMode.WORKSPACE_WRITE)
        self.agent.tool_runner.set_readonly_mode(False)
    else:
        raise ValueError(f"未知姿态：{mode}")   # 端点转 422；绝不接受 full-access/plan
    self._mode = mode
```

- **`info().mode` 报当前会话姿态**（P4.4 的常量 `read-only` 升级为真实 `self._mode`）；`/tools` 的 `blocked` 随姿态动态（read-only 拦写/shell；workspace-write 放行、但 `raw/`/config 仍被层①拦，`/tools` 文案另注「`raw/`+配置仍只读」避免误导）。
- **单份 `AGENTAO.md` = full 能力模板，不做文件切换**：「能不能写」由两点置位 enforce、「写到哪（raw/config 只读）」由 `PolicyFileSystem` wrapper 独立 enforce、二者皆与散文解耦（CLI 同构）。`init` 默认 `AGENTAO.md` = 该模板、永不覆盖既有；read-only 下 Agent 即便读到「可跑 shell」也被 `tool_runner` 拦下并如实回报（决策P4.5-1）。

## 5. 单写者与并发（决策P4.5-6）

P4 的读写分线假设「唯一写者 = ingest 子进程」；P4.5 引入**第二类写者 = 可写 chat turn**（写 `wiki/`/`workspace/`）。另有一个**既有却漏网的宿主写者 = `GET /graph`**（`app.py:413` 直接 `build_and_write_graph` 写 `graph/`，既不走 JobQueue、又不在任何锁下——评审 P4.5 #2 第二条）。为不破单写者：

- **两把锁，别混**（关键，评审 P4.5 #2）：
  - **`JobQueue._lock`（既有、保持不动）**：只护作业表/计数器 `_jobs`/`_counter`（`jobs.py:54`），临界区**极短**，`enqueue`/`get_job` 用它；**绝不**让它包住长时间的写作业 `fn()`（`fn()` 本就在 `_run` worker 里、`_lock` 之外执行，`jobs.py:103`）。若把它升成长写锁，`enqueue`/`get_job` 会被在飞 ingest 卡死——正好破下面「job 轮询/UI 不阻塞」。
  - **`write_lock`（新增、注入共享）**：一把**进程级 `threading.Lock`**，**只包真正的写执行**——JobQueue worker 跑 `fn()` 前后、可写 chat turn 起跑/收尾、**「撤销本轮写」回放前后**（§3，撤销是写、须乐观校验后写回）、**`GET /graph` 的 `build_and_write_graph` 调用前后**都 acquire/release。**任意时刻至多一个写者**，`wiki/`/`raw/`/`graph/` 写永不交错、`raw/` 快照窗口不被并发写互踩（沿用决策P4-5 精神）。
- **`/graph` 纳入 `write_lock`、但不纳入 `active_writable_turns` 423 互斥**（评审 P4.5 #2 第二条）：`/graph` 是同步写 `graph/` 的宿主写者，故 `app.py:413` 的 `await anyio.to_thread.run_sync(build_and_write_graph, …)` 须先异步取 `write_lock`（同 chat turn）——既防两个 `/graph` 并发写 `graph/` 互踩/截断，又防 ingest 写 `wiki/` 时 `/graph` 读到撕裂的 `wiki/` 出残图。**但 `/graph` 不受层③ 423 时序互斥**：`graph/` 是可幂等重建的**派生物**、非 immutable 敏感面，agent `curl /graph` 至多触发一次派生重建（无 `raw/` 风险、且已被 `write_lock` 串行），故无须 423 拒（拒了反而挡人在写会话中看图，得不偿失）。**单写者范围 = `raw/`+`wiki/`+`workspace/`+`graph/` 四树的宿主/Agent 写者 + 「撤销本轮写」回放，全经 `write_lock` 串行。**
- **报告读端点（`/api/report/{check,health,lint}`，`app.py:403`）= best-effort 读、不取 `write_lock`**（评审 Medium）：它们 `run_check`/`run_health`/`run_lint(wiki)` 与 `/graph` 同样读 `wiki/`，并发写下既可能读到「文件 A 新、文件 B 旧」的不一致集，**也可能读到单个文件的中间态**——agentao 的 `LocalFileSystem.write_text` 是 `open(path,"w")` 截断后直写、**非临时文件 + `os.replace`**（`agentao/capabilities/filesystem.py:117`），故无原子性保证，读者可撞上半截内容/坏 frontmatter。但与 `/graph` 不同，**报告是 P3 既定的 advisory（非门禁）**，且是状态面板高频轮询——若也加 `write_lock` 会让面板卡在 30s ingest 之后，得不偿失。故**刻意不锁，但理由是「容忍而非排除瞬态」**：报告须把并发写下的中间态/不一致**归类为 transient findings**（写飞行中可能瞬时多/少几条或解析失败、下次轮询自愈），实现上**对单文件解析异常不崩、降级为一条 transient 提示**，**UI 不得把它当稳定门禁**。真正的 `wiki/` 门禁是可写 turn 收尾那道**在 `write_lock` 内**跑的写后 `check`（§3），它读的是自己刚写完、锁内一致的状态。`/graph` 取锁是因为它**写** `graph/`（派生物互踩/截断），报告**纯读**、不污染盘上状态——最坏只是这一次回包是瞬态。
- **`write_lock` 跨线程 + async-safe 获取**：它跨「JobQueue 后台线程写者」与「async chat 写者」共享，故**不能用 `asyncio.Lock`**（JobQueue 线程拿不到）。可写 turn **必须异步地拿这把阻塞锁**——`await anyio.to_thread.run_sync(write_lock.acquire)`（或 `loop.run_in_executor`）、收尾 `write_lock.release()`，**绝不在 event loop 线程上直接 `acquire()` 阻塞**（否则 read-only turn / job 轮询 / UI 全卡死）。
- **本地单用户、争用罕见**：可写 turn 等在飞 ingest 之后（反之亦然），偶发数十秒，可接受（同 P4.1 投喂排队语义）。read-only turn **不**取写锁（纯读、可与写者并发，零阻塞）。
- **统一锁序 `conversation.lock` → `write_lock`，所有会话态写操作同序（评审 High）**：可写 turn **与撤销端点**都先取本会话 `self.lock`（同会话各轮 + 撤销彼此串行，P4.2 既有）、再异步取进程 `write_lock`。**撤销绝不反序**——若撤销「先 `write_lock` 再会话锁」、而另一轮 chat 已持会话锁正等 `write_lock`，即交叉死锁。撤销取会话锁还顺带让它与**同会话在飞 turn 串行**（撤销读/回放的是会话内 undo token/写日志，须与正在追加该日志的 turn 互斥，避免竞态）。补「undo 与同会话在飞 turn 不死锁」测试。

## 6. 会话恢复语义（决策P4.5-7/8）

- **恢复用当前进程默认姿态、不回放盘上姿态**：盘上快照不存运行时姿态；`restore` 与新建**同一构造**，开局姿态 = 进程 `--mode` 默认（与决策P4.2-4「不恢复出越权姿态」一脉）。恢复后人可再 `/mode` 翻（决策P4.5-8）。
- **冷会话 `/mode` 翻转回 409**：姿态是 per-会话 live agent 的运行时态；仅盘上的冷会话无 agent 可翻 → `POST /api/chat/{id}/mode` 回 `409`「会话未激活，先续聊一轮恢复再切」，前端据此提示（沿用 P4.4 冷会话语义，决策P4.5-7）。`GET .../info` 对冷会话仍出 `cold_info` 部分信息（P4.4 §3）。
- **可写 turn 的落盘**：可写 turn 可能改 `agent.messages`（同任何 turn），按 P4.2 成功轮 off-loop 落盘；落的是会话历史（`.agentao/sessions/`），**不**落运行时姿态。

在 P4.4 的 `GET /api/info` + `GET /api/chat/{id}/info` 之上加：

|------|------|------|------|------|

- **宿主写端点时序互斥（层③，决策P4.5-10）**：`POST /api/raw`/`upload`/`ingest`/`heal` 入口先查 `active_writable_turns`，>0 → `423 Locked`「可写会话进行中，请稍后」（**用 `423` 不与 `/api/raw` 既有 `409`不覆盖冲突 混淆**；其余 404/4xx 语义不变，仅加此前置闸）。read-only turn 不计数、不触发。
- **`GET /graph` 纳入 `write_lock`（决策P4.5-6）**：`build_and_write_graph` 调用前后异步取/放 `write_lock`，与 ingest/heal/可写 turn 串行（防并发写 `graph/` 互踩、防读撕裂 `wiki/`）；**不**受 `active_writable_turns` 423 互斥（派生物、可幂等重建、已被锁串行）。失败仍 `500`、成功仍 `302`（既有契约不变）。
- **撤销端点带乐观并发 + 统一锁序（决策P4.5-13，评审 High）**：撤销本轮写经一个写动作，**按 `conversation.lock` → `write_lock` 取锁**（与可写 turn 同序，§5——绝不反序，否则与在飞 turn 交叉死锁；取会话锁还让撤销与同会话在飞 turn 串行、不竞态写日志）；逐文件比对当前哈希 == 本 turn 写后哈希，相等才还原；某文件已被后续写改动 → 该文件**跳过、回 `409` conflict**「已被后续写改动，未撤销」（前端标红），其余照常还原。undo token 只对最近一个可写 turn 有效、per-path 被后续写失效。
- **报告读端点 = best-effort（决策对应 §5）**：`/api/report/{check,health,lint}` 纯读 `wiki/`、**不取 `write_lock`**（避免面板卡在长写之后）；并发写下结果为 transient findings（下次轮询自愈），前端**不当稳定门禁**。权威 `wiki/` 门禁是 §3 锁内写后 `check`。
- **`set_mode`/`info` 逻辑落 `chat.py`，`app.py` 只序列化/分流**（决策P4.5-9，同 P4.4-2 / P4.3）。

## 9. 决策记录

- **\[决策P4.5-1] Web 双姿态对齐 agentao 枚举（`read-only`/`workspace-write`），绝不 `full-access`/`plan`；单份 `AGENTAO.md`、不做文件切换。** 「能不能写」由运行时两点置位 enforce、「raw/config 只读」由 `PolicyFileSystem` wrapper 独立 enforce、皆与 `AGENTAO.md` 散文解耦（CLI 同构）。`full-access` 绕全部 preset 规则、`plan` 是内部态——端点 `422` 拒、测试断死。
- **\[决策P4.5-2] `raw/`+`AGENTAO.md` 硬只读层① = 构造期注入的 `PolicyFileSystem` wrapper（姿态无关、装一次）。** 包住 `agent.filesystem`，对经该 capability 的结构化写（`write_file`/`replace` 终调 `write_text`）做**叶解引用包含判定**，命中 immutable 集（`raw/` ∪ `{AGENTAO.md}`——**`SCHEMA.md` 不在内，决策P4.5-12**）即拒。采上游 [`host-fs-policy.md`](../../agentao/docs/design/host-fs-policy.md) 的 interim 配方——P4.5 命中其「cwd 内只读子集」facet，**今天零 agentao 改动、原生 `write_file` 直接生效**。比旧 regex deny-rules 严格更安全（包含语义、不靠精确 arg 名、挡软链 clobber 的 fail-open）；immutable 集两姿态恒等 → wrapper 无需热切换，**顺带消掉 `add_run_rules` 无 remove/clear、`set_mode` 不装卸的 hack**。wrapper 还在 immutable 判定**前**先防御性断「目标在 kb 根内」（`t == kb or kb in t.parents` 否则拒），**不把 kb-containment 委托给上游 `contain_file`**——挡未来直调 `write_text(Path("/tmp/x"))` 绕预检的工具（评审 P4.5 #3）。**地基须先验（升为 C0 前置 spike，非脚注，评审反馈 D）**：①`build_from_environment`/`Agentao(filesystem=…)` 能把 wrapper 透传到 `agent.filesystem`；②内建写工具（`write_file`/`replace` 及任何 patch/notebook 类）**全部**终经 `write_text`、无旁路；③`_effective_target` 叶解引用、勿复用 fail-open 的 `contain_file`（建议推上游 `PathPolicy.contain_any` 归口）。**C0 验不过则可写模式不落地**——层① 是唯一「阻止写」的层，它不成立则只剩层③+约束，撑不起「确定性只读」。shell/MCP 够不着层①——`AGENTAO.md` shell 直写由层②还原、`raw/` shell 直写是残留、`SCHEMA.md` 放行、嵌入 agent 不挂 MCP 写工具。
- **\[决策P4.5-3 / (c) 重切] 层② 只对 `AGENTAO.md` 一个文件做快照+自动还原；`raw/` 树不 per-turn 扫、其 shell 直写并入残留。** 反向评审 A/B 的结论：在没有 sandbox、对抗性 agent 已封不住的前提下，为抓「shell 直写 `raw/`」这条堵不住的旁路而每条可写消息前置 O(`raw/`) 哈希+拷贝（`raw/` 可数百 MB），成本与价值不匹配。故**砍掉 F2 的 `raw/` 内容快照+回滚+symlink/目录/special 对象模型**；`raw/` 的 shell 直写降级为残留（与 kb-外写、派生后台进程同桶，sandbox 才堵）。**保留**的是 `AGENTAO.md` 单文件的快照+还原（行为硬约束、固定小、近免费、被改＝提权最该复原）：`snapshot_agentao` 存原字节、`restore_agentao` 把被旁路改/删的写回，`done`/`error` 带 `immutable_mutated` 标注「已还原」。`SCHEMA.md` 不在层②（agent 可写，决策P4.5-12）。read-only turn 与 `raw/` 树均零开销。**`raw/` 内容回滚不作为 P4.5 Done 条件**（验收降级，§10）。
- **\[决策P4.5-4] `wiki/` 写后 `check`：surface + 人触发「修复 / 撤销本轮」，软门禁、不硬阻断。** 可写 turn 收尾跑 `check.run_check`，findings 经 SSE surface，UI 给两键：①「让 Agent 修复」以 `REPAIR_PROMPT` 驱动下一轮；②「撤销本轮写」（`wiki/` + `SCHEMA.md`）——**复用层① wrapper 的 per-turn 写日志**（wrapper 本就拦每次 `write_text`，`wiki/` 页与 `SCHEMA.md` 都经它 → 都进同一份日志，记 **`(路径, 写前字节, 写后哈希)`**——写后哈希供撤销时乐观校验，决策P4.5-13），回放还原、O(被改文件)，不需全树快照（评审反馈，纠正「wiki 欠保护 / 过度保护 raw/」的反向分配）。**撤销范围含 `SCHEMA.md`**是决策P4.5-12「SCHEMA 可写可撤销」论证的落地（只覆盖 `wiki/` 则论证落空，评审 Medium）。**check 触发与撤销解耦**：写后 check **默认每条可写 turn 无条件跑**（覆盖 shell 直写）、不靠此日志、**也不用 mtime/size 指纹**（同大小/`touch -r` 会漏检；若要优化只能内容哈希，评审 High/Medium，§3）；撤销缺口=shell 直写 `wiki/`/`SCHEMA.md` 不入日志（check 仍 surface、但撤销盖不住），兜底（开 turn 前拍 `wiki/`+`SCHEMA.md` 快照）与阻断式合并门禁均列后续/单独相位。**撤销可用性 = 本轮写日志非空、独立于 `check`**：`done.undo:{available,token,paths}` 驱动 UI 撤销键，`SCHEMA.md`-only 改 / check 通过的 `wiki/` 改都仍可撤销，撤销键不挂 `check.violations` 分支（评审 Medium）。复用 `check`/`gate.REPAIR_PROMPT` 原语。撤销的并发与锁序见决策P4.5-13。
- **\[决策P4.5-13] 撤销带乐观并发控制，不盲回放（评审 High）。** 撤销按钮在 UI 留存到下一次写，期间 turn B / ingest / `/api/raw` 可能改同一文件；盲目回放「写前字节」会**覆盖这些后续写**。故 per-turn 写日志记 **`(路径, 写前字节, 写后哈希)`**，撤销时：① **取 `write_lock`**（撤销是写，§5 单写者范围含它）；② 逐文件**乐观校验**当前内容哈希 == 本 turn 写后哈希，相等才还原、否则**跳过并报 `409` conflict**「已被后续写改动，未撤销」（SSE note + UI 标红）；③ undo token **仅指向最近一个可写 turn 的日志**，任一后续写（chat/ingest/raw/heal）落到某 path 即令该 path 的撤销项失效（per-path 失效），下个可写 turn 收尾替换日志；④ **锁序与可写 turn 同：`conversation.lock` → `write_lock`**（评审 High）——撤销同时碰会话内 undo 日志与全局写锁，**绝不反序**（先 `write_lock` 再会话锁会与「已持会话锁、等 `write_lock` 的在飞 turn」交叉死锁），取会话锁也让撤销与同会话在飞 turn 串行、不竞态写日志。撤销=「把本 turn 这几处改回去」、非全局时间旅行；据此决策P4.5-12 的 `SCHEMA.md` 「可撤销」安全论证只在「撤销前该文件仍是本 turn 留下的状态」时成立，否则退化为人工 git 回退（仍可见、随 git）。
- **\[决策P4.5-11] 威胁模型显式化：prompt-injection 为唯一现实不信任源；确定性层 = 抬门槛/阻意外/挡同步·结构化旁路，非对抗性封堵；真封堵 = sandbox（deferred）。** 嵌入 agent 是用户助手但会 ingest 可被注入的 `raw/` 源。确定性层挡得住：结构化写（层①，`raw/`+`AGENTAO.md`）、可写 turn 内同步打写端点（层③）、`AGENTAO.md` 还原（层②）。挡不住：shell 直写 `raw/`、shell 写 kb 外、`curl … &` 派生后台进程——三类残留只 sandbox 能堵。由此**砍掉不成比例的 `raw/` per-turn 快照/回滚（决策P4.5-3）**，并把「为何不堆更多层」钉死在「层数补不了没有 sandbox 这个事实」上：sandbox 落地前，可写会话定位为「**受限信任**」模式，目标/验收/契约均不宣称「任何路径不可变」。
- **\[决策P4.5-12] `SCHEMA.md` 不在只读集，agent 可写（人驱动）——区别于 `AGENTAO.md`。** 二者性质不同：`AGENTAO.md` 是 **agent 行为硬约束（宪法）**，agent 改它＝自解约束＝**提权**，故硬只读（层①拒 + 层②还原）；`SCHEMA.md` 是**本库领域/页面类型/自定义规则**，是 agent 的**路由指引**、**不被 `check`/`lint` 门禁消费**（`pages.py:51` 明排除），skill 早定为「**人与 Agent 共同演进 / 随 markdown 走**」（`SKILL.md:26`）。把它捆进硬只读集本就与既有设计打架（P4.4 评审 #3「raw 硬 config 软」的正确解是「raw 硬 / AGENTAO 硬 / SCHEMA 共演」，而非三者一刀切）。**安全性**：门禁不读它 → 不存在「改 schema 让 check 放水」的循环信任攻击；它非提权面；最坏情况（注入 agent 改乱路由）可见、随 git、且经层① wrapper 进 per-turn 写日志 → **「撤销本轮」一键回退**（§3 撤销范围显式含 `SCHEMA.md`，与 `wiki/` 同口径——评审 Medium：否则此论证落空）。故 `SCHEMA.md` 归 wiki 级软门禁（写后 surface + 可撤销）。**caveat**：`health.py:27` 暗示未来或有「SCHEMA 覆盖」类检查读 `SCHEMA.md`——**若哪天 `check`/`lint` 把它当门禁输入**，循环信任回归，那时正解是「SCHEMA 改动须人审批后才生效于门禁」（参 P4.6 的「agent 提案→人背书→宿主落」模式），**而非退回永久冻结**。当下门禁不读它，agent 可写安全。
- **\[决策P4.5-5] `/mode` 运行时真切换（只翻两点置位），不重建 agent、不改文件、不动 wrapper。** 经 `POST /api/chat/{id}/mode` 调 `set_mode`，持会话锁与在飞 turn 串行。layer① wrapper 姿态无关、构造期装一次，`/mode` 只翻「能不能写」的 Mode 两点、不触 wrapper 的「写到哪」。新会话开局姿态 = 进程 `--mode`（默认 read-only）。
- **\[决策P4.5-6] 单写者用两把锁，别混（评审 P4.5 #2）。** ①`JobQueue._lock` 保持原样、只护作业表 `_jobs`/`_counter`（`jobs.py:54`，临界区极短，`enqueue`/`get_job` 用），**绝不**包长写作业 `fn()`（`fn()` 在 `_run` worker、`_lock` 之外，`jobs.py:103`）——否则 `enqueue`/`get_job` 被在飞 ingest 卡死。②新增注入的进程级 `write_lock` **只包真正写执行**：JobQueue worker 跑 `fn()` 前后、可写 chat turn 起跑/收尾、**`GET /graph` 的 `build_and_write_graph` 前后**都 acquire/release，互斥 → 任意时刻至多一写者、`wiki/`/`raw/`/`graph/` 写永不交错、快照窗口不被并发写互踩。**单写者范围 = `raw/`+`wiki/`+`workspace/`+`graph/` 四树写者**；`/graph` 这个既有同步写者（`app.py:413`，原先无锁——评审 P4.5 #2）一并收入 `write_lock`，但**不**纳入层③ 423 互斥（派生物、可幂等重建、已被锁串行，agent `curl /graph` 无 `raw/` 风险）。`write_lock` 跨「后台线程写者」与「async chat 写者」共享，故**不能用 `asyncio.Lock`**；可写 turn **异步获取阻塞锁**（`anyio.to_thread.run_sync(write_lock.acquire)`/executor，**绝不在 event loop 线程直接 `acquire` 阻塞**，否则 read-only turn/job 轮询/UI 全卡死）。read-only turn 不取 `write_lock`（可与写者并发、零阻塞）。**会话锁→`write_lock` 固定取序防死锁，可写 turn 与撤销端点同序**（撤销也碰会话内 undo 日志 + 写锁，反序即与在飞 turn 交叉死锁，评审 High，决策P4.5-13）。
- **\[决策P4.5-10] 硬只读层③ = 可写 turn 活跃期间宿主写端点时序互斥（兜 shell 经 localhost 旁路，评审 P4.5 #1）。** Web 宿主在 `127.0.0.1` 暴露的写端点（`/api/raw`/`upload`/`ingest`/`heal`）是**不经 `agent.filesystem` 的宿主合法写**，层① 够不着、层② 又不扫 `raw/` 树 → 可写 turn 里 agent `curl` 即可让宿主替它写 `raw/`，击穿 P4.6「源由人背书是架构保证」。对策：宿主维护 `active_writable_turns` 原子计数（**独立于 `write_lock`/`JobQueue._lock`**），计数 >0 时这些写端点 `423 Locked` 拒（不与 `/api/raw` 既有 `409`不覆盖 混淆）；agent shell 仅在可写 turn 内执行，curl 在 turn 内到达即被拒、根本不入队，同步竞态掐死。**不用 token/CSRF**（同机 same-uid shell 能抓页内 token/重放 cookie，非真边界；真边界需 Jupyter 式带外启动 token，本相位不引入）。**残留**（`curl … &` 派生后台进程在 turn 结束后才打端点、shell 写 kb 外）只有 shell 网络/localhost sandbox 能堵，列纵深防御后续。
- **\[决策P4.5-7] 冷会话 `/mode` 翻转回 409（先续聊恢复再切）；`info` 仍出 `cold_info` 部分信息。** 姿态是 live agent 运行时态，冷会话无 agent 可翻。沿用 P4.4 冷会话语义。
- **\[决策P4.5-8] 恢复用当前进程默认姿态、不回放盘上姿态。** 盘上不存运行时姿态；恢复与新建同一构造、开局进程默认，人可再 `/mode` 翻（与决策P4.2-4 一脉）。
- **\[决策P4.5-9] 不新增 SSE event 种类、不新增退出码；端点逻辑落 `chat.py`。** 可写 turn 的 `check`/`immutable_mutated` 是既有 `done`/`error` 帧的**可选字段**（非新事件类型）。`set_mode`/`info`/写后校验编排在 `chat.py`，`app.py` 只序列化/分流（Web 零业务智能，同 P4.3/P4.4）。
