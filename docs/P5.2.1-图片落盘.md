# P5.2.1 转换图片随源落盘 `guanlan convert`（图片随转换文件一并落 `raw/`，重写引用 · 引擎无关 · 仍零 LLM · 仍只写 `raw/`）

> **定位收窄**：P5.2.1 是 [`P5.2-多格式摄入.md`](P5.2-多格式摄入.md) 的一个**补漏半阶段**，**不是**新主流程、**不**改 P5.2
> 的任何既有契约（命令名 / 退出码 / 两步默认 / `rawio` 归口 / ingest 逐字不动 / 「源、非派生物」定性全部沿用）。它只补
> 一个具体的洞：**P5.2 的 `convert_to_markdown` 内核只取产物 `.md` 文本、把转换器抽出的图片连同 temp 树一并丢弃**——
> mineru/marker 转 PDF 时抽出的插图全部被扔掉，落进 `raw/<slug>.md` 的 `![](…)` 引用因此**全部悬空**（指向已被删除的
> temp 目录）。P5.2.1 让**图片随转换文件一起落进 `raw/`**、并**重写 markdown 里的图片引用**指向落盘后的新路径，使
> `raw/<slug>.md` 自洽可读。
>
> 目标（用户口径，逐字钉死）：`convert <filename.ext>` 时——
> - **markdown 文件**直接放到 `raw/` 根目录：`raw/<filename>.md`（与 P5.2 一致，不变）；
> - **图片文件**放到 `raw/images/<filename>/` 目录下，并**统一命名为 `<filename>-<number>.<ext>`**，`number` 按
>   markdown 内**首次出现序**从 **1** 开始编号；
> - **修订 markdown 里对图片的引用**，指向落盘后的新相对路径。
>
> 仍是**零 LLM 宿主写**（guanlan 不内嵌 LLM/密钥，只 shell out 到既有 `pdf-to-markdown` skill 当转换引擎，决策P5.2-4）、
> **只写 `raw/`**（md + `images/` 子树，不碰 `wiki/`/`log.md`、不起 Agentao、不取快照，决策P5.2-3）、图片是**源、非派生物**
> （逐字拷贝、不重编码、不承诺字节确定性，沿决策P5.2-4）。文中 **\[决策P5.2.1-N]** 集中记录在 §4。

## 0. 补的是 P5.2 内核「丢图片」那个洞

P5.2 的 `convert_to_markdown(src, …) -> str`（`convert.py:65`）在一个 `tempfile.TemporaryDirectory` 里跑 skill
`convert.py`，**只 `read_text()` 出产物 `.md` 的文本就返回**，`with` 块一退出整个 temp 树（含转换器抽出的全部图片）即被
删除（决策P5.2-10 的 temp 暂存防污染机制顺手也把图片清掉了）。结果：

- mineru 把插图抽到 `<stem>/<method>/images/<hash>.jpg`，md 里写 `![](images/<hash>.jpg)`；
- marker 把插图抽到 `<stem>/_page_X_Figure_Y.jpeg`（与 md 平级），md 里写 `![](_page_X_Figure_Y.jpeg)`；

两种引用都是**相对产物 md 自身所在目录**的相对路径。P5.2 落盘后只剩 `raw/<slug>.md` 一个文件，这些相对引用**指向的图片
已不存在**——`raw/<slug>.md` 里所有插图全悬空，ingest / 本地 markdown 渲染器读到的都是坏图链接（Web 预览能否显示图另见
§3.1 口径收窄）。

P5.2.1 在内核读出 md 文本的**同一个 `with` 块内**（temp 销毁之前）把被引用的图片**一并收集进内存**，连同**重写后的
markdown** 交给命令壳，由命令壳落进 `raw/images/<slug>/`。**引擎无关**：不写 mineru/marker 分支，统一按「相对产物 md
父目录解析每条引用」处理两种布局（决策P5.2.1-3）。

### 0.1 两引擎实测布局（已核实，驱动「引擎无关收集」）

| 引擎 | 产物 md | 图片落点 | md 内引用形态 |
|---|---|---|---|
| **mineru** | `<tmp>/<stem>/hybrid_auto/<stem>.md` | `<tmp>/<stem>/hybrid_auto/images/<hash>.jpg`（`images/` 子目录） | `![](images/<hash>.jpg)` |
| **marker** | `<tmp>/<stem>/<stem>.md` | `<tmp>/<stem>/_page_X_Figure_Y.jpeg`（与 md 平级） | `![](_page_X_Figure_Y.jpeg)` |

> `hybrid_auto` 是 mineru 实际 `<method>` 子目录名（随 `-b`/`-m` 后端而变，如 auto/txt/ocr/vlm/hybrid_*）——**不可硬编码**；
> P5.2 既有的 `find_markdown(root, stem)`（skill `convert.py:112`，`rglob("*.md")` 优先 `<stem>.md`）已归口这点。**关键不变式**：
> 两引擎的图片引用**都相对产物 md 自身目录**，故按相对 `produced_md.parent` 解析（`realpath` 后须落 temp 树内，§3.1 准入）
> 对两者都命中真实图片文件——这正是
> 「引擎无关收集」成立的依据（决策P5.2.1-3）。

## 1. 命令契约（不变；行为新增三条副作用）

命令签名、参数、退出码**全部不变**（`convert <src> [--name --origin --overwrite --dry-run --ingest --backend]`，无新
退出码、不引 `--model`）。仅新增三条**落盘副作用**（用户目标三条逐字落地）：

```bash
guanlan -C <kb> convert 报告.pdf
# → 写 raw/报告.md（图片引用已重写）
# + 写 raw/images/报告/报告-1.jpg、报告-2.jpg …（按 md 内首次出现序编号）
# 回执：✓ 已写 raw/报告.md（origin: 报告.pdf，N 字节，M 张图片 → raw/images/报告/）
```

- **`--dry-run`**：转换 + 收集 + 引用重写照跑，但**连图片一并零落盘**（不写 md、不建 `raw/images/<slug>/`），只把**重写后的
  markdown** 打 stdout，stderr 注记「含 M 张图片，`--dry-run` 未落盘；引用已重写为 `images/<slug>/<slug>-N.ext`」（决策P5.2.1-7）。
- **`--overwrite`**：覆盖 `raw/<slug>.md` 的同时**整盘替换 `raw/images/<slug>/`**（经 staging-swap、旧图保 `.bak` 直到换成，
  §3.2），避免上一次转换残留的旧图片串味（决策P5.2.1-6/9）。失真告警文案沿用 P5.2-8。
- **`--name <slug>`**：图片目录 / 文件名跟随**最终 slug**（`raw/images/<slug>/<slug>-N.ext`），与 md 文件名对齐（决策P5.2.1-2）。

## 2. 模块与归口（改 `convert.py`，**不碰 `rawio.py`**）

### 2.1 `convert_to_markdown` 返回类型升级（str → 结构化 `ConvertResult`）

图片字节**必须在 `TemporaryDirectory` 销毁前读入**——故收集逻辑只能落在内核 `with` 块内，内核返回类型从 `str` 升为携带
图片字节的结构化结果（决策P5.2.1-4）：

```python
@dataclass(frozen=True)
class ConvertedImage:
    name: str    # 落盘后文件名 <stem>-<n>.<ext>（n 从 1，ext 取原图后缀小写）
    data: bytes  # 图片字节，逐字拷贝、不重编码（决策P5.2.1-8）

@dataclass(frozen=True)
class ConvertResult:
    markdown: str                            # 引用已重写为 images/<stem>/<stem>-<n>.<ext>
    images: tuple[ConvertedImage, ...] = ()  # 按 md 内首次出现序；无图 → 空 tuple
    skipped: int = 0                         # 未准入/未搬运的引用数（原样保留），供命令壳 stderr 计数

# 容量上限（决策P5.2.1-11）——保护 convert 入内存 + 下游 ingest 的 snapshot_raw 递归 hash 整个 raw/。
IMAGE_EXTS = {".jpg", ".jpeg", ".png", ".gif", ".webp", ".bmp", ".tif", ".tiff", ".svg"}  # 图片扩展白名单
MAX_IMAGE_BYTES = 20 * 1024 * 1024          # 单图 ≤ 20 MiB
MAX_IMAGES_TOTAL_BYTES = 200 * 1024 * 1024  # 单次转换全部图累计 ≤ 200 MiB
MAX_IMAGE_COUNT = 500                        # 单次转换张数 ≤ 500（任一超限 → ConvertError → EXIT_USAGE）

def convert_to_markdown(src: Path, *, stem: str, backend: str = "auto",
                        cwd: Path | None = None) -> ConvertResult:
    """P5.2 内核 + P5.2.1 图片收集。新增参数 `stem` = 最终 raw slug（命令壳传 target.stem），
    用于图片重命名与引用重写口径对齐 md 文件名。
    既有步骤（temp 暂存绝对 staged / cwd=KB root 保 .env / 子进程调 skill / 解析产物路径）不变（P5.2-§2.1）；
    新增：在读出产物 md 文本的**同一 with 块内**（`tmp_root` = 该次 TemporaryDirectory 绝对根）——
      1. 扫描 markdown 图片语法 ![alt](url ["title"]) 的 url，按文档序；
      2. **准入硬约束（决策P5.2.1-5，安全敏感，逐条 AND）**——任一不过即 **原样保留引用、不收集、不编号**：
         (a) url 仅可为**相对路径**：拒绝绝对路径（`/…`、Windows `X:\…`）、`~`、`http(s)://`/协议相对 `//`/`file:`/
             `data:` 等任何带 scheme 的 url；
         (b) `urllib.parse.unquote` 解码（挡 `%2e%2e` 等编码穿越）后，按相对 `produced_md.parent` 解析；
         (c) `os.path.realpath`（解 symlink）后必须 `relative_to(tmp_root)`——**任何指向 temp 树外的引用（含 temp 内
             symlink 逃逸到 /etc/passwd、`../../secret`）一律拒收**；
         (d) 落点须是**真实普通文件**（`is_file()` 且非 symlink 本身、非 FIFO/设备/目录）；
         (e) 后缀（小写）须在**图片扩展白名单** `IMAGE_EXTS`（.jpg/.jpeg/.png/.gif/.webp/.bmp/.tif/.tiff/.svg，与 skill
             `SUPPORTED_EXTENSIONS` 图片子集对齐）；
      3. 按解析后 realpath **去重**：同一图片多次引用复用同一序号/文件名（决策P5.2.1-5）；
      4. 按**首次出现序**从 1 编号：new_name = f"{stem}-{n}{ext}"（ext = 解析图后缀小写）；**逐张校验容量上限**
         （决策P5.2.1-11）：单图 > `MAX_IMAGE_BYTES` / 累计 > `MAX_IMAGES_TOTAL_BYTES` / 张数 > `MAX_IMAGE_COUNT`
         → raise `ConvertError`（命令层 EXIT_USAGE，非静默丢弃）；过限前 read_bytes() 入内存；
      5. 重写每条被收集引用的 url 段为 `images/{stem}/{new_name}`（POSIX 相对，保留 alt/title）；
    返回 ConvertResult(重写后 md, images)。无图片/全部被准入挡掉 → images=()，md 对应引用原样（向后兼容）。"""
```

> **为何 `stem` 进内核**：图片文件名与引用路径都要对齐**最终落盘的 md slug**（用户目标第二条），而 slug 由命令壳经
> `rawio.safe_raw_target` 算出。把它作内核入参传入，内核就能在 temp 内一次性把引用重写到最终形态，命令壳只负责把
> `images` 字节落盘、不必二次改写文本（决策P5.2.1-4）。

### 2.2 命令壳 `run_convert` 落盘改造（图片落 `raw/images/<slug>/`，md 仍走 `rawio`）

```python
def run_convert(src, *, root, name=None, origin=None, overwrite=False,
                dry_run=False, do_ingest=False, backend="auto") -> int:
    # 1) 先算 target / slug（与转换无关，可提前；顺带对坏名快失败）
    target = safe_raw_target(root, name if name is not None else Path(src).stem)  # raw/<slug>.md
    stem = target.stem
    # 2) 早存在性预检（默认不覆盖 → 连转换/收集都不跑，快失败，决策P5.2.1-6）
    if target.exists() and not overwrite: return EXIT_USAGE  # 文案沿 P5.2
    # 3) 转换 + 收集（cwd=root 保 .env；stem 传入用于命名/重写）
    result = convert_to_markdown(src_path, stem=stem, backend=backend, cwd=root)
    # 4) 文本准入 + provenance（对 result.markdown，与 P5.2 同）
    check_text_admission(result.markdown); content = apply_origin(result.markdown, origin_value)
    # 5) --dry-run：连图片一并零落盘，仅打印重写后 md + stderr 注记 M 张未落盘（决策P5.2.1-7）
    if dry_run: print(content); return EXIT_OK
    # 6) 落盘（sibling-staging + 「图目录先换、md 末步提交」，决策P5.2.1-9）：所有「可失败的字节写」都先落
    #    staging，real 图目录只被「rename」这种不分配空间、近乎不可失败的元数据操作触碰；**md 严格最后提交**，
    #    故任何成功落盘的新 md 其引用的图必已就位（注意：images 落点是 target.parent/"images"，即 raw/，决策P5.2.1-2）。
    images_root = target.parent / "images"         # = <kb>/raw/images/（target = raw/<slug>.md → parent = raw/）
    real = images_root / stem                       # raw/images/<slug>/
    staging = images_root / f".{stem}.staging-<uniq>"  # sibling，同一文件系统 → rename 原子
    #    (a) 把全部 result.images 写进 staging（mkdir(parents); 写 ConvertedImage.data；
    #        任何 OSError → 删 staging、real 与旧 md 原封不动 → except OSError → EXIT_USAGE，决策P5.2.1-10）
    #    (b) 换图目录（仅 rename，**在 md 提交之前**）：若 real 存在 → os.rename(real, .{stem}.bak-<uniq>)；
    #        os.rename(staging, real)
    #    (c) **末步提交 md**（权威写 + 不覆盖门禁，零改；md 是最后落盘的资源）：
    code = atomic_write_raw(target, content, overwrite)
    #        若 code != EXIT_OK（罕见 TOCTOU 抢建）或字节写 OSError → **回滚图目录**（fresh: rmtree real；
    #        overwrite: 移除 real 后 os.rename(.bak, real) 复位旧图）→ 返回 code / EXIT_USAGE（旧 md+旧图全复位）
    #    (d) 成功 → rm .{stem}.bak
    #    若 result.images 为空 → 跳过 (a)(b)(d)，仅 (c) 写 md（无图文档零建 images 目录，向后兼容）
    # 7) 回执含图片数；--ingest 串联不变
```

- **`rawio.py` 一字不改**（决策P5.2.1-1）：图片落盘是 `convert` 命令侧专属逻辑——**P4.6 Web 投喂/晋级与 `rawio` 都不涉
  图片**（它们写的是人手贴/晋级的纯文本 `.md`）。把图片处理塞进 `rawio` 会**破坏 P5.2-6 的「零行为变更抽出」硬门**。故
  图片收集在 `convert.py` 内核、落盘在 `convert.py` 命令壳，md 文本仍**逐字复用 `rawio.atomic_write_raw`**（无覆盖门禁
  /409-500 分流契约全不动）。
- **图片落盘路径归口（注意是 `raw/images/`，不是 `<kb>/images/`）**：新增一个 `convert.py` 内的小 helper——以 **md 目标
  的父目录**为锚：`_images_dir(target) -> target.parent / "images" / target.stem`（`target` = `safe_raw_target(...)` =
  `<kb>/raw/<slug>.md`，故 `target.parent` = `<kb>/raw/` → 落点恒为 `<kb>/raw/images/<slug>/`，与 md 内 `images/<slug>/…`
  相对引用对齐）。**绝不写成 `root/"images"/stem`**（那会落到 `<kb>/images/<slug>/`、与 md 引用断裂）。`stem` 已是 slug-safe
  （出自 `safe_raw_target`），`<stem>-<n><ext>` 天然安全、无越界。

## 3. 行为细则

### 3.1 收集 + 编号 + 重写（引擎无关，决策P5.2.1-2/3/5）

- **扫描**：markdown 图片语法 `![alt](url ["可选title"])`，url 取到空白或 `)` 为止；按**文档先后序**遍历。**仅 markdown
  图片语法**——HTML `<img src=…>` 不在范围（mineru/marker 产物用 markdown 语法，§0.1 实测；HTML 图属极端少数，列入边界 §6）。
- **准入硬约束**（决策P5.2.1-5，**安全敏感**，五条 AND，任一不过即原样保留引用、不收集）：
  - **(a) 必须相对路径**：拒绝绝对路径（`/etc/passwd`、Windows `C:\…`）、`~`、以及任何带 scheme 的 url
    （`http(s)://`、协议相对 `//`、`file:`、`data:`）——它们不是 temp 树里的本地产物图；
  - **(b) 先 `urllib.parse.unquote` 解码再解析**：挡 `%2e%2e%2f`（`../`）等百分号编码穿越；
  - **(c) `realpath` 解 symlink 后必须落在 temp 根内**：以本次 `TemporaryDirectory` 绝对根 `tmp_root` 为界，
    `os.path.realpath((produced_md.parent / ref))` 须 `relative_to(tmp_root)`——**`![](../../secret)`、`![](/etc/passwd)`、
    或 temp 内一个指向库外的 symlink，统统拒收**（杜绝「把任意可读文件拷进 `raw/images/`」）；
  - **(d) 必须真实普通文件**：`is_file()` 且**自身非 symlink**、非目录/FIFO/设备；
  - **(e) 后缀（小写）∈ `IMAGE_EXTS` 白名单**（§2.1）：非图片扩展不搬。
  任一不过 → **原样保留引用、不编号**（防御坏/恶意引用，stderr 计数告知未搬运数，**不让转换整体失败**）。
- **去重**：按解析后 **realpath** 去重——同一张图被 md 引用多次时复用**同一序号/文件名**（决策P5.2.1-5）。
- **编号 + 容量校验**：按**首次出现序**从 **1** 递增（用户目标第二条逐字）；`new_name = f"{stem}-{n}{ext}"`，`ext` =
  解析图文件后缀**小写**（逐字保留原扩展、不归一 `.jpeg→.jpg`，决策P5.2.1-8）。逐张累计校验 §2.1 三上限（单图/累计/
  张数），任一超限 → `ConvertError` → `EXIT_USAGE`（决策P5.2.1-11，**非静默丢弃**）。
- **重写**：把被收集引用的 **url 段**替换为 `images/<stem>/<new_name>`——这是 `raw/<slug>.md`（在 `raw/`）到
  `raw/images/<slug>/<new_name>` 的**正确相对路径**；**保留 alt 文本与可选 title**，只换 url。远程/未准入引用整条文本不动。

> **「raw 文件系统内自洽」（口径收窄，决策P5.2.1-12）**：md 在 `raw/<slug>.md`、图在 `raw/images/<slug>/<slug>-N.ext`，
> 引用 `images/<slug>/<slug>-N.ext` 从 md 所在目录 `raw/` 解析即命中——**任意本地 markdown 渲染器/编辑器/ingest 读到的
> 都是有效本地图链**。但**现有 Web 不保证可加载该图**：`/api/raw/file` 只返回**渲染后的 HTML**、当前**无 `raw/images/**`
> 的只读图片端点/静态服务**。故本阶段只承诺**「raw 文件系统内自洽」**，**不**承诺「Web 预览能显示图」——后者需新增一个
> 只读图片端点，属 P4.6 机会性优化（非本阶段，§6 / 决策P5.2.1-12）。

### 3.2 落盘顺序、staging-swap 与覆盖语义（决策P5.2.1-6/9）

**问题**（评审 #2）：若「先清旧图目录 → 写新图 → 最后写 md」，而最后 `atomic_write_raw` 因磁盘满/权限失败，会落得「旧
`raw/<slug>.md` 仍在、但图目录已被清空/替换」→ **坏图链接**，与「成功落盘的 md 永不指向缺失图片」自相矛盾。

**解法：sibling-staging + 「图目录先换、md 末步提交」**（决策P5.2.1-9）。前一版（评审第一轮）把 md 提交放在图目录 rename
**之前**——评审 #2（第二轮）正确指出这留了个窗：「`atomic_write_raw` 成功后、图目录 rename 前」崩溃 → **新 md 已落、指向新图
名，但 real 目录仍是旧图/不存在**，承诺破。**根因**：一个 md 文件 + 一个目录无法跨资源真原子，故**谁最后落盘，谁的引用必须
已就位**。因此把 **md 调到最后提交**——其引用的图在它落盘前就已 rename 到位：

1. **写 staging**：`<kb>/raw/images/.<slug>.staging-<uniq>/` ← 全部 `ConvertedImage.data`（**这里才是可能 ENOSPC/权限失败的
   地方**）。失败 → 删 staging、**real 与旧 md 原封不动** → `except OSError` → `EXIT_USAGE`（决策P5.2.1-10）。
2. **换图目录（仅 rename，在 md 提交之前）**：`os.rename(real, .<slug>.bak-<uniq>)`（仅当 real 存在，即 overwrite）→
   `os.rename(staging, real)`。两次都是**同一文件系统内、不分配空间的元数据操作**，在步骤 1 已成功的前提下近乎不可失败。
3. **末步提交 md**：`atomic_write_raw(target, content, overwrite)`（`os.replace`，原子；仍是**权威写 + 不覆盖门禁**，零改）。
   返回非 `EXIT_OK`（罕见 TOCTOU 抢建）或字节写 `OSError` → **回滚图目录**（fresh: `rmtree(real)`；overwrite: 移除 real 后
   `os.rename(.bak, real)` 复位旧图）→ 返回 `code` / `EXIT_USAGE`，**旧 md + 旧图完整复位**。成功 → `rm .<slug>.bak`。
- **承诺现在成立**：**md 是最后落盘的资源**——`os.replace` 提交成功的那一刻，其引用的 `images/<slug>/<slug>-N.ext` 早在步骤
  2 就 rename 到位。故**任何成功落盘的新 `raw/<slug>.md` 永不指向缺失图片**（评审 #2 的核心诉求）。
- **门禁单点 + 快失败**：用户面的「同名已存在且无 `--overwrite`」门禁前移为 §2.2 步骤 2 的**早存在性预检**——命中即
  `EXIT_USAGE`，**连转换/图片收集都不跑**（省一次重型转换）。`atomic_write_raw` 内部不覆盖复检保留为纵深防御。
- **唯一残余窗（诚实标注，仅涉旧 md / 不涉任何成功提交的新 md）**：overwrite 下，残余窗**从步骤 2 的 `rename(real→bak)`
  开始、到步骤 3 的 md 成功提交（或异常回滚完成）为止**——评审 #1（第三轮）正确指出它**不止「两次 rename 之间」**，至少含
  两个 `kill -9` 子区间：(i)「`rename(real→bak)` 之后、`rename(staging→real)` 之前」→ **real 缺失** + 旧 md 仍在（旧 md 指向
  已不存在的旧图目录）；(ii)「`rename(staging→real)` 之后、md 提交之前」→ **real 已是新图** + **旧** md 仍在（旧 md 可能断图
  /串图，因新图按 `<slug>-N` 重编号、与旧引用未必对齐）。但两个子区间的共性是：①受影响的恒是**旧 md**，且整次 overwrite
  **未报成功**（**无任何「成功落盘的新 md」被破**——承诺面不动）；②`.bak` 在 md 成功提交前**始终完整保留旧图**，崩溃后可手工
  /重跑 `--overwrite` 从 `.bak` 复位；③`.bak`/`.staging-*` 以 `.` 前缀、**不被 ingest/检索扫描**（非 `raw/*.md`），下次同 slug
  `--overwrite` 清理之。这是「一个 md 文件 + 一个目录」无法跨资源真原子的固有残余，**整窗仅波及旧 md、可由 `.bak` 复位**，
  不再为它引两阶段提交。
- **`--overwrite` 整盘替换图片目录**（决策P5.2.1-6）：覆盖经上述 swap **整盘换掉** `<kb>/raw/images/<slug>/`——否则上次转换
  抽 8 张、这次抽 5 张，残留的旧 `<slug>-6/7/8.ext` 会串味。只换**该 slug 自己的子目录**，绝不动 `raw/images/` 下其它源目录。

### 3.3 `--dry-run`：连图片一并零落盘（决策P5.2.1-7）

转换 + 收集 + 引用重写照跑（让人预览**最终形态**的 md），但**不写 md、不建 `raw/images/<slug>/`、不落任何图片字节**；
stdout 是重写后的归一 markdown（含 provenance frontmatter），stderr 注记「含 M 张图片，`--dry-run` 未落盘；引用已重写为
`images/<slug>/<slug>-N.ext`」。退 `EXIT_OK`。`--dry-run --ingest` 仍互斥（沿 P5.2-8）。

### 3.4 仍是源、非派生物；仍只写 `raw/`（沿决策P5.2-3/4）

- 图片**逐字拷贝、不重编码、不缩放、不改格式**（决策P5.2.1-8）：它们和 md 一样是**人一次性背书的源**，不是必须可幂等
  重建的派生物；后端跨版本/是否启用 LLM 增强导致图片集变化，可接受（沿决策P5.2-4「转换不承诺字节确定性」）。
- 仍**只写 `raw/`**（md + `images/` 子树），不碰 `wiki/`/`log.md`、不起 Agentao、不取 raw 快照（沿决策P5.2-3）。
- 仍**零 LLM 宿主写**：guanlan 不内嵌 LLM/密钥，图片由外部 skill 转换器抽出，guanlan 只搬运字节（沿决策P5.2-4）。

## 4. 决策（沿仓库 `决策PX-N` 体例）

- **决策P5.2.1-1（图片落盘归口在 `convert.py`，`rawio.py` 一字不改）**：图片处理是 convert 专属——P4.6 Web 投喂/晋级与
  `rawio` 都只处理纯文本 `.md`、不涉图片。把图片逻辑塞进 `rawio` 会破 P5.2-6 的「零行为变更抽出」硬门。故收集在内核、
  落盘在命令壳，md 文本仍逐字复用 `rawio.atomic_write_raw`（不覆盖门禁 / 409-500 web 分流契约全不动）。
- **决策P5.2.1-2（目录结构 `<kb>/raw/images/<slug>/<slug>-<n>.<ext>`，编号按 md 内首次出现序从 1，与 md slug 对齐）**：逐字
  落地用户三条目标。**落点锚在 md 目标的父目录**（`target.parent/"images"/stem`，`target` = `<kb>/raw/<slug>.md` → 父目录
  `<kb>/raw/`），**不是 `<kb>/images/`**（评审 #1：误写 `root/"images"/stem` 会落到 `<kb>/images/<slug>/`、与 md 内
  `images/<slug>/…` 相对引用断裂）。「一个源一子目录」便于整体删除/覆盖（决策P5.2.1-6）、与 md 文件名对齐便于人肉对应；
  `<slug>-<n>` 统一命名抹掉 mineru 的 hash / marker 的 `_page_X_Figure_Y` 等引擎内部命名噪声。
- **决策P5.2.1-3（引擎无关收集：按「相对产物 md 父目录解析引用」统一处理 mineru 与 marker，不写引擎分支）**：§0.1 实测两
  引擎引用**都相对产物 md 自身目录**，按相对 `produced_md.parent` 解析（叠加 §3.1 准入安全闸）对两者皆命中真实图——故**不**判断 backend、
  **不**硬编码 `images/` 或 `_page_*` 命名。新增第三方后端只要遵循「md 内相对引用」即自动兼容。
- **决策P5.2.1-4（内核返回类型 str → `ConvertResult`，图片字节须在 temp 销毁前入内存；`stem` 进内核驱动命名/重写）**：图片
  只活在 `TemporaryDirectory` 里，必须在 `with` 块内 `read_bytes()` 入内存，故收集逻辑落内核、返回类型升级为携带
  `tuple[ConvertedImage,…]` 的 dataclass。`stem`（最终 slug）作内核入参，使内核一次性把引用重写到最终形态、命令壳只搬字节。
  **既有测试适配**：`tests/test_convert.py` 现把 `convert_to_markdown` mock 成返回 `str`（`test_convert.py:127` 等十余处），
  P5.2.1 落实现时需改成返回 `ConvertResult(markdown=…, images=())`——这是 spec 已预告的破坏性返回类型变更（§8）。
- **决策P5.2.1-5（图片引用准入是安全边界：只收 temp 树内、相对路径、真实图片普通文件；越界/scheme/symlink-逃逸/非白名单
  扩展一律拒收）**：评审 #1 指出「命中现存文件就收集」会被 `![](/etc/passwd)`、`![](../../secret)`、temp 内 symlink 逃逸
  利用成「把任意可读文件拷进 `raw/images/`」。故钉死五条 AND 准入（§3.1）：(a) 仅相对路径、拒一切 scheme/绝对/`~`；
  (b) `unquote` 后再解析挡编码穿越；(c) `realpath` 解 symlink 后必须 `relative_to(tmp_root)`；(d) 须真实普通文件（非
  symlink 自身/目录/FIFO/设备）；(e) 扩展 ∈ `IMAGE_EXTS` 白名单。任一不过 → 原样保留引用、不收集（坏/恶意引用 stderr
  计数告知，不让整次转换失败）。重写仍只换 url 段、保留 alt/title；按 realpath 去重、同图多引复用同号。
- **决策P5.2.1-11（图片容量上限：单图 / 累计 / 张数三道闸 → 超限 `EXIT_USAGE`，非静默丢弃）**：评审 #3 指出
  `rawio.MAX_RAW_BYTES` 只护 markdown 文本、不护图片，而下游 `snapshot_raw` 会**递归 hash 整个 `raw/`**——大 PDF 抽出大量
  高分辨率图会把图片全 `read_bytes()` 入内存拖垮 convert、并让后续 ingest 的 raw 快照门禁变慢。故设 `MAX_IMAGE_BYTES`
  (20 MiB) / `MAX_IMAGES_TOTAL_BYTES` (200 MiB) / `MAX_IMAGE_COUNT` (500) 三道闸（§2.1，值可调），逐张累计校验，任一超限
  → `ConvertError` → `EXIT_USAGE` + 明确文案（指出哪道闸、建议拆分/降分辨率/`--dry-run` 排查），**绝不静默丢图**（丢图会让
  md 引用悬空，违 §3.1 自洽）。
- **决策P5.2.1-12（落盘自洽口径收窄为「raw 文件系统内自洽」；Web 预览显示图属 P4.6 机会性优化、非本阶段）**：评审 #4 指出
  现有 Web `/api/raw/file` 只返回渲染 HTML、**无 `raw/images/**` 只读端点/静态服务**，`images/<slug>/…` 对本地 markdown
  自洽、对现有 Web 不一定可加载。故本阶段**只承诺「raw 文件系统内自洽」**（本地渲染器/编辑器/ingest 命中），**不**承诺
  「Web 预览显示图」；把只读图片端点纳入范围属 P4.6 机会性优化（与决策P5.2-9 的「Web 半 = P4.6 机会性优化」一致），不在
  P5.2.1 验收内（§6）。
- **决策P5.2.1-6（`--overwrite` 经 staging-swap 整盘替换 `raw/images/<slug>/`，旧图保 `.bak` 直到换成）**：覆盖须整盘换掉
  该 slug 图目录，否则旧转换的多余图残留串味；但**不**「先清后写」（评审 #2：清后若 md 写失败 → 坏图链接）——改为 §3.2 的
  staging-swap：旧图保留为 `.bak` 直到新图 swap 成功，失败可完整复位。默认不覆盖时早存在性预检（连转换都不跑）省一次重型
  转换；门禁仍单点（`atomic_write_raw` 复检为纵深防御）。
- **决策P5.2.1-7（`--dry-run` 连图片一并零落盘，仅打印重写后 md + stderr 注记 M 张未落盘）**：dry-run 的契约是「`raw/` 零写」
  （P5.2-8），图片同属 `raw/` 写、故一并不落；但仍跑收集+重写让人预览最终形态的 md 与图片数。
- **决策P5.2.1-8（图片逐字拷贝、不重编码；ext 取原后缀小写、不归一）**：图片是源、非派生物（沿决策P5.2-4），不缩放/转码/
  改格式以免失真；`.jpeg` 不强归 `.jpg`，逐字保留原扩展（仅小写化）。
- **决策P5.2.1-9（落盘用 sibling-staging + 「图目录先换、md 末步提交」，非「先清后写」、亦非「先提交 md 后换图」）**：评审第一
  轮把「清旧图→写新图→写 md」改成 staging-swap，但仍把 md 提交放在图 rename **之前**；评审 #2（第二轮）正确指出这留了
  「md 已提交、图未换」的窗（新 md 指向新图名、real 仍旧图/不存在）。**根因**：md 文件 + 图目录无法跨资源真原子，故**最后落盘
  者的引用必须先就位**。终版：①可失败字节写**全隔离到 sibling staging**（ENOSPC/权限只可能在此，real 与旧 md 未被碰）；
  ②**先 `rename` 换图目录**（同 fs 不分配空间、近乎不可失败；overwrite 旧图存 `.bak`）；③**`atomic_write_raw` 末步提交 md**
  （仍是不覆盖门禁单一权威点，零改复用 rawio；失败则从 `.bak`/`rmtree` 回滚图目录、旧态完整复位）。**承诺「成功落盘的新 md
  永不指向缺失图片」据此成立**（md 最后提交、图先就位）。唯一残余是 overwrite「**从 `rename(real→bak)` 起、到 md 成功提交或
  异常回滚完成止**」（含 `rename(staging→real)` 后、md 提交前那段）被 `kill -9` 时短暂破**旧** md（整次未报成功、`.bak` 始终
  保留旧图可复位），**不触及任何成功提交的新 md**（§3.2 诚实标注），不引两阶段提交。
- **决策P5.2.1-10（无新退出码；图片落盘 IO 失败走 `run_convert` 的 `except OSError` → `EXIT_USAGE`）**：沿 P5.2-6/7——
  写图片字节的 `OSError`（磁盘满/权限）由命令壳统一 `except OSError` 转码、打印「写 raw/images 失败」、返回 `EXIT_USAGE`，
  不外抛 traceback（镜像 `graph_entrypoint`）。**不引新退出码。**

## 5. 与 P5.2 / ingest / Web 的关系（加法，最小改动）

- **P5.2 既有契约全不变**：命令名 / 参数 / 退出码 / 两步默认 / 「源、非派生物」/ cwd=root 保 `.env` / temp 防污染 /
  `rawio` md 写 / ingest 逐字不动——P5.2.1 只在内核加「图片收集」、在命令壳加「图片落盘 + 引用已在内核重写」。
- **ingest 仍逐字不动**：它读 `raw/<slug>.md` 的 `.md` 文本建页；图片作为源的一部分静躺 `raw/images/<slug>/`，ingest 是否
  在生成页里引用图片由 Agent/skill 决定，**不在 P5.2.1 范围**（P5.2.1 只保证 `raw/` 内引用自洽）。
- **Web 半仍是 P4.6 的机会性优化（非阶段，沿决策P5.2-9）**：将来若把 P4.6 解析步替成直调 `convert_to_markdown` 内核，可
  连图片一并复用（内核已返回 `ConvertResult`）；本阶段不实现。**MCP 仍明确不做 convert**（写 `raw/` 与只读相悖）。

## 6. 边界（明确不做）

- **不处理 HTML `<img>` / 其它非 markdown 图片语法**——mineru/marker 产物用 markdown `![](…)`（§0.1 实测）；HTML 图属极端
  少数，留后；
- **不重编码 / 缩放 / 转格式图片**——逐字拷贝、源非派生物（决策P5.2.1-8）；
- **不搬运远程 / `data:` URI / 绝对路径 / temp 树外 / symlink-逃逸 / 非白名单扩展的引用**——准入五条 AND 拒收，原样保留
  引用（决策P5.2.1-5，安全边界）；
- **不收集 md 未引用的旁路产物**（mineru 的 `*_layout.pdf`/`*_middle.json`/`*_content_list.json` 等）——只搬被 md `![](…)`
  引用且能解析到的图片；
- **不搬运超容量图片集**——单图/累计/张数任一超限即 `EXIT_USAGE`、**不静默丢图**（决策P5.2.1-11）；
- **不新增 Web `raw/images/**` 只读图片端点 / 静态服务**——本阶段只承诺「raw 文件系统内自洽」，Web 预览显示图属 P4.6 机会性
  优化（决策P5.2.1-12）；
- **不改 `rawio.py` / 不碰 P4.6 Web 投喂晋级 / 不动 ingest gate**（决策P5.2.1-1）；
- **不在 ingest 生成页里自动插图**——P5.2.1 只保证 `raw/` 内引用自洽，wiki 页是否用图是 ingest/skill 的事；
- **不承诺图片集字节确定性 / 幂等重建**——产物是源（沿决策P5.2-4）；
- **不引新退出码 / 不改命令签名 / 不加 `--model`**（沿 P5.2）。

## 7. JSON 契约 / 退出码（全不变）

`convert` 仍是宿主写命令、**无 stdout JSON 契约**。`--dry-run` 的 stdout 仍是纯 markdown（**引用已重写**）。退出码沿用
P5.2-§7 矩阵**逐条不变**，仅在「raw 写盘 IO 失败 → `EXIT_USAGE`」一行的语义里**追加「图片字节落盘 IO 失败」也走同一
`except OSError` → `EXIT_USAGE`**（决策P5.2.1-10）。**无新退出码。**

## 8. 测试要点（扩 `tests/test_convert.py`；回归 `tests/test_web.py`/`test_convert.py`）

- **mineru 形态收集**：构造 `<stem>/<method>/<stem>.md`（含 `![](images/<hash>.jpg)`）+ `images/<hash>.jpg` 图 → 落盘后
  `raw/images/<slug>/<slug>-1.jpg…`，`raw/<slug>.md` 引用重写为 `images/<slug>/<slug>-1.jpg…`，序号 = 出现序。
- **marker 形态收集（引擎无关）**：构造平级 `![](_page_X_Figure_Y.jpeg)` + 同级图 → **同样结果**（验证不写引擎分支，决策P5.2.1-3）。
- **编号从 1 按首次出现序 / 同图多引复用同号**（决策P5.2.1-5）：md 内重复引用同一文件 → 只落一份、复用同 `<slug>-n`。
- **ext 保留小写不归一**（决策P5.2.1-8）：`.JPG`/`.jpeg` → 落盘为 `.jpg`/`.jpeg`（小写、不强归一）。
