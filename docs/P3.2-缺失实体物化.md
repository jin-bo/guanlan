## 1. 目标与范围

**P3.2 = 把 `lint.missing_entity`（被 ≥2 页引用却无页的高价值断链）按需用 LLM 物化成 entity 页。**

> **后修订（概念分类）**：heal 不再只建 `entities/`，而是按目标的实体/概念属性落 `wiki/entities/`（人物/组织/产品/系统）或 `wiki/concepts/`（方法/理论/术语/战法等）。写集允许写入面相应从「新建 `entities/`」放宽为「新建 `entities/`∪`concepts/`」（决策P3.2-11 的口径不变，只是把允许新建目录从一个扩到两个）；`HEAL_PROMPT`/SKILL 加一步实体/概念分类（拿不准当实体）。下文凡「新建 entity 页 / `wiki/entities/`」均按此理解为「`entities/`∪`concepts/`」。
这是 broken-link 笔记里观澜「最积极一档」的有限落地：检测确定性、生成才用 LLM、且**只对高频断链**动手。

参考实现是 llm-wiki-agent 的 `tools/heal.py`（频次分级 → 引用上下文召回 → LLM 建页），但它的实现方式
**几乎每条都踩在观澜刻意立的边界上**——脚本自带 `litellm` / 自管密钥、裸 `write_text` 无门禁、无幂等/
无校验。P3.2 保留它的**产品形态**，重做**实现路径**：检测走 P3 图层、生成走 P2 写门禁，**零脚本 LLM**。

**核心立场：检测确定性、生成有门禁、默认非自动、按需且有界。** 断链仍是 Karpathy 模式的生长信号
（决策8），heal **不塞进 ingest、不自动跑**；它是用户显式触发的「把已稳定的高频缺页一次性补齐」命令。

sources / aliases 三类阻断性违规）**全部原样复用**，这是 P3.2 仍能这么薄的根因。

范围外的能力（自动 heal、语义资料缺口、Web 宿主、规范标题页等）统一列在末节 **§11 后续工作**。

## 2. 检测与读取：heal worklist（确定性，零 LLM）

heal 的「补什么」完全是算术，**复用 P3 的图层**，不引入任何新解析口径：

```
build_graph(wiki).broken            # 未解析 [[…]] 边（与 check.wikilink.broken 同口径，决策P3-6）
  → 按 target 聚合引用它的**不同**页集合
  → 取引用数 ≥ MISSING_ENTITY_MIN_REFS 的目标       # lint.py:30，默认 2
  → 按 (引用数降序, target 升序) 稳定排序
  → 截到 --limit
```

- **worklist 与 `lint.missing_entity` 同源**（**\[决策P3.2-1]**）：同一份 `build_graph().broken`、同一个阈值
  `MISSING_ENTITY_MIN_REFS`（`lint.py:30`）、同一聚合逻辑（`lint.py:55-73` 的 `refs_by_target`）。默认 `--min-refs` 下 worklist
  与 `guanlan lint` 报的 `missing_entity` 集**逐项相等**；`--min-refs > 2` 时 worklist 是其**真子集**（更保守）、
  `--limit` 时执行批次又是其子集——但两命令对「哪些算高价值缺页」的判据永不分叉。
  **落地必须复用 lint 的聚合，不能筛 `run_lint` 的 `Finding`**：`missing_entity` 的 `Finding` 是
  `Finding("", "lint.missing_entity", "[[X]] 被 N 页引用…")`（`lint.py:67-73`）——page 为空串、target 只埋在
  **人读 message** 里、且**不带 `ref_pages`**，筛它既要从文案抠 target（撞决策P3.2-10）又拿不到引用页清单
  （决策P3.2-2 必需）。故应把缺失实体聚合抽成 `lint.py` 的一个**结构化访问器**（返回 `target` / `ref_count` /
  `ref_pages`），lint 与 heal **共享同一份代码**——「同口径同阈值、永不分叉」由代码强制，而非两处平行聚合自觉对齐。
- **`target` 即归一断链键**（**\[决策P3.2-14]**）：`graph.py:120` 在建边时已 `target = link_stem(raw)`
  （剥 `|别名`/`#锚点`/`.md`、取末段、`.lower()`），故 `build_graph().broken` 里**只有归一键、拿不回原始
  显示名**。worklist 的 `target` 因此**就是这个归一键**，新页文件名也按它生成——`[[原引用]]` 经同一
  `link_stem` 必落到该 stem，天然解析。**不为保留显示名去改 `Graph.Edge`**（那会污染 P3 图层契约）。
- **附带引用页清单**（**\[决策P3.2-2]**）：worklist 每项除 `target` 外，带上**引用它的页面路径列表**
  （从 `graph.broken` 的 `edge.source` 经 `node_path` 还原）。这是**确定性事实**（谁 `[[X]]` 了），不是
  intelligence，由 wrapper 算好喂给 skill，把 agent 锁在真实引用上下文里——对应 `heal.py` 的
  `search_sources`，但在零-LLM 读层做、不在脚本里截 800 字符。**读哪几页、怎么合成**留给 skill。
- **频次阈值压幻觉**：≥2 页引用才补（术语被复述过 = 真实体的最低信号），单次断链留作生长信号、不补。
  `--min-refs` 可上调（更保守，使 worklist 成 lint.missing_entity 真子集），但默认对齐 lint 的单一常量，
  使二者在默认阈值下逐项相等。
- **结构化 worklist 契约**（**\[决策P3.2-10]**）：heal 内部传结构化对象、`--json` 直接序列化它，
  **不从人读文案反推状态**；正确性判定全部来自重算 graph（§5），不依赖 agent 输出的任何结构。

```python
@dataclass(frozen=True)
class HealWorkItem:
    target: str              # 归一断链键（== graph Edge.target == pages.link_stem(raw)）；亦即新页 stem
    ref_count: int           # 不同引用页数量
    ref_pages: tuple[str, ...]  # 相对 root 的 wiki/... 路径，稳定排序
    postponed: bool = False  # 因 --limit 推迟；dry-run 报告用
```

`--limit` 返回的执行批次只含 `postponed=False` 项；预览 / dry-run 同时返回推迟项，避免消费侧误以为
本轮覆盖了全部缺失实体。

## 3. 写路径：复用 P2 单写者门禁（不另开门禁）

heal **是写操作**，必须走与 ingest 完全相同的一条写路径——这正是 broken-link 笔记 §6-1 强调的
「自动建页是写操作，不能塞进读路径」。落地即一行 `run_guarded_write_result`（§1 决策P3.2-13 的结果版）：

```
heal worklist（§2，零 LLM）
  → 空？→ 短路 EXIT_OK，**不触 Agentao**（零 LLM 成本）        # 决策P3.2-6
  → snapshot wiki/（写集审计基线，wrapper 层）                  # 决策P3.2-11
  → run_guarded_write_result(kb, HEAL_PROMPT, model):          # gate.py 加性重构，与 ingest 同编排核心
        snapshot raw/ → Agentao(workspace-write) + skill →
        enforce_write_result（raw 快照 + check + 增量门禁）→
        有界自愈（frontmatter/sources 违规回喂，≤2 轮）→
        返回 GuardedWriteResult(exit_code, final_text, gate)    # 不向 stdout 打印门禁报告（↻ 自愈进度仍走 stderr）
  → 写后确定性回执（§5，零 LLM；重算 graph + wiki/ 写集 diff，不依赖 agent 输出结构）
```

- **与 ingest 同构**（**\[决策P3.2-3]**）：heal 与 ingest 共用同一编排核心，唯一差别是 prompt 与「ingest 取
  退出码、heal 取整个结果版自渲染」；门禁、自愈、退出码全在结果版收口，**绝不复制一份门禁**。`raw/` 不可变
  （heal 只写 `wiki/`，碰 `raw/` 即 `raw_mutated` 判死）、frontmatter/sources 阻断性增量判定、断链仍作
  警告——一切既有契约自动继承。报告打印则由 wrapper 自己渲染回执（§5），不再走 `report_outcome`。
- **frontmatter 由门禁兜底**：`heal.py` 裸信任 LLM 输出的 frontmatter；观澜不。新建 entity 页若
  frontmatter 不合法或 `sources` slug 不存在，`check` 报阻断性违规 → 有界自愈回喂 `REPAIR_PROMPT` 就地修，
  仍不过则 `EXIT_CHECK_FAILED` 硬失败、改动留盘供人工修。**生成内容永远过同一道确定性闸**。
- **写集审计按「有害写」判，不按「文件名==target」判**（**\[决策P3.2-11]**）：门禁负责全局合法性，但 heal
  还要在 wrapper 层记录 `wiki/` before/after diff。`unexpected_writes` 只收**有害写**：
  - **修改**已有非目标页面、**删除**任何页面、**覆盖**已有页（含已有 entity）。

  **新建** `wiki/entities/`∪`wiki/concepts/` 下的页（按实体/概念分类；即便 stem≠target，如 agent 自选了更规范名
  `大语言模型.md`）与**追加** `wiki/log.md` 都**不算越界**——前者是新内容、不毁旧数据，是否真解析了 `[[target]]` 由 §5 重算 graph
  独立裁决（解析靠该页 stem 或其 `aliases` 收编 target）。这样写集约束就不会把一次成功的「规范名 + aliases」
  物化误报为非预期写。`unexpected_writes` 是**批次级**（§5，非逐目标 status）；实现上 P3.2 先**报告不改
  退出码**、不在 `gate.py` 新增裁决；若有害写同时引入 frontmatter/sources 阻断性违规，仍由 P2 门禁以
  `EXIT_CHECK_FAILED` 裁决。
- **Web 留待 P4.x**（见 §11）：单写者 FIFO（`web/jobs.py:43` 的 `JobQueue`）已是通用的
  `enqueue(kind, thunk)`，未来把 heal 作为新写 job kind 即可零新写基础设施接入；P3.2 首版不实现，
  故本节只锚定写路径本身。

## 4. prompt 与 skill 工作流

承 P2 纪律：wrapper 的 prompt **薄**，真正步骤在 skill。`HEAL_PROMPT` 列出本批目标名 + 各自的引用页清单，
指示 skill 走 `guanlan-wiki` 的 `heal` 工作流：

> 对下列缺失实体（已是归一键，可直接作文件名），**只读所列引用页**与 `wiki/index.md` 建上下文；若上下文
> 足以确认其为实体，则在 `wiki/entities/<目标>.md` 合成一页 entity 定义（frontmatter 齐全）；把正文里的
> 术语转 `[[wikilink]]`；向 `log.md` 追加一条 `## [<日期>] heal | <目标>`。
> 若上下文不足、目标明显是概念/主题页、或疑似已有页别名，**跳过该目标**（正常人读说明即可，无需特定格式）。
> **永不修改 `raw/`、不新建 `raw/` 资料、不臆造引用页里没有的事实、不运行 shell 或 `guanlan check`。**

- **命名天然收敛**：worklist 的 `target` 已是 `link_stem` 归一键（决策P3.2-14），文件名直接用它，则
  `[[原引用]]` 经同一 `link_stem` 必解析到该页——**无需 agent 自行对齐命名**。仅当 skill 想用更规范的
  标题当 stem（如把 `大模型` 建成 `大语言模型.md`）时，才需在 frontmatter `aliases` 收编原断链键，这是
  **可选优化**、不是正确性前提；无论哪种，§5 的回执都重算 graph **确定性核验**——不盲信。
- **「不臆造」**：只准从引用上下文合成，缺则宁可跳过或写桩页（`health` 会另行标记桩页），不得编造来源。
  `sources` 列若引用页有出处可顺延，否则留空（合法）——避免 `heal.py` 那种 frontmatter 失真。
- **跳过是 agent 的自由，但不强求机器信号**（**\[决策P3.2-12]**）：heal 是「物化高价值缺页」，不是强制
  把每条断链都变成页面。skill 可因上下文不足 / 非实体 / 疑似别名而跳过某目标，用**人读说明**交代即可。
  **不要求 agent 产出固定 JSON 状态块**：正确性只来自写后重算 graph（§5），跳过与「建了但没解析」在图上
  都表现为「目标仍断」，统一归 `still_broken`，下次 heal 仍会看到它（细分跳过原因见 §11）。
- **skill 是 intelligence 所在**：读哪段、如何归纳、转哪些 `[[link]]` 全在 skill 工作流，wrapper 只给
  「目标 + 引用页路径」这点确定性事实。符合「`guanlan/` 不携带业务智能」的总分工。

## 5. 写后确定性回执（heal 是否真修好了，零 LLM 可验证）

`heal.py` 写完即结束、不核验 LLM 到底建对没有。观澜把它变成确定性事实并报告（**\[决策P3.2-4]**）：
写门禁通过后**重算 `build_graph`**，对本批每个目标给出 status（**纯由图判定，与 agent 输出无关**）：

| 回执 | 含义 |
|------|------|
| `resolved` | 目标现已解析到某页（断链消除）——heal 对该目标成功 |
| `still_broken` | 目标仍在 `graph.broken`（agent 没建 / 命名不符 / 主动跳过，图上无从区分）——未达效 |

非预期写集是**批次级事实**（不是某个目标的状态），单列：

| 批次字段 | 含义 |
|------|------|
| `unexpected_writes: list[str]` | 本轮越界的 wiki 写路径（修改非目标页 / 删除页 / 覆盖已有页），列出供人工审计 |

报告形如「物化 N 页：M 个目标现已解析，K 个仍断；另有 U 处非预期写」。这把「LLM 是否真的修好了」从
「信它说的」变成「重算图一比即知」，是观澜相对 `heal.py` 的关键加固。回执**不改退出码**（门禁已裁决写入
合法性）；`still_broken` 是质量提示，下次 heal 会再纳入 worklist。

```python
@dataclass(frozen=True)
class HealReceipt:        # 逐目标
    target: str           # 归一键（== worklist.target）
    status: Literal["resolved", "still_broken"]  # v1 仅图判定二值；未来可扩 "skipped"（需 agent 信号/持久化）
    resolved_to: str | None      # wiki/... 路径；仅 resolved 必有
    created_path: str | None     # 本轮新建页；未知/未建则 None
    reason: str                  # 人读短因

@dataclass(frozen=True)
class HealResult:         # 批次级
    receipts: tuple[HealReceipt, ...]
    unexpected_writes: tuple[str, ...] = ()  # 越界写路径（批次级，非逐目标）
    changed_paths: tuple[str, ...] = ()      # 本轮 wiki/ 写集 diff 全量
    exit_code: int = 0
```

`HealResult` 是 CLI（及未来 Web）共用的机器契约；人读输出只是它的渲染。状态优先级：P2 门禁退出码裁决
整体成败；`build_graph` 逐目标裁决 `resolved` / `still_broken`；wiki/ 写集 diff 给出 `unexpected_writes`。
**全程不解析 agent 输出**。若门禁失败（`raw_mutated` / `check_failed` / `agent_error`），仍尽量输出已能
确定的 `changed_paths`，但最终裁决以退出码为准。

## 6. CLI 契约

```bash
guanlan -C <库> heal                 # 物化 lint.missing_entity（默认上限 --limit 条），走写门禁
guanlan -C <库> heal --dry-run       # 仅打印 worklist（纯读、零 LLM、必 EXIT_OK），不写不触 Agentao
guanlan -C <库> heal --limit 5       # 本批最多物化 5 个（按引用频次降序取高价值者）
guanlan -C <库> heal --min-refs 3    # 提高阈值（更保守）；默认对齐 lint MISSING_ENTITY_MIN_REFS=2
guanlan -C <库> heal --model <m>     # 覆盖 Agentao 模型（同 ingest/query）
guanlan -C <库> heal --json          # 输出 worklist/receipt 的结构化 JSON（人读输出仍为默认）
```

- **退出码复用写路径四值**（**\[决策P3.2-5]**），**不新增退出码**（同 P4）：
  `EXIT_OK` / `EXIT_RAW_MUTATED` / `EXIT_CHECK_FAILED` / `EXIT_AGENT_ERROR`，全由 `run_guarded_write_result`
  给出（与 `run_guarded_write` 同值，仅多回传结构化结果）。
- **空 worklist 短路 `EXIT_OK`**（**\[决策P3.2-6]**）：无缺失实体（图已充分连通）→ 打印「无可物化的缺失实体」
  并直接返回，**不启动 Agentao**——零 LLM 成本，对应 `heal.py` 的「Graph is fully connected」。
- **`--dry-run` 纯读**：复用 §2 worklist 计算，打印「将物化哪些目标、各被几页引用、引用页是谁、本次因
  `--limit` 推迟了哪些」，**绝不写、必 `EXIT_OK`**。让用户在付 LLM 成本前预览。
- **`--limit` 不静默截断**（**\[决策P3.2-1]**）：超额目标在报告里显式列为「本次推迟（重跑续补）」，
  不假装「全部补齐」——遵 DESIGN 的「有界即声明」原则。
- **`--json` 输出稳定契约**：dry-run 输出 `{worklist, postponed}`；真实 heal 输出
  `{worklist, postponed, receipts, unexpected_writes, changed_paths, exit_code}`（`changed_paths` = 本轮
  wiki/ 写集全量，供 CI / `check` 重跑重建审计轨迹）。JSON 面向自动化，字段来自 §2 / §5 dataclass，
  不从 stdout 文本切片（靠 `run_guarded_write_result` 的非打印结果版隔离门禁人读报告）。

（Web 契约不在本相位，见 §11。）

## 7. 幂等与不破坏性

- **天然幂等、目标上绝不覆盖**（**\[决策P3.2-7]**）：worklist 只含**当前断链**目标（已有页者其 `[[X]]`
  已解析、不在 `graph.broken`），故已物化的实体**自然不再入选**；prompt 与写集审计都要求不得覆盖现页
  （同 `init` 的非破坏性）。全部补齐后重跑 → 空 worklist → 短路 EXIT_OK。
- **多次 heal 收敛**：每轮补一批（受 `--limit` 限），断链单调减少；命名不符的 `still_broken`（§5）下轮再试。
- **半成品可审计**：Agentao 写文件不是原子批事务；若进程中断或自愈失败，wiki/ 半成品会留盘（同 P2 写门禁）。
  heal 不自动回滚，而是通过写集 diff / receipt 报出本轮触及路径。下次重跑时，已有页若让 `[[X]]` 解析则自然
  退出 worklist；若仍不解析则继续以 `still_broken` 进入 worklist，供下一轮或人工修。
- **与生长信号不冲突**：低频断链（<阈值）从不入 worklist，仍作前向引用随 ingest 自然消除——heal 只收割
  **已稳定的高频**缺页，不扼杀生长。

- **\[决策P3.2-5]** 退出码复用写路径四值，**不新增退出码**（同 P4）；`--dry-run` 纯读、零 LLM、必 EXIT_OK。

- **\[决策P3.2-9]**（**受控偏离 backlog**）heal 默认**显式按需、非自动**：不内联进 ingest、不做定时富化

- **\[决策P3.2-10]** worklist / `HealResult` 是结构化机器契约，`--json` 直接序列化；人读输出只是渲染。
  **正确性判定全程不解析 agent 输出**——不要求 agent 产出 `HEAL_STATUS` 之类固定状态块。
- **\[决策P3.2-11]** heal 在 P2 门禁之外增加写集审计：允许新建目标 entity 页与追加 `log.md`，非预期 wiki
  写入收进**批次级** `unexpected_writes`（非逐目标 status），P3.2 先报告不新增退出码、**不在 `gate.py`
  新增裁决逻辑**（写集 diff 在 wrapper 层算，与决策P3.2-13 的结果版重构无关）。
- **\[决策P3.2-12]** heal 可跳过单个目标（上下文不足 / 非 entity / 疑似别名），用人读说明交代即可；**不
  要求机器跳过信号**——跳过与「建了但没解析」在图上都表现为 `still_broken`，下次 heal 仍会看到它。
  `HealReceipt.status` 留作可扩展，细分跳过原因见 §11。
- **\[决策P3.2-13]**（**P3.2 唯一的 `gate.py` 改动**）`run_guarded_write` 抽出**不向 stdout 打印门禁报告、
  返回结构化结果**的 `run_guarded_write_result(...) -> GuardedWriteResult(exit_code, final_text, gate)`；
  原函数退化为「调结果版 → `report_outcome` 打印 → 返回 exit_code」的薄壳，**ingest 路径行为/字节不变**。
  「不打印」**仅约束 stdout 的门禁结论报告**（`report_outcome`）——自愈进度行 `↻ 门禁未过…`（`gate.py:321`）
  本就走 stderr、属编排核心、照常保留，不影响 `--json` 的 stdout 洁净。理由：heal 要在 `--json` 下产出不被
  门禁报告污染的结构化回执、并自渲染人读报告，而原函数只回传 `int` 且把 `final_text` 直接 print 到 stdout。
  这是加性重构，门禁判定/自愈/`REPAIR_PROMPT` 全部原样复用。
- **\[决策P3.2-14]** worklist/receipt 的 `target` **即归一断链键**（`pages.link_stem`）：因 `graph.py:120`
  建边时已归一、`build_graph().broken` 拿不回原始显示名。新页文件名按此键生成，`[[原引用]]` 经同一
  `link_stem` 天然解析；**不为保留显示名改 `Graph.Edge`**（避免污染 P3 图层契约）。

## 9. 测试要点

- `gate.py` 加性重构（决策P3.2-13，零 LLM 回归）：
  - `run_guarded_write` 经现有 ingest 测试**字节级不回归**（薄壳输出与旧实现一致）。
  - `run_guarded_write_result` 在假 runner 下回传 `exit_code` / `final_text` / `gate`，且**stdout 无门禁报告**
    （`report_outcome` 不触发）；触发自愈时 `↻` 进度仍可见于 stderr（断言 stdout 干净、不约束 stderr）。
- `heal` worklist（零 LLM，可不调 LLM 单测）：
  - 默认 `--min-refs` 下 worklist 与 `lint.run_lint` 的 `missing_entity` 集逐项相等（同源回归）；
    且两者复用同一聚合访问器（构造断链场景断言二者目标集合一致）。
  - 引用页清单 == 实际 `[[target]]` 的来源页集合；按 (频次降序, 名升序) 稳定排序。
  - 低于阈值的断链不入选；`--min-refs` 上调后相应目标退出 worklist（worklist 成 `missing_entity` 真子集）。
  - 空 worklist（无缺失实体）→ 短路 EXIT_OK、**未调用 runner**（注入假 runner 断言零调用）。
  - `--limit` 截断：返回前 N，且推迟项被记录（非静默丢弃）。
  - `--json` 输出结构化 `worklist` / `postponed`，不依赖人读文案。
- `heal` 写路径（注入假 `AgentRunner`，仿 P2 测试）：
  - 假 runner 把目标 `target` 建成 `entities/<target>.md`（合法 frontmatter）→ 门禁过、EXIT_OK；回执报
    `resolved` 且 `resolved_to` 指向该页（验证「文件名=归一键 → `[[原引用]]` 天然解析」，决策P3.2-14）。
  - 假 runner 改 `raw/` → `EXIT_RAW_MUTATED`。
  - 假 runner 写坏 frontmatter → 触发有界自愈；修好 → OK，修不好 → `EXIT_CHECK_FAILED` 且改动留盘。
  - 假 runner 建了页但文件名归一**不**等于目标键、又**未**用 `aliases` 收编（或干脆没建/主动跳过）→
    门禁过（断链是警告）但回执统一报 `still_broken`（图无从区分跳过与未达效，决策P3.2-12）。
  - 假 runner 建**新**的规范标题页（stem≠target）并在 `aliases` 收编原断链键 → 回执报 `resolved`，
    且该新页**不**进 `unexpected_writes`（锁定决策P3.2-11 的「有害写」口径：新建页不算越界）。v1 prompt 不要求
    走这条路，但审计须容忍它。
  - 假 runner **修改**非目标已有页 / **删除**页 / **覆盖**已有页 → 批次级 `unexpected_writes` 含这些路径，
    且**不**污染任何目标的 `status`（仍各按 graph 判定）。
  - 回执判定**与 agent 文本无关**：runner 的 `final_text` 任意（含/不含任何状态块）→ status 只随 graph 变。
  - `--dry-run` → 不调 runner、不写盘、EXIT_OK、打印 worklist。
- 幂等：物化后重跑 heal → worklist 空、EXIT_OK、现页未被覆盖（mtime/内容不变）。
- 半成品：runner 中途失败或自愈耗尽 → 退出码按 P2 门禁，`HealResult` 尽量列出 `changed_paths`，重跑不覆盖现页。
- （Web-heal 测试随 Web 宿主一并后移 P4.x，见 §11，P3.2 不含。）

## 10. 风险

- **命名不符导致空补**：worklist 给的就是归一键、文件名同源，本不该不符；但若 LLM 擅自改名（如建成
  `大语言模型.md` 却未在 `aliases` 收编 `大模型`），断链未消。§5 回执确定性识别并报 `still_broken`，
  **修复依赖下一轮 heal 或人工**。文件名直接用归一键（§4）已把此风险降到最小。
- **批量 LLM 成本 / 质量**：一次 guarded write 内建多页，目标多则上下文大、质量降。`--limit` 是唯一闸门；
  默认值需保守（建议起步 ≤10），并在报告里始终显示「推迟了多少」。
- **桩页泛滥**：引用上下文太薄时合成的 entity 页接近空壳。skill 可径直跳过（v1 表现为 `still_broken`，
  下次仍会重试——这是已知折中，见决策P3.2-12）；已写出的薄页交由 `health` 的桩页检测兜底（heal 不自评
  质量），避免 heal 既当运动员又当裁判。
- **不可愈目标反复重试**：因不持久化跳过决策，真·非实体或永远上下文不足的目标每轮都重入 worklist、
  重付 LLM 成本。当前缓解：`--limit` 限批 + `--min-refs` 收紧 + dry-run 预览；根治见 §11。
- **非预期 wiki 写入**：P2 门禁不阻止合法但越界的 wiki 改动；P3.2 用写集 diff 报批次级 `unexpected_writes`，
  暂不新增硬失败码，避免与 `gate.py` 分叉。若实测误写频繁，再考虑把写集约束升级为阻断门禁。
- **与 ingest 写并发**：CLI 下两个写命令同时跑无锁保护（P2 既有限制，非 P3.2 引入）；Web 下由单 worker FIFO
  天然串行化。并发导致 worklist 在运行期间变化时，以写后 graph receipt 为准；文档化「写命令请勿并发」
  即可，不在 P3.2 引入跨进程锁。
