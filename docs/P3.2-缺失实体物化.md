> 「`heal` 类工具（参考 llm-wiki-agent `heal.py`）」的**实现级细化**，是 P3.2 编码的落地依据。
> 高层理念与原则以 `DESIGN.md` 为准。文中标注的 **\[决策P3.2-N]** 是对 `DESIGN.md` / backlog /

## 1. 目标与范围

**P3.2 = 把 `lint.missing_entity`（被 ≥2 页引用却无页的高价值断链）按需用 LLM 物化成 entity 页。**
这是 broken-link 笔记里观澜「最积极一档」的有限落地：检测确定性、生成才用 LLM、且**只对高频断链**动手。

参考实现是 llm-wiki-agent 的 `tools/heal.py`（频次分级 → 引用上下文召回 → LLM 建页），但它的实现方式
**几乎每条都踩在观澜刻意立的边界上**——脚本自带 `litellm` / 自管密钥、裸 `write_text` 无门禁、无幂等/
无校验。P3.2 保留它的**产品形态**，重做**实现路径**：检测走 P3 图层、生成走 P2 写门禁，**零脚本 LLM**。

**核心立场：检测确定性、生成有门禁、默认非自动、按需且有界。** 断链仍是 Karpathy 模式的生长信号
（决策8），heal **不塞进 ingest、不自动跑**；它是用户显式触发的「把已稳定的高频缺页一次性补齐」命令。

### 纳入 P3.2

| 改动 | 文件 | 性质 |
|------|------|------|
| heal worklist 计算（复用 `build_graph().broken`，与 `lint.missing_entity` 同口径同阈值） | `guanlan/heal.py`（新） | 零 LLM（读） |
| 薄 prompt + 经 `run_guarded_write` 走 P2 单写者门禁 | `guanlan/heal.py`（新） | LLM，写门禁 |
| 写后**确定性回执**（重算 graph 核对哪些目标已解析 / 仍断） | `guanlan/heal.py`（新） | 零 LLM（读） |
| `guanlan heal`（`--limit` / `--min-refs` / `--dry-run` / `--model`） | `guanlan/cli.py` | 编排 |
| `heal` 工作流（读引用页 → 合成 entity 页 → 联链 → log） | `skills/guanlan-wiki/SKILL.md` | 约定 |
| 引用上下文建页的命名/frontmatter 纪律 | `skills/guanlan-wiki/references/conventions.md` | 约定 |
| Web `POST /api/heal` 走**同一**单写者 FIFO 队列 + dry-run 只读预览 | `guanlan/web/app.py` | 复用，零新写基础设施 |

**`gate.py` 不改**：heal 与 ingest 共用 `run_guarded_write`（raw 快照 + check + 有界自愈），连
`REPAIR_PROMPT` 都复用（它已覆盖 frontmatter / sources / aliases 三类阻断性违规）。这是 P3.2 能这么薄的根因。

### 不纳入 P3.2（仍按 §8 / backlog 推后，需实证驱动）

- **自动 heal**（ingest 内联补页、nightly 定时富化）——属 DESIGN §7 E3「自动富化」，P5 之后；P3.2 只做**显式按需**。
- **语义资料缺口**（需引入**新** `raw/` 资料才能写的页）——heal 只物化**能从现有引用上下文合成**的实体，不抓新源。
- **concept / synthesis 自愈**——`missing_entity` 目标是实体性名词，P3.2 只建 `entities/`；其余页型留给 ingest/query。
- **向量检索先消解同义假断链**再物化——CJK 笔记第 3 条，heal 的上游降噪，另算。

## 2. 检测与读取：heal worklist（确定性，零 LLM）

heal 的「补什么」完全是算术，**复用 P3 的图层**，不引入任何新解析口径：

```
build_graph(wiki).broken            # 未解析 [[…]] 边（与 check.wikilink.broken 同口径，决策P3-6）
  → 按 target 聚合引用它的**不同**页集合
  → 取引用数 ≥ MISSING_ENTITY_MIN_REFS 的目标       # lint.py:30，默认 2
  → 按 (引用数降序, target 升序) 稳定排序
  → 截到 --limit
```

- **worklist ≡ `lint.missing_entity` 集**（**\[决策P3.2-1]**）：同一份 `build_graph().broken`、同一个阈值
  `MISSING_ENTITY_MIN_REFS`（`lint.py:30`）、同一聚合逻辑（`lint.py:55-67`）。故 `guanlan lint` 报的每条
  `missing_entity`，正是 heal 会处理的目标——两命令对「哪些是高价值缺页」永不分叉。落地上 heal 直接
  调 `lint.run_lint` 并筛 `kind == "lint.missing_entity"`，或复用其内部聚合（择一，避免逻辑二写）。
- **附带引用页清单**（**\[决策P3.2-2]**）：worklist 每项除目标名外，带上**引用它的页面路径列表**
  （从 `graph.broken` 的 `edge.source` 经 `node_path` 还原）。这是**确定性事实**（谁 `[[X]]` 了），不是
  intelligence，由 wrapper 算好喂给 skill，把 agent 锁在真实引用上下文里——对应 `heal.py` 的
  `search_sources`，但在零-LLM 读层做、不在脚本里截 800 字符。**读哪几页、怎么合成**留给 skill。
- **频次阈值压幻觉**：≥2 页引用才补（术语被复述过 = 真实体的最低信号），单次断链留作生长信号、不补。
  `--min-refs` 可上调（更保守），但默认对齐 lint 的单一常量，保持 worklist ≡ lint.missing_entity。

## 3. 写路径：复用 P2 单写者门禁（不另开门禁）

heal **是写操作**，必须走与 ingest 完全相同的一条写路径——这正是 broken-link 笔记 §6-1 强调的
「自动建页是写操作，不能塞进读路径」。落地即一行 `run_guarded_write`：

```
heal worklist（§2，零 LLM）
  → 空？→ 短路 EXIT_OK，**不触 Agentao**（零 LLM 成本）        # 决策P3.2-6
  → run_guarded_write(kb, HEAL_PROMPT, model):                  # gate.py:285，与 ingest 同函数
        snapshot raw/ → Agentao(workspace-write) + skill →
        enforce_write_result（raw 快照 + check + 增量门禁）→
        有界自愈（frontmatter/sources 违规回喂，≤2 轮）→ 退出码
  → 写后确定性回执（§5，零 LLM）
```

- **与 ingest 同构**（**\[决策P3.2-3]**）：heal 与 ingest 唯一差别是 prompt；门禁、自愈、报告、退出码
  全在 `run_guarded_write` 收口，**绝不复制一份门禁**。`raw/` 不可变（heal 只写 `wiki/`，碰 `raw/` 即
  `raw_mutated` 判死）、frontmatter/sources 阻断性增量判定、断链仍作警告——一切既有契约自动继承。
- **frontmatter 由门禁兜底**：`heal.py` 裸信任 LLM 输出的 frontmatter；观澜不。新建 entity 页若
  frontmatter 不合法或 `sources` slug 不存在，`check` 报阻断性违规 → 有界自愈回喂 `REPAIR_PROMPT` 就地修，
  仍不过则 `EXIT_CHECK_FAILED` 硬失败、改动留盘供人工修。**生成内容永远过同一道确定性闸**。
- **Web 复用单写者 FIFO**（**\[决策P3.2-3]**）：P4 的 `JobQueue` 是通用的 `enqueue(kind, thunk)`
  （`web/app.py:136/223`）。heal 作为**第二个写 job kind** 即时入队、与 ingest **同一个单 worker 串行**
  （单写者不变式不破），前端轮询 `/api/jobs/{id}`。**零新写基础设施**，不新增 SSE、不新增退出码。

## 4. prompt 与 skill 工作流

承 P2 纪律：wrapper 的 prompt **薄**，真正步骤在 skill。`HEAL_PROMPT` 列出本批目标名 + 各自的引用页清单，
指示 skill 走 `guanlan-wiki` 的 `heal` 工作流：

> 对下列每个缺失实体，**只读所列引用页**与 `wiki/index.md` 建上下文，在 `wiki/entities/<名>.md` 合成一页
> entity 定义（frontmatter 齐全；文件名须使 `[[原引用]]` 解析得到该页——见 `pages.link_stem` 归一口径）；
> 把正文里的术语转 `[[wikilink]]`；向 `log.md` 追加一条 `## [<日期>] heal | <实体名>`。
> **永不修改 `raw/`、不新建 `raw/` 资料、不臆造引用页里没有的事实、不运行 shell 或 `guanlan check`。**

- **命名收敛是正确性关键**：新页文件名经 `pages.link_stem` 归一后须 == 原断链目标的归一键，否则
  `[[X]]` 写完仍断。这点 prompt 强约束、conventions 记纪律，并由 §5 的回执**确定性核验**——不盲信。
- **「不臆造」**：只准从引用上下文合成，缺则宁可写桩页（`health` 会另行标记桩页），不得编造来源。
  `sources` 列若引用页有出处可顺延，否则留空（合法）——避免 `heal.py` 那种 frontmatter 失真。
- **skill 是 intelligence 所在**：读哪段、如何归纳、转哪些 `[[link]]` 全在 skill 工作流，wrapper 只给
  「目标 + 引用页路径」这点确定性事实。符合「`guanlan/` 不携带业务智能」的总分工。

## 5. 写后确定性回执（heal 是否真修好了，零 LLM 可验证）

`heal.py` 写完即结束、不核验 LLM 到底建对没有。观澜把它变成确定性事实并报告（**\[决策P3.2-4]**）：

heal 写门禁通过后，**重算 `build_graph`**，对本批每个目标确定性判定：

| 回执 | 含义 |
|------|------|
| `resolved` | 目标现已解析到某页（命名正确，断链消除）——heal 成功 |
| `still_broken` | 目标仍在 `graph.broken`（agent 没建 / 文件名归一不符）——heal 名义成功但未达效 |

报告形如「物化 N 页：M 个目标现已解析，K 个仍断（命名不符或未建，可重跑或人工修）」。这把
「LLM 是否真的修好了」从「信它说的」变成「重算图一比即知」，是观澜相对 `heal.py` 的关键加固。
回执**不改退出码**（门禁已裁决写入合法性）；`still_broken` 是质量提示，下次 heal 会再纳入 worklist。

## 6. CLI 契约

```bash
guanlan -C <库> heal                 # 物化 lint.missing_entity（默认上限 --limit 条），走写门禁
guanlan -C <库> heal --dry-run       # 仅打印 worklist（纯读、零 LLM、必 EXIT_OK），不写不触 Agentao
guanlan -C <库> heal --limit 5       # 本批最多物化 5 个（按引用频次降序取高价值者）
guanlan -C <库> heal --min-refs 3    # 提高阈值（更保守）；默认对齐 lint MISSING_ENTITY_MIN_REFS=2
guanlan -C <库> heal --model <m>     # 覆盖 Agentao 模型（同 ingest/query）
```

- **退出码复用写路径四值**（**\[决策P3.2-5]**），**不新增退出码**（同 P4）：
  `EXIT_OK` / `EXIT_RAW_MUTATED` / `EXIT_CHECK_FAILED` / `EXIT_AGENT_ERROR`，全由 `run_guarded_write` 给出。
- **空 worklist 短路 `EXIT_OK`**（**\[决策P3.2-6]**）：无缺失实体（图已充分连通）→ 打印「无可物化的缺失实体」
  并直接返回，**不启动 Agentao**——零 LLM 成本，对应 `heal.py` 的「Graph is fully connected」。
- **`--dry-run` 纯读**：复用 §2 worklist 计算，打印「将物化哪些目标、各被几页引用、引用页是谁、本次因
  `--limit` 推迟了哪些」，**绝不写、必 `EXIT_OK`**。让用户在付 LLM 成本前预览。
- **`--limit` 不静默截断**（**\[决策P3.2-1]**）：超额目标在报告里显式列为「本次推迟（重跑续补）」，
  不假装「全部补齐」——遵 DESIGN 的「有界即声明」原则。

## 7. 幂等与不破坏性

- **天然幂等、绝不覆盖**（**\[决策P3.2-7]**）：worklist 只含**当前断链**目标（已有页者其 `[[X]]` 已解析、
  不在 `graph.broken`），故已物化的实体**自然不再入选**，heal 永不覆盖现页（同 `init` 的非破坏性）。
  全部补齐后重跑 → 空 worklist → 短路 EXIT_OK。
- **多次 heal 收敛**：每轮补一批（受 `--limit` 限），断链单调减少；命名不符的 `still_broken`（§5）下轮再试。
- **与生长信号不冲突**：低频断链（<阈值）从不入 worklist，仍作前向引用随 ingest 自然消除——heal 只收割
  **已稳定的高频**缺页，不扼杀生长。

## 8. 决策记录

- **\[决策P3.2-1]** heal worklist ≡ `lint.missing_entity` 集：同一 `build_graph().broken`、同一阈值

- **\[决策P3.2-5]** 退出码复用写路径四值，**不新增退出码**（同 P4）；`--dry-run` 纯读、零 LLM、必 EXIT_OK。
- **\[决策P3.2-6]** 空 worklist 短路 EXIT_OK、**不启动 Agentao**（零 LLM 成本），对应 `heal.py` 的

- **\[决策P3.2-7]** 天然幂等不覆盖：worklist 只含当前断链目标，已物化实体自然跳过，heal 绝不覆盖现页；

- **\[决策P3.2-8]**（**受控偏离 `heal.py`**）生成 frontmatter 不被信任：新页一律过 `check`，违规则有界自愈

- **\[决策P3.2-9]**（**受控偏离 backlog**）heal 默认**显式按需、非自动**：不内联进 ingest、不做定时富化
  （那属 §7 E3 自动富化，P5 之后）；只物化能从**现有**引用上下文合成的实体，不抓新 `raw/` 资料。

## 9. 测试要点

- `heal` worklist（零 LLM，可不调 LLM 单测）：
  - worklist 与 `lint.run_lint` 的 `missing_entity` 集逐项一致（同口径回归）。
  - 引用页清单 == 实际 `[[target]]` 的来源页集合；按 (频次降序, 名升序) 稳定排序。
  - 低于阈值的断链不入选；`--min-refs` 上调后相应目标退出 worklist。
  - 空 worklist（无缺失实体）→ 短路 EXIT_OK、**未调用 runner**（注入假 runner 断言零调用）。
  - `--limit` 截断：返回前 N，且推迟项被记录（非静默丢弃）。
- `heal` 写路径（注入假 `AgentRunner`，仿 P2 测试）：
  - 假 runner 写出合法 entity 页 → 门禁过、EXIT_OK；回执报 `resolved`。
  - 假 runner 改 `raw/` → `EXIT_RAW_MUTATED`。
  - 假 runner 写坏 frontmatter → 触发有界自愈；修好 → OK，修不好 → `EXIT_CHECK_FAILED` 且改动留盘。
  - 假 runner 建了页但文件名归一不符 → 门禁过（断链是警告）但回执报 `still_broken`。
  - `--dry-run` → 不调 runner、不写盘、EXIT_OK、打印 worklist。
- 幂等：物化后重跑 heal → worklist 空、EXIT_OK、现页未被覆盖（mtime/内容不变）。
- `web`（有 `web` extra 时）：`POST /api/heal` 入队返回 job_id；与 ingest 共用单 worker 串行（不并发写）；
  dry-run 预览为只读端点不入写队列。

## 10. 风险

- **命名归一不符导致空补**：LLM 把 `[[大模型]]` 建成 `entities/大语言模型.md`，断链未消。§5 回执确定性识别
  并报 `still_broken`，但**修复仍依赖下一轮 heal 或人工**；可与 P3.1 别名联动（建页时声明 `aliases` 收编同义名）。
- **批量 LLM 成本 / 质量**：一次 guarded write 内建多页，目标多则上下文大、质量降。`--limit` 是唯一闸门；
  默认值需保守（建议起步 ≤10），并在报告里始终显示「推迟了多少」。
- **桩页泛滥**：引用上下文太薄时合成的 entity 页接近空壳。交由 `health` 的桩页检测兜底（heal 不自评质量），
  避免 heal 既当运动员又当裁判。
- **与 ingest 写并发**：CLI 下两个写命令同时跑无锁保护（P2 既有限制，非 P3.2 引入）；Web 下由单 worker FIFO
  天然串行化。文档化「写命令请勿并发」即可，不在 P3.2 引入跨进程锁。
