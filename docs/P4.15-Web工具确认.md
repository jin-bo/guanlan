## 1. 目标与范围

**P4.15 = 让嵌入式可写 Web 会话的 ASK 决策与模型 `ask_user` 提问从「transport 默默放行/返回哨兵」变成「经 SSE 弹给用户、阻塞等应答、回传运行时」；以超时+取消+默认拒绝兜住「无人应答」，全程不破 P4.5 的层①②③与写锁不变量，且明确「人点允许 ≠ 绕过 immutable 硬墙」。**

### 1.1 现状（已核对源码，这是出发点）

嵌入式会话在 `conversation.py:114` 构造 transport 时**只接了 `llm_text_callback`**：

```python
transport=build_compat_transport(llm_text_callback=self._on_token),
```

故 `confirmation_callback` / `ask_user_callback` 都是 `None`：

- `SdkTransport.confirm_tool` 走默认 `return True`（`agentao/transport/sdk.py:104`）→ **ASK 决策被静默自动放行**；
- `ask_user` 返回哨兵串 `"[ask_user: not available …]"`（同文件）→ 模型拿到无效答复、自行决断。

而 agentao 运行时**确实会调** `confirm_tool`——`runtime/tool_runner.py:210-231` 的 **Phase 2: Confirmation（sequential, interactive）**：

```python
for _plan in _plans:
    if _plan.decision == ToolCallDecision.ASK:                 # ASK → "prompt"（tool_runner.py:34）
        self._transport.emit(AgentEvent(EventType.TOOL_CONFIRMATION, {"tool": _fn, "args": ...}))
        _confirmed = self._transport.confirm_tool(_fn, _plan.tool.description, _plan.function_args)  # ← 同步阻塞
        _plan.decision = ToolCallDecision.ALLOW if _confirmed else ToolCallDecision.CANCELLED
```

`arun`（`agent.py:1092`）自陈 "bridges through `run_in_executor`"（1129/1133 行 `loop.run_in_executor`）。**三个对落地至关重要的事实**：

1. **同步阻塞**：`confirm_tool` / `ask_user` 的返回值决定后续，运行时**就地等**——正是「阻塞等用户」所需契约（`transport/base.py` 明示 confirm/ask 是 "Request-response interactions (blocking)"）。
2. **在 executor 线程**：Phase 2（及 `ask_user`）跑在 `arun` 的 `run_in_executor` 内（`conversation.py` 的 `_on_token` 注释「在 arun 的 executor 线程里跑」同此线程）→ **阻塞的是线程池线程、不是事件循环**（能用、且必须正确桥接的根因）。
3. **顺序、一次一个（confirm）**：Phase 2 是顺序 for 循环，一个 ASK 没确认完不会问下一个 → 同一会话同一时刻至多一个待确认。`ask_user` 发生在工具执行期（Phase 3）、与 confirm 不同期，故 pending 槽按 **interaction_id 键控**以兼容二者（§3）。

> 子进程路径（`ingest`/`query`）不在本相位：它用 `agentao run --interaction-policy reject` → `NonInteractiveTransport`，ASK **当场 abort（退出码 5）**，因为子进程背后无人在终端（见 [`agentao-shell权限与permissions.json.md`](agentao-shell权限与permissions.json.md)）。那条路靠 `~/.agentao/permissions.json` 预放行，**与 UI 无关**。P4.15 只动**进程内嵌入**的 Web chat。

### 1.2 纳入 P4.15

- **`confirm_tool` 接 UI**：构造期给 `build_compat_transport` 补 `confirmation_callback`；回调在 executor 线程把待确认经 SSE 推前端、阻塞等应答、回传布尔（§3/§4）。
- **`ask_user` 接 UI**（本期纳入，\[决策P4.15-11]）：补 `ask_user_callback`；模型提问经 SSE 弹一帧（可带 `options`/`multiple`/`allow_custom`），收回用户答案串。与 confirm 同构（executor 阻塞 + SSE + 端点回传），但返回 **string**、且 **read-only 也会触发**（提问非写，§9）。
- **新端点**：`POST /api/chat/{id}/confirm {interaction_id, decision}`（布尔，`decision` 三值）与 `POST /api/chat/{id}/answer {interaction_id, answer}`（字符串）→ 解阻塞；`POST /api/chat/{id}/confirm-mode {confirm_mode}` 翻本会话「问/不问」（§5）。
- **confirm 气泡三选项 + 安全版「本会话起自动放行」**（\[决策P4.15-13]）：允许 / **本会话起自动放行（仍 workspace-write + 层①②③，非 full-access）** / 拒绝；对标 CLI 三选项但**剔除其 full-access 翻转**（§5.1/§6）。
- **超时 + 取消 + 默认拒绝**：等待须同时观测**墙钟超时**与**停止令牌**，任一触发即返回拒绝（confirm→`False`；ask→约定的取消串），杜绝「人走开 → 写锁永占」（§4.2/§4.3）。
- **触发集钉死 + 与层①②③关系**：明确哪些决策走 confirm、确认通过**不**放宽 immutable（§2/§6）。
- **`--confirm` 进程开关**：`guanlan web --confirm {ask,auto}`，定 ASK 是「弹给人」还是「沿用 P4.5 静默放行」；默认 `ask`（\[决策P4.15-7]）。`--confirm-timeout`（默认 120s）调超时。
- **前端**：确认气泡（工具名 + 命令/参数 + 允许/拒绝/超时倒计时）+ 提问气泡（问题 + 选项/自由文本）+ `/tools` 文案随姿态。

### 1.3 不纳入 P4.15（推后）

- **逐次「记住此命令/会话内放行」白名单**：会把「人在环」退化成新的静默放行面，须单独评估其威胁模型，先不做（§9）。
- **多用户/远程确认**：Web 自 P4 起 `127.0.0.1`-only 单用户，不变。
- **改 `--interaction-policy` 子进程路径**：子进程无人值守，不接 UI（§1.1 注）。

## 2. 触发集：到底哪些调用会弹确认（钉死）

**唯一进入 `confirm_tool` 的是 `ToolCallDecision.ASK`。** 据 agentao 权限求值（`permissions.py` 预设 + `tool_runner` 两点姿态），各姿态下的去向：

| 姿态 | 写工具（`write_file`/`replace` 到可写区） | 写到 immutable（`raw/`、`AGENTAO.md`） | 只读 shell（无操作符） | 带操作符/管道 shell | 带 `requires_confirmation` 的工具 | `ask_user`（模型提问） |
|------|------|------|------|------|------|------|
| **read-only** | DENY（`tool_runner.readonly_mode`） | DENY | DENY | DENY | DENY | **会触发**（非写、不持写锁，§9） |
| **workspace-write** | **ALLOW（自动）** | **层①拒**（`PolicyFileSystem.write_text` 抛 `PermissionError`，§6） | ALLOW（预设白名单） | **ASK → confirm_tool ✅** | **ASK → confirm_tool ✅** | **会触发**（持写锁，§9） |

读出三条要点（\[决策P4.15-1]）：

1. **confirm 的主战场是 workspace-write 的 shell**：带管道/重定向/`$()`/`&&` 的命令落到预设兜底 `{"tool":"run_shell_command","action":"ask"}`（见 [`agentao-shell权限与permissions.json.md`](agentao-shell权限与permissions.json.md) 对该兜底的剖析）→ ASK → confirm。**这正是「跑 shell 需确认」那一刻。**
2. **写可写区不弹确认**：`wiki/`/`workspace/` 的结构化写在 workspace-write 是 ALLOW（自动）——P4.5 已用「写后 check + 撤销本轮」兜其质量，不在 confirm 范围（否则 agent 每写一页都打断人，UX 崩）。
3. **写 immutable 不靠 confirm 拦、而靠层①**：agent 结构化写 `raw/`/`AGENTAO.md` 不是 ASK、是层① wrapper 在 `write_text` 处直接 `PermissionError`（§6）——**confirm 通过与否都不放宽它**。这是本相位安全论证的支点（§6）。

> 若装了 `~/.agentao/permissions.json`（如 KB 主为放行只读管道 shell 加的那份），它**先于预设求值**（first-match-wins）→ 被它 allow 的命令**不进 ASK、也就不弹确认**。即 confirm 触发集 = 「过了 hardline、又没被用户规则/预设 allow、落到 ASK 兜底」的那批——**user permissions.json 与 P4.15 确认是叠加关系**，前者放行的不再问人。

## 3. 线程桥与时序（核心）

`confirm_tool` / `ask_user` 在 **executor 线程**阻塞，而 SSE 推送与端点处理在**事件循环线程**——必须双向桥接。观澜已有全部原语（`turn()` 里 `loop.call_soon_threadsafe` 的 thread→loop 桥、`conversation.py:239`）。

```
executor 线程（arun→Phase2 / Phase3 ask）        事件循环线程
─────────────────────────────────────          ─────────────────────────
confirm_tool / ask_user 被调
  │ 0. （仅 confirm）若 self._confirm_mode=="auto" → 直接 return True（不弹、不建 pending）  ← 气泡②/--confirm auto
  │ 1. 生成 interaction_id；在 _pending[id] 建一个 queue.Queue(1)
  │ 2. loop.call_soon_threadsafe(emit, "confirm_request"|"ask_request",
  │        {interaction_id, …, deadline_epoch})  ──────►  SSE 帧 → 浏览器渲染气泡
  │ 3. q.get(...) 阻塞（带短轮询，见 §4.3）                       用户点 允许/拒绝 / 填答案
  │                                            POST /api/chat/{id}/confirm|answer {interaction_id, …}
  │                                              端点处理（loop 线程，取 _pending_lock）：
  │  ◄───────────────────────────────────────    找 _pending[id].queue.put_nowait(payload)
  │ 4. q.get 返回 → confirm 回 bool / ask 回 str；从 _pending 摘除 id
  ▼
Phase 2 据 bool ALLOW/CANCELLED；ask 把 str 交回模型
```

- **桥的两半都现成**：thread→loop 用 `call_soon_threadsafe`（同 `_on_token`）；loop→thread 用 `queue.Queue`（`Queue.put_nowait` 本身线程安全，端点在 loop 线程直接 put，无需再跳线程）。
- **interaction_id 关联**：推送帧带 `interaction_id`（新 uuid）；前端 POST 原样回传，端点据它在 `_pending` 找对应槽。agentao 的 `TOOL_CONFIRMATION` 事件（`tool_runner.py:216`）**不带** call_id，故 id 由我们这层自生、不依赖 agentao（\[决策P4.15-2]）。
- **pending 槽按 id 键控、且每项记 `kind` 与完整 envelope**（`self._pending: dict[str, _Pending]`，`_Pending = {kind, queue, envelope}`，\[决策P4.15-9 修订]）：confirm 是 Phase 2 顺序问（同时至多一个），但 `ask_user` 在 Phase 3 执行期触发、理论上可与并行执行的别的工具叠加，故用 **id 键控字典**（兼容「极少数同时两个待应答」）。**`kind` 不可省**（安全，§5.2/\[决策P4.15-12]）：`/confirm` 只许消费 `kind=="confirm"`、`/answer` 只许消费 `kind=="ask"`，跨类一律 `409`——否则 `/answer` 把任意非空串塞进 confirm 的队列，`confirm_tool` 返回该 **truthy 串被当 `True`**（`tool_runner.py:231` `ALLOW if _confirmed`）→ 等于没点「允许」却放行 shell。**envelope 不可省**（可恢复性，§5.3）：整份请求帧存下供 `info()` 回放、断线重连重渲染。`/confirm`/`/answer` 对不上 id 即 `409`（陈旧/错乱点击）。
- **emit 复用 turn 的 `thread_safe_emit`**：confirm/ask 帧是本 turn SSE 流上的又一种帧（与 `token`/`heartbeat`/`done` 并列），前端 `event_stream`（`app.py:1005`）已逐帧 `yield _sse(kind, data)`，加帧种即可、零流式改造（\[决策P4.15-3]）。

## 4. 锁、超时、取消（三个硬点，必须同时处理）

### 4.1 写锁：confirm 阻塞 = 写锁被人类犹豫时间占着

workspace-write turn **全程持进程级 `write_lock`**（`conversation.py:253` acquire、`turn()` finally release）。confirm_tool 阻塞发生在 turn 内 → **等待用户期间写锁一直被持有**：

- ingest/heal/投喂/`GET /graph`/撤销 全部排在这把锁后（`jobs.py:46-61` WriteGate 语义）→ **人不点，它们就排队等**；
- 层③ `active_writable_turns > 0` 持续 → `/api/raw`·`ingest`·`heal` 端点持续 `423`（`jobs.py:57`）。

这是**可接受的代价但必须有上界**：单用户本地台，写者排在「人正在审批一条 shell」之后是合理的；但绝不能「人关了标签页 → 写锁永占 → 全站写死」。故 **§4.2 超时是硬需求、非可选**（\[决策P4.15-4]）。

> **`ask_user` 在 read-only 不持写锁**（§2 表）：只读 turn 本就不取 write_lock，故 read-only 提问阻塞**只**占 `conv.lock`（挡同会话下一轮）、不挡全站写者——比 workspace-write confirm 轻。但超时/取消/线程桥**一视同仁**（无人答的提问同样要超时收场）。

> **`/confirm`/`/answer` 端点绝不取 write_lock**：只 `q.put_nowait`、跑在 loop 线程、瞬时返回。若它也排队取 write_lock 就会和「持锁等它的 turn」死锁。这与 `app.py:908` 既有的「层③自死锁修复」同一类陷阱，须在端点显式注释（\[决策P4.15-5]）。

### 4.2 超时：无人应答 → 默认拒绝

等待带**墙钟超时** `CONFIRM_TIMEOUT_S`（默认 120s，可 `--confirm-timeout` 调）。超时即：

- confirm → 返回 `False` → 该工具 **CANCELLED**（运行时按拒绝继续，模型收到「工具被拒」、自行改道或收尾）；ask → 返回约定的取消串（如 `"[ask_user: timed out, no answer]"`），模型据此自决；
- 经 SSE 补一帧 `interaction_resolved {interaction_id, decision:"timeout"}`，前端把气泡标「⏱ 超时已拒绝/未答」；
- turn 继续跑到自然收尾 → finally 释放 write_lock（§4.1 上界被超时兜住）。

**默认拒绝而非默认放行**（\[决策P4.15-6]）：本相位的全部价值就是「不再静默放行」；超时回退到放行等于在最危险的「无人看管」场景下恢复旧的弱姿态。拒绝是安全侧的默认。

### 4.3 取消：停止按钮 / 客户端断开必须能解阻塞

executor 线程不能在 `q.get()` 上无限阻塞而对停止令牌失聪。用**短轮询**兼顾两信号：

```python
def _wait_interaction(q, token, deadline):
    while True:
        if token is not None and token.is_cancelled:   # 停止按钮（conversation.py:281 的同一令牌）
            return CANCELLED                            # confirm→False / ask→取消串；让正常取消链照常抛
        remaining = deadline - monotonic()
        if remaining <= 0:
            return TIMEOUT                              # 超时拒绝（§4.2）
        try:
            return q.get(timeout=min(0.2, remaining))   # 用户应答了 → 立即返回
        except queue.Empty:
            continue
```

- **停止按钮**：`request_stop()`（`conversation.py:442`）cancel 同一 `CancellationToken` → 轮询下一拍读到 `is_cancelled` → 返回拒绝 → arun 后续检查点抛 `AgentCancelledError` → turn 走 `stopped` 帧（`app.py:994`）。**已流出的 token 留在前端气泡**，与现状一致。
- **客户端断开**（评审 Medium，修正原 shield 误述）：断开时 `event_stream` finally 执行 `task.cancel()`（`app.py:1028`），取消的是**外层** `_run_turn`——它在 `await asyncio.shield(conv.turn(...))`（`app.py:980`）处收 `CancelledError`、走自己的 finally `conv.end_turn()`（清 `_inflight`→0）；但 `shield` 保护的**内层** `conv.turn` **继续在后台跑**，仍持 `conv.lock` + `write_lock` + 一个**无人能应答的 pending**。`shield` 让内层跑完、**不**让外层等它，故「会话已标 idle（`_inflight==0`）、内层却仍持锁等应答」会同时成立。现状（无 P4.15）turn 由 LLM/工具速度有界、可接受；**P4.15 引入人类长度的 pending 等待 + 全程持写锁**，若仍只靠超时，则每次断开都白占写锁达 `CONFIRM_TIMEOUT_S`、且 idle 会话可能被回收/`close()` 与后台内层并发——二者叠加是真问题。**故 P4.15a 必做**（非 deferred）：断开时除 `task.cancel()` 外，**若该会话有未决 pending 则 `conv.request_stop()`**（trip 同一取消令牌）——内层 `_wait_interaction` 下一拍读到 `is_cancelled` 立即返回拒绝、agent 抛 `AgentCancelledError` 收尾、**promptly 释放写锁**，不空等到超时。**无 pending 的普通 turn 不动**（保留 shield「后台跑完并落盘」的既有语义，不因断开误杀正常长轮）（\[决策P4.15-8 修订]）。
- **断线处理的精确执行序（落地钉死，评审复审提醒）**：必须**先实际 trip 取消令牌、再取消外层 task**——**不能**只 `task.cancel()` + `gather` 等外层自然结束（外层秒回、内层仍阻塞在 pending 等待）。改写 `event_stream` 的 finally（`app.py:1026-1029`）为：

  ```python
  finally:
      # ① 断线先 trip 令牌（仅当有未决 pending）——否则内层 shielded turn 仍卡在 confirm 等待、
      #    持写锁达 CONFIRM_TIMEOUT_S。request_stop() 幂等且线程安全（conversation.py:442），
      #    在事件循环线程直接调即可；必须**早于** task.cancel()/gather。
      if conv.has_pending():            # _pending 非空（新增廉价 helper）
          conv.request_stop()           # → 内层 _wait_interaction 下一拍返回拒绝 → arun 抛 AgentCancelledError → 释写锁
      # ② 再取消外层 _run_turn（它收 CancelledError、走 finally end_turn；不负责解内层阻塞）。
      if not task.done():
          task.cancel()
      await asyncio.gather(task, return_exceptions=True)
  ```

  - **顺序不可换**：`gather` 等的是**外层** task，它在 `await shield(...)` 处被取消后**立即**返回——`gather` 完成**不代表**内层 turn 结束。真正解开内层阻塞的是 ①的 `request_stop()`，故它必须在前、且独立于外层 task 的命运。
  - **`has_pending()` 守门**：只在确有未决 pending 时 trip（保留无 pending 普通长轮的 shield「后台跑完落盘」语义，不误杀）。`has_pending()` 是 `Conversation` 上新增的廉价纯读 helper（`bool(self._pending)`，持 `_pending_lock`）。
  - **窄竞态由超时兜底**：若断线与「turn 刚要建 pending」擦肩（`has_pending()` 判 False 后内层才阻塞），①不触发，但 §4.2 的 `CONFIRM_TIMEOUT_S` 仍是最终回退——request_stop 是快路径、超时是慢兜底，二者并存（\[决策P4.15-8 修订]）。
- **轮询粒度 0.2s**：对人类应答无感、对取消足够灵敏、CPU 可忽略（一个会话至多一个等待线程在转）。

## 5. 端点与 SSE 契约

### 5.1 新 SSE 帧（本 turn 流内，复用 `event_stream`）

| 帧 `kind` | data | 时机 |
|------|------|------|
| `confirm_request` | `{interaction_id, tool, args, description, mode, deadline_epoch}` | Phase 2 调 confirm_tool 时由 executor 线程经 `call_soon_threadsafe` 推 |
| `ask_request` | `{interaction_id, question, header, options, multiple, allow_custom, deadline_epoch}` | Phase 3 模型调 `ask_user` 时推 |
| `interaction_resolved` | `{interaction_id, decision: allow\|allow_session\|deny\|answered\|timeout\|stopped}` | 决定落定后推（让 UI 收掉气泡、记审计行） |

`args` 须**原样**带上（尤其 `run_shell_command` 的 `command` 全文）——人要看见**将要跑的命令**才能判断；这是「人在环」相对「静默放行」的全部信息增益。前端对 `args.command` / `question` 做纯文本展示（不渲染、不执行）。

**confirm 气泡的用户可选项（钉死，\[决策P4.15-13]）**——三选一 + 两个非按钮终局：

| 选项 | 语义 | 回传 / 副作用 | 本轮 | 后续 ASK |
|------|------|------|------|------|
| **① 允许** | 仅放行这一个工具 | put `True` | 继续 | 仍逐次问 |
| **② 本会话起自动放行** | 放行这一个 + 本会话不再逐次问 | put `True` **且** `conv._confirm_mode="auto"` | 继续 | **静默放行**（仍 workspace-write + 层①②③，**非 full-access**，§6） |
| **③ 拒绝** | 仅这一个工具不跑 | put `False` → 工具 CANCELLED | **继续**（模型收「工具被拒」、自行改道/收尾） | 仍逐次问 |
| ⏱ 超时（不点，120s） | 默认拒绝（§4.2） | put `False` | 继续 | 仍逐次问 |
| ⏹ 停止（turn 停止按钮） | 拒绝**且中止整轮** | trip 取消令牌 → turn 走 `stopped` | **中止** | — |

- **②是 CLI 第 2 项的「安全版」**：CLI 的 `2. Yes, allow all tools during this session`（`cli/transport.py:71-77`）会把会话翻 `FULL_ACCESS`、绕全部 preset 规则——**Web 严禁**（P4.5 决策P4.5-1，§6）。Web 的②**只把本会话 `confirm_mode` 从 `ask` 切到 `auto`**（粗粒度、可逆、不碰姿态/层），UI 另给「**恢复逐次确认**」一键切回（`POST /confirm-mode {confirm_mode:"ask"}`，§5.2）。
- **③拒绝 ≠ ⏹停止**：拒绝只毙这一个工具、模型本轮继续；停止中止整轮（已流出 token 留气泡）。停止按钮是 P4 既有、不在 confirm 气泡内，但用户面对一条待确认 shell 时实际是「允许 / 自动放行 / 拒绝（继续）/ 停止（整轮）」的选择面。前端须把③与⏹**做成两个不同动作**，不可混（评审提醒）。
- **不提供的回复**：CLI 式 `full-access` 翻转（②已是其安全替代）、「允许但改命令」（`confirm_tool` 是 `-> bool`，给不了改参数）、逐条命令白名单（§1.3，退化人在环）。

### 5.2 新 HTTP 端点

```
POST /api/chat/{id}/confirm        body: {interaction_id: str, decision: "allow" | "allow_session" | "deny"}
POST /api/chat/{id}/answer         body: {interaction_id: str, answer: str}
POST /api/chat/{id}/confirm-mode   body: {confirm_mode: "ask" | "auto"}        # 仅切未来模式（无 interaction_id、不放行当前 pending）
```

- **kind 必须匹配（安全，评审 High，\[决策P4.15-12]）**：`/confirm` 仅当 `_pending[id].kind=="confirm"` 才消费、put 布尔；`/answer` 仅当 `kind=="ask"` 才消费、put 答案串。**跨类一律 `409`**——否则 `/answer` 命中一个 confirm pending 会把任意非空 `answer` 串塞进 confirm 队列，`confirm_tool` 拿到 **truthy 串当 `True`** 处理（`tool_runner.py:231` `ALLOW if _confirmed`），**等价于用户没点「允许」却放行了 shell**。
- **`decision` 三值（\[决策P4.15-13]）**：`"allow"` → put `True`；`"deny"` → put `False`；`"allow_session"` → **先**置 `conv._confirm_mode="auto"`（持 `_pending_lock`）**再** put `True`——即放行当前工具 + 本会话后续 ASK 由 `confirmation_callback` 直接 `return True`（不再 emit 帧、不再建 pending，§3）。`allow_session` **绝不**触碰 `permission_engine`/`readonly_mode`/姿态——它不是 full-access（§6）。
- **`/confirm-mode` 只切「未来模式」、不碰在飞 pending（评审 Medium，\[决策P4.15-13]）**：把本会话 `_confirm_mode` 切 `ask`（「恢复逐次确认」）或 `auto`（**无 pending 时**主动关确认的设置式开关）。它**没有** `interaction_id`、**不 put 任何 pending 的 True**——故**不等价气泡②**：②（`/confirm {decision:"allow_session"}`）是「**放行当前这一个工具** + 翻 auto」二合一；`/confirm-mode {auto}` 是「仅以后不再问」，**当前若正有 pending，它仍悬着**，须另点②/允许/拒绝才落定（也因此「翻 auto」和「放行当前」是两件事，不可用一个无 id 的端点冒充两者）。**仅改本会话**（镜像 `/mode` 的 per-conversation 语义、不动进程默认）；冷会话/无会话 → `404`。
- 命中会话 + `interaction_id` 对得上当前 pending **且 kind 相符** → 按上 put → `200 {ok:true}`；
- 会话无此 pending / id 对不上 / **kind 不符**（陈旧点击、已超时、已被停止、打错端点）→ `409 {detail:"无匹配的待应答"}`（前端忽略或提示「已失效」）；
- 未知会话 → `404`；`confirm` 的非法 `decision`（非三值之一）→ `422`；`answer` 的 `allow_custom:false` 却给了表外答案 → 端点不校验语义（交回模型容错，前端按 `options` 约束 UI 即可）。
- **锁序**：端点取**独立的轻量 `_pending_lock`（`threading.Lock`）**只护 `_pending` 字典与 `_confirm_mode` 的读改，**不**取 turn 此刻正持有的 `conv.lock`（否则与阻塞在应答上的 turn 死锁），也**不**取 write_lock（§4.1）。`Queue` 本身线程安全，put 不需要额外锁（\[决策P4.15-9]）。

### 5.3 `info()` / `/tools` 自省

`conversation.py:508` 已报每工具 `requires_confirmation`。P4.15 在 `info()` 加 `confirm_mode`（`ask`/`auto`）与 `pending`（当前 pending 列表，**带完整请求 envelope**：`[{interaction_id, kind, tool, args, description, question, options, multiple, allow_custom, deadline_epoch}]`，缺项按 kind 省略）。**只回 `{interaction_id, kind}` 不够**（评审 High）：前端丢了 SSE 流后，要重渲染确认/提问气泡**并让用户决策**，必须拿到 `tool`/`args`/`description`（confirm）或 `question`/`options`/`deadline_epoch`（ask）——故整份 envelope 存进 pending（§3）、由 `info()` 原样返回，断线重连即拉 `info()` 重建气泡。envelope 已在 §3 为安全（kind）+ 可恢复性（这里）双重需要而落在 pending，`info()` 只是把它读出。`confirm_mode` 是**每会话**状态（`conv._confirm_mode`，开局 = 进程默认 `--confirm`，经气泡②或 `/confirm-mode` 翻；镜像 `/mode` 的 per-conversation 语义），UI 据它显示「逐次确认中 / 本会话已自动放行（可恢复）」。

## 6. 与层①②③的关系：确认是正交的第四道闸，不替代 immutable 硬墙

**这是本相位最该说清的安全论证**（\[决策P4.15-10]）。confirm 加在 **ASK 决策点、执行之前**，与 P4.5 三层各管各的：

| 层 | 管什么 | confirm 通过后它还在不在？ |
|----|--------|------------------------|
| **层①**（`PolicyFileSystem` wrapper，P4.5 §2.2） | agent 经 `agent.filesystem` 结构化写 `raw/`/`AGENTAO.md` → `PermissionError` | **仍在、不放宽**。即便人对一条 shell 点了「允许」，agent 随后用 `write_file` 写 `raw/` 仍被层①拒——那是结构化写、不经 confirm。 |
| **层②**（AGENTAO.md 快照还原，P4.5 §2.3） | shell 直写/换形态 `AGENTAO.md` → turn 收尾自动还原 | **仍在、不变**。confirm 放行的 shell 若 `echo x > AGENTAO.md`，turn finally 照样还原。 |
| **层③**（时序互斥 → 423，P4.5 §2.4） | 可写 turn 活跃期间宿主写端点 423，兜 `curl localhost/api/raw` | **仍在、且应答等待期间持续生效**（turn 仍活跃）。 |
| **P4.15 confirm/ask**（新） | workspace-write 下 **ASK 类 shell / `requires_confirmation` 工具**的执行准入 + 模型提问 | 把「静默放行/返回哨兵」换成「人决定」。 |

**关键不变量**：**「人点允许」≠「绕过 immutable」。** confirm 只决定一个 ASK 工具**跑不跑**；它**不**给 agent 开任何新写路径。原因：

- 人类会被参数里埋的恶意路径骗（prompt-injection 的源文档诱导出一条看似无害、实则 `… > raw/secret` 的命令）。所以 `raw/`/`AGENTAO.md` 的只读**必须**继续由**确定性**层①②扛，**不能**降级成「让人点确认」。confirm 是**抬门槛**（人能看见命令全文、挡住大多数明显恶意），**不是**对抗性封堵。
- 反过来，confirm 也**没有扩大**攻击面：workspace-write 现状本就**自动放行**这些 ASK shell（§1.1）。P4.15 只是把同一批调用从「悄悄跑」变「跑前给人看一眼」——**严格收紧、不放松**。
- **三类残留照旧 deferred 到 sandbox**（P4.5 §1）：shell 直写 `raw/`、写 kb 外、`curl … &` 后台派生。confirm 对它们的作用**仅是抬门槛**（恶意 shell 在被人点「允许」前要先骗过人眼），真封堵仍需网络/文件 sandbox。**P4.15 改善的是「prompt-injection 经 shell 偷跑」的可见性，不声称封死对抗性 shell。**
- **气泡②「本会话起自动放行」严防退化成 full-access（\[决策P4.15-13]）**：CLI 第 2 项（`cli/transport.py:71-77`）会 `set_mode(FULL_ACCESS)` + `allow_all_tools` 绕全部 preset 规则、作废层①②③——**Web 严禁**（决策P4.5-1）。Web 的②**只**把本会话 `confirm_mode` 切 `auto`：confirm_tool 直接返 True，但姿态仍 workspace-write、**层①②③ 一个不少、`raw/`/`AGENTAO.md` 依旧硬只读**。即②放宽的仅是「ASK shell 跑不跑要不要问人」、**绝不**放宽「写到哪」。故②在最坏情况（人被一条命令骗了点②）下，损失面与 `--confirm auto` 完全相同、且**远小于** full-access（后者连结构化写 `raw/` 都放）。可逆（「恢复逐次确认」）+ `info().confirm_mode` 可见，是其安全护栏。

> 一句话定位：层①②③是**确定性边界**（机器拦，对抗性有效），P4.15 confirm 是**人在环抬门槛**（人拦，挡意外/明显恶意、不挡精心伪装）；气泡②只松「问不问」这一档、不松「写到哪」。二者叠加、各补盲区，谁都不替代谁。

## 7. 默认值与开关（已定）

`guanlan web --confirm {ask,auto}`（\[决策P4.15-7]，**默认 `ask`**）：

- **`ask`（默认）**：workspace-write 下 ASK 弹给人。理由：现状静默放行是 P4.5 自己承认的弱姿态（§1.1），而本相位的全部价值就在翻转它；单用户本地台，偶尔确认一条带管道的 shell 是合理摩擦。**首次**弹确认时前端附一句「可 `--confirm auto` 关闭逐次确认」，让重度用户自助降噪。
- **`auto`（逃生舱）**：保持 P4.5 现状（静默放行），给「我就要 agent 一路跑、不想被打断」的批量维护场景。等价于不接 `confirmation_callback`。

**进程默认 vs 会话级覆盖（同一 `ask↔auto` 轴）**：`--confirm` 只定**进程开局默认**；每会话 `conv._confirm_mode` 开局继承它、之后可被**气泡②**或 `POST /confirm-mode` 翻（镜像 `--mode`/`/mode` 的「进程默认 + per-conversation 覆盖」结构，§5.2/§5.3）。即「本会话起自动放行」= 会话内把这一档从 `ask` 拨到 `auto`、可一键拨回——**粒度是「问/不问」，不是姿态、不是 full-access**（§6）。故无需在 §7 首次弹窗里硬推 `--confirm auto`：直接点气泡②即可会话内降噪、且可逆。

read-only 会话**与本开关无关**（其写/shell 本就 DENY、不到 ASK；唯 `ask_user` 提问跨姿态触发，§9）。`--confirm-timeout`（默认 **120s**，已定）调 §4.2 超时。

## 8. 测试（落地时）

- **触发集**：构造 workspace-write 会话，喂一条带管道 shell（mock 模型产出该 tool_call）→ 断言 `confirm_request` 帧出现、turn 阻塞；POST allow → 工具执行；POST deny → 工具 CANCELLED、模型收到拒绝。
- **写工具不弹**：`write_file` 到 `wiki/` → 无 confirm 帧、直接 ALLOW（确认触发集边界）。
- **immutable 不被 confirm 放宽**：人 allow 一条 shell 后，agent `write_file('raw/x')` 仍被层① `PermissionError`（§6 不变量回归）。
- **ask_user 往返**：mock 模型调 `ask_user(question, options=[…])` → 断言 `ask_request` 帧带 options；POST answer → 模型收到该串；read-only 会话同样触发且**不持写锁**（断言并发写者不被它挡）。
- **超时**：把 `CONFIRM_TIMEOUT_S` 调 0.5s、不应答 → 断言自动 deny/取消串、`interaction_resolved{decision:"timeout"}`、write_lock 最终释放（无泄漏，复用 P4.5 写锁泄漏回归口径）。
- **取消**：弹气泡后 POST `/stop` → 等待返回拒绝、turn 走 stopped、写锁释放。
- **端点不死锁**：turn 阻塞在应答时并发 POST `/confirm`/`/answer` → 端点 200 且**不**取 write_lock/conv.lock（§5.2/§4.1）。
- **陈旧点击 / 键控**：interaction_id 对不上 → 409；无 pending → 409；同时两个 pending（confirm + ask）各自按 id 解锁、不串。
- **跨类拒（安全回归，决策P4.15-12）**：turn 阻塞在一个 **confirm** pending 时 POST `/answer {同一 id, answer:"yes"}` → 断言 `409` **且工具未被放行**（shell 未执行）；反向 `/confirm` 打到 ask pending 同样 409。这是 High #1 的定向回归。
- **断开按需释锁（决策P4.15-8）**：workspace-write turn 阻塞在 confirm 时断开客户端 → 断言写锁在 ~轮询粒度内释放（**不**等满 `CONFIRM_TIMEOUT_S`）、后续 ingest 立即可起；对照组：无 pending 的普通长轮断开仍后台跑完落盘（不被误杀）。
- **info().pending 回 envelope**：有未决 confirm 时 `GET /api/chat/{id}/info`（**会话级**自省，`app.py:1036`；非 app 级 `GET /api/info`）→ 断言 `pending[0]` 带 `tool`/`args`/`deadline_epoch`（足以重渲染气泡）；有未决 ask 时带 `question`/`options`。
- **auto 模式**：`--confirm auto` 下同一带管道 shell 静默放行、无 confirm 帧（回归 P4.5 行为）。
- **气泡②本会话起自动放行（决策P4.15-13）**：第一条 ASK 弹气泡 → POST `/confirm {decision:"allow_session"}` → 断言①当前工具执行、②**下一条 ASK shell 静默放行、无 `confirm_request` 帧**、③`info().confirm_mode=="auto"`、④**姿态仍 `workspace-write`、`permission_engine` 未被翻 `FULL_ACCESS`、`readonly` 不变**（关键安全断言：②≠full-access）、⑤agent `write_file('raw/x')` 仍被层① `PermissionError`（层未失效）。
- **②可逆**：`allow_session` 后 POST `/confirm-mode {confirm_mode:"ask"}` → 断言下一条 ASK **又弹** `confirm_request`、`info().confirm_mode=="ask"`。
- **并发写者排队**：confirm 等待期间提交 ingest → 它排在写锁后、应答放行/拒绝后才轮到；层③ `/api/raw` 在等待期 423。

## 9. `ask_user`：模型主动提问（本期纳入）

`transport.ask_user(question, header, options, multiple, allow_custom) -> str`（`base.py`）是**另一条通道**：模型想问人澄清问题（非工具准入）。与 confirm 同构（executor 阻塞 + SSE 弹 + 端点回传），关键差异：

- **返回 string**（非 bool）：端点 `/answer` 收答案串、原样交回模型；`options`/`multiple`/`allow_custom` 是**呈现提示**（前端可渲染成单/多选 + 自由文本），但答案最终是单串（`base.py` 契约）。
- **跨姿态触发**：提问不是写，**read-only 也会触发**（§2 表）；read-only 下阻塞**只占 `conv.lock`、不持写锁**（§4.1 注），比 workspace-write confirm 轻。
- **未接也不崩**：现状返回哨兵串，模型自决——故 P4.15a（confirm）先落、P4.15b（ask_user）紧随，二者解耦但同属本期（§10）。

**逐次白名单 / 会话内记住放行**（“这条命令别再问”）**不在本期**——它会把人在环退化成新的静默面，须单独评估威胁模型（§1.3）。

## 10. 子相位拆分（落地顺序，均属本期）

- **P4.15a**：`confirmation_callback` 接线（含 `_confirm_mode=="auto"` 短路，§3 步 0）+ 线程桥 + `confirm_request`/`interaction_resolved` 帧 + `POST /confirm`（**三值 decision + kind 匹配**，§5.2）+ `POST /confirm-mode`（②切回）+ 每会话 `_confirm_mode`（继承 `--confirm` 默认）+ pending 存 `{kind, queue, envelope}`（§3）+ 超时/取消/默认拒绝 + **断开即按需 trip 取消令牌**（§4.3，非 deferred）+ `info().pending`/`confirm_mode` 自省 + 前端**三选项**确认气泡（含「恢复逐次确认」+ 断线重连重渲染）。**核心闭环**。
- **P4.15b**：`ask_user` 接 UI（§9）——`ask_user_callback` + `ask_request` 帧 + `POST /answer`（**kind 匹配**）+ 提问气泡（选项/自由文本）。复用 P4.15a 的线程桥/pending/超时/取消/断开骨架。

## 11. 决策记录

- **\[决策P4.15-1]** 触发集 = workspace-write 下的 `ASK`（带操作符 shell + `requires_confirmation` 工具）。写可写区不弹（P4.5 写后 check 兜）；写 immutable 不靠 confirm 而靠层①。
- **\[决策P4.15-2]** `interaction_id` 由宿主层自生（uuid），不依赖 agentao 的 call_id（`TOOL_CONFIRMATION` 不带它）。
- **\[决策P4.15-3]** confirm/ask/resolved 是本 turn SSE 流上的新帧种，复用既有 `event_stream` 逐帧转发，零流式改造。
- **\[决策P4.15-4]** 等待持 write_lock（workspace-write），故**超时是硬需求**、有明确上界。
- **\[决策P4.15-5]** `/confirm`·`/answer` 端点绝不取 write_lock（只 `put_nowait`，loop 线程瞬返），防与持锁等待的 turn 死锁。
- **\[决策P4.15-6]** 超时/取消的默认决定是**拒绝**（安全侧），绝不默认放行。
- **\[决策P4.15-7]** `--confirm {ask,auto}` 开关，**默认 `ask`**（已定）；`auto` = P4.5 现状逃生舱。`--confirm-timeout` 默认 **120s**（已定）。
- **\[决策P4.15-8 修订]** 客户端断开时**若会话有未决 pending 则 `conv.request_stop()`**（trip 取消令牌），令后台内层 turn 立即收尾、promptly 释放写锁，不空等 `CONFIRM_TIMEOUT_S`；无 pending 的普通 turn 保留 shield「后台跑完落盘」语义。**P4.15a 必做、非 deferred**——因 P4.15 的 pending 等待全程持写锁、断开后白占写锁达超时是真问题（评审 Medium，§4.3）。
- **\[决策P4.15-9 修订]** pending 槽 = **interaction_id 键控字典**，每项 `{kind, queue, envelope}` + 独立 `_pending_lock`（threading.Lock）护，**不**用 turn 持有的 `conv.lock`（否则端点与阻塞 turn 死锁）。键控兼容 confirm（Phase 2）/ask_user（Phase 3）并存；记 `kind` 供端点跨类拒（决策P4.15-12）；记 `envelope` 供 `info()` 回放、断线重连重渲染（§5.3，评审 High）。
- **\[决策P4.15-10]** confirm 与层①②③正交：**人点允许 ≠ 绕过 immutable**；immutable 只读永远由确定性层①②扛，confirm 只决定 ASK 工具跑不跑，不开新写路径、不声称封死对抗性 shell。
- **\[决策P4.15-11]** `ask_user` **本期纳入**（P4.15b），与 confirm 同骨架；逐次白名单/会话内放行不纳入（退化人在环）。
- **\[决策P4.15-12]** `/confirm`·`/answer` 端点**必须校验 pending.kind**：confirm 端点只消费 confirm pending、answer 端点只消费 ask pending，跨类一律 `409`。否则 `/answer` 的非空 `answer` 串被 `confirm_tool` 当 truthy `True`（`tool_runner.py:231`）→ 绕过「点允许」放行 shell（评审 High，安全）。
- **\[决策P4.15-13]** confirm 气泡 **三选项**：**允许** / **本会话起自动放行** / **拒绝**（+ 非按钮终局：超时→拒绝、停止→拒绝并中止整轮）。对标 CLI `1. Yes / 2. Yes, allow all this session / 3. No`（`cli/transport.py:59-61`），但 **②取安全版**：CLI 第 2 项 `set_mode(FULL_ACCESS)`+`allow_all_tools` 绕全部规则、作废层①②③（Web 决策P4.5-1 严禁）；Web ②**只**把本会话 `_confirm_mode` 切 `auto`（confirm_tool 直返 True），**姿态仍 workspace-write、层①②③ 不动、`raw/` 仍硬只读**，且**可逆**（`/confirm-mode {ask}`「恢复逐次确认」）+ `info().confirm_mode` 可见。`_confirm_mode` 是每会话状态、开局继承 `--confirm` 进程默认。**不提供** CLI 式 full-access 翻转、改命令、逐条白名单（§5.1）。
