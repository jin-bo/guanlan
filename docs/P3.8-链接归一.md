## 0. 缺口与立场

观澜已在**解析期非破坏地**消解了 OpenKB 在*修复期*改写正文才做的事——`link_stem`（`pages.py:179`）剥
`|#.md`+小写，P3.1 别名（`link_target_stems`）让 `[[别名]]` 命中拥有页，**全程零正文改写**。**残余真缺口窄**：
`link_stem` **不**折叠 `_`↔`-`、不做 Unicode 归一 → `[[multi_head_attention]]` 对页 `multi-head-attention`
（且无别名覆盖）仍**误判断链**，`[[Café]]` 的 NFD/NFC、全角/半角变体同理。

> **立场**：把变体**像别名一样在解析期非破坏命中**——零正文改写、守"markdown 唯一真相"、无 P3.7 式
> "谁写内容页"张力。**不**做 OpenKB 式 `fix_broken_links` 改写规范形 / strip 去括号（决策8：断链是会随
> 后续资料自我消除的前向引用，strip 掉正是删待填占位）。**关键约束**：`link_stem` 被 `alias_index`/check/
> graph/heal/Web 渲染共用——**绝不动它的语义**（一旦它直接 fold，`foo_bar` 先变 `foo-bar`，"raw 精确优先"当场就丢了）。

## 1. 三步路线（不另起解析框架）

### 1.1 加 `fold_stem` / `link_fold_stem`，`link_stem()` 一字不动

```python
# pages.py：link_stem 保持原样；新增两个纯函数
def fold_stem(stem: str) -> str:
    """把已归一的 stem 进一步折叠掉常见等价变体（NFKC → casefold → `_`→`-`）。"""
    return unicodedata.normalize("NFKC", stem).casefold().replace("_", "-")

def link_fold_stem(raw: str) -> str:
    """[[raw]] 的折叠键 = fold_stem(link_stem(raw))。fold 永远叠在 link_stem 之上、绝不替换它。"""
    return fold_stem(link_stem(raw))
```

- **NFKC**：折全角/半角、兼容字符、组合记号（`Café`NFD → `café`NFC 同形）。
- **casefold**（非 `.lower()`）：Unicode 大小写折叠更全；对纯 ASCII/CJK 与 `.lower()` 一致、无回归。
- **`_`→`-`**：kebab-case 为主、`_` 是最常见同义变体。**仅此一条替换**——不折重复 `-`、不剥首尾 `-`（过激会误并真实独立名，决策P3.8-1）。

### 1.2 单一解析表 `link_resolution_index`（base + 安全 fold variant → owner path）

把现有 `link_resolution_index`（`pages.py:254-268`，raw stem 含 config + 别名、stem 优先 → owner 相对 posix 路径）
保留为 **base**，**只叠加无冲突的 fold variant**（决策P3.8-3/4）：

```python
def link_resolution_index(wiki: Path) -> dict[str, str]:
    base = _base_resolution_index(wiki)        # = 原 link_resolution_index 函数体（raw stem/别名 → owner path）
    groups: dict[str, set[str]] = defaultdict(set)
    for key, owner in base.items():
        fk = fold_stem(key)
        if fk not in base:                     # fk 撞任意已有 raw/别名键 → 不新增（"撞则不折叠"，自动成立）
            groups[fk].add(owner)
    for fk, owners in groups.items():
        if len(owners) == 1:                   # fold group **唯一拥有者**才生成 variant；≥2 → 撞名，丢弃
            base[fk] = next(iter(owners))
    return base
```

- **机械规则（决策P3.8-4）**：variant 键 = `fold_stem(base_key)`，**当且仅当**「`fk` 不在 base 键集」**且**「该 `fk` 的 fold group 拥有者恰 1 个」。raw/别名永远由 base 处理；variant 只补「base 没有、且无歧义」的新键。
- variant 键与 base 键**天然不相交**（`fk not in base` 门），故整表每键唯一 owner、无须区分"raw 命中 vs fold 命中"。
- `foo_bar.md` + `foo-bar.md` 同存：两者 base 键 `foo_bar`/`foo-bar` 各在表中；fold 键 `foo-bar` 已是 base 键 → 不新增 → **两页各由精确键解析、零串台**（撞名自动不折叠，无需额外检测）。

### 1.3 `resolve_owner` 归口 + 四处复用（解析表，不是布尔探针）

收敛成**一个返回 owner 的解析函数**（决策P3.8-2/3），而非「key 集 + 布尔 `target_resolves` + 另一套 owner 映射」三层：

```python
def resolve_owner(raw: str, idx: Mapping[str, str]) -> str | None:
    """[[raw]] → 拥有页相对 posix 路径。raw 精确(link_stem)优先、fold(link_fold_stem)兜底；皆不中 → None。"""
    k = link_stem(raw)
    if k in idx:
        return idx[k]            # raw stem / 别名 精确命中（优先）
    fk = link_fold_stem(raw)
    if fk in idx:
        return idx[fk]           # fold variant 兜底
    return None
```

**必须改到的四处调用点**（任一漏改，`[[multi_head_attention]]` 这类 fold 命中会在旧的 `link_stem(raw) in 键集` / `resolution.get(raw)` 路径上漏掉，退回断链）：

- **check**（`_check_wikilinks`，`check.py:92`）：**入参由 `link_targets`(键集) 改为 `idx`(解析表)**，判据由 `link_stem(raw) in link_targets` 改为 `resolve_owner(raw, idx) is None` → `wikilink.broken`。改签名能让旧调用点漏改时直接编译/类型暴露，不静默走老路。
- **graph**（`build_graph` 边解析，`graph.py:83-152`）：`owner = resolve_owner(raw, idx)`；建 `content_path_to_id = {node.path: node.id}`（与 `idx` 的 owner 值同基，皆相对库根 posix）→ `owner in content_path_to_id` 建 resolved 边、`owner` 非空但非内容页（config）→ 丢弃、`owner is None` → broken 边。**三分类不变、`graph.broken ≡ check.wikilink.broken` 不破**（决策P3-6）。
- **heal 写后回执**（`_build_receipts`，`heal.py:354`）：现为 `resolution.get(w.target)`——**精确查表会漏 fold 命中**：若 `w.target="foo_bar"`、Agent 建了 `foo-bar.md`，回执会误报 `still_broken`。改为 `resolved_to = resolve_owner(w.target, resolution)`，与门禁/lint 同口径判"是否真补上"。
- **Web 渲染**（`_resolve_wikilink`）：`[[..]]` → `resolve_owner(raw, idx)` 取路径（原本就读 `link_resolution_index`，改为同样的两段探针）。**仅 `_resolve_wikilink` 走 fold**；行内 code 引用（`_code_ref_target`，精确路径/文件名判定）**不放宽**——见 §4 边界。

> `link_target_stems`（`pages.py:244`）退化为 `frozenset(link_resolution_index(wiki))`，**仅供需要键集的旧调用方做兼容**——
> **断不可再当断链判据**（`link_stem(raw) in targets` 会漏掉所有 fold 兜底命中）。断链判定一律走 `resolve_owner`。

## 2. 决策（沿用仓库 `决策PX-N` 体例）

- **决策P3.8-1（最小折叠集）**：仅 NFKC + casefold + `_`→`-`；不折重复 `-`、不剥首尾、不做同义/拼写。过度归一误并真实独立名。
- **决策P3.8-2（`link_stem` 不动、显式两段探针）**：新增 `fold_stem`/`link_fold_stem`；"精确优先 fold 兜底"由 `resolve_owner` **显式先查 `link_stem` 再查 `link_fold_stem`** 实现——**绝不**把 fold 塞进 `link_stem`（会丢精确优先 + 波及全体 `link_stem` 消费者）。
- **决策P3.8-3（单一解析表 + 单一 owner 归口）**：`link_resolution_index`（key→owner path）+ `resolve_owner` 一处；check（owner 是否 None）、graph（owner→节点）、heal（写后回执判 still_broken）、Web（owner→路径）**全复用**，不各写 owner 逻辑。比"键集 + 布尔探针 + 拥有者映射"少一层概念。
- **决策P3.8-4（机械 variant 规则）**：见 §1.2——`fk not in base` 且 fold group 唯一拥有者，二者皆成立才生成。撞名（≥2 拥有者，或 fk 已是 base 键）→ 自动不生成 variant、不影响解析。
- **决策P3.8-5（零正文改写）**：fold 纯解析期；**绝不** auto-rewrite/strip（撞决策8 前向引用生长信号）。markdown 唯一真相不破。
- **决策P3.8-6（v1 不加 `lint.fold_collision`）**：既然"撞则不折叠"已保证撞名**不影响解析**，第一版**只做解析收益**，撞名仅由测试保证"不新增 variant"。不新开 lint finding（免改 lint 契约/文案/JSON 测试/与拓扑 finding 并列），收益（提示统一命名）单列 **P3.8.x** 再说（见 §4）。
- **决策P3.8-7（无新命令/退出码）**：纯改解析归口。`check`/`graph`/`heal`/`lint`/Web 自动受益（误断链减少），不引新命令、不动退出码。

## 3. 测试要点

- `[[multi_head_attention]]` 命中 `multi-head-attention.md`；`[[Café]]`(NFD) 命中 `café.md`（NFKC）；全角/半角变体命中。
- **撞名按行为断言（非查表内部）**：`foo_bar.md` + `foo-bar.md` 同存时，`resolve_owner("foo_bar", idx)` → `wiki/.../foo_bar.md`、`resolve_owner("foo-bar", idx)` → `wiki/.../foo-bar.md`，**各归各页、无串台、两页都不丢**。（不去断言"表里有没有 variant 项"——dict 不记录键是 base 还是 variant，且 `foo-bar` 本就作为 base 键存在，那种断言既脆又测不准。）
- `graph.broken ≡ check.wikilink.broken` 在含 fold 兜底后仍恒等（同 `resolve_owner` 口径回归）。
- 别名 fold：alias `multi_head` 让 `[[multi-head]]` 命中拥有页（variant 唯一拥有者才叠）。
- config 兜底：`[[index]]`/fold 撞 config 键 → 不新增 variant、check 仍不报断链、graph 仍丢弃（行为不变）。
- **heal 写后回执 fold 命中**：`w.target="foo_bar"`、Agent 实际建 `foo-bar.md` → receipt 判 **resolved**（不误报 `still_broken`，走 `resolve_owner` 而非 `resolution.get`）。
- **Web code-ref 不 fold**：库内存在 `foo-bar.md` 时，正文行内 code `` `foo_bar` `` **不**被折叠成指向该页的链接；**只有** `[[foo_bar]]` 走 fold 兜底（`_code_ref_target` 不接 `fold_stem`）。
- 零正文改写：跑前后所有 `.md` 字节不变。
- 向后兼容：纯 ASCII/CJK 既有库 fold 行为同 `.lower()`，原 check/graph/lint/Web 测试全绿；`_base_resolution_index` 拆出后无 variant 命中的库表内容与原 `link_resolution_index` 一致。

## 4. 边界（明确不做）+ 后续

- 不改任何正文（决策P3.8-5）；不加 `heal`/`fix-links` 改写命令；不做同义词表/拼写/分词（仍守 `cjk-retrieval-enhancements.md` 戒律，留 E1）。
- **fold 只放宽 `[[wikilink]]` 解析**：行内 code 引用（Web `_code_ref_target`，按精确路径/文件名判定）**不接 `fold_stem`**——代码标识符里 `_`/`-` 语义不同，折叠会把普通代码误链成页面引用。fold 的入口仅 `resolve_owner`，code-ref 路径不碰。
- 不碰 `raw/`、不起 LLM、不引依赖（`unicodedata` 标准库）。
- **不在本阶段碰检索**：P3.8 是 wikilink **解析层**增强，**不**承诺 P5.0 `build_doc` 的 alias/title 召回面变化——P5 检索索引另有其 `tokenize` 路径，未接入 `fold_stem`。若 P5 后续选择复用本解析表/`fold_stem`，可获同类归一收益，但那是 P5 的事、非本阶段承诺。
- **P3.8.x（可选后续，决策P3.8-6）**：若实测"`_`/`-` 孪生页"确有困扰，再加一条 advisory（`lint` 或 `check`）提示统一命名。第一版不做——撞名已安全（不折叠、不丢页），advisory 纯属锦上添花。
