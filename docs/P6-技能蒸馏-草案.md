# P6 — 技能蒸馏（distill）· 技术路线【草案 · 暂缓进实现】

> 把 `wiki/` 蒸馏成一份**面向未来分发的 Anthropic Skill 草稿**，补上 Karpathy 线缺的「生成层」终点（"drop in a book, out comes an expert"）。本文是 `../llm-wiki/OpenKB` Skill Factory 反向评审后的落地设计——**只借形状、不借实现**。
>
> **状态**：草案，**暂缓进实现**。设计已自洽可实现，但更稳的排序是先让 P3/P4/P5 的「维护—检索—宿主」闭环跑出真实使用、确认 distill 有需求再建。CLAUDE.md 那句 "no roadmap spec remains" 是"无待还的债"，不是"必须开新里程碑"的理由。
>
> **本次范围**：P6 第一版 = 「蒸馏 → 本地 `dist/skills/<name>/` 草稿 → 零-LLM 结构校验」**两段**。marketplace 收录与分发**移入 backlog（`P6-MKT`）、本次不实现、且不作为 P6 前置**（§7）。产物只称**「需人审的草稿」**，不是"装上即用的专家"——「专家/可分发」措辞门控在 P6.1 eval 之上。

---

## 1. 定位

`guanlan distill <name> "<intent>"`：一条 **LLM 蒸馏（经 Agentao+skill）→ 零-LLM 结构校验** 的命令。读既有 `wiki/`（用 P5.0 确定性召回），把某主题蒸馏成一份**句法自洽**的 skill 草稿落到 `<kb>/dist/skills/<name>/`，写后零-LLM 校验其结构。

**边界（诚实声明）**：
- **只是草稿**：零-LLM validator 只保**结构**（有 frontmatter、无外链、不超大），保不了内容是真专家规则还是"主题罗列/自信胡说"——那要 P6.1 的 eval。
- **「自洽」只到句法层**：validator 只查 `[[...]]` 写法（无指向本库 wiki 的外链、`[[references/…]]` 能解析），**不查正文语义完整**（正文可引用未拉进 references 的概念却照样过校验）。

**为什么不照搬 OpenKB**：它把蒸馏器做成 in-process agent 循环——自持 LLM client + 在 wrapper 里编码"怎么蒸馏"的业务智能 + 嵌套 `query_wiki` LLM 检索 + 默认覆盖产物，四处都撞 guanlan 红线。本设计把蒸馏**工作法**落进 skill 工作流（由 Agentao 跑），wrapper 只出一条薄 PROMPT + 受限 posture + 零-LLM validator——这正是 ingest/convert 已验证的分工。

## 2. 架构：复刻既有"两件套"

| 层 | 既有（ingest/convert） | P6 distill | 复用点（已验证存在） |

## 3. 命令与产物

```
guanlan -C <kb> distill <name> "<intent>"            # 蒸馏 → dist/skills/<name>/（LLM 写 + 写后零-LLM 校验）
guanlan -C <kb> distill <name> "<intent>" --dry-run  # 零落盘：打印 wrapper 召回的候选页列表，不触 Agentao、不写 dist/
guanlan -C <kb> distill validate <name>              # 零-LLM 纯只读重校验（--json 稳定契约 / --strict 把 warning 也计入）
```

- **契约对齐 `convert`**：默认**不覆盖**（`dist/skills/<name>/` 已存在 → `EXIT_USAGE` 不动盘）、**无 `--overwrite`**（覆盖须 staging/rollback 才安全，留 P6.2）。
- **`validate` 是保留子命令字**：`distill <name>` 的 `<name>` 不能取 `validate`（否则 `distill validate "<intent>"` 与校验子命令二义）——argparse 上 `validate` 解析为子命令，故 `<name>=validate` 须显式拒：归 `name.invalid` 或 CLI usage 错（`EXIT_USAGE`）。
- **产物**：`dist/skills/<name>/{SKILL.md, references/*.md, (scripts/)}`——一份**句法自洽**的 skill（不含指向本库 wiki 的外链，只许 `[[references/<slug>]]`）。`dist/` 是 `raw/`/`wiki/`/`graph/` 之外新增的第四个顶层产物区，opt-in（只在跑 `distill` 后出现），可删可再生。

**退出码（全复用 `errors.py`，不新增）**：

| 码 | 触发 |
|---|---|
| `EXIT_OK`(0) | 成功 |
| `EXIT_USAGE`(1) | 用法/IO（含产物已存在的不覆盖拒绝；仿 convert `except OSError`） |
| `EXIT_CHECK_FAILED`(3) | 蒸馏期 `wiki/`/`AGENTAO.md`/`SCHEMA.md` 漂移（不该碰） |
| `EXIT_RAW_MUTATED`(4) | `raw/` 被改 |
| `EXIT_AGENT_ERROR`(5) | 蒸馏运行失败（Agentao 子进程错误） |
| `EXIT_LINT_FINDINGS`(6) | validator error 级 finding（或 `validate --strict` 的 warning） |

## 4. 拱心石：受限写 posture（读 wiki、保护 raw、只写 dist/）

distill 的写姿态是第三种：**读 `wiki/`、写 `dist/`、`raw/` 全程只读保护**（既非 query 纯只读、也非 ingest 可写 wiki）。一期工作法只要求读候选 wiki 页 + `wiki/index.md`；`raw/` 不是必读面（必要时可读 source provenance，但**不作为一期工作法要求**），只是被快照护住不许改。

**编排（仿 `run_guarded_write_result`，更轻）**：
```
before_raw = gate.snapshot_raw(root)                       # 直接复用既有 raw 门禁（只调用、不改 gate.py）
before_ro  = snapshot_readonly(wiki/, AGENTAO.md, SCHEMA.md)  # 轻量只读快照（逐文件 size+sha256）
run_agent_task(DISTILL_PROMPT, permission_mode="workspace-write", model=…, runner=…)
写后零-LLM 裁决：
  gate.diff_raw 非空  → EXIT_RAW_MUTATED(4)
  wiki/ 或 config 变   → EXIT_CHECK_FAILED(3)
  都没变             → distill.validate(dist/skills/<name>)
                        error → EXIT_LINT_FINDINGS(6)（产物留盘供修）
                        通过  → EXIT_OK（止于本地草稿）
```

- **raw/ 直接复用既有门禁**：调用 `gate.snapshot_raw` / `gate.diff_raw`（**只调用、不改 `gate.py`**）——raw 不可变的完整防绕过口径（符号链接/空目录/特殊文件）就地复用，既不重写保命码、也不复制它的复杂遍历与重复测试。
- **wiki/ + config 用轻量只读 helper**：`snapshot_readonly(...)` 逐文件取 `(size, sha256)`，只为"是否被改"的检测——它不是 raw 那种对抗性不可变边界（Agent 是受 prompt 约束的合作方、非攻击者），无需复刻 raw 门禁的防绕过严苛口径。
- **为什么也盯 `AGENTAO.md`/`SCHEMA.md`**：它们跨次驱动 Agent 行为，被改会持久影响后续 ingest/query/distill；P4.11 刚立"wiki 内容即数据、可能夹带注入"的信任边界，让一个**读 wiki**（潜在被注入）的生成器静默改这两个 config 正好踩线。代价近乎零（一个遍历子树+单文件的纯函数）。
- **诚实边界**：快照只确定性证明 `raw/`+`wiki/`+两个 config 零改动；其余顶层目录（`.agentao/`、`graph/`、其他 `dist/skills/*`）靠 prompt 约束、非门禁保证。**不声称**"唯一允许增量是 dist/" 的强不变量；全库 allowlist 快照留 P6.x（出现真实需求再做）。
- **无"谁来写"张力**：产物正文是 LLM 蒸馏的散文，本就该 Agent 写；wrapper 只事后做零-LLM 校验——完全落在"Agent 写内容、wrapper 写 config/派生"的既有线内。

## 5. 零-LLM 结构校验（`distill.validate`，仿 `check.run_check`）

每条检查 → 一个带 `severity` 的 finding（`{page, kind, detail, severity}`，仿 `Violation` 但加一字段）。OpenKB `skill/validator.py` 的清单与正则（`FOREIGN_WIKILINK_RE` 等）可直接借——它本就是零-LLM 正则/IO。

**error 级（主命令阻断、返 `EXIT_LINT_FINDINGS`、产物留盘供修）**：
- `skill.missing`：无 `SKILL.md`。
- `frontmatter.unparsable`：缺 frontmatter 块 / YAML 解析失败 / 非 mapping——用**严格档** `split_frontmatter` + `parse_frontmatter`（`guanlan/pages.py`），**不**复用 `pages.load_page`（容错档会把缺块/YAML 错/非 mapping 都吞成 `meta=None`，区分不出 unparsable），也**不**复用 `check._check_frontmatter` 的 wiki 页字段规则（skill 字段集不同：只关心 name/description）。
- `name.invalid`：`name` 缺失 / 非 kebab-case（`^[a-z0-9]+(-[a-z0-9]+)*$`）/ 与目录名不符 / >64 / **取了保留子命令字 `validate`**（见 §3）；或 `description` 缺失 / >1024 / 含 `<>`。
- `wikilink.foreign`（**allowlist 拦，非 denylist**）：任意 `[[…]]` 只要目标不是 `references/<slug>` 即报——含最常见的**裸 `[[某概念]]`**、以及 `[[concepts/…]]`/`[[sources/…]]`/`[[entities/…]]`/`[[syntheses/…]]`（它们装到消费端都是死链）。**只有** `[[references/<slug>]]` 放行。
- `reference.missing`：目标是 `references/<slug>`、但 `references/<slug>.md` 不存在。
- **`<slug>` 口径（安全相关、须钉死）**：放行的 `[[references/<slug>]]` 里 `<slug>` 用与 `name` 同一套 kebab-case 正则（`^[a-z0-9]+(-[a-z0-9]+)*$`），**不得含 `/`、`.`、`|`、`#`**——故 `[[references/../x]]`（路径穿越）、`[[references/a/b]]`（嵌套）、`[[references/foo|label]]`（别名修饰）、`[[references/foo#bar]]`（锚点）一律**不放行 → `wikilink.foreign`**；放行项精确对应 `references/<slug>.md`，validator 不被路径穿越/wikilink 修饰语带偏。

**warning 级（只打印、不阻断；`--strict` 才计入 `EXIT_LINT_FINDINGS`，与 health/lint 一致）**：
- `skill.too_large`（SKILL.md >50 KiB）/ `description.thin`（<20 字符）/ `reference.too_large`（>100 KiB）。`scripts/` 仅作存在提示 + 大小限制，**不做依赖/import 审计**（非 stdlib import 判定涉 AST/相对导入/版本差异，与"结构草稿"关系弱，留到真要分发/安装时再做）。

**JSON 契约**：`distill.report_dict(findings)`（仿 `pages.report_dict` 归口，每条带 `severity`），CLI `--json` 与未来 Web/MCP 共用一口径。

## 6. 蒸馏工作法（skill 工作流，薄 PROMPT）

`DISTILL_PROMPT` 薄（仿 `QUERY_PROMPT`/`HEAL_PROMPT`），真步骤写进 `guanlan-wiki` skill 工作流（**Option A**：扩既有维护引擎，不新发布 `guanlan-distill` skill——避免改 `runtime.py`/`skill.py`/force-include 的安装·激活面；posture 隔离的专用 skill 留 P6.x）。

**召回由 wrapper 确定性完成**（决策见 D2）：编排壳先调 `search.search_pages()`（P5.0 零-LLM）算好候选页列表、写进 prompt，Agent 只读这些页 + `wiki/index.md` 摸清范围再蒸馏。CLI `run_agent_task` 子进程**没有** Web 内嵌 chat 的 `guanlan_search` host tool 注入面，故 P6 **不依赖任何 host tool / 不嵌套 `query_wiki` LLM 调用**——召回入口收敛到这一条最短路线；`--dry-run` 复用同一候选列表、只打印不触 Agentao。

> **性能边界**：`search_pages()` 是 P5.0 的**冷算入口**——会建 corpus + 算反链图、全库冷扫一次（`guanlan/search.py`）。P6 **接受这一次冷扫、复用 CLI 冷算口径，不引入 `CorpusCache` 或后台预热**——distill 是显式命令、非长驻进程，长驻侧的缓存/预热优化（P5.1/P5.4 Web·MCP 那侧的事）不在 P6 范围；`--dry-run` 同样吃这次冷扫。

要点：
1. 读 prompt 里 wrapper 已注入的候选页 + `wiki/index.md` 摸清范围（召回已在 wrapper 侧确定性完成，见上）。
2. 蒸馏出 **5–12 条实践者决策规则**（"遇 X 优先 Y 而非 Z——因为…"），而非主题罗列。
3. 写 `dist/skills/<name>/SKILL.md`：frontmatter(name/description) + 何时用 + 决策规则 + 方法 + References + 已知缺口；深度材料拆进 `references/<slug>.md`。
4. **句法自洽 + 尽量语义自洽**：引用只用 `[[references/<slug>]]`，绝不写指向本库 wiki 的外链；正文提到的概念要么拉进 `references/`、要么就地讲清（语义悬空 validator 查不出，靠 Agent 自觉）。
5. **只写 `dist/`、永不改 `wiki/`/`raw/`**；不自跑校验（wrapper 强制校验）。

蒸馏 ≈ query 的"读相关页 + 综合"再加"写成自洽 skill 结构"——Agent 已会两者，distill 只是把它定向到"产出草稿"。

## 7. marketplace 收录 / 分发（未来 feature，本次不实现）

原设计的第三件「host 确定性写」：零-LLM 扫 `dist/skills/*/` 重生 `<kb>/.claude-plugin/marketplace.json`（仿 `reindex`）+ `distill marketplace` 显式重生子命令 + 「一行可装」分发链路。**整体推迟**，理由：

- **它是 P6 唯一的"未验证价值出口"**：marketplace manifest schema（顶层 `name`/`owner`/`metadata`、plugin 的 `source`/`skills` 等）在 OpenKB 与本草案里**都是逆向猜**，没人拿真实 Claude Code 插件规范核对过；「一行可装」未端到端测。把这块未验证依赖留在 P6 会拖累整个里程碑的可信度。
- **P6 核心价值不依赖它**：自洽草稿落 `dist/skills/<name>/` 本身就是交付物，可人审、手动复制安装。收录成 marketplace 只是"最后一公里分发"，与草稿质量正交。

**整块移到 backlog（`P6-MKT`），与 P6 排期完全解耦——不作为 P6 前置，连 schema 调研也不放进 P6 关键路径。** 真要做时第一步是调研、非编码：查官方插件文档把 manifest 字段钉死（顶层 vs plugin 层、`source` 语义、安装命令真实形态），端到端验证"推 GitHub → 别的 agent 装上并触发"。OpenKB `skill/marketplace.py` 是形状起点但非已验证规范。

**两条硬依赖**：
1. **P6.1 eval 门控 P6 的定位措辞**——P6 单独发布只能称"草稿"。
2. **P6.2 覆盖依赖 staging/rollback**——故 P6 第一版干脆不提供 `--overwrite`。

## 10. 风险

- **🔴 蒸馏质量难自动判**：validator 只保结构，保不了"真专家 vs 自信胡说"——需 P6.1 eval。缓解：产物只称"草稿"，`--help`/回执显式写"未经 eval、需人审"。
- **🔴「自洽」只是句法、非语义**：validator 不查正文语义完整。一期不解（语义判定本身要 LLM，归 P6.1 coverage），但**必须如实标、不得宣称语义自洽**。
- **wiki/config 只用轻量只读快照**：不复刻 raw 门禁的防绕过严苛口径——够用，因 Agent 是受 prompt 约束的合作方、非对抗者（raw 那条仍走完整 `gate.snapshot_raw`，不打折）。
- **门禁只护 `raw/`+`wiki/`+两个 config，非全库**：其余目录靠 prompt 约束。须在文档/`--help` 如实标，不 over-claim"只写 dist/"。
- **Option A 混 posture**：维护与蒸馏两种工作法并存一份 SKILL.md。缓解：distill 是显式命令、`DISTILL_PROMPT` 钉死 posture、工作法分节；`conventions.md` 记一笔"`dist/` 是 distill 产物区、勿手当 wiki 维护"。

## 11. 测试要点（仿 `tests/test_convert.py` + `tests/test_check.py`）

- **`validate()` 分级（纯单测、无需 Agentao）**：缺 SKILL.md→`skill.missing` / 缺 frontmatter·YAML 错·非 mapping→`frontmatter.unparsable`（严格档，非 `load_page` 容错）/ 裸 `[[某概念]]` 与 `[[concepts/x]]` **均**→`wikilink.foreign`（allowlist）、`[[references/y]]` 放行、但 `[[references/../x]]`·`[[references/a/b]]`·`[[references/foo|l]]`·`[[references/foo#z]]` **均**→`wikilink.foreign`（slug 不含 `/.|#`、挡路径穿越/修饰语）/ `[[references/y]]` 无 `references/y.md`→`reference.missing` / name 非 kebab·与目录不符·取保留字 `validate`→`name.invalid`（均 error）/ description<20→`description.thin`、超大文件→warning / 全合规→空 findings·`EXIT_OK`；任一 error→`EXIT_LINT_FINDINGS`，warning 默认不计、`--strict` 才计；`report_dict` 含 `severity` 稳定。
- **主命令阻断（mock runner）**：产物含 `wikilink.foreign`（或缺 SKILL.md）→`EXIT_LINT_FINDINGS`、产物留盘；全合规→`EXIT_OK`、草稿落 `dist/skills/<name>/`。
- **posture（mock runner）**：只写 `dist/`→通过、validate 自动跑 / 改 `raw/`→`EXIT_RAW_MUTATED`(4) / 改 `wiki/` 或 config→`EXIT_CHECK_FAILED`(3)（断言轻量只读快照 diff 非空被拦）/ 空召回→Agent 产"已知缺口"占位、validate 不崩。
- **快照分工**：`gate.py` 字节不变（distill 只调用、不改）；改 `raw/`→`gate.diff_raw` 检出→`EXIT_RAW_MUTATED`；改 `wiki/` 或 config→轻量只读快照检出→`EXIT_CHECK_FAILED`（symlink/特殊文件防绕过仍由 `gate.snapshot_raw` 负责，不在 distill 重测）。
- **不覆盖**：`dist/skills/<n>/` 已存在→`EXIT_USAGE` 零落盘、不触 runner。
- **召回（wrapper 确定性）**：`search_pages()` 候选注入 prompt（mock 校验候选写进 DISTILL_PROMPT）。
- **`--dry-run`**：复用同一候选列表、只打印、不触 runner、必 `EXIT_OK`、不写 `dist/`。
- **`validate` 只读零副作用**：跑前后 `dist/` 字节不变；坏产物返非零但不动盘。
- **端到端（mock runner）**：`distill my-skill "意图"`→产物落盘、validate 通过、`EXIT_OK`；`wiki/`+`raw/`+config 写后快照零改动。
