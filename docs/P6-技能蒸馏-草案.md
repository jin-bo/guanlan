# P6 — 技能蒸馏（distill）落地小设计 ·【草案 / DRAFT，含 1 项未决】

> 本文是 `../llm-wiki/OpenKB` 反向评审后的产物：把 OpenKB **Skill Factory**（`openkb skill new <name> "<intent>"` —— 把 wiki 子集蒸馏成一个**可分发的 Anthropic Skill**）的**形状**，落进 guanlan 既有的「Agentao+skill 跑 LLM 工作流 / wrapper 只做零-LLM 校验与 host 确定性写」骨架。**尚未实现，仅供评审**。
>
> 缘起：CLAUDE.md 现状行成文写着——「no further roadmap spec remains unimplemented」。P4/P5 把"建库—维护—检索—宿主"四段跑通后，**生成层是 Karpathy 那条线尚未落地的终点**：知识复利到一定密度后，应能**蒸馏成一个别的 agent/人可直接安装的专家**（"drop in a book, out comes an expert"）。本草案开**新里程碑 P6「蒸馏/生成层」**，先只跑通 **skill 蒸馏**一个 worked example（仿 P3.7 只跑通 source-drift），deck/report 等其余生成器留 P6.x。
>
> **⚠️ 未决（OPEN，需拍板才能进实现）**：蒸馏工作法**住哪个 skill**——扩既有 `guanlan-wiki` 维护引擎加一个 distill 工作流（Option A，本草案暂按 A 行文）还是新发布一个专用 `guanlan-distill` skill（Option B）。这是全文**唯一**未定项，详见 **未决P6-1**（§9）；已登记 backlog 跟踪：[`backlog/notes/p6-技能蒸馏-工作法归属未决.md`](backlog/notes/p6-技能蒸馏-工作法归属未决.md)。除此点外，决策P6-2…7 均已定稿。

## 0. 一句话

`guanlan distill <name> "<intent>"`：一条**LLM 蒸馏（经 Agentao+skill）→ 零-LLM 结构校验 + host 确定性写 marketplace**的命令，读既有 `wiki/`（用 P5.0 确定性召回）把某主题蒸馏成一个**自洽、可分发**的 skill 落到 `<kb>/dist/skills/<name>/`，再零-LLM 校验其结构、重生 `<kb>/.claude-plugin/marketplace.json`。**精确复刻 guanlan 既有三件套**——LLM 工作流归 skill（同 ingest/query）、零-LLM 校验归 wrapper（同 check/lint）、host 确定性写归 wrapper（同 convert→raw / reindex→index）——只是把产物从"维护我的 wiki"换成"产出一个给别人的专家 skill"。

## 1. 目标与范围

### 为什么不照搬 OpenKB

OpenKB 的 Skill Factory 把蒸馏器做成一个 **in-process 的 OpenAI Agents SDK agent 循环**：`skill/creator.py:build_skill_create_agent` 手工装配工具（`list_wiki_dir`/`read_wiki_file`/`get_page_content`/`query_wiki`/`write_skill_file`）+ `max_turns=80` + 一份 `prompts/skill_create.md` 系统提示，直连 LiteLLM。这套**对 guanlan 有三处致命冲突**：

1. **它要在 wrapper 里重建一个 agent 循环 + 持 LLM client + 编码"怎么蒸馏"的业务智能** —— 三重违反 guanlan 红线（`guanlan/` 不携带业务智能 / 脚本不持 LLM client、API key / LLM 只经 Agentao 运行时，CLAUDE.md 架构 §1 + 不变量）。"怎么从一堆 concept 页蒸馏出实践者决策规则"本身就是语义活，本该归 LLM 工作流（skill），不该塞进确定性 wrapper。
2. **它的检索用 `query_wiki`（嵌套 LLM 调用）**。guanlan 已有 P5.0 零-LLM `guanlan search` / 宿主 `guanlan_search` 工具——召回该走**确定性入口**（同 `query.py:QUERY_PROMPT` 的传输中立召回措辞，决策P5.1-6），不嵌套二次 LLM。
3. **它 `skill new` / `recompile` 默认覆盖既有产物**（`recompile` 注释明说 "manual edits overwritten"），靠事后存 iteration 兜底。这与 guanlan 的写纪律相悖。

所以本设计**只借形状、不借实现**：蒸馏的**工作法**（survey→读 summaries→读 sources→蒸馏 5–12 条决策规则→写自洽 SKILL.md）落进一份 **skill 工作流**（§6），由 Agentao 跑；**wrapper 只出一条薄 PROMPT（仿 `QUERY_PROMPT`/`HEAL_PROMPT`）+ 一个受限 posture（§4）+ 零-LLM 结构 validator（§5）+ 零-LLM marketplace 重生（§7）**。这正是 guanlan 在 ingest/convert 已验证过的分工。

### 纳入 P6（本草案落地的）

- **`guanlan distill <name> "<intent>"`** 命令：LLM 蒸馏（受限 posture）→ 写后零-LLM 结构校验（自动，仿 convert 链式）→ host 确定性重生 marketplace。契约对齐 `convert`（默认不覆盖、`--overwrite`、`--dry-run`）。
- **产物形态**：`<kb>/dist/skills/<name>/{SKILL.md, references/*.md, (scripts/)}` —— 一个**自洽**（不含指向用户 wiki 的外链）、可被 Claude Code / Codex / Gemini 直接安装的 skill（§3）。
- **新零-LLM 模块 `guanlan/distill.py`**：结构 validator（§5）+ marketplace 重生（§7）+ 产物目录生命周期 + `distill_entrypoint`。**不持 LLM client、不调 Agentao**——纯确定性脚本（同 `check.py`/`reindex.py`）。
- **受限写 posture（拱心石，§4）**：蒸馏期 `dist/` 可写、`wiki/`+`raw/` **只读**——复用 `gate.snapshot_raw` 的快照-比对机制证明二者零改动。

### 不纳入 P6（同框架兄弟，留给 P6.x）

- **触发准确率 eval（P6.1）**：generator-LLM 造 should-trigger/should-not 测例 + grader-LLM 打分 + body coverage 复核（OpenKB `skill/evaluator.py` 是现成形状）。它要再起一条**独立 LLM 路径**，与 distill 同框架但单列命令 `distill eval <name>`。**高价值**（验证 description 到底触不触发），但非蒸馏闭环必需，跑通 distill 之后接入。
- **history/rollback 工作区（P6.2）**：覆盖前存 `iteration-N/` + 结构 diff（OpenKB `skill/workspace.py`）。**零-LLM、确定性**，是加法；但 P6 核心默认**不覆盖**（决策P6-5），history 是"允许覆盖后"才需要的安全网，故随 `--overwrite` 的常态化一起推迟。
- **deck / 其他生成器 + Generator 抽象（P6.3）**：OpenKB 把 skill 与 deck 抽象成统一 `Generator`。guanlan**先只跑通 skill 蒸馏一个 worked example**，**不预建抽象**——第二个生成器出现前，抽象是负债（仿 P3.7 §1 只跑 source-drift、不预建多疑点框架）。
- 不引入 in-process agent 循环、不持 LLM client、不用嵌套 `query_wiki` 检索、不引入 LiteLLM 多 provider（绑 Agentao）。
- 不新增退出码、不碰 `check`/`raw/` 门禁语义、不动既有 `wiki/`。

## 2. 架构对齐（复用既有"三件套"，不另起炉灶）

```
        既有 ingest / convert                     本草案 P6 distill
  ┌───────────────────────────┐        ┌───────────────────────────────┐
  │ LLM 工作流经 Agentao+skill  │   ⇒    │ LLM 工作流经 Agentao+skill      │
  │ ingest: raw → wiki          │        │ distill: wiki → dist/skills/<n> │
  │   (run_guarded_write)       │        │   (run_agent_task + 受限 posture)│
  └─────────────┬─────────────┘        └─────────────┬─────────────────┘
  ┌─────────────┴─────────────┐        ┌─────────────┴─────────────────┐
  │ 写后零-LLM 校验：check 门禁  │   ⇒    │ 写后零-LLM 校验：distill validate│
  │ (raw 快照 + frontmatter)    │        │ (skill 结构 + 禁外链 + 大小)     │
  └─────────────┬─────────────┘        └─────────────┬─────────────────┘
  ┌─────────────┴─────────────┐        ┌─────────────┴─────────────────┐
  │ host 确定性写              │   ⇒    │ host 确定性写                  │
  │ convert→raw / reindex→index │        │ distill→.claude-plugin/market…  │
  └───────────────────────────┘        └───────────────────────────────┘
```

逐一对齐：

- **LLM 工作流归 skill、薄 PROMPT 归 wrapper**：`DISTILL_PROMPT`（仿 `query.py:QUERY_PROMPT`、`heal.py:HEAL_PROMPT`，薄；真步骤在 skill 工作流）。Agent 经 `runtime.run_agent_task` 跑——**与 ingest/query 同一运行时入口、同一 model/runner 注入**。
- **零-LLM 校验归 wrapper**：`distill.validate(skill_dir)`（仿 `check.run_check`，返回 `Violation` 风格 findings）。distill **不复用 `run_guarded_write`**——那条是"写 `wiki/` + 跑 check"的门禁，而 distill **不写 `wiki/`**，写的是 `dist/`，故另起一条更轻的编排（§4），但**复用 `snapshot_raw` 证明 raw/+wiki/ 不变**。
- **host 确定性写有先例**：`reindex` 早已让 wrapper 确定性写 `wiki/index.md`（config 页），`convert` 让 wrapper 确定性写 `raw/`。distill 的 `marketplace.json` 是同性质的 host 写（决策P6-6），**不越"Agent 写内容、wrapper 写 config/派生"的既有线**——这正是它**不含 P3.7 那种未决**的原因（§4.4）。

## 3. 命令与产物形态

### 3.1 CLI 契约（逐项对齐 `convert`）

```
guanlan -C <kb> distill <name> "<intent>"      # 蒸馏 → dist/skills/<name>/（LLM 写 + 零-LLM 校验 + 重生 marketplace）
guanlan -C <kb> distill <name> "<intent>" --overwrite   # 显式覆盖既有同名产物（默认拒，仿 P5.2）
guanlan -C <kb> distill <name> "<intent>" --dry-run     # 零落盘：打印蒸馏计划/候选页，不触 Agentao、不写 dist/
guanlan -C <kb> distill validate <name>        # 零-LLM 重校验已蒸馏产物（--strict → EXIT_LINT_FINDINGS）
guanlan -C <kb> distill validate <name> --json # 稳定 JSON 契约（仿 check --json）
```

- 退出码：复用现有集合——`EXIT_OK` / `EXIT_AGENT_ERROR`（蒸馏运行失败）/ `EXIT_USAGE`（用法/IO，仿 `convert` 的 `except OSError → EXIT_USAGE`）/ `EXIT_LINT_FINDINGS`（`validate --strict` 有 findings，复用 health/lint 的 advisory 码）。**不新增退出码**（决策P6-7）。
- 默认**不覆盖**：`dist/skills/<name>/` 已存在且非 `--overwrite` → `EXIT_USAGE` 不动磁盘（仿 `convert` 决策P5.2）。
- 校验**默认 advisory 不阻断**：蒸馏完自动跑一遍 `validate` 并打印（仿 `convert --ingest` 链式 + ingest 后的门禁回执），但有 findings **不删产物、不返非零**（除非 `validate --strict`）——产物留盘供人审/修，与 check"wiki 改动留盘供修正后重跑"同精神。

### 3.2 产物：一个**自洽**的可分发 skill（决策P6-3）

```
<kb>/dist/skills/<name>/
├── SKILL.md              # frontmatter(name/description) + 何时用 + 决策规则 + 方法 + References + 已知缺口
├── references/           # 按需加载的深度材料
│   ├── <slug>.md
│   └── …
└── (scripts/)            # 可选；intent 隐含计算时才有，仅 stdlib
```

- **自洽是硬要求**：产物是给**别的机器/人**安装的，故**不得**含指向用户 wiki 的外链 `[[concepts/…]]`/`[[sources/…]]`/`[[entities/…]]`（在消费端是死链）——只允许 `[[references/<slug>]]` 指向随 skill 一并发布的文件。**这条由 §5 的零-LLM validator 强制**（OpenKB `skill/validator.py:FOREIGN_WIKILINK_RE` 同款检查）。
- `dist/` 是 `raw/`、`wiki/`、`graph/` 之外**新增的第四个顶层产物区**，专放**可分发生成物**。它是 `wiki/` 的**二阶 LLM 派生物**（§8 论证其与"markdown 唯一真相"不冲突）。

## 4. 拱心石：受限写 posture（蒸馏期 `dist/` 可写、`wiki/`+`raw/` 只读）

distill 的写姿态既不是 query 的纯只读、也不是 ingest 的"可写 wiki/+快照 raw/"，而是**第三种**：**读 `wiki/`+`raw/`、只写 `dist/`**。

### 4.1 编排（仿 `run_guarded_write_result`，但更轻）

```
snapshot_before = snapshot_raw(root)          # 复用 gate.snapshot_raw
wiki_before     = snapshot_subtree(root/"wiki")  # 同机制推广到 wiki/ 子树（见 4.2）
run_result = run_agent_task(DISTILL_PROMPT, working_directory=root,
                            permission_mode="workspace-write", model=…, runner=…)
# 写后裁决（零 LLM）：
#   raw/ 变  → 复用 EXIT_RAW_MUTATED（同 ingest 兜底）
#   wiki/ 变 → 蒸馏不该碰 wiki/，判失败（按门禁失败处理，不新增退出码）
#   都没变   → 跑 distill.validate(dist/skills/<name>)，advisory 打印
```

- Agent 仍以 `workspace-write` 起（那是唯一能写文件的 posture），但**门禁只放行 `dist/` 落地**：靠**写前/写后快照 `raw/`+`wiki/` 双子树**，任一改动即判违规（`raw/` 复用既有 `EXIT_RAW_MUTATED`；`wiki/` 漂移按门禁失败、复用现码）。这与 gate.py「快照 `raw/` 证明不可变」**同一招、同一份 `snapshot_raw`/`diff_raw` 机制**，只是把保护面从 `raw/` 推广到 `raw/`+`wiki/`。
- 蒸馏**绝不写 `wiki/`**——连 `log.md` 都不追（决策P6-4）。它是纯"读 wiki 产 dist"的生成器，不回灌用户 wiki。这让它的不变量极简：**唯一允许的磁盘增量是 `dist/skills/<name>/` 与 `marketplace.json`**。

### 4.2 `snapshot_subtree`：一行推广，不重写机制

`gate.snapshot_raw` 现在硬编码遍历 `root/"raw"`。本阶段把"遍历哪棵子树"抽成参数（`snapshot_subtree(dir)`），`snapshot_raw` 成 `snapshot_subtree(root/"raw")` 的别名——**零行为变化的纯重构**（仿 P5.2 把 raw-write 抽进 `rawio.py` 的零-behavior-change 重构，决策P5.2-6）。distill 用它同时盯 `raw/` 与 `wiki/`。符号链接/空目录/特殊文件的既有防绕过逻辑**原样继承**。

### 4.3 为什么这是只读 wiki（而非可写）

蒸馏是**从** wiki 提炼，不是**维护** wiki。让 Agent 在蒸馏途中顺手改 wiki 既无必要、又会把"生成"与"维护"两种 posture 混在一次调用里（OpenKB 的 `add` 会同时写 wiki 与产物，guanlan 有意分开）。只读 wiki 还让 distill 能在**任意时刻安全跑**（不与 ingest 的单写者抢锁、不改库状态、可在 CI/只读挂载上跑 `--dry-run`）。

### 4.4 它**没有** P3.7 那种"谁来写"的未决

P3.7 的 `raw_digest` 卡在"Agent 写还是 wrapper 写"（因 hash 是 Agent 算不准的、却要落进**内容**页）。distill **没有**这个张力：**产物正文是 LLM 蒸馏的散文，本就该 Agent 写**（用其文件工具写 `dist/`）；**wrapper 只在事后做零-LLM 校验 + 写 `marketplace.json`（config/派生，非内容）**——完全落在"Agent 写内容、wrapper 写 config/派生"的既有线内（同 reindex 写 index.md）。故 §9 的唯一未决是工作法**归属**（哪个 skill），不是写入主体。

## 5. 零-LLM 结构 validator（`guanlan/distill.py`，仿 `check.run_check`）

蒸馏产物的结构正确性**纯确定性可判**，故归零-LLM 脚本档（OpenKB `skill/validator.py` 的检查清单直接可借，**这部分实现也借**——它本就是零-LLM 正则/IO，不碰红线）。检查项（每条 → 一个 `Violation`-风格 finding，`kind` 见括号）：

- `SKILL.md` 存在（`skill.missing`）、大小 ≤ 上限（`skill.too_large`，OpenKB 取 50 KiB）。
- frontmatter 是合法 YAML mapping（`frontmatter.unparsable`，复用 `pages.load_page` 容错读）；`name` 必备、kebab-case、与目录名一致、≤64（`name.invalid`）；`description` 必备、≤1024、不含 `<>`（破坏解析器）、≥20 字符（`description.thin`，warning）。
- **禁外链**（`wikilink.foreign`）：正文/description 含 `[[concepts|sources|entities|summaries/…]]` → 阻断（消费端死链，§3.2 自洽硬要求）。
- `references/<slug>` 被 `[[references/…]]` 引用却不存在（`reference.missing`）；`references/*.md` 超大（`reference.too_large`，OpenKB 取 100 KiB，warning）。
- `scripts/*.py` 非 stdlib import（`script.nonstdlib`，warning，仅 `--strict` 计入）。

- **advisory 语义**：findings 默认只打印（`EXIT_OK`），`validate --strict` 才把 findings（含 warning）计入 `EXIT_LINT_FINDINGS`——与 `health`/`lint` 的 `--strict` 完全一致（决策P3-4 复用）。
- **JSON 契约**：`distill.report_dict(findings)`（仿 P4.10 抽出的 `pages.report_dict` 归口），CLI `--json` 与（未来）Web/MCP 共用一口径。

## 6. 蒸馏工作法（skill 工作流，薄 PROMPT）

`DISTILL_PROMPT`（仿 `QUERY_PROMPT`/`HEAL_PROMPT`，薄；真步骤在 skill）：

```
请按 `guanlan-wiki` skill 的 distill 工作流，把下述意图蒸馏成一个**自洽、可分发**的 skill：
意图：{intent}
产物目录：dist/skills/{name}/（**只写此目录**；wiki/ 与 raw/ 只读，禁改）
步骤：
 1) 用**可用的 search 入口**（宿主 `guanlan_search` 工具 / `guanlan search "<关键词>"` CLI）召回与意图相关的页，读这些 concept/entity/source 页 + wiki/index.md 摸清范围；
 2) 蒸馏出 5–12 条**实践者决策规则**（"遇到 X 优先 Y 而非 Z——因为…"），而非主题罗列；
 3) 写 dist/skills/{name}/SKILL.md：frontmatter(name={name} 的 kebab-case、description 一句话准确触发面) + 何时用 + 决策规则 + 方法 + References + 已知缺口；深度材料拆进 references/<slug>.md；
 4) **自洽**：引用只用 [[references/<slug>]]，**绝不**写 [[concepts/…]]/[[sources/…]] 等指向本库 wiki 的外链（消费端是死链）。
遵循 AGENTAO.md 硬约束。**只写 dist/，永不改 wiki/ 与 raw/。** 不要运行 shell 改文件、不要自跑校验（wrapper 会强制校验）。
完成后用一两句说明蒸馏了哪些页、产出几条决策规则。
```

- **召回走 P5.0 确定性入口**（传输中立措辞，复用 `QUERY_PROMPT` 的成熟表述）——**不嵌套 `query_wiki` LLM 调用**（决策P6-2，与 OpenKB 分道）。
- **不教 LLM 新把戏**：蒸馏 ≈ query 的"读相关页 + 综合"再加"写成自洽 skill 结构"，二者 Agent 都已会；distill 只是把它定向到"产出可分发专家"。
- 工作法**写进哪个 skill 见 未决P6-1**——Option A 加进 `guanlan-wiki` 的工作流清单；Option B 新建 `guanlan-distill` skill（须进 `BUNDLED_SKILL_NAMES` + pyproject force-include + 决定激活方式）。

## 7. marketplace 重生（零-LLM host 确定性写，仿 `reindex`）

蒸馏成功后，wrapper 扫 `dist/skills/*/SKILL.md`、确定性重生 `<kb>/.claude-plugin/marketplace.json`（OpenKB `skill/marketplace.py` 形状直接可借）：

- 字段：`name`（库名）/`owner`（`git config user.name|email`，缺则降级占位）/`plugins[].skills`（`./dist/skills/<n>` 列表）/`version`（取 `SCHEMA.md` 或缺省 `0.1.0`）。
- **纯确定性、可幂等重建**：它**完全**从 `dist/skills/*/` 扫描而来，是不折不扣的"派生索引"（§8）——删了重跑一字不差。这正是它能由 wrapper host 写、无需 Agent 的理由（同 reindex 从磁盘页重建 index.md）。
- 让用户的整个 KB 一行可装（`npx skills add <org>/<repo>` / `/plugin marketplace add`）。**分发到 GitHub 的步骤是文档，不是代码。**

## 8. 不变量 / 写后回执

- **markdown 唯一真相**：`dist/skills/<name>/SKILL.md` 是 `wiki/` 的**二阶 LLM 派生物**——与 `wiki/` 本身同性质（`wiki/` 就是 `raw/` 的一阶 LLM 派生物，非确定性可重建，却是 guanlan 公认的"内容真相"）。它是**生成物/产物**，不是需独立重建的索引，故不要求确定性可重建（同 wiki/ 页不要求）。**真正的派生索引 `marketplace.json` 则严格可幂等重建**（§7）。不变量不破。
- **零-LLM 脚本 vs LLM 工作流**：蒸馏（语义）全经 Agentao；validator + marketplace + 目录生命周期（结构/IO）零 LLM。`guanlan/distill.py` **不持 LLM client**。红线全守。
- **raw/ 不可变 + wiki/ 蒸馏期只读**：复用 `snapshot_raw`/`snapshot_subtree` 双子树快照证明（§4）。
- **写后确定性回执**（零 LLM 可验证）：蒸馏后 wrapper 跑 `validate` + 确认 `dist/skills/<name>/SKILL.md` 真落盘（仿 `convert`/heal 的写后回执，不靠 LLM 自述"我写了"）；raw/+wiki/ 快照比对回执"二者零改动"。
- **幂等 / 不覆盖**：默认 `dist/skills/<name>/` 存在即拒（决策P6-5）；`--overwrite` 才整盘替换该 name 目录（仿 P5.2.1 `--overwrite` 整盘换图目录）。

## 9. 决策记录

- **决策P6-1**：开**新里程碑 P6「蒸馏/生成层」**，先只跑通 **skill 蒸馏**一个 worked example。理由：它是 Karpathy 线尚未落地的终点、且 guanlan 已有 skill 安装基建（`skill.py`）+ Agentao + P5.0 召回，落地阻力最小；deck/Generator 抽象等第二个生成器再说（§1 不纳入）。
- **决策P6-2**：召回走 P5.0 零-LLM 入口（`guanlan_search`/`search`，传输中立措辞），**不嵌套 `query_wiki` LLM 调用**。与 OpenKB `creator.py` 分道，省一层 LLM 成本、对 CJK 无偏。
- **决策P6-3**：产物**自洽**——禁指向本库 wiki 的外链，只许 `[[references/<slug>]]`；由 §5 零-LLM validator 强制（借 OpenKB `FOREIGN_WIKILINK_RE`）。理由：产物是给消费端安装的，外链是死链。
- **决策P6-4**：蒸馏**只写 `dist/`、绝不改 `wiki/`/`raw/`**（连 `log.md` 都不追）。posture = `workspace-write` 起、但靠 `raw/`+`wiki/` 双子树快照把落地面**硬限在 `dist/`**（§4）。理由：蒸馏是"读 wiki 产 dist"的生成器，分开"生成"与"维护"两种 posture。
- **决策P6-5**：默认**不覆盖**既有产物（仿 `convert` 决策P5.2），`--overwrite` 才整盘替换。history/rollback 随 `--overwrite` 常态化一起留 P6.2（§11）。
- **决策P6-6**：`marketplace.json` 由 **wrapper 零-LLM 确定性重生**（从 `dist/skills/*/` 扫描）。理由：它是不折不扣的派生索引，落在"wrapper 写 config/派生"既有线内（同 reindex 写 index.md），**故本阶段无 P3.7 式"谁来写"未决**（§4.4）。
- **决策P6-7**：复用 `run_agent_task` 运行时 + `snapshot_raw`/`snapshot_subtree` 快照 + 现有退出码集合（`EXIT_OK`/`EXIT_AGENT_ERROR`/`EXIT_USAGE`/`EXIT_LINT_FINDINGS`），**不新增退出码、不复用 `run_guarded_write`**（distill 不写 wiki/、不跑 check 门禁，是更轻的另一条编排）。
  - **🔴 未决P6-1（OPEN，需拍板才能进实现）**：蒸馏工作法**住哪个 skill**？
    - **Option A（本草案暂按此行文）——扩既有 `guanlan-wiki` 维护引擎**，加一个 `distill` 工作流。优：单一 skill、不增安装/可发现性面（`SKILL_NAME=guanlan-wiki` 的强激活门禁原样可用）。缺：把"维护我的 wiki"（写 wiki、就地标矛盾）与"产出给别人的自洽 skill"（只读 wiki、禁外链）两种**相反 posture** 的指令塞进一份 SKILL.md，可能扰乱 Agent 判断。
    - **Option B——新发布专用 `guanlan-distill` skill**。优：posture 干净隔离（OpenKB 正是用专门的 `prompts/skill_create.md`）。缺：`BUNDLED_SKILL_NAMES` + pyproject `force-include`（`skills/guanlan-distill → guanlan/_skill/guanlan-distill`）都要加，且须决定激活方式（构造期强激活 like guanlan-wiki，还是按需 like pdf-to-markdown——distill 是显式命令触发，倾向后者）。
    - 倾向：**B（posture 隔离，长期更干净）**，但因其扩了安装面 + 含一项激活方式子决策，**留作未决待你拍板**；定了再把 §6 的"按 `guanlan-wiki` skill"措辞与 §1 纳入清单收敛到所选项。

## 10. 测试要点（仿 `tests/test_convert.py` + `tests/test_check.py`）

- **`distill.validate()`（零 LLM，纯单测，无需 Agentao）**：缺 SKILL.md → `skill.missing` / 含 `[[concepts/x]]` → `wikilink.foreign` 阻断 / `[[references/y]]` 但无 `references/y.md` → `reference.missing` / name 非 kebab 或与目录不符 → `name.invalid` / description 超 1024 含 `<>` → 阻断、<20 → `description.thin` warning / 超大 SKILL.md/reference → 对应 finding / 全合规 → 空 findings、`EXIT_OK`；`--strict` 把 warning 计入 `EXIT_LINT_FINDINGS`；`report_dict` 字段稳定。
- **posture（mock runner，仿 test_convert 的 mock skill backend）**：Agent 只写 `dist/` → 通过、validate 自动跑 / Agent 试改 `raw/` → 复用 `EXIT_RAW_MUTATED` / Agent 试改 `wiki/` → 判门禁失败（断言 wiki 子树快照 diff 非空被拦）/ 空召回（库无相关页）→ Agent 产"已知缺口"占位、validate 不崩。
- **`snapshot_subtree` 纯重构**：`snapshot_raw(root)` == `snapshot_subtree(root/"raw")` 逐字节（防回归 gate 既有行为）；符号链接/空目录/特殊文件防绕过用例对 `wiki/` 子树同样成立。
- **不覆盖 / overwrite**：`dist/skills/<n>/` 已存在且无 `--overwrite` → `EXIT_USAGE` 零落盘 / `--overwrite` → 整盘替换该 name 目录、别的 name 目录一字不动。
- **`--dry-run` 纯读零 LLM**：不触 runner、必 `EXIT_OK`、不写 `dist/`、不重生 marketplace。
- **marketplace 重生（零 LLM）**：两个产物 → `plugins[].skills` 含两条、按 name 升序稳定；删一个产物重跑 → 该条消失（幂等可重建）；无 git config → owner 降级占位、不崩。
- **端到端（mock runner）**：`distill X "意图"` → 产物落 `dist/skills/X/`、validate 通过、marketplace 含 X；`wiki/`+`raw/` 写后快照零改动。

## 11. 风险

- **蒸馏质量难自动判**：零-LLM validator 只保**结构**正确（有 frontmatter、无外链、不超大），保不了**内容**是不是"真专家规则"还是"主题罗列"——那需 P6.1 的 eval（trigger + coverage 双档 LLM 打分）。缓解：P6 核心如实只承诺"结构自洽 + 可安装"，质量判断显式标为 P6.1 范围；`--help`/文档讲清，避免"validate 过了就是好 skill"的错觉（同 P3.7 §11.1.0 显式划边界的姿态）。
- **`dist/` 是新顶层产物区**：用户 KB 多一个目录、多一个 `.claude-plugin/marketplace.json`。但二者都**只在用户跑 `distill` 后才出现**（opt-in），不污染纯 ingest/query 用户；且都是可删可重建的产物（marketplace 严格可重建，dist/ 重跑可再生）。
- **posture 推广触碰 gate 核心**：把 `snapshot_raw` 抽成 `snapshot_subtree` 是零-behavior-change 重构，但它动的是 P2 门禁最敏感的一段码——须以"`snapshot_raw` 逐字节等价"的回归测试钉死（§10），比纯加法 CLI 重一点，是它该单独评审的原因。
- **工作法归属是未决项**（🔴 未决P6-1）：Option B 扩安装面 + 含激活方式子决策；Option A 混 posture。无论哪个，都需 `conventions.md`/`examples/SCHEMA.md` 记一笔"`dist/` 是 distill 产物区、勿手当 wiki 维护"。这是它排在 §0 "唯一需先拍板"的原因。

## 12. 后续工作（P6 之后，同框架兄弟）

### 12.1 触发准确率 + body coverage eval（P6.1 — 高价值兄弟，独立 LLM 命令）
`distill eval <name>`：generator-LLM 据 skill **全文**造 should-trigger/should-not 测例 → grader-LLM 仅看 description 判触发（trigger accuracy）+ 仅看 body 判是否兑现（coverage）。OpenKB `skill/evaluator.py` 是现成形状（含 `asyncio.Semaphore` 限并发、`return_exceptions` 容错、按错误剔除分母）。它**对 guanlan 自己的 `guanlan-wiki` skill 描述调优也直接有用**，是 P6 之后最值得接的一项。

### 12.2 history / rollback 工作区（P6.2 — 零-LLM，随 `--overwrite` 常态化）
覆盖前存 `dist/skills/<name>-workspace/iteration-N/` + 结构 diff（description 变更/文件增删/SKILL.md 行数 delta），`distill rollback <name> --to N` 还原。OpenKB `skill/workspace.py` 形状可借，纯零-LLM 确定性。

### 12.3 deck / report 生成器 + Generator 抽象（P6.3 — 等第二个实例再抽象）
出现第二个生成器（deck/report/podcast）时，再把"读 wiki → 受限 posture 产物 → 零-LLM 校验 → host 收尾"抽成统一编排（OpenKB `skill/generator.py` 的 `Generator` 抽象 + skill 自带 `od.mode`/`output_path_template` frontmatter 契约）。**P6 不预建**——单实例时抽象是负债。
