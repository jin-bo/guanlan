# P3.6 图论桥与割点 `bridge / cut-vertex`（零 LLM 确定性割边/割点）

> 半阶段增强，不是新里程碑（P5 仍 = 多格式+检索）。接着 P3.5 把 `graph` 这块「维护仪表」再深一层：
> 在 P3.5 已建好的 `undirected_adjacency` 上算**确定性图论割边（bridge）与割点（cut vertex）**——
> 即「删之即断」的单点故障，富化 `graph.json`/`graph.html` 并新增两条 `lint` 拓扑建议。
> **纯零 LLM、零新依赖**，落在 P3「健康与图谱」主题内——是 P3.5 `thin_intercommunity_link` 的**正交补强**。

## 0. 与 P3.5 `thin_intercommunity_link` 的边界（关键）

P3.5 决策P3.5-13 已为本半阶段留门：

> 真正的删边连通性检测（确定性 BFS）若需要可另开，但属更重特性，本半阶段不做。

二者**问的是不同问题、正交、不去重**（决策P3.6-2）：

| | `thin_intercommunity_link`（P3.5） | `bridge_edge` / `cut_vertex`（P3.6） |
|---|---|---|
| 看什么 | **社区结构密度**：一对社区间仅靠单条直接互链 | **真连通性**：删掉该边/点后连通分量是否增加 |
| 判据 | 跨社区边计数 == 1 | 图论割边/割点（Tarjan low-link） |
| `A—B` 单边但 `A—C—B` 仍通 | **报**（跨社区引用稀疏） | **不报**（有旁路，删之不断） |
| 同社区内的命脉边 | 不报（不跨社区） | **报**（删之即断） |
| 语义 | "这两块联系太少，补交叉引用" | "这条线断了就有一整块知识从图谱脱钩" |

一条边可能**同时**是 `thin_intercommunity_link` 与 `bridge_edge`（两社区恰由唯一一条边相连，且该边是割边）——
此时两条 finding 都报，**不合并**：它们是两个独立诊断视角（决策P3.6-2）。一条边也可能只占其一
（同社区命脉边只报 bridge；有旁路的稀疏跨社区边只报 thin）。

## 3. 行为细则

### 3.1 确定性图论割边/割点（手写迭代式 Tarjan）

割边（bridge）= 删除后连通分量增加的边；割点（cut vertex / 关节点）= 删除后连通分量增加的节点。
标准 Tarjan 一趟 DFS 用发现序 `disc` 与回溯低点 `low` 同时判二者。P3.6 用**确定性 + 迭代**变体（决策P3.6-4）：

1. 在 `undirected_adjacency`（resolved、无自环、断链不算连通——与 P3.5 同口径，决策P3.6-1）上算；
2. **根按 node_id 升序**遍历各连通分量，每个节点**邻居按 id 升序**展开 → DFS 树唯一确定；
3. 用**显式栈**模拟递归（非 `sys.setrecursionlimit`）：`disc[u]`=发现序、`low[u]`=u 子树经至多一条回边可达的最小 `disc`；
4. 树边 `(u→v)`：`low[v] > disc[u]` ⇒ `(u,v)` 是**割边**；
5. 割点：非根 `u` 若存在子 `v` 使 `low[v] >= disc[u]` ⇒ `u` 是割点；根 `u` 若有 **≥2** 棵 DFS 子树 ⇒ 是割点；
6. **输出按 (u,v) / id 稳定排序**。`disc/low` 仅由结构与遍历序决定，遍历序确定 ⇒ 同图两次产出字节一致（§3.5）。

### 3.2 噪声控制：只报「保护非平凡一块」的割边/割点（决策P3.6-5）

裸图论割边/割点在**稀疏 wiki（近似树）上极吵**：一条链 `A–B–C–D` 每条边都是割边、每个内点都是割点；
一颗叶子页（度 1）由单边挂在主体上——那条边技术上是割边，但删掉只孤立**一张叶子页**，是 `orphan` 量级的小事，
不是值得预警的"命脉"。故 P3.6 复用 P3.5「小库不被噪声淹没」纪律（决策P3.5-8），加一道**最小保护规模**过滤：

- **统一判据**：删除该边/点后产生的诸连通分量里，**第二大的那块 ≥ 阈值**才报。
  - 割边：删后恰两块，尺寸 `s` 与 `comp−s`；第二大 == `min(s, comp−s)` == **较小一侧**。`≥ BRIDGE_MIN_SIDE(2)` 才报。
  - 割点：删后多块；**第二大块 ≥ CUT_MIN_COMPONENT(2)** 才报（最大块视作"主体"，第二大块即"被切出去的非平凡一块"）。
- 效果：
  - 链 `A–B–C–D`：仅中间边 `B–C`（两侧各 2）报割边；`A–B`/`C–D`（切出单叶）静默；无割点（切叶被压掉）。
  - 星形（枢纽 + 叶）：所有边切出单叶 → **零割边**；中心删后全是单叶 → **零割点**（该枢纽已由 P3.5 `hub_node` 覆盖）。
  - 两个团由单边相连：那条边报割边（两侧各 ≥2）、两端点报割点 —— 正是"命脉"场景。
  - 环（`A–B–C–D–A`）：无割边、无割点（恒有旁路）——与 `thin_intercommunity_link` 形成对照（决策P3.6-2）。

### 3.3 `graph.json` / `graph.html` 富化（additive，守决策P3-7）

**`graph.json`**（决策P3.6-6，镜像既有 `stats.orphans` 的"计数进 stats、明细不进 node/edge"先例）：

```jsonc
{ "generated_from": "wiki/",
  "stats": { "nodes": 312, "edges": 540, "broken": 7, "orphans": 18, "communities": 9,
             "bridges": 3, "cut_vertices": 5 },   // ← 新增两计数，接在 communities 之后
  "nodes": [ … ],   // 不变（不加 is_cut 标记）
  "edges": [ … ] }  // 不变（不加 is_bridge 标记）
```

> 为何不在 node/edge 上标 `is_bridge`/`is_cut`？`stats.orphans` 早已确立"少数特例只计数、明细留给 html/lint"的先例
> （node 字典从不带 `orphan` 标记，孤儿仅在 html `[孤儿]` 标注）。割边/割点同属少数特例，沿用同一归口最省回归面，
> 且不改既有 node/edge 字节序（决策P3.6-6）。社区号 `community` 进 node 是因其为**每节点固有属性**，性质不同。

**`graph.html`**（仍守决策P3-7：零 JS、零第三方库、纯静态列表，决策P3.6-7）：在 P3.5「拓扑提示」段尾部加两块——
**「割边（单点故障边）」**、**「割点（关节点）」**，确定性列出（与 lint 同数据，文字呈现）；顶部摘要行加 `· 割边 3 · 割点 5`。

### 3.4 两类拓扑 `Finding`（建议非门禁）

在 `run_lint` 已建好的 `g` 上、接在 P3.5 三类拓扑 finding 之后追加（保既有顺序）：

- **`lint.bridge_edge`**（割边 / 单点故障边）：**全局 finding**（`page=""`，跨两页，同 `thin_intercommunity_link` 体例）：
  `Finding("", "lint.bridge_edge", "[[u]]—[[v]] 是割边：删之 wiki 图谱即断为两块（单点故障），建议补冗余交叉引用")`。
- **`lint.cut_vertex`**（割点 / 关节点）：记在该页（`page=该页路径`，同 `hub_node` 体例）：
  `Finding(page, "lint.cut_vertex", "本页是割点：删之 wiki 图谱断为多块（关节点），建议为其邻接页补旁路链接")`。

阈值不达 / 图太小 / 近似树的叶边叶点经 §3.2 过滤后自然静默——小库不被噪声淹没（决策P3.6-5）。

### 3.5 确定性与幂等

固定根序（id 升序）+ 固定邻居展开序（id 升序）⇒ `disc/low` 唯一；所有输出按 `(u,v)`/`id` 稳定排序 ⇒ 同 wiki 两次：
割边/割点集一致、`graph.json` 字节一致、`graph.html` 字节一致、lint findings 顺序一致。与 `graph`/`lint` 既有幂等承诺同级。

## 4. 决策（沿用仓库 `决策PX-N` 体例）

- **决策P3.6-1（复用 P3.5 无向邻接单一归口）**：割边/割点在 `undirected_adjacency`（resolved、无自环、断链不算连通）上算，
  与社区/枢纽/孤岛同口径同归口，不二次建图（接续决策P3.5-3/9/10/11）。
- **决策P3.6-2（与 thin_intercommunity_link 正交、不去重）**：见 §0。前者问"社区间引用是否过稀"、后者问"删之是否真断"；
  `A–B` 单边但 `A–C–B` 仍通时只前者报、同社区命脉边只后者报、唯一跨社区且为割边时两者皆报——两个诊断视角，不合并。
- **决策P3.6-3（拓扑分析单一归口 `fragile_topology`）**：割边与割点是同一趟 Tarjan DFS 的产物，合为一个
  `NamedTuple` 返回、一次计算供 lint/graph/html 共用，不拆成两个各跑一趟 DFS 的函数。
- **决策P3.6-4（迭代式确定性 Tarjan、零依赖）**：显式栈非递归（避大库长链爆栈）；固定遍历序 + 输出稳定排序 + 无 RNG，
  守 `graph.json` 字节稳定；手写纯 Python，不引 networkx。
- **决策P3.6-5（最小保护规模过滤 + 常量化）**：`BRIDGE_MIN_SIDE`/`CUT_MIN_COMPONENT` 常量化（默认 2）；统一判据
  "删后第二大连通块 ≥ 阈值"——压掉近似树的叶边/叶点噪声，兑现「小库不被噪声淹没」（接续决策P3.5-8）。
- **决策P3.6-6（graph.json 只加计数、镜像 stats.orphans）**：见 §3.3。只增 `stats.bridges`/`stats.cut_vertices` 两计数，
  node/edge 字典一字不动；明细由 html 拓扑提示段 + `lint --json` 承载——与孤儿"只计数、明细在 html"同先例。
- **决策P3.6-7（graph.html 守决策P3-7）**：仍零 JS / 零第三方库 / 纯静态列表，只在 P3.5 拓扑提示段加两块文字注记，
  不引力导向布局。

## 6. 边界（明确不做）

- **不做 LLM 推断边**（接续决策P3.5-1，破坏 graph 可重建性，另开）；

## 7. 测试要点（扩 `tests/test_graphstats.py`，回归 `tests/test_graph.py`）

- **确定性**：同一 wiki 连跑两次，割边/割点集一致、`graph.json`/`graph.html` 字节一致。
- **割边正确性**：两团由单边相连 → 那条边报 `lint.bridge_edge`，两端点报 `lint.cut_vertex`；
  **环（`A–B–C–D–A`）零割边零割点**（证明是真图论判断、有旁路即不报，与 thin link 对照）。
- **链**：`A–B–C–D` 仅中间边 `B–C` 报割边（两侧各 2）；端边 `A–B`/`C–D`（切单叶）**不报**；无割点。
- **噪声过滤**：团 + 一张单边挂接的叶子页 → 那条挂接边**不报**割边（较小侧 = 1 < `BRIDGE_MIN_SIDE`）；叶子走 `orphan`/低度。
- **星形**：中心是 `hub_node`（P3.5）但**非** `cut_vertex`（删后全单叶，第二大块 = 1）；辐条边非割边。
- **正交不去重**：唯一跨社区且为割边的那条边，**同时**出 `thin_intercommunity_link` 与 `bridge_edge` 两条 finding（决策P3.6-2）。
- **additive 契约**：`graph.json` stats 新增 `bridges`/`cut_vertices` 计数，**既有键/顺序/排序与 node/edge 字典一字不动**
  （回归 test_graph 全绿）；orphan/broken/missing_entity 及 P3.5 三类 findings 一字不变。
- **`--strict`**：含割边/割点建议时退 `EXIT_LINT_FINDINGS(6)`；默认退 0。
- **小图/空图/单节点/无边**：不崩、不报噪声 finding；计数为 0。
- **大库不爆栈**：长链路（远超递归上限）下 `fragile_topology` 正常返回（验证迭代式实现，决策P3.6-4）。
- **自环/断链口径**：自链页（自环）、断链不参与连通——与 `undirected_adjacency` 同口径（接续决策P3.5-11）。
