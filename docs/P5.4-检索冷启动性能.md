# P5.4 检索冷启动性能：首搜 ~1 分钟，后续瞬时

**编号**：P5.4（DESIGN §7 P5 行 / §8 检索线已登记）。**状态**：**已实现**（候选① 启动预热 + 推荐(a) singleflight 构建锁）。属检索增强线（承 P5.0 内核 / P5.1 Web 接入 / P5.3 backlink 重排）的**性能**子线，与 cjk-retrieval-enhancements.md 的「检索质量」增强正交。

## 现象

Web 检索（`GET /api/search`）实测：**第一次检索耗时接近 1 分钟，后续几次都瞬间出结果**。

## 根因（确定，已核对代码）

`CorpusCache` 是**进程内 mtime memo，懒构建**：

- `create_app` 处 `app.state.search_cache = CorpusCache()` 建的是**空 cache**（`guanlan/web/app.py:330`，注释「即空、首搜按需重建」）。进程启动**不预热**。
- 首次 `/api/search` 一次性付清两笔全库扫描：
  1. `build_corpus(wiki)` — 全 content 页读盘 + 分词建 `DocBag`（较轻）。
  2. `CorpusCache.backlinks(...)` → `graph.build_graph` — **全库 `yaml.safe_load` + 别名索引 + 逐 body `[[…]]` 解析**（P5.3 决策P5.3-6 已点明 build_graph **比 build_corpus 重**，是「额外一次全库读」）。
- 2 次起：两笔都按 content 页 `(mtime_ns,size)` 署名 memo 命中（`search.py:478` / `525`），省掉「读盘+分词」和「`build_graph`」两笔重活 → 当前库规模下**观感瞬时**。

> ⚠️ **热路并非真·O(1)，仍有 O(N) 地板**（勿被「瞬时」误导，未来扩库时这是下一个瓶颈）：即便全 memo 命中，`corpus()` 每次仍 `iter_pages` + 全库 `stat`（`search.py:460`），`backlinks()` 每次仍按全 docs 聚合 O(N) 签名（`search.py:525`）。且 memo 是**两套不同粒度**：DocBag 是**逐页** memo（`corpus()` 只重建变更页）；但 backlink 是**整库签名**——任一 content 页改 → 签名变 → `build_graph` **全量**重算（决策P5.3-6「整库签名→全量即够」，**非**「只重建变更页」）。故「几次后瞬时」是**当前规模**的经验值，不是渐近承诺。

故「~1 分钟」= 冷算两笔全库扫描 + macOS 首读冷盘 I/O，全部压在**用户首搜的关键路径**上。

> ⚠️ 待测：1 分钟对当前库规模是否**过慢**（疑超出纯 corpus+graph 构建应有量级）。应先 profile 区分「真·扫描耗时」vs「进程/import 冷启动、首次 GIL/线程池预热、潜在 O(n²)」。不要盲目优化未测量的热点。

## 优化候选（由轻到重，均守 P5.0 stateless 边界：markdown 唯一真相、索引可重建、无盘上派生）

1. **启动预热（首选，最便宜）—— ✅ 已实现**：长驻进程（`guanlan web` / `guanlan mcp`）起服时起一个 daemon 后台线程跑一次 `search_cache.prewarm(wiki)`（内部 `corpus()`+`backlinks()`），把 1 分钟移出用户关键路径到启动期。零新派生物、零契约变更、零新 CLI 旗标。reader/单库、`workers=1` 下只预热挂载的那个库；预热失败由 `prewarm` 内部静默吞掉（不阻塞 serve、不污染 MCP stdout）。
   - ⚠️ **并发首搜会与预热重算撞车** —— **已按推荐 (a) 收口**：原 `corpus()`/`backlinks()` 的重活都在锁外跑（last-writer-wins、benign 但**不去重**），若用户在预热跑完前发起 `/api/search` 会**两边各跑一遍冷全扫**、I/O/CPU 翻倍。现补了一道**按库根分桶的 singleflight 构建锁**（`CorpusCache._build_locks`）：冷/变更构建（`build_doc` / `build_graph`）**只在有 miss 时入锁**串行，进锁后**复查**缓存——并发 builder（含预热线程）已填好的部分变 hit、本次不再重复 build，预热与首搜**经同一把锁天然合流**。hot path（全 memo 命中）**不入构建锁**，并发首搜互不阻塞。备选 (b)「warming 就绪态短路 503」/ (c)「显式承认撞车退化」均未采，因 (a) 同时收口了**预热竞态**与**P5.1 起就存在的并发首搜重复建图**，且不破 stateless 边界。
2. **先 profile 再动手（未做）**：若 1 分钟远超扫描应有量级，可能是 import/线程池/某处 O(n²)，预热只是搬运而非治本。先量。预热已把成本移出关键路径，profile 留作后续按需深挖。
3. **拆分 corpus 与 backlink 两笔（未做）**：首搜可只付 `build_corpus`（轻）先出 BM25 结果，backlink 先验**异步补算**、第二次起才重排。代价是首搜名次与后续不一致（破坏 P5.3「四处共享名次」承诺）。预热令此取舍多余——首搜已是热路。
4. **backlink 增量化（未做，E1 邻域）**：当前签名变更 → `build_graph` **全量**重算（P5.3 决策P5.3-6 明确「整库签名→全量即够」）。大库可考虑文件级 diff 增量，但 P5.3 已显式**不做** incremental/inverted-edge/persisted index。
5. **持久化索引（E1，最重，越界，未做）**：落盘倒排/向量索引消除冷启动，但**破坏 P5.0「无盘上派生」承诺**，须整体进 E1（向量/rerank/持久索引）一并设计，不在本线。

## 实现（已落）

- **`CorpusCache.prewarm(wiki) -> bool`**（`guanlan/search.py`）：一次性付清 `corpus()` 全库 `build_doc` + `backlinks()` 的 `build_graph`，幂等、只读 `wiki/`、零写盘、与首搜结果字节等价；**任何异常都吞掉**返回 `False`（优化非正确性前提）。
- **singleflight 构建锁**（决策P5.4-2）：`CorpusCache._build_locks: dict[root_posix, threading.Lock]` + `_build_lock(bucket)` 归口；`corpus()` 重构为「锁外算 entries → 分类 → 有 miss 才入构建锁、进锁复查再 build → 合并剪枝」，`backlinks()` 同理「签名 miss → 入构建锁、复查签名 → build_graph」。与护纯字典短临界区的 `self._lock` **分两把**，构建锁可长持、`self._lock` 永远短持即放，无嵌套等待、无死锁。`_classify(entries, snapshot)` 抽出锁前/锁内复查共用。
- **宿主接线**：Web `serve()` 在 `uvicorn.run` 前起 `_start_prewarm(app)` daemon 线程；MCP `serve_mcp()` 自建 `CorpusCache` + daemon 预热线程后注入 `build_mcp(search_cache=…)`。两宿主经**同一把** singleflight 锁与首搜合流。
- **决策**：
  - **P5.4-1**：预热只搬运冷算、不改语义（与冷算 `search_pages` 字节等价），失败静默降级为懒构建；无新 CLI 旗标/退出码/契约（零契约变更）。
  - **P5.4-2**：反转 P5.0-19「放弃双重-build 抑制」的权衡，补 singleflight 构建锁——但**只在有 miss 时入锁**，故 hot path 仍全程无构建锁、并发不互阻；同时治好预热竞态与既有并发首搜重复建图，副带把并发冷启的 `build_doc`/`build_graph` 从 N× 降到 1×。
- **测试**（`tests/test_search.py`）：预热后零 build / 预热≡冷算 / 预热幂等 / 预热容错不抛 / singleflight 并发冷启 build_graph 仅一次 + 名次≡冷算 / singleflight 并发 corpus 每页仅建一次。既有 51 检索用例与 Web/MCP/graph 全绿。
