## 1. 目标与范围

**P4.3 = 把 `guanlan heal`（缺失实体物化）放进浏览器：先零-LLM 预览本批将物化的高频缺页，再一键经写门禁物化、轮询看确定性回执——全程不离开浏览器、不旁路任何 P2/P3 门禁。**

现状（P4 + P4.1/P4.2）：浏览器能从 `raw/` 选篇 `ingest`、能投喂新源、能只读多轮问答，却**唯独缺 `heal`**——
要补高频缺页仍须回命令行跑 `guanlan heal`。P4.3 用一对端点补齐它：

- **`GET /api/heal/preview`**：零-LLM 算 worklist（== `heal --dry-run`），返回「本批将物化哪些目标、各被几页引用、引用页是谁、因 `--limit` 推迟了哪些」。**纯读、不入队、不触 Agentao**。
- **`POST /api/heal`**：把 heal 作为**新写作业 kind** 入 P4 既有单写者 FIFO `JobQueue`（提交方式**对标 ingest 的 `enqueue`、非投喂的 `submit_and_wait`**），即时返回 `job_id`，前端**轮询** `/api/jobs/{id}`。作业内调 heal 的结构化 core（走 P2 写门禁 + 有界自愈），完成后把 `HealResult` 经 `Job.result` 回传。

heal 在 Web 端**仍是写操作**：它经 `run_guarded_write_result` 走 P2 子进程 + `raw/` 快照门禁 + 有界自愈
（与 ingest 同一条写路径），故归到**写线**——单 worker FIFO 串行、轮询拿结果，与 P4 的 ingest 端点同构。
Web 层**零业务智能**：worklist、门禁、回执的结论全来自被复用的 `heal.py` core，宿主只「收 HTTP → 调既有
函数 → 序列化/转发」。

### 纳入 P4.3

- `GET /api/heal/preview?limit=&min_refs=` → `{worklist, postponed}`（零 LLM，复用 `heal.compute_worklist`，经 `to_thread` 卸载）。
- `POST /api/heal {limit?, min_refs?, model?}` → `{job_id}`（入单写者队列 `enqueue`，与 ingest / 投喂同 worker 串行）。
- **`Job` 加可选 `result` 字段**承载 heal 的**纯机器回执**（P3.2 §11 点名的那处）：`GET /api/jobs/{id}` 对 heal 作业多回一个 `result == heal_result_dict(heal_result, postponed)`（即 P3.2/P3.3 既有六字段 `{worklist, postponed, receipts, unexpected_writes, changed_paths, exit_code}`），ingest/投喂作业为 `null`。**agent 散文走 `job.output`、不进 `result`**（决策P4.3-1：`result` 只放机器回执、不掺 UI 展示补充）。
- **`heal.py` 加性重构**：抽出**不打印、返回结构化结果**的 core `run_heal_result(...) -> HealRun`（仿 P3.2 决策P3.2-13 把 `run_guarded_write` 抽出结果版的手法），`run_heal` 退化为「调 core → 渲染 → 返回退出码」的薄壳，**CLI 行为/字节不变**（含 dry-run / 空 worklist 的现有特判，§3.1）。
- **前端**：顶栏 `heal` 动作 → 浮层先拉 preview（可调 `--limit`/`--min-refs`）→「物化」按钮 → `POST /api/heal` → 轮询渲染回执（resolved / still_broken / 非预期写 / 推迟；agent 散文取自 `job.output`）。

### 不纳入 P4.3（仍按路线图推后）

- **细分 / 持久化跳过决策**（真·非实体每轮重入 worklist）→ P3.2 §11 / P3.3 §10 已列，属 core 能力、与 Web 无关，不在本相位。
- **自动 heal / nightly 富化** → DESIGN §7 E3，P5 之后；P4.3 仍是**用户显式按需触发**（点按钮才跑），不内联进 ingest、不定时。
- **heal 作业的 token 级流式（SSE）** → 不做：与写作业 ingest 同口径**只轮询**（决策P4-6：飞行中无流可推、轮询对单用户足够）。heal 的散文进度由 core 的 `↻` 自愈行经 worker 捕获进 `job.output`，回执经 `job.result`，无需 SSE。
- **客户端指定 target 列表** → **不做**（决策P4.3-3）：客户端只传 `limit`/`min_refs`/`model`，**绝不传目标页名**；worklist 由作业在服务端**重算**，杜绝陈旧/伪造目标越过确定性聚合。

## 2. 端点契约（`guanlan/web/app.py`）

| 方法 路径 | 入参 | 复用 | 返回 | 错误 |
|------|------|------|------|------|
| `GET /api/heal/preview` | `limit: int=10 (ge=1)`、`min_refs: int=2 (ge=1)`（query） | `heal.compute_worklist` | `{worklist:[{target,ref_count,ref_pages}], postponed:[…]}`（== `heal --dry-run --json` 体；零 LLM、`to_thread` 卸载） | `422` 参数越界（FastAPI `Query(ge=1)`，对齐 CLI `positive_int` 的「须 ≥ 1」） |
| `POST /api/heal` | `HealBody {limit:int=10(ge=1), min_refs:int=2(ge=1), model:str\|None=None}` | `enqueue` + `heal.run_heal_result` | `{job_id}`（**即时返回**，前端轮询；唯一写入口之一，与 ingest 并列） | `422` 体非法 |
| `GET /api/jobs/{id}` | — | 作业表 | `{id,kind,state,exit_code,output, result}`——**新增 `result`**：heal 作业为 `heal_result_dict(heal_result, postponed)`（六字段机器回执），ingest/投喂为 `null`；agent 散文在 `output` | `404` 未知 id（沿用 P4） |

- **默认值与 CLI 同源**：`limit` 默认 `heal.DEFAULT_LIMIT(=10)`、`min_refs` 默认 `lint.MISSING_ENTITY_MIN_REFS(=2)`——**直接 import 这两个常量**，不在 Web 端硬编码，避免与 CLI 漂移。
- **preview 是 read-only 读端点**：只算 worklist（`compute_worklist` → `missing_entities` → `build_graph`，纯读 `wiki/`），**不取 `raw/` 快照、不入队、零 LLM**，与 `/api/report/*` 同类——经 `anyio.to_thread.run_sync` 卸离事件循环（决策P4-2）。
- **`POST /api/heal` 是写端点**：入 §4 的单写者 `JobQueue`、**轮询**（决策P4-5/P4-6）。退出码透传进 `job.exit_code`（写路径四值 `0/4/3/5`，决策P3.2-5），前端据此显示徽标；**P4.3 不新增退出码**（同 P4）。
- **`result` 是 heal 专属、对 ingest 向后兼容**：`/api/jobs/{id}` 多回一个 `result` 键，ingest/投喂作业取值 `null`、前端忽略；heal 作业取 `heal_result_dict(heal_result, postponed)`（§3，复用 P3.2/P3.3 既有序列化器、不另造结构）。既有 ingest 轮询逻辑不受影响（只多了个它不读的键）。

## 3. 承载结构化结果：`heal.py` 加性重构 + `Job.result`——本设计的核心　**\[决策P4.3-1]**

ingest 作业的结果是**散文**（触及页面 + 门禁报告），`job.output` 捕获 stdout 文本即够（决策P4-7）。
**heal 不同**：它的价值在**结构化回执**——逐目标 `resolved`/`still_broken`、批次级 `unexpected_writes`、
`changed_paths`、推迟项。这些是 `HealResult` 的字段（P3.2 §5），**不能从 stdout 文本切片反推**（决策P3.2-10：
正确性判定全程不解析人读文案）。故 P4.3 要把结构化结果**原样**带回前端，需要两处加性改动：

### 3.1 `heal.py`：抽出不打印的 core（仿 P3.2 决策P3.2-13）

现 `run_heal(...)` 把 worklist 计算、写门禁、回执**与打印耦合**（`_print_human`/`_dumps` 直写 stdout），
作业线程拿不到干净的 `HealResult`。仿照 P3.2 当初把 `run_guarded_write` 抽出**结果版** `run_guarded_write_result`
的同一手法，把 heal 也抽一层：

```python
@dataclass(frozen=True)
class HealRun:                       # 一轮真实 heal 的结构化产物（CLI 渲染 + Web 回执共用的进程内对象）
    result: HealResult               # P3.2/P3.3 既有批次契约（receipts/unexpected_writes/changed_paths/exit_code）
    postponed: tuple[HealWorkItem, ...]   # 因 --limit 推迟项（worklist 级，HealResult 本就不含）
    had_batch: bool                  # False = 空 worklist 短路（未触 Agentao，决策P3.2-6），供 CLI 选空批次特判渲染
    final_text: str = ""             # agent 散文（来自 GuardedWriteResult.final_text）；空 worklist 短路时为 ""

    @property
    def exit_code(self) -> int:
        return self.result.exit_code


def run_heal_result(*, root, limit, min_refs, model, runner) -> HealRun:
    """heal 真实写路径的**不打印**编排核心：require_kb_root → compute_worklist →
    （空 batch 短路 EXIT_OK，had_batch=False）/（snapshot → run_guarded_write_result →
    _writeset → _build_receipts → HealResult）。**不碰 stdout、不处理 dry_run/json_output**。"""
    ...

def run_heal(*, root, limit, min_refs, model, dry_run, json_output, runner=None) -> int:
    """退化为薄壳，但**保留 dry_run 与空 worklist 的现有特判**（字节不变是硬约束）：
       dry_run → _print_preview（纯读预览，2 键 JSON / 人读）；
       否则 run = run_heal_result(...)；
         not run.had_batch → 现有空批次特判（2 键 `{worklist:[], postponed}` / 人读消息，用 run.postponed）；
         else → 现有真实 heal 渲染（_print_human(run.final_text, …) / _dumps(heal_result_dict(run.result, …)))。
       返回 run.exit_code。**CLI 行为/字节不变**（既有 test_heal 全程回归）。"""
```

- **`HealRun` 不带 `to_json_dict`、不掺 UI 字段（决策P4.3-1）**：Web 的 `result` 由端点直接用既有序列化器 `heal_result_dict(run.result, run.postponed)` 产出（六字段机器回执），**不在 `HealRun` 上挂 `final_text`/`skipped_llm` 等展示补充**——`result` 是「机器回执」单一职责，散文与短路语义另有去处（散文 → `job.output`，§3.2；短路 → 空 `receipts` + `exit_code=EXIT_OK` 已自明）。`final_text`/`had_batch` 只是**进程内** `HealRun` 字段，供 CLI 薄壳渲染与 thunk 取散文，不进任何 wire 契约。
- **序列化器升为公开、CLI/Web 共用（决策P4.3-1）**：现私有 `heal._result_dict(result, postponed)` **改名为公开 `heal.heal_result_dict(result, postponed)`**（入 `__all__`），CLI（`_dumps(heal_result_dict(...))`）与 Web 端点共用同一份 wire 契约——避免 `web/app.py` 跨模块调用 `heal.py` 的私有 helper。**纯重命名、不改结构、不改字段**（六字段不增不减），只是把既定契约显式化。
- **不要用 `HealRun` 统一渲染 CLI、以免空批次回归（点名风险，对应点 2）**：当前 `run_heal --json` 在**空 worklist** 时只输出 `{worklist, postponed}` **两键**（非六字段），dry-run 同理。故 CLI 薄壳**必须保留这两条特判**、**只对非空真实 heal** 走 `_dumps(heal_result_dict(...))` 的六字段渲染——**切勿**把空/dry-run 也塞进六字段统一渲染（那会把 CLI `--json` 从 2 键涨成 6 键，破坏「字节不变」）。**Web 的 `result` 与 CLI 空批次形态可以不同**：Web 一律返回 `heal_result_dict(run.result, run.postponed)`（空批次即空 `receipts` 的六字段），它是 **Web 自己的回执契约、不声称与 CLI 空批次对齐**；CLI 六字段契约（P3.3 §6）专指**非空真实 heal**，一字不增。
- **dry_run 不进 core**：preview 走 `compute_worklist`（§2），`run_heal_result` 只管真实写路径，无 `dry_run` 形参——职责更清。
- **`require_kb_root` 归属**：core 内仍调（`writable=True`）；Web 路径其实在 `serve` 时已校验过库根，但 core 保留校验做纵深防御。core 内若抛 `GuanlanError` 由作业 worker 归一为失败（§4），CLI 薄壳维持现有 try/except 打印行为。

### 3.2 `jobs.py`：`Job` 加可选 `result`，worker 鸭子分流（P3.2 §11 点名项）

```python
@dataclass
class Job:
    id: str; kind: str; state: JobState = "queued"
    exit_code: int | None = None
    output: str = ""
    result: object | None = None         # 新增：进程内结构化结果（heal 的 HealRun）；ingest/投喂为 None。端点经 heal_result_dict 序列化
    done_event: threading.Event = field(default_factory=threading.Event)

def _run(self):
    while True:
        job, fn = self._queue.get(); job.state = "running"; buf = io.StringIO()
        try:
            with contextlib.redirect_stdout(buf), contextlib.redirect_stderr(buf):
                outcome = fn()
            if isinstance(outcome, int):     # ingest/投喂：fn 返回退出码（口径不变）
                job.exit_code = outcome
            else:                            # heal：fn 返回带 .exit_code 的结构化结果（HealRun）
                job.result = outcome
                job.exit_code = outcome.exit_code
        except Exception as exc:             # worker 绝不死，异常归一失败（沿用）
            job.exit_code = EXIT_AGENT_ERROR; buf.write(f"\n作业执行异常：{type(exc).__name__}: {exc}")
        finally:
            job.output = buf.getvalue(); job.state = "done"; job.done_event.set()
```

- **`jobs.py` 仍域无关**：worker 只碰 `.exit_code`（鸭子类型）、把 `outcome` 整体存进 `job.result`，**不 import heal 类型**；`result` 的序列化（`heal_result_dict(job.result.result, job.result.postponed)`）落在 `/api/jobs/{id}` 端点（app.py），逻辑留在 `heal.py`。`enqueue`/`submit_and_wait`/`get_job` 签名不变。
- **对既有作业零回归**：ingest（`run_ingest` 返回 int）、投喂（`_atomic_write_raw` 返回 int）走 `isinstance int` 旧分支，`result` 恒为 `None`。只有 heal 作业的 thunk 返回 `HealRun` 走新分支。
- **app.py 的 heal 作业 thunk**（与 ingest 端点同构）：

```python
@app.post("/api/heal")
async def heal(body: HealBody) -> dict:
    def _job() -> HealRun:               # 返回结构化结果（非 int）→ worker 走新分支存 job.result
        run = run_heal_result(root=root, limit=body.limit, min_refs=body.min_refs,
                              model=body.model or model, runner=runner)
        if run.final_text:               # agent 散文 → worker 的 redirect 捕获进 job.output（与 ingest 同口径）
            print(run.final_text)
        return run                       # worker 取 run.exit_code、整体存 job.result（端点经 heal_result_dict 序列化）
    return {"job_id": jobs.enqueue("heal", _job)}   # enqueue（异步轮询），非 submit_and_wait

# /api/jobs/{id} 端点对 heal 作业序列化机器回执（ingest/投喂 result=null）：
#   result = heal_result_dict(job.result.result, job.result.postponed) if isinstance(job.result, HealRun) else None
```

- **职责分离（决策P4.3-1）**：`job.result`（HealRun）→ 端点经 `heal_result_dict` 序列化为**机器回执**；`job.output` 承载**人读散文**（thunk 打印的 `final_text` + core 内 `run_guarded_write_result` 走 stderr 的 `↻` 自愈进度，被 worker 一并捕获）——与 ingest 的 `job.output=散文` 同口径。两者各司其职，`result` 不掺展示文本。
- **`redirect_stdout` 红线不破**：`run_heal_result` core **本身不向 stdout 打印**（这正是抽 core 的目的，保结构化回执干净）；散文由 thunk 在 worker 的 redirect 块内显式 `print`，捕获仍只在单 worker 串行发生、无跨线程竞态（决策P4-5）。

> **为何不走「`run_heal(json_output=True)` 捕获 stdout 再 `json.loads`」**（被否的备选）：那把结构化结果
> 退化成「序列化→捕获文本→反序列化」的脆弱往返，且 `agent` 散文（`final_text`）在 `--json` 分支被丢；
> 还让 Web 正确性隐性依赖 stdout 文本格式（撞决策P3.2-10「不从人读文案反推状态」）。抽不打印的 core
> 返回真对象，是与 P3.2 `run_guarded_write_result` 一脉相承的正解。

## 4. 读写分线与单写者串行（复用 P4 §4.1/§4.2，不引新基础设施）

heal 在 P4 的读写分线里归**写**（它过写门禁、改 `wiki/`），故落点完全沿用 ingest 的写路径设施，**零新写基础设施**（P3.2 §3 决策P3.2-3 早已为此埋好：`JobQueue` 是通用的 `enqueue(kind, thunk)`）：

| | preview（`GET /api/heal/preview`） | 物化（`POST /api/heal`） |
|--|--|--|
| 性质 | 零 LLM 读 | LLM 写（过 P2 门禁） |
| 集成 | `compute_worklist`，`to_thread` 卸载 | `enqueue` 入单写者 FIFO worker，`run_heal_result` |
| `raw/` 快照 | 不取（纯读） | 由 `run_guarded_write_result` 内部取（与 ingest 同） |
| 并发 | 可并发（不写、无竞态） | 单 worker FIFO 串行（与 ingest / 投喂同 worker） |
| 拿结果 | 同步返回 | 即时返回 `job_id` → 轮询 `/api/jobs/{id}` |

- **单写者假设不破**：heal 与 ingest、投喂在**同一条 worker** 上 FIFO 串行——三类写作业永不并发，`raw/` 前后快照不被并发写互踩（决策P4-5）。这正是 P3.2 §10「与 ingest 写并发」风险在 Web 下的天然解（CLI 下无锁，Web 下单 worker 串行化）。
- **heal 排在前序写作业之后**：若提交时正有 ingest/投喂在飞，heal 作业 pending 到它跑完才轮到——与投喂同款排队语义（P4.1 §3.1），但 heal 用 `enqueue`（异步轮询）而非 `submit_and_wait`（同步等待），因为 heal 自身慢（LLM 子进程，同 ingest），轮询体验恰当（决策P4.3-2）。
- **heal 用子进程 runner、无嵌入四坑**：heal 经 `run_guarded_write_result` 走 `agentao run` 子进程（与 ingest 同），**不是**只读嵌入会话（§4.4 那条）。故 P4-8 的「`model=None` 盖掉 `.env` 模型触发 ValueError」嵌入坑**不适用**——heal 把 `model`（含 `None`）直接透传给子进程 runner，与 ingest 一致。`runner=app.state.runner` 供测试注入 fake（不打真实 LLM），同 ingest。
- **安全姿态沿用 P4 §8**：heal 只写 `wiki/`（`entities/` 新页 + `index.md`/`log.md` catalog + 既有内容页**纯追加别名**收编，P3.3 §3 窄缝），**永不碰 `raw/`**——由 P2 快照门禁 + heal `_writeset` 写集审计双重兜底（core 既有，Web 不放宽、不旁路）。端点无 `path`/`name` 入参（只 `limit`/`min_refs`/`model`），故**无路径穿越面**。

## 5. 前端（`guanlan/web/static/`）

## 6. 决策记录

- **\[决策P4.3-1] heal 的结构化结果经「`heal.py` 抽不打印 core（`run_heal_result -> HealRun`）+ `Job.result` 承载」回传，且 `result` 收窄为纯机器回执。** 仿 P3.2 决策P3.2-13 把 `run_guarded_write` 抽出结果版的同一手法：`run_heal` 退化为薄壳（CLI 字节不变），core 返回真 `HealResult`；`jobs.py` 给 `Job` 加可选 `result`、worker 鸭子分流（`int`→exit_code / 否则→结构化结果，只碰 `.exit_code`，不 import heal 类型）。兑现 P3.2 §11「承载 `HealResult` 需给 `Job` 加可选 `result` 字段」。**`result` 单一职责 = 机器回执**：端点用既有 `heal_result_dict(run.result, run.postponed)`（六字段）序列化，**不掺 `final_text`/`skipped_llm` 等 UI 展示补充**——agent 散文走 `job.output`（thunk 在 worker redirect 块内 `print`，与 ingest 同口径），短路语义由空 `receipts`+`EXIT_OK` 自明。这避免 `result` 退化成「回执 + 展示」的混合对象。否决「`run_heal(json_output=True)` 捕获 stdout 再 `json.loads`」：脆弱往返 + 隐性依赖人读格式（撞决策P3.2-10）。
- **\[决策P4.3-2] heal 入单写者 FIFO `JobQueue`、用 `enqueue`（异步轮询）而非 `submit_and_wait`（同步等待）。** heal 是写、且自身慢（LLM 子进程，同 ingest），故对标 ingest 的「入队即返回 `job_id` + 轮询」（P3.2 §11 明确「对标 ingest 的 `enqueue`、非 raw 的 `submit_and_wait`」）。与 ingest/投喂同 worker 串行 → 单写者假设不破、`raw/` 快照不被并发写互踩（决策P4-5）。**零新写基础设施**：`JobQueue` 本就是通用 `enqueue(kind, thunk)`，heal 只是新 job kind。
- **\[决策P4.3-3] worklist 由作业在服务端重算，客户端只传 `limit`/`min_refs`/`model`、绝不传 target 列表。** 与 CLI 一致：哪些算高价值缺页由确定性聚合（`compute_worklist` → `lint.missing_entities`，决策P3.2-1）裁决，不容客户端指定/伪造目标越过它。preview 与 POST 各自独立重算（其间若库变，以作业运行时的重算为准）——preview 是预览、非冻结订单。
- **\[决策P4.3-4] preview 是零-LLM 只读端点（`to_thread`、不入队、不取 `raw/` 快照），与 `/api/report/*` 同类。** 它只算 worklist，绝不写、绝不触 Agentao（== `heal --dry-run`）。POST 才是写、才入队、才过门禁。读写两端口分明，呼应 P4 §4.1 读写分线。
- **\[决策P4.3-5] heal 走子进程 runner，复用 P2 门禁；嵌入四坑（决策P4-8）不适用。** heal 经 `run_guarded_write_result` → `agentao run` 子进程（与 ingest 同），非只读嵌入会话；故 `model`（含 `None`）直接透传子进程、无 `.env` 覆盖坑，`runner` 可注入 fake 供测试。门禁/自愈/`REPAIR_PROMPT`/写集审计全继承 core，Web 端零新增。
- **\[决策P4.3-6] 不新增退出码、不做 heal SSE、不内联自动化。** P4.3 沿用 P4 退出码面（进程 `EXIT_OK`/`EXIT_USAGE`，作业退出码只进 `job.exit_code`）；写作业只轮询、不流式（决策P4-6）；heal 仍**显式按需触发**（点按钮才跑），不入 ingest、不定时（决策P3.2-9）。
