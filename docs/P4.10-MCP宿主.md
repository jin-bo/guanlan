# P4.10 MCP 宿主 `guanlan mcp`（只读，可选叠加层）

> P4「宿主层」家族的第二种传输：Web 把命令搬进**浏览器**（HTTP+SSE），MCP 把**只读核心**搬进**任意 Agent**
> （Claude Code / Codex / Cursor，MCP stdio）。与 P4.9 reader **同姿态、异传输**——同一只读核心、同一零写契约，
> 只换协议。**零 LLM 为主、零业务智能、不引新退出码。**

## 0. 定位：为什么是 P4.10，而不是新里程碑 / E2

DESIGN §1 第17条把 P4 的里程碑身份明写成"叠加的**可选宿主层**……同一文件库的**另一个入口**"——"Web 宿主"只是
当时唯一的传输。MCP 宿主正是"另一个入口"，与 Web 宿主**同构**：可选、依赖门控、零业务智能、复用既有只读核心，
只换传输（MCP stdio ⇄ HTTP+SSE）。故归 **P4 家族**，接 P4.9 之后为 **P4.10**；里程碑主题从「Web 宿主」扩为
「**可选宿主层**：Web 与 MCP 两种传输」（与 P5 扩为「多格式+检索」同款收口）。

- **为什么不是新里程碑/P6**：为一个只读宿主新开顶层里程碑过重；半阶段就该挂在 P4 宿主主题下。
- **为什么不是 E2**：多租户 MCP-over-HTTP+OAuth（gbrain 那种）才属 E2（多租户权限）。`guanlan mcp` 如本文
  = **本地、单客户端、只读、stdio**，正是 P4 宿主画像。它和 P4.9 一样：**本体 P4 半阶段，同时作"远程/scoped
  MCP"的 E2 前哨**；真正动数据模型的 source 级作用域排 P 轨之后。

### 0.1 必须澄清的同名反向陷阱

DESIGN §1 第22条 / §5.1「**Tool 注入**留作 P4 之后按需项、`.agentao/*.json` 需 MCP 时再加」指的是**反方向**：
**Agentao 作 MCP 客户端**消费外部工具、把 Tool 注入 Agent。本文是**相反方向**：**guanlan 作 MCP 服务端**，把 wiki
只读暴露给外部 Agent。两者同名「MCP」、方向相反（决策P4.10-6）——文档与帮助文案须写明区分。

## 1. 命令契约

```bash
guanlan -C <kb> mcp                 # 在 stdio 上起只读 MCP 服务（前台长驻，由调用方 Agent 作子进程拉起）
guanlan -C <kb> mcp --model <M>     # 覆盖 ask 工具的 Agentao 模型（仅 ask 用；其余工具零 LLM）
```

调用方（Claude Code 等）在其 MCP 配置里把 `guanlan -C <kb> mcp` 注册为一个 stdio server，即可把本 wiki 当**原生
只读工具**用。典型 `~/.claude.json` / `mcp.json` 片段在 §5 给出。

- **stdio 单客户端单进程**（决策P4.10-1）：MCP 本地集成标准；比 Web 还省心——无端口、无 `127.0.0.1` 暴露、无
  `workers=1` 之虑（一客户端一进程）。HTTP/远程/多租户 = E2，不做。**注意**：单进程 ≠ 单 in-flight——MCP 是
  异步 JSON-RPC，客户端可并发发多个工具调用，故仍有事件循环阻塞与共享 `CorpusCache` 临界区之虑（决策P4.10-11/15）。
- **stdio 通道洁净**（决策P4.10-13，**比零写更要命的不变量**）：stdio MCP 的 **stdout 即 JSON-RPC 传输帧**——
  server 路径**绝不向 stdout 写任何非协议字节**（一个杂散 `print` 即破帧、整服务挂）。本仓库 entrypoints 满地
  `print(format_report(...))`（health.py:145、lint.py:153 …），故 MCP 路径**只调核函数（返对象）、绝不调
  `*_entrypoint`/`format_report` 打印壳**，所有降级/诊断/日志一律走 **stderr**。`ask` 的 agentao 子进程 stdout 被
  `run_agent_task` **捕获解析**（runtime.py，不继承父 stdout），故子进程安全；要守的是**父进程自身零 stdout 写**。
- **只读、KB 零字节写入**（决策P4.10-3）：`require_kb_root(writable=False)`（只需 `wiki/` 在）；不写
  `agentao.log`、不落 session——镜像 P4.9 reader 零写契约。**不注册任何写工具**。
- **无新退出码**：正常停服 `EXIT_OK`；非知识库根 / 缺 mcp extra / 用法错 `EXIT_USAGE`。工具内错误走 MCP
  **in-band tool error**（返回给调用方 Agent），**不**映射进程退出码（决策P4.10-7）。

## 2. 模块与归口

新增 `guanlan/mcp/` 子包，与 `guanlan/web/` 并列、**同样零业务智能**——只把核心模块的只读能力包成 MCP 工具：

```
guanlan/mcp/
  __init__.py     # from .server import serve_mcp（缺 mcp extra → ImportError，CLI 优雅降级，镜像决策P4-2）
  server.py       # serve_mcp(root, *, model): 建 FastMCP server + 长驻 CorpusCache + 注册工具(各带返回类型注解→自动 schema) + 跑 stdio 事件循环 → EXIT_OK
  tools.py        # 各工具：薄壳，直接调既有核心函数，绝不复制业务逻辑；统一 try/except → in-band tool error（决策P4.10-16）
```

**SDK 钉死（决策P4.10-14）**：用官方 `mcp` Python SDK 的 **FastMCP**（`mcp.server.fastmcp.FastMCP` + `stdio` 传输），
extra 声明 `guanlan-wiki[mcp] = ["mcp>=1.27,<2"]`：官方 Python SDK **v1.x 为当前 stable、v2 尚 alpha**，须加 `<2` 上界
防将来破坏性升级（来源：官方 SDK README）；下界 `1.27` 取已验证支持结构化输出（协议 ≥2025-06-18）的稳定版。
**不**用 low-level `Server` 手搓 schema。**结构化输出由返回类型注解驱动**（见决策P4.10-10/14）：每个工具以
`TypedDict`/`Pydantic`/`dataclass` 作**返回类型注解**，FastMCP 据此自动生成 output schema + `structuredContent`——
**不是**在 `@mcp.tool()` 上手填 `outputSchema` 参数（无该常规入口）；手写 `CallToolResult` 时用
`Annotated[CallToolResult, Model]` 校验。

**复用点（全部既有、零新业务逻辑）**：

| MCP 工具 | 复用核心 | LLM? |
|---|---|---|
| `search` | `search.search_result_dict(search.score(cache.corpus(wiki), q, limit=limit))`（P5.1 同款长驻 `CorpusCache`，非冷算 `search_pages`；P5.0/P5.1 单一归口） | 零 |
| `read_page` | `pages.load_page` + `page_title`（≈ web `/api/page` 内核，app.py:427） | 零 |
| `list_pages` | `pages.iter_pages` + `page_title`/`page_type`（≈ web `/api/pages`，app.py:423） | 零 |
| `graph` | `graph.build_graph` + `graph_to_dict`（≈ web `/graph/graph.json`） | 零 |
| `health` / `lint` | `pages.report_dict(...)`（新抽产-dict 核，由 `run_health`/`run_lint` 结果喂入；`report_json = json.dumps(report_dict(...), ensure_ascii=False, indent=2)`，决策P4.10-10）——**不**走 `json.loads(format_report(...))` 字符串往返 | 零 |
| `ask`（唯一 LLM 工具） | `runtime.run_agent_task(..., permission_mode="read-only")` + `query.QUERY_PROMPT`（即 CLI query 只读路径，query.py:53） | LLM |

> **路径口径（统一钉死，决策P4.10-9）**：所有 MCP 工具的页面 path **= 相对库根、带 `wiki/` 前缀**（如
> `wiki/entities/DeFi.md`），与 P5.0 `SearchHit.page`、web `/api/pages`/`/api/page` **同口径**（search.py:70、
> app.py:204）。`read_page(path)` 直接吃 `search().results[i].page`（**search→read 链路无须拼接/剥前缀**），
> 内部走 web 同款 `_safe_wiki_file(root, path)`：`(root/path).resolve().relative_to((root/"wiki").resolve())`
> 夹断越界。**严禁**把 path 当作 `wiki/` 下的相对路径（否则会读成 `wiki/wiki/...`）。
>
> **结构归口（决策P4.10-10，修正后）**：MCP 工具结果是 **content blocks**，不是裸 dict。FastMCP 据工具**返回类型注解**
> （`TypedDict`/`Pydantic`/`dataclass`）自动生成 output schema，协议 ≥2025-06-18 时回 `structuredContent`（对象），
> **同时附一个 JSON 文本块**给不读结构化输出的老客户端兜底——故"绝不回传 JSON 文本"在线缆层并不成立；要保的是
> **两路同源、parsed-equal**。归口仍是字段同形：
> `search` 复用 `search_result_dict`（CLI/Web/MCP 三处同形，search.py:356）；`health`/`lint` 复用**新抽的产-dict 核
> `pages.report_dict`**（把 `report_json` 拆成 `report_json = json.dumps(report_dict(...), ensure_ascii=False, indent=2)`，镜像 heal 的
> `run_heal_result`/`run_heal` 拆分），**直接拿 dict**，不再 `json.loads(format_report(...))` 绕 JSON 字符串往返。
> 每个工具**必须给返回类型注解**（让 FastMCP 生成 schema），否则结构化输出退化为纯文本串（与 P5.1 `guanlan_search`
> 的 `json.dumps` 同形，可用但失去对象契约）。

> **关键简化：MCP 宿主完全不碰 `chat.py`**（决策P4.10-4）。Web 之所以有 `Conversation`/`ConversationStore`/
> session/idle-reclaim，是因为**浏览器**是瘦客户端、对话状态须落服务端。**MCP 客户端（调用方 Agent）自持对话**——
> 每次工具调用无状态、单次往返。故 MCP 宿主无会话表、无 session 落盘、无多轮服务端状态，是本提案相对 Web 的
> 最大减法。

## 3. 工具集（两类：零 LLM 检索 + 唯一 LLM `ask`）

### 3.1 零 LLM 检索工具（无模型、可离线）

调用方本身就是 LLM——给它原料、让它自己综合，是 MCP 的自然分工（nashsu/llm_wiki 的 MCP 即纯检索）。故核心是
**零 LLM 检索面**，无模型/无 API key 也能跑：

- **`search(query, limit=10)`** → `search_result_dict(...)` 信封 `{ok, query, pages_searched, results:[{page,title,
  type,score,snippet}]}`：确定性 BM25+2-gram（复用 P5.0 + P5.1 单一归口，与 CLI/Web 字段同形）。**走 server 长驻
  `CorpusCache`**——`search_result_dict(score(cache.corpus(wiki), query, limit=limit))`，与 P5.1 Web 同款（决策
  P4.10-11），**不**用冷算 `search_pages`（那会每次工具调用全库读盘+分词，与既有性能路径漂移）。`page` 带 `wiki/`
  前缀，可直接喂 `read_page`。**`limit` 工具层自钳**（决策P4.10-12）——MCP 参数来自客户端/LLM、无 HTTP
  `Query(ge=1)` 兜底，而 `score` 对 `limit < 1` **raise ValueError**（P5.0 决策P5.0-15）；写成
  `try: lim = max(1, int(limit)); except (TypeError, ValueError): lim = 10`——`int()` 对坏类型（`None`/不可转字符串）
  会 raise `TypeError`/`ValueError`，故**不能**裸 `max(1, int(limit))`，须 `try/except` 兜回默认 10，绝不让异常冒泡成
  未定义工具失败（与 P5.1 `guanlan_search` 同源、更严格补全坏类型路径）。**`query` 同样防坏类型**：客户端/LLM
  可填 number/array/null，而 `score`→`tokenize`→`re.finditer` 遇非 str 抛 `TypeError`（search.py:100），故调 `score`
  前 `if not isinstance(query, str): query = "" if query is None else str(query)`（与 web `chat.py:212` 同款；空/纯
  标点 query 不必额外管——`score` 对零 query-token 短路回空命中、安全）。
- **`read_page(path)`** → `{path,title,content}`：读单页正文（容错档，坏 frontmatter 不崩）。`path` = 相对库根带
  `wiki/` 前缀（即 `search().results[i].page`），经 web 同款 `_safe_wiki_file` 防越界到 `wiki/` 外。
- **`list_pages()`** → `{pages:[{path,title,type}]}`：非 config 页清单（≈ index.md 目录，但从磁盘实时枚举）。`path`
  同样带 `wiki/` 前缀。
- **`graph()`** → `graph_to_dict(build_graph(wiki))`：节点/边/stats（含 P3.5 社区，若已落）。
- **`health()` / `lint()`** → `pages.report_dict(...)` 直接产出的 **dict**（信封 `{ok, pages_checked, findings}`，
  决策P4.10-10）：零 LLM 体检/结构 lint 的结构化结果，供 Agent 判断 wiki 健康度。**不**经 `json.loads(format_report(...))`
  字符串往返。

> **载荷上限（决策P4.10-10 附）**：只有 `search` 有 `limit`。`list_pages`/`graph`/`read_page` 当前**无上限**——大 wiki 上
> `graph()` 倾倒全图、`list_pages()` 全量枚举、`read_page` 回超长正文，可能撑爆调用方 context。v1 接受（与 CLI/Web 既有
> 行为一致、不引新截断口径），但工具描述须**明示无分页**，让调用方 LLM 自行决定是否先 `search` 收窄；`graph` 的体量上限
> 留作后续（E1 检索升级时一并给 summary 档）。

### 3.2 唯一 LLM 工具 `ask`（复用 CLI query 只读路径）

- **`ask(question, model?)`** → `{answer}`：把问题交给观澜自己的只读 Agentao 综合出**带 `[[引用]]`** 的答案
  （wiki 有自己的综合/引用约定，值得提供）。
- **与 §3.1 纯检索分工的张力（须自觉）**：§3.1 主张"调用方本身就是 LLM、给它原料让它自综合"才是 MCP 自然分工，
  而 `ask` 反向做**服务端综合**、且每次拉起整个 agentao 子进程（慢 + 费 token）。保留 `ask` 的唯一理由是 wiki 的
  `[[引用]]`/`## ⚠️ 矛盾与存疑` 综合约定调用方无从复刻；**它不是检索面的替代，是重路径补充**——工具描述应引导
  "先 `search`+`read_page`，需要观澜式带引用综合时才 `ask`"，避免调用方默认走最贵的路。
- **复用 CLI query 只读子进程路径**（`run_agent_task(QUERY_PROMPT.format(...), permission_mode="read-only")`，
  query.py:53 同款）：故 **P4-8 嵌入四坑不适用**（走子进程，与 heal/backfill 同理），且**无 session 状态**
  （单次无状态调用）。只读姿态双重兜底拦写、不取 raw 快照、不跑 check——与 `guanlan query` 字节同源。
- `model` = `ask` 工具入参 `model` `or` 启动 `--model` `or None`（可解析到 None，由 Agentao 兜底；MCP 无 HTTP body，
  注意别照搬 Web 的 `body.model` 口径）。`run_agent_task` 把**所有**失败
  归一为 `AgentRunResult(ok=False)`——无模型、agentao 不在 PATH、输出不可解析、LLM API marker 等同列（runtime.py:26/40）。
  故 `ask` 一律 `ok=False → in-band tool error`（无模型只是其中一种），**不**影响 §3.1 零 LLM 工具可用。

### 3.3 （可选增强）页面暴露为 MCP resources

MCP 原生有 tools 与 **resources** 两面。把 wiki 页暴露为只读 resources（uri 如 `guanlan://wiki/entities/DeFi.md`）
让客户端**浏览/附加**页面，是 `read_page` 的自然补充。**tools 优先**；resources 作附加增强，可随后补（不阻塞 P4.10
主体）。

## 4. 决策（沿用仓库 `决策PX-N` 体例）

- **决策P4.10-1（stdio 单客户端单进程）**：MCP 本地集成标准；无端口/无网络暴露/无 workers=1 之虑。HTTP/远程/
  多租户 = E2，不做。
- **决策P4.10-2（依赖门控 + 新子包）**：新 `guanlan-wiki[mcp]` extra（mcp SDK），缺失 `ImportError` → CLI 优雅降级
  引导安装（镜像 web 决策P4-2）；新 `guanlan/mcp/` 子包零业务智能，复用核心模块。
- **决策P4.10-3（只读 + KB 零字节写入）**：`require_kb_root(root, writable=False)`（paths.py:23，只验存在）；不写
  agentao.log、不落 session、不注册任何写工具——镜像 P4.9 reader 零写契约。**注意 `ask` 不照搬 P4.9**：P4.9 reader 是
  **进程内嵌入**、靠 `session_persist=False` + agentao.log 写探针保零写；`ask` 走的是 **CLI query 子进程**路径（异姿态），
  agentao 子进程以 `<kb>` 为 working_directory 跑时**可能自落** `agentao.log`/`.agentao/sessions/`。故"KB 零字节写"对
  `ask` 是**待实证**项：实现期须实测确认零写，或对其加等价于 P4.9 的写抑制/探针（§7 的"只读保证"用例须覆盖跑过 `ask` 后
  KB 仍零字节变动），不得仅以"镜像 P4.9"一笔带过。
- **决策P4.10-4（无会话/无 session 状态，不碰 chat.py）**：MCP 客户端自持对话，工具调用无状态。Web 的
  Conversation/session/idle-reclaim 全不需要——本提案相对 Web 的最大减法。
- **决策P4.10-5（工具分两类）**：零 LLM 检索工具（search/read_page/list_pages/graph/health/lint，离线可用）+
  唯一 LLM 工具 `ask`（复用 CLI query 只读子进程，无嵌入坑、无 session）。
- **决策P4.10-6（方向澄清：服务端，非 Tool 注入）**：见 §0.1。guanlan 作 MCP **服务端**；区别于 DESIGN §1.22 的
  Agentao 作 MCP **客户端**（Tool 注入），同名反向。
- **决策P4.10-7（无新退出码；工具错走 in-band）**：进程级只有 `EXIT_OK`/`EXIT_USAGE`；工具内错误（坏 path、
  无模型、读盘失败）作 MCP tool error 返回调用方，不映射退出码。
- **决策P4.10-8（search 软依赖 P5.0）**：`search` 工具复用 P5.0 检索内核，故 P4.10 落在 P5.0 后能直接暴露；
  不等也可先上其余工具，`search` 随 P5.0 补（软依赖，非硬阻塞）。
- **决策P4.10-9（页面 path 单一口径：相对库根带 `wiki/` 前缀）**：所有 MCP 工具（search/read_page/list_pages）的
  `page`/`path` 字段统一为相对库根、带 `wiki/` 前缀，与 P5.0 `SearchHit.page` / web 端点同口径。`read_page` 直接
  消费 `search().results[i].page`（链路无拼接/剥前缀），内部走 web 同款 `_safe_wiki_file`。**禁止**把 path 解释为
  `wiki/` 下相对路径（否则 `wiki/wiki/...`）。
- **决策P4.10-10（结构化输出：structuredContent + 文本块兜底，非"纯 dict"）**：MCP 工具结果是 content blocks，不是
  裸 dict。FastMCP 据工具返回类型注解（`TypedDict`/`Pydantic`/`dataclass`）自动生成 schema、协议 ≥2025-06-18 时回
  `structuredContent` 对象，**并同时附 JSON 文本块**
  兜底老客户端——故"绝不回传 JSON 文本"不成立，要保的是两路 parsed-equal。归口：`search` 复用 `search_result_dict`
  （CLI/Web/MCP 三处同形）；`health`/`lint` 复用**新抽的 `pages.report_dict` 核**（`report_json = json.dumps(report_dict(...), ensure_ascii=False, indent=2)`，
  **无尾随换行**——须原样保留 pages.py:270 既有序列化参数，否则破 CLI/Web JSON 字节契约；镜像 heal 的
  `run_heal_result` 拆分），直接拿 dict、**不**走 `json.loads(format_report(...))` 字符串往返。每个工具
  **必须给返回类型注解**（FastMCP 据此生成 schema），否则退化为纯文本串（与 P5.1 `guanlan_search` 同形，失对象契约）。
- **决策P4.10-11（search 走长驻 `CorpusCache`，非冷算）**：MCP 是长驻 stdio server，`search` 工具持 server 级
  `CorpusCache`、按 `score(cache.corpus(wiki), q, limit=limit)` 算（与 P5.1 Web 同款性能路径），**不**用一次性
  `search_pages` 冷算——否则每次工具调用全库读盘+分词、与既有路径漂移。`CorpusCache` 按 (mtime_ns,size) memo、
  只重建变更页、与冷算字节等价（P5.0 决策P5.0-14）。**并发临界区靠 `CorpusCache` 自持锁**兜底（与 P5.1 Web 同款，
  app.py §3.3）——单进程 ≠ 单 in-flight：客户端可并发发多个 `search`、在共享 cache 增量重建上交错，故**不能**声称"无
  临界区之虑"（措辞已修正，详见决策P4.10-15）。
- **决策P4.10-12（`limit` 工具层自钳）**：MCP 工具参数由客户端/LLM 填、无 HTTP `Query(ge=1)` 门，而 `score` 对
  `limit < 1` raise ValueError（P5.0 决策P5.0-15）。故 `search` 工具体 `try: lim = max(1, int(limit)); except
  (TypeError, ValueError): lim = 10`——`int(None)`/`int("abc")` 会 raise，**不能**裸 `max(1, int(limit))`；`try/except`
  同时兜 0/负值（clamp）与坏类型（回默认 10），绝不让异常冒泡成未定义失败（与 P5.1 `guanlan_search` 同源、补全坏
  类型路径，不复用 CLI `positive_int`）。**`query` 同防坏类型**：非 str 一律 `str()` 归一、`None`→空串（否则
  `tokenize`→`re.finditer` 抛 `TypeError`），与 web `chat.py:212` 同款。
- **决策P4.10-13（stdio 通道洁净，最高优先不变量）**：stdout 即 JSON-RPC 传输帧——server 路径绝不向 stdout 写非协议
  字节。只调核函数（返对象），绝不调 `*_entrypoint`/`format_report` 等**打印壳**；降级/诊断/日志一律 stderr。`ask` 子
  进程 stdout 被 `run_agent_task` 捕获、不继承父 stdout（安全），守的是父进程自身零 stdout 写。§7 须有"stdout 只含合法
  JSON-RPC 帧"用例。
- **决策P4.10-14（MCP SDK 钉死：FastMCP + 返回类型注解驱动 schema）**：用官方 `mcp` SDK 的 `FastMCP`（stdio 传输），
  extra `guanlan-wiki[mcp] = ["mcp>=1.27,<2"]`（v1.x stable、v2 alpha，须加 `<2` 上界；下界取已验证支持结构化输出的稳定版，
  来源官方 SDK README）。不用 low-level `Server` 手搓。**结构化输出由返回类型注解（`TypedDict`/`Pydantic`/`dataclass`）
  驱动**——FastMCP 自动生成 output schema，使 §3 各信封落 `structuredContent`（决策P4.10-10）；**非** `@mcp.tool()` 手填
  `outputSchema`，手写 `CallToolResult` 时用 `Annotated[CallToolResult, Model]` 校验。
- **决策P4.10-15（handler 异步姿态 + `ask` 卸载 + 并发锁；修正 P4.10-11 的"无临界区"）**：MCP server 跑单事件循环、
  客户端可并发多 in-flight。故 (a) **`ask` 的阻塞子进程须 `anyio.to_thread.run_sync` 卸离事件循环**（与 Web app.py 同
  款），否则一次 `ask`（可数十秒）饿死所有其他工具调用与取消/心跳；零 LLM 工具的读盘/分词同理卸线程；(b) 并发 `search`
  对共享 `CorpusCache` 的临界区**靠 cache 自持锁**保证，不靠"单进程"假设。原 P4.10-11"单客户端单进程下无并发临界区之
  虑"作废。
- **决策P4.10-16（全工具统一 in-band error 总壳）**：决策P4.10-7 的"工具内错误走 in-band"须落成 **`tools.py` 的统一
  `try/except Exception` 总壳**——任何核函数意外抛出（`read_page` 不可读、`graph`/`build_graph` 遇坏 frontmatter、
  `run_health`/`run_lint` 异常）都收敛为 MCP tool error 返回调用方，**绝不让异常冒泡杀 server / 破 stdio 帧**。`search`
  的 `limit`/`query` 坏类型防御（决策P4.10-12）是这层之内的细化，不替代总壳。

## 5. 客户端注册（文档示例，非代码）

```jsonc
// Claude Code / 兼容客户端的 MCP 配置
{ "mcpServers": {
    "guanlan": { "command": "guanlan", "args": ["-C", "/path/to/kb", "mcp"] }
} }
```

注册后调用方 Agent 即获 `search`/`read_page`/`list_pages`/`graph`/`health`/`lint`/`ask` 七个只读工具，把本 wiki
当原生知识后端用——无需起 Web、无需 `guanlan query` 来回 shell。

## 6. 边界（明确不做）

- **不暴露任何写工具**（ingest/heal/backfill/raw/upload 全不注册，决策P4.10-3）；
- **不做 HTTP / 远程 / 多租户 / OAuth**（E2，决策P4.10-1）；
- **不做服务端会话状态 / 多轮**（客户端自持，决策P4.10-4）；
- **不做 Tool 注入方向**（那是 Agentao 作客户端，反向，非本提案，§0.1）；
- **不做向量/重排**（`search` 是 P5.0 确定性 BM25；向量属 E1）；
- **不向 stdout 写任何非协议字节**（stdout 是 JSON-RPC 帧；杂散 print 即破帧，决策P4.10-13）；
- **不引新退出码、不写 KB、`ask` 外零 LLM、绝不碰 `raw/`**。

## 7. 测试要点（`tests/test_mcp.py` 主体在装有 mcp extra 时跑，缺 extra 整体 skip，镜像 test_web）

- **依赖门控（放在不依赖 SDK 的 CLI 测试文件，不随主体 skip）**：缺 `guanlan-wiki[mcp]` 时 CLI `guanlan mcp` 优雅
  降级提示安装、返回 `EXIT_USAGE`（不吐 traceback）。为在**装有** mcp extra 的 CI 也覆盖此路径，用 `monkeypatch`
  模拟 `from guanlan.mcp import serve_mcp` 抛 `ImportError`（镜像 web）——**不能**只靠"实际缺 SDK 的环境"，否则该环境
  整组 skip、降级路径永不被测。
- **零 LLM 工具正确性**：`search`/`read_page`/`list_pages`/`graph`/`health`/`lint` 返回与对应核心函数
  （`search_result_dict∘search_pages`/`load_page`/`build_graph`/`run_health`/`run_lint`）**结构一致**——证明只是薄壳、
  无逻辑分叉。`search` 体与 CLI/Web 经 `search_result_dict` 解析后字段同形；`health`/`lint` 经新 `pages.report_dict`
  核直接产 dict（断言 `report_json == json.dumps(report_dict(...), ensure_ascii=False, indent=2)` 且**无尾随换行**，
  证拆分对 CLI/Web 既有 JSON 字节契约零变更）。
- **结构化输出契约（决策P4.10-10/14）**：经 MCP 客户端实跑（或 FastMCP 测试夹具），断言工具结果含 `structuredContent`
  对象**且**附带的 JSON 文本块 parsed-equal；每个工具据**返回类型注解**自动生成的 schema 与实际返回字段对齐（漏注解
  会退化为纯文本、本用例须能捕获）。
- **stdout 通道洁净（决策P4.10-13）**：跑完整套工具（含 `ask` 的 mock 子进程路径）后，捕获进程 stdout，断言**只含合法
  JSON-RPC 帧**、无任何核函数/降级文案泄漏（防 `format_report` 等打印壳混入破帧）。
- **error 总壳（决策P4.10-16）**：令核函数抛异常（`read_page` 指向不可读文件、`graph` 喂坏 frontmatter、`run_health`
  注入抛错）→ 工具回 in-band tool error、**server 不崩、stdio 帧不破**；非仅 `search` 的 limit/query 坏类型。
- **并发不踩 cache（决策P4.10-15）**：并发发多个 `search`（+ 一个长 `ask`）→ 结果均正确、`CorpusCache` 无错乱重建；
  断言 `ask` 经 `to_thread` 卸载、运行期间廉价 `search` 仍能及时返回（不被事件循环阻塞饿死）。
- **path 单一口径 + search→read 链路**：`search().results[i].page` 带 `wiki/` 前缀，**直接**传 `read_page` 能读到
  同一页（无拼接/剥前缀）；断言不会读成 `wiki/wiki/...`。`list_pages().pages[i].path` 同口径。
- **search 走长驻 cache（决策P4.10-11/15）**：连续多次 `search` 复用同一 server `CorpusCache`（监视 `build_corpus`/
  `build_doc` 调用数：首搜建库、后续命中 memo 不重建未变页），且结果与冷算 `search_pages` **字节等价**——证明
  长驻路径不引入逻辑分叉、只省读盘+分词。
- **`limit` 自钳（决策P4.10-12）**：`search(q, limit=0)` / 负数 / 坏类型（`None` / 不可转字符串 `"abc"`）经
  `try/except (TypeError, ValueError)` 归一——均不 raise、不冒泡未定义失败。断言 `limit=0`/负数 → clamp 到 1（结果
  **至多 1 条**，避免空命中 fixture 下断言不稳）；`None`/`"abc"` → 回默认 10、不崩。
- **`query` 坏类型自防（决策P4.10-12）**：`search(query=None)` / 数字 / 数组 → 经 `str()` 归一（`None`→空串）后
  调 `score`，**不**让 `tokenize`→`re.finditer` 的 `TypeError` 冒泡；断言工具回正常空命中信封而非崩。
- **只读保证（含 `ask` 子进程，决策P4.10-3）**：跑完整套工具**包括 `ask`** 后，`raw/` 与 `wiki/` 字节零变动、无
  `agentao.log`、无 `.agentao/sessions/` 写入（KB 零字节写）。**`ask` 子进程零写须实证**——若实测发现 agentao 子进程
  落 log/session，则补写抑制/探针后此用例方可绿（不得跳过 `ask` 只测零 LLM 工具）。
- **写工具不存在**：MCP `tools/list` 里**无** ingest/heal/backfill/raw/upload 任何写工具（不是注册后拒绝，是
  根本不注册）。
- **`ask` 路径**：有模型时返回带 `[[引用]]` 的答案（mock runner）；**任意 `AgentRunResult(ok=False)` → in-band tool
  error**（参数化覆盖：无模型 / agentao 不在 PATH / 输出不可解析 / LLM API marker，无模型只是其一），均不崩、且
  **不**影响零 LLM 工具可用。
- **路径越界**：`read_page("../outside.md")` / 绝对路径 → in-band 错误，不读到 `wiki/` 外。
- **方向不混**：帮助文案/README 明确"服务端"，与 DESIGN 的"Tool 注入"区分（防同名误解）。
